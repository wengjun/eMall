# 容量验证说明

[文档索引](README.md) | [运维配置索引](../ops/README.md)

本文用于说明当前工程如何验证容量边界。工程目标可以按 10 亿用户、1 亿 DAU、100 万峰值并发来做架构设计，
但真实容量必须用集群资源、压测数据、数据库容量、Redis 容量、Kafka 积压和可观测指标证明。

## 验证范围

当前验证重点不是在单机上证明 100 万并发，而是证明核心交易链路具备可压测、可观测和可解释的容量模型。

- 下单链路：网关、订单、价格、营销、库存、Outbox。
- 支付链路：支付创建、回调、订单支付确认、优惠券核销、库存确认。
- 热点链路：热点 SKU、秒杀排队、库存桶、单用户频率保护。
- 异步链路：Outbox 发布、Kafka 消费、消费者幂等、失败重试、死信。
- 读链路：商品、搜索、库存和价格读请求。

## Java 压测工具

工程已经提供 Java 17 压测模块 `loadtest`，不再新增 JavaScript 压测脚本。

构建压测工具：

```powershell
mvn -pl loadtest -am package -DskipTests
```

执行核心下单压测：

```powershell
java -jar loadtest/target/loadtest-0.1.0-SNAPSHOT.jar http://localhost:8080 100 60 200 checkout
```

参数含义：

- `http://localhost:8080`：网关地址。
- `100`：目标 QPS。
- `60`：持续秒数。
- `200`：最大并发线程数。
- `checkout`：压测场景。

支持的场景：

- `checkout`：普通下单链路。
- `hot-sku`：热点 SKU 下单链路。
- `read-heavy`：商品、搜索、库存和价格读多链路。
- `payment-callbacks`：支付回调链路。
- `mq-backlog`：商品变更和 MQ 消费积压链路。
- `flash-sale-hotspot`：秒杀热点排队链路。

## 结果记录

每次压测都应同步填写 `../ops/loadtest/p2-capacity-baseline-template.md`。

必须记录的指标：

- QPS、成功数、失败数、错误率。
- P50、P95、P99 延迟。
- 网关 429、5xx、超时数量。
- MySQL 连接数、慢 SQL、锁等待、主从延迟。
- Redis CPU、内存、命中率、慢命令。
- Kafka 生产速率、消费速率、消费者 lag、DLT 消息数量。
- JVM CPU、堆内存、GC 次数、GC 暂停、线程数。
- Outbox 待发布数量、发布失败数量、补偿任务积压。

## 容量结论写法

容量结论必须按证据写，不写“代码已支持 100 万并发”这种不可验证结论。

推荐写法：

```text
在 8C16G 单机开发环境、MySQL/Redis/Kafka 本地容器部署下，checkout 场景稳定承载 X QPS，
P95 为 Y ms，P99 为 Z ms，错误率 A%，瓶颈出现在 B。

要达到 100 万峰值并发，需要按网关、订单、库存、支付、Kafka、Redis、MySQL 分片和多可用区部署做水平扩容，
并通过分阶段压测确认每一层容量。
```

## 100 万并发扩展路径

- 网关层：按客户端、用户、IP、设备、SKU、活动维度限流，热点流量提前削峰。
- 订单层：幂等表、订单表和路由表分库分表，单用户频率保护，Outbox 本地事务落库。
- 库存层：热点 SKU 使用库存桶和秒杀队列，避免单行库存被打满。
- 营销层：优惠券使用预占、确认、释放，避免重复核销。
- 支付层：回调幂等、签名校验、对账补偿和状态机。
- MQ 层：按业务事件分 topic，按 aggregate key 分区，消费者落库去重。
- 数据层：交易库只承载核心写入，搜索、分析和数仓走异步同步。
- 可观测层：用 SLO、错误率、延迟、积压和补偿指标判断是否继续放量。
