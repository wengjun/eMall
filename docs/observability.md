# 可观测性

本文说明 eMall 的指标、日志、链路追踪和 trace 验证策略。

## 指标

基础指标包括：

- HTTP 请求量、错误率、延迟分位数。
- 下单成功率、库存预占失败率、支付回调失败率。
- Outbox 待发布数量、失败数量、重试数量。
- MQ 消费延迟和死信数量。
- 熔断状态、限流命中数、降级次数。
- P8 控制面指标，例如隔离单元数量、阻塞的 readiness gate、运行中的灰度、关键安全信号。

Prometheus 规则位于 `ops/prometheus`，Grafana 看板位于 `ops/grafana`。

## 日志

日志目标：

- 每个请求都有 `X-Trace-Id`。
- 结构化输出，方便日志平台检索。
- 关键业务日志包含订单号、支付单号、SKU、用户、请求 ID。
- 内部运维操作记录操作人、trace ID、请求参数摘要和结果。
- 不在日志中输出明文手机号、密钥、令牌和支付敏感信息。

本地和生产候选日志平台采用 ELK：

- Elasticsearch 保存结构化应用日志索引。
- Logstash 负责接收、解析和补充日志字段。
- Kibana 负责按 `traceId`、`requestId`、`orderId`、`userId` 和错误码检索。
- 应用继续输出 JSON 日志，交易链路不直接依赖日志平台可用性。

## 链路追踪

Servlet 服务会透传 `X-Trace-Id`，内部 HTTP 调用也应继续传递该字段。服务可以通过 OpenTelemetry OTLP
导出 trace。

重点关注链路：

- 下单：网关 -> 订单 -> 价格 -> 营销 -> 库存。
- 支付：支付渠道 -> 支付 -> 订单 -> 库存确认。
- 商品同步：商品 -> Outbox -> Kafka -> 搜索。
- 补偿：内部运维 -> 订单/库存/支付 -> Outbox。

## Trace 验证

验证目标：

- 网关响应必须包含 `X-Trace-Id`。
- 服务之间的内部调用必须透传同一个 trace ID。
- 错误日志、业务日志和指标标签应能关联到同一请求。
- 补偿和重试任务要生成或继承可追踪的 trace ID。

## 生产扩展

真实生产还需要：

- ELK 日志平台和 trace 平台统一关联。
- 告警路由和值班机制。
- 指标基线和异常检测。
- 关键业务大盘，例如下单漏斗、支付成功率、库存失败率、退款对账差异。
