# 面试补强共享框架

[返回按分类学习面试题](README.md)

本文档集中整理逐题答案中高度重复的补强方法。逐题答案保留题目本身的精讲内容，
通用的复习、白板、取舍、验收、规模化和成本治理方法统一引用本文档。

## 使用方式

学习某一道题时，先看逐题答案正文，再用本文档补齐以下七个维度：

- 深度理解：面试官真正想确认什么能力。
- 专项落点：这道题在 eMall 项目里落到哪个链路或模块。
- 实战追问：面试官会从哪些方向继续追问。
- 架构取舍：为什么选这个方案，为什么不选其他方案。
- 生产验收：上线前、灰度中、回滚时要看什么。
- 规模成本：10 亿用户、1 亿 DAU、100W 峰值并发下如何估容量和控成本。
- 白板自测：能否在 30 秒内讲出结论、边界、风险和验证方式。

## 通用回答骨架

专家级面试不要只回答定义。更稳的结构是：

```text
结论 -> 业务场景 -> 实现方案 -> 失败路径 -> 指标验证 -> 取舍演进
```

如果是系统设计题，可以按下面顺序讲：

```text
目标和非目标 -> 容量估算 -> 架构和数据流 -> 一致性 -> 容灾降级 -> 成本治理
```

如果是编码题，可以按下面顺序讲：

```text
输入输出 -> 数据结构 -> 核心代码 -> 复杂度 -> 边界用例 -> 生产化缺口
```

如果是行为题，可以按下面顺序讲：

```text
Situation -> Task -> Action -> Result -> 量化指标 -> 复盘机制
```

## 能力域框架

### Java 语言和工程基础

推荐图解：[Java 工程能力从语法到生产设计](assets/java-engineering-model.svg)

- 重点不是背语法，而是说明语言特性如何降低误用、提升建模清晰度。
- 回答时要把例子落到金额、时间、状态、DTO、异常、公共库和领域对象。
- 取舍时说明新语法不是越多越好，公共 API 更重视兼容性和可读性。
- 验收时关注代码评审问题、单元测试覆盖、兼容性破坏和缺陷逃逸。

### JVM、GC、性能诊断

推荐图解：[JVM 运行时内存模型](assets/jvm-runtime-memory.svg)

- 先量化现象：P99、CPU、内存、GC 暂停、线程阻塞和错误率。
- 再收集证据：GC 日志、JFR、线程栈、堆转储、容器指标和业务指标。
- 不要直接调大内存，要先证明瓶颈确实在运行时。
- 规模化时关注实例容量、对象分配速率、堆大小和容器资源限制。

### 并发、线程池、分布式锁

推荐图解：[并发治理模型](assets/concurrency-governance.svg)

- 先找共享资源和临界区，再决定锁、CAS、队列、线程池或幂等。
- 高并发下必须说明超时、拒绝、取消、重试、隔离和降级。
- 锁和队列都要有上限，不能把请求无限堆积到内存里。
- 验收时看线程池活跃数、队列长度、拒绝次数和锁等待时间。

### Spring Boot、API、HTTP

推荐图解：[Spring 服务分层](assets/spring-service-stack.svg)

- 先明确 API 契约、版本、错误码、幂等键和参数校验。
- Controller 不应该承载业务规则，应用服务负责流程编排。
- HTTP 客户端要配置连接池、超时、重试、熔断和限流。
- 验收时看接口错误率、下游超时率、连接池等待和契约测试。

### 微服务、配置、可观测

推荐图解：[可观测平台](assets/observability-platform.svg)

- 服务拆分后，要回答调用链、配置灰度、日志、指标和 Trace。
- traceId 要贯穿网关、服务、日志、指标标签和下游调用。
- 控制指标标签基数、日志量、Trace 采样率和存储保留周期。
- 验收时看 Trace 覆盖率、告警准确率、MTTR 和配置回滚记录。

### 事务、MQ、一致性

推荐图解：[一致性补偿闭环](assets/consistency-compensation-loop.svg)

- 先明确强一致还是最终一致，不能默认所有场景都用分布式事务。
- 典型闭环是本地事务、Outbox、消息投递、消费幂等、补偿和对账。
- 失败路径要覆盖重复、乱序、丢失、积压、死信和人工介入。
- 验收时看最终一致延迟、补偿成功率、幂等拦截数和对账差异。

### 数据库、库存、分片、SQL

推荐图解：[索引设计](assets/index-design.svg)

- 先说明访问模式，再设计主键、唯一键、二级索引和状态字段。
- 高并发更新要讲事务隔离、行锁、乐观锁、库存扣减和幂等。
- 分库分表前先治理慢 SQL、热点行、归档和读写分离。
- 验收时看执行计划、锁等待、慢 SQL、索引命中和分片热点。

### Redis、Kafka、搜索读模型

推荐图解：[数据缓存消息模型](assets/data-cache-mq.svg)

- 数据库是事实源，缓存、搜索、推荐和读模型都是派生模型。
- 必须说明版本号、TTL、回放、重建、乱序和降级读取。
- 缓存和索引都要控制对象大小、字段数量、保留周期和冷热分层。
- 验收时看缓存命中率、回源 QPS、索引延迟和消息滞后量。

### 安全、风控、开放平台

推荐图解：[开放平台安全链路](assets/openapi-security.svg)

- 区分认证、授权、验签、风控、审计和合规留痕。
- 安全边界默认拒绝，权限最小化，密钥可轮换，操作可审计。
- 风控要关注误杀率，不能只追求拦截率。
- 验收时看验签失败率、越权拦截数、误杀恢复和审计覆盖。

### 运维、告警、Kubernetes、发布

推荐图解：[Kubernetes 运维模型](assets/kubernetes-operations.svg)

- 先判断用户影响，再决定扩容、限流、降级、回滚或切流。
- 发布要覆盖 readiness、liveness、优雅停机、连接排空和预热。
- 告警必须绑定 SLO、owner、Runbook 和升级路径。
- 验收时看 SLO 达成率、错误预算、回滚耗时和告警到确认时长。

### 测试、CI、工程质量

推荐图解：[代码质量闭环](assets/coding-patterns.svg)

- 先判断风险，再选择单元测试、集成测试、契约测试、烟测或压测。
- 覆盖率不是目标，能阻止生产事故才是目标。
- 测试越多越好是误区，关键是反馈速度、稳定性和风险覆盖。
- 验收时看流水线通过率、flaky 测试数、缺陷逃逸率和关键路径覆盖。

### 系统设计和平台设计

推荐图解：[电商系统架构](assets/ecommerce-architecture.svg)

- 先澄清目标、非目标、规模、延迟、可用性、一致性和成本。
- 架构图要画入口、服务、存储、缓存、消息、任务和观测。
- 同步链路要短，异步链路要可靠，治理能力要平台化。
- 验收时看可用性、P99、容量水位、成本和数据一致性延迟。

### Amazon L6 行为面试

推荐图解：[电商系统架构](assets/ecommerce-architecture.svg)

- 只讲执行容易像 L4/L5，L6 要体现模糊问题中的判断和影响。
- 故事必须包含反对意见、权衡过程、跨团队影响和量化结果。
- 不能只讲团队做了什么，要讲清你具体做了什么。
- 复习时用 STAR，但表达时重点突出 ownership、judgment、scope 和 impact。

### 现场编码和设计实现

推荐图解：[代码质量闭环](assets/coding-patterns.svg)

- 先复述输入输出和边界，再解释数据结构、复杂度和失败用例。
- Java17 代码要清晰表达核心思路，不要为了炫技牺牲可读性。
- 生产化要补线程安全、容量限制、监控、日志、测试和降级。
- 验收时看复杂度、边界测试、并发安全和可维护性。

## 白板复述检查单

复习时可以逐题按下面清单自测：

- 我能不能 30 秒给出结论和适用边界？
- 我能不能举出 eMall 项目里的模块落点？
- 我能不能画出主路径、失败路径、回滚路径和 owner？
- 我能不能说出关键指标、告警条件和验收证据？
- 我能不能解释为什么不选更简单或更复杂的方案？
