# Java 专家级分布式服务开发面试题库

[返回面试专题首页](README.md) | [返回学习手册首页](../learning/README.md) | [返回技术能力地图](../technical-skill-map.md)

本文整理 Java 专家级分布式服务开发面试中可能被问到的问题。
它不是背诵清单，而是复习地图。
真正高评价的回答需要能结合项目代码、生产故障、指标、容量估算和技术取舍讲清楚。

逐题详细答案见 [Java 分布式服务专家级面试答案手册](answers/README.md)。
按专题复习见 [面试答案分类索引](answers/categories/README.md)。

使用方式：

- 第一遍按分类扫盲，标记完全不会的问题。
- 第二遍结合 eMall 的 `order`、`inventory`、`payment`、`common`、`gateway` 模块回答。
- 第三遍对每个问题准备“背景、方案、失败场景、监控、取舍”五段式答案。
- 面试前重点训练系统设计、并发、JVM、数据库、MQ、缓存、稳定性和行为面试。

## 如何把题库转成高评价答案

回答任何专家级问题时，不要只给定义。建议使用下面结构：

```text
1. 这个问题在生产中为什么重要。
2. 常见方案有哪些。
3. 我会选择哪个方案，原因是什么。
4. 这个方案的失败场景是什么。
5. 如何监控、告警、恢复和验证。
6. 如果规模继续扩大，下一步如何演进。
```

示例：

```text
问题：支付成功但订单更新失败怎么办？

回答：这是跨服务一致性问题。支付服务不能和订单服务做全局事务。
我会先在支付服务本地事务中把支付单置为成功，写支付流水和 Outbox 事件。
随后调用订单服务确认订单。如果调用失败，支付单保留 orderConfirmed=false，
补偿任务按 paymentId 重试确认订单。订单服务的支付确认接口必须幂等。
监控上关注 succeeded_but_unconfirmed 数量、最老未确认时间和补偿成功率。
如果长时间失败，内部运维入口可以人工触发重试，并通过审计记录操作人和 traceId。
```

这种回答体现的不只是知识点，而是生产系统设计能力。
