# 240 幂等记录保留多久？

[返回按分类学习面试题](../README.md)

## 题目

幂等记录保留多久？

## 先给面试官的短答案

幂等记录保留多久取决于重复请求可能到达的最长时间、业务风险、审计要求和存储成本。
普通 API 幂等记录可能保留几小时到几天，支付、退款、库存和资金类记录通常要保留更久，并满足审计要求。

保留期不能拍脑袋，要基于重试窗口、消息保留期和业务追责周期设计。

## 影响因素

因素包括：

- 客户端重试最长时间。
- MQ 消息保留时间。
- 死信重放窗口。
- 对账周期。
- 业务撤销和退款周期。
- 合规审计要求。
- 存储成本。

不同业务不应使用同一个固定保留期。

## 常见策略

策略：

- 普通查询或低风险写操作短期保留。
- 订单创建保留到订单生命周期稳定。
- 支付和退款长期保留。
- MQ 去重记录至少覆盖消息最大重投时间。
- 历史记录归档到低成本存储。

高风险业务宁愿多保留，也不要误重复执行。

## 过期后的风险

记录过期后，如果旧请求或旧消息再次到达，系统可能无法识别重复。

所以过期策略要和：

- 客户端重试策略。
- MQ 保留策略。
- 死信重放策略。
- 业务状态机。

一起设计。

## 在 eMall 项目中怎么讲？

支付幂等记录至少要覆盖支付回调、渠道重试和对账周期，不能只保留几分钟。

普通购物车更新的幂等记录可以短一些，因为重复风险和业务损失较低。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
幂等记录保留期要根据重复请求可能到达的最长窗口设计，包括客户端重试、MQ 保留、死信重放、对账周期和审计要求。
普通低风险接口可以短期保留，支付、退款、库存等高风险业务应长期保留或归档。

记录过期后仍要依靠业务状态机和唯一约束作为最后防线。
```

## 回答评分点

高分答案应该覆盖：

- 保留期取决于重复到达窗口。
- 支付退款等高风险业务保留更久。
- 要考虑 MQ、死信和对账周期。
- 过期记录可归档。
- 状态机和唯一约束是最后防线。
