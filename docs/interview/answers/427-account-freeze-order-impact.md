# 427 账号冻结对下单链路有什么影响？

[返回按分类学习面试题](../README.md)

## 题目

账号冻结对下单链路有什么影响？

## 先给面试官的短答案

账号冻结会影响登录、下单、支付、优惠券使用、售后和履约变更。下单链路必须在关键节点校验账号
状态，不能只在登录时校验一次。

账号状态变更还要考虑缓存失效、正在进行的订单和用户提示。

## 校验点

校验点：

- 登录时。
- 创建订单前。
- 使用优惠券前。
- 发起支付前。
- 支付回调处理时。
- 申请退款和售后时。

冻结后不应允许继续发起高风险交易。

## 缓存问题

账号状态常被缓存。

冻结账号后要：

- 删除用户状态缓存。
- 提高 token version。
- 使旧 token 失效。
- 通知相关服务。
- 对关键接口回源校验。

否则冻结可能延迟生效。

## 已有订单

要区分：

- 已支付订单是否继续履约。
- 未支付订单是否关闭。
- 售后是否允许。
- 风控冻结是否需要人工审核。

不同冻结原因策略不同。

## 在 eMall 项目中怎么讲？

eMall 用户因盗号风险被冻结后，下单接口和支付接口都应拒绝新交易，并返回明确提示。

如果用户已有已支付订单，履约是否暂停应由风控策略决定，不能简单删除订单或破坏支付状态。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
账号冻结不是只影响登录，它会影响下单、支付、优惠券、售后和履约。系统要在关键交易节点校验账号
状态，并让冻结状态及时失效缓存和 token。

已有订单要按冻结原因区分处理。未支付订单可以关闭，已支付订单是否继续履约或人工审核要有风控
策略，不能破坏订单状态机。
```

## 回答评分点

高分答案应该覆盖：

- 冻结影响多个交易节点。
- 不能只登录时校验。
- 缓存和 token 要失效。
- 已有订单要按状态处理。
- 风控策略要可解释。
