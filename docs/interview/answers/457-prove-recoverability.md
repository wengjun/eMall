# 457 如何证明系统具备可恢复能力？

[返回按分类学习面试题](../README.md)

## 题目

如何证明系统具备可恢复能力？

## 先给面试官的短答案

证明可恢复能力不能靠口头说明，要靠演练记录、恢复指标、备份恢复验证、故障注入结果、Runbook、
RTO/RPO 达成情况和真实事故复盘证据。

能恢复的系统，必须经过验证。

## 证据类型

证据：

- 定期恢复演练报告。
- 备份恢复校验结果。
- 故障演练记录。
- RTO 和 RPO 实测数据。
- 自动化回滚记录。
- 事故复盘行动项。
- 监控和告警覆盖率。

这些比架构图更有说服力。

## 恢复能力维度

维度：

- 服务实例恢复。
- 数据库恢复。
- 缓存恢复。
- MQ 积压恢复。
- 下游恢复后平滑放量。
- 数据一致性修复。
- 人员响应恢复。

恢复不是只重启服务。

## 验证方式

方式：

- 定期灾备演练。
- Chaos 工程。
- 灰度故障注入。
- 备份恢复抽检。
- Runbook 演练。
- 指标回放和复盘。

验证要周期化。

## 在 eMall 项目中怎么讲？

eMall 可以用订单库备份恢复演练证明 RPO/RTO，用 Redis 故障演练证明缓存降级，用 Kafka 积压演练
证明补发和幂等能力。

每次演练记录恢复时间、数据校验结果和改进项。

## 深度增强：Kubernetes 运维治理图

![Kubernetes 生产运行和故障治理](../assets/kubernetes-operations.svg)

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
系统可恢复能力要靠证据证明，包括备份恢复演练、故障注入、RTO/RPO 实测、Runbook 演练、自动化
回滚记录和事故复盘。

真正的恢复能力覆盖服务、数据、缓存、消息、下游依赖和人员流程。只有定期验证并记录结果，才能
证明系统不是纸面高可用。
```

## 回答评分点

高分答案应该覆盖：

- 不能只靠口头承诺。
- RTO/RPO 要实测。
- 备份恢复和故障演练。
- 恢复包括数据一致性。
- 演练结果要记录和复盘。
