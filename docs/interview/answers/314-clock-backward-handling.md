# 314 时钟回拨如何处理？

[返回按分类学习面试题](../README.md)

## 题目

时钟回拨如何处理？

## 先给面试官的短答案

时钟回拨会让依赖时间生成 ID 或判断顺序的系统出现重复、乱序和过期判断错误。
处理方式包括检测回拨、小幅回拨等待、大幅回拨拒绝发号、切换 workerId、使用逻辑时钟或号段服务，并加强 NTP 监控。

不能在时钟回拨时继续无保护发号。

## 小幅回拨

如果回拨很小，例如几毫秒：

- 可以等待时间追上。
- 阻塞发号一小段时间。
- 记录告警。

这种方式简单，但会影响短时间可用性。

## 大幅回拨

如果回拨较大：

- 拒绝发号。
- 切换备用 workerId。
- 使用逻辑时钟补偿。
- 告警人工处理。

继续发号可能产生重复 ID。

## 工程治理

需要：

- NTP 同步监控。
- 禁止手工改生产时间。
- 发号服务记录最近时间戳。
- 机器重启后校验时间。
- 多机房时钟偏差监控。

时钟是基础依赖，需要运维治理。

## 在 eMall 项目中怎么讲？

订单 ID 服务发现当前时间小于上次发号时间时，先计算回拨差值。

小于阈值则等待，大于阈值则拒绝发号并告警，避免生成重复订单 ID。

record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {

## 专家级完整回答

```text
时钟回拨对 Snowflake 这类时间型 ID 很危险，可能导致重复 ID 或顺序异常。
生产系统应记录上次发号时间并检测回拨。小幅回拨可以等待，大幅回拨应拒绝发号、切换 workerId 或进入保护模式。

同时要做好 NTP 监控、禁止手工改时钟和多机房时钟偏差治理。
```

## 回答评分点

高分答案应该覆盖：

- 时钟回拨可能导致重复 ID。
- 小幅回拨可以等待。
- 大幅回拨应拒绝或切换。
- 要记录上次发号时间。
- NTP 和运维治理很重要。
