# 517 设计支付系统

[返回按分类学习面试题](../README.md)

## 题目

设计支付系统。

## 先给面试官的短答案

支付系统的核心是安全、幂等、状态正确和可对账。它负责创建支付单、调用支付渠道、处理同步返回和异步回调、
验签、金额校验、状态机更新、通知订单、退款和渠道对账。资金链路不能依赖单次回调，必须通过幂等、对账和补偿兜底。

## 核心模型

支付系统通常包含支付单、支付流水、退款单、渠道流水、回调记录和对账记录。订单和支付单是一对一或一对多关系，
支付流水记录每次渠道交互。

金额必须使用整数分或高精度类型，不能用浮点数。

## 支付流程

订单服务创建订单后请求支付服务生成支付单。支付服务选择渠道，生成支付参数或二维码，返回给前端。

用户支付后，渠道发送异步回调。支付服务验签、校验金额、校验商户号和订单号，然后幂等更新支付单状态。

支付成功后通过 Outbox 或消息通知订单服务，订单服务幂等更新订单为已支付。

## 幂等和状态机

支付回调可能重复、乱序和延迟。支付服务必须基于渠道交易号、支付单号和回调 ID 做幂等。

状态机要防止成功状态被失败回调覆盖。已成功的支付单不能回退为失败。已取消订单收到成功回调时，应进入退款或人工处理。

## 安全和风控

所有渠道回调必须验签，签名要覆盖关键字段。金额、币种、商户号和订单号必须和本地支付单一致。

高风险支付要接入风控，例如异常用户、异常设备、频繁失败、黑名单和大额交易。

## 对账和补偿

支付不能只依赖回调。每天或准实时拉取渠道账单，与本地支付单和订单状态对账。
如果渠道成功本地失败，要补单；如果本地成功渠道失败，要人工处理或冲正。

## 在 eMall 项目中怎么讲？

eMall 的 `payment` 模块应聚合渠道适配、回调验签、支付状态机、Outbox 通知、退款和对账。它和 `order` 通过事件最终一致，
和 `risk` 协同做支付风控。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

## 专家级完整回答

```text
支付系统设计的第一原则是资金正确，第二原则是可追溯和可恢复。

我会把支付单、支付流水、回调记录、退款单和对账记录建模清楚。支付回调必须验签、校验金额，
并通过幂等和状态机防止重复入账和状态回退。

支付成功后通过可靠事件通知订单。因为回调可能丢失或延迟，所以还要做渠道对账和补偿。
在资金系统里，不能把一次接口调用成功当成最终真相，对账才是兜底闭环。
```

## 回答评分点

高分答案应该覆盖：

- 覆盖支付单、流水、回调、退款和对账模型。
- 能讲清创建支付、渠道回调、通知订单流程。
- 知道验签、金额校验和幂等状态机。
- 能处理重复、乱序、延迟和已取消订单回调。
- 强调渠道对账和补偿是资金兜底。

## 深度完善：系统设计高分展开

围绕「设计支付系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。

深度完善标记：系统设计题已补容量、数据流、失败路径和验收口径。
