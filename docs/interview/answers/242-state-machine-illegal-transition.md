# 242 状态机如何防止非法状态跳转？

[返回按分类学习面试题](../README.md)

## 题目

状态机如何防止非法状态跳转？

## 先给面试官的短答案

状态机通过定义合法状态、事件和转移规则，拒绝不符合当前状态的操作。
工程上要用状态转移表、乐观锁、条件更新、幂等校验和审计日志，避免订单从已取消跳到已支付这类非法变化。

状态机是分布式一致性的核心防线。

## 基本模型

状态机包含：

- 当前状态。
- 触发事件。
- 目标状态。
- 转移条件。
- 转移动作。
- 失败处理。

只有规则表中存在的转移才允许执行。

## 数据库防护

常用条件更新：

```sql
UPDATE orders
SET status = 'PAID', version = version + 1
WHERE id = ?
  AND status = 'PENDING_PAYMENT'
  AND version = ?;
```

如果更新行数为 0，说明状态已变化或事件非法。

## 处理重复事件

如果订单已经是 `PAID`，再次收到支付成功事件，可以幂等返回成功。

如果订单是 `CANCELED`，收到支付成功事件则要进入异常处理或退款流程，不能直接改成已支付。

## 在 eMall 项目中怎么讲？

订单允许 `PENDING_PAYMENT -> PAID`，但不允许 `CANCELED -> PAID`。

支付事件到达时，订单服务用状态机和版本号更新，非法转移进入异常单。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
状态机通过显式定义状态、事件和合法转移来防止非法跳转。实现时不能只在内存判断，
还要用数据库条件更新、版本号和唯一约束保证并发下也正确。

重复事件按幂等处理，非法事件进入异常处理。状态转移要记录审计，便于追踪一致性问题。
```

## 回答评分点

高分答案应该覆盖：

- 状态机要定义合法转移。
- 数据库条件更新防并发问题。
- 重复事件和非法事件处理不同。
- 版本号或乐观锁很重要。
- 状态变更要审计。
