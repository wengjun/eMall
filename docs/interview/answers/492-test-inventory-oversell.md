# 492 如何测试库存防超卖？

[返回按分类学习面试题](../README.md)

## 题目

如何测试库存防超卖？

## 先给面试官的短答案

测试库存防超卖要验证高并发扣减时，成功预占数量不能超过可售库存，库存不能变成负数，失败请求要明确返回库存不足。
需要覆盖数据库乐观锁、行锁、Redis 预扣、唯一预占记录、重复请求和补偿释放。

## 基本测试思路

先准备一个库存很小的 SKU，例如可售库存为 10。然后并发发起 100 个预占请求，等待全部完成后断言：

- 成功请求数等于 10。
- 失败请求数等于 90。
- 数据库库存不小于 0。
- 预占记录数量等于成功请求数。
- 同一个订单或用户重复请求不会重复预占。

这类测试必须使用真实数据库或 Redis，不能只靠 mock。

## 需要覆盖的场景

边界库存：库存为 0、库存为 1、库存刚好等于请求数。

并发扣减：多个线程同时扣同一个 SKU，验证锁或条件更新是否有效。

重复预占：同一个订单重复请求，验证幂等键或唯一索引。

释放补偿：订单取消或支付超时释放库存，不能多释放或少释放。

回滚场景：库存扣减成功但订单创建失败时，要验证补偿或事务一致性。

## SQL 级验证点

如果使用数据库条件更新，关键是类似 `where available_quantity >= ?` 的原子条件。测试要证明并发下只有满足条件的请求成功。
如果使用乐观锁，要验证版本号冲突处理。如果使用 Redis 预扣，要验证最终数据库落库和 Redis 数量能对齐。

## 在 eMall 项目中怎么讲？

eMall 的 `inventory` 模块应有并发集成测试。秒杀 SKU 可以使用 Redis 预扣提升吞吐，但最终必须有数据库库存和预占记录兜底，
并通过对账修正 Redis 和数据库差异。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 专家级完整回答

```text
库存防超卖测试一定要模拟并发，而不是只测单线程扣减成功。

我会准备小库存 SKU，发起远大于库存的并发请求，然后断言成功数不超过库存、库存不为负、
预占记录数量正确、重复请求不会重复占用。

实现上要根据方案验证关键语义。数据库方案要验证条件更新、行锁或乐观锁；Redis 方案要验证 Lua 原子性、
落库一致性和补偿释放。对电商系统来说，防超卖测试是生产级集成测试的核心之一。
```

## 回答评分点

高分答案应该覆盖：

- 说明要用并发测试证明成功数不超过库存。
- 覆盖库存为 0、1、刚好够等边界。
- 知道要断言库存、预占记录和失败数量。
- 能说明数据库条件更新、乐观锁、Redis Lua 的验证点。
- 能覆盖释放补偿和重复请求。
