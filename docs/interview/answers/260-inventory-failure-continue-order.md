# 260 库存服务故障能不能继续下单？

[返回按分类学习面试题](../README.md)

## 题目

库存服务故障能不能继续下单？

## 先给面试官的短答案

库存服务故障时能否继续下单取决于是否能可靠确认库存和是否允许超卖。
普通实物商品通常不能在库存不可确认时继续成功下单，否则可能超卖；可以进入排队、稍后重试或缺货提示。

如果是虚拟商品、预售或可补货商品，可以按业务规则接受更宽松的一致性。

## 普通实物商品

普通商品库存不可确认时：

- 不应直接下单成功。
- 可以提示库存确认失败。
- 可以排队等待恢复。
- 可以保留购物车。
- 可以稍后通知用户。

正确性优先于成功率。

## 可继续的场景

可能继续：

- 预售商品。
- 虚拟商品。
- 可无限供给服务。
- 可延期发货商品。
- 商家允许超卖并承担履约。

这些要有明确业务规则和用户提示。

## 技术兜底

可以使用：

- 库存预占。
- 本地库存缓存。
- 秒杀令牌。
- 库存桶。
- 限流保护。
- 恢复后对账。

但缓存库存不能无限信任，必须有有效期和扣减边界。

## 在 eMall 项目中怎么讲？

秒杀商品库存服务故障时，应停止秒杀下单或排队等待，不能继续放单。

预售商品可以在页面明确说明发货时间后继续下单，因为它的业务语义允许库存后置履约。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
库存服务故障时不能一概继续下单。对普通实物和秒杀商品，库存不可确认就继续成功下单会造成超卖，
应快速失败、排队或提示稍后重试。

对预售、虚拟商品或允许延期履约的场景，可以按业务规则继续，但要明确用户提示和后续履约保障。
```

## 回答评分点

高分答案应该覆盖：

- 库存不可确认时不能默认成功。
- 普通商品和秒杀要防超卖。
- 预售和虚拟商品可按业务规则放宽。
- 本地缓存和令牌要有边界。
- 用户提示和履约保障很重要。
