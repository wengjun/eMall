# 425 如何做接口权限审计？

[返回按分类学习面试题](../README.md)

## 题目

如何做接口权限审计？

## 先给面试官的短答案

接口权限审计要记录谁在什么时间、从哪里、以什么身份、访问了哪个接口、操作了什么资源、结果如何。
高危操作还要记录审批单、变更前后值和 trace ID。

审计日志要完整、可检索、防篡改，并和告警联动。

## 审计字段

字段包括：

- 主体 ID。
- 主体类型。
- 角色和权限。
- 接口和方法。
- 资源 ID。
- 来源 IP。
- user agent。
- 操作结果。
- trace ID。
- 时间戳。

高危接口需要更详细字段。

## 审计范围

范围：

- 登录和登出。
- 权限变更。
- 数据导出。
- 批量操作。
- 支付和退款。
- 改价和库存调整。
- 运维接口。
- 开放平台调用。

不是所有请求都需要同等审计粒度。

## 使用方式

审计用于：

- 事后追溯。
- 异常检测。
- 合规证明。
- 权限复核。
- 事故分析。

审计日志本身也要保护访问权限。

## 在 eMall 项目中怎么讲？

eMall 批量退款接口要记录操作人、审批单号、订单列表摘要、退款金额、变更前后状态和 trace ID。

如果某个账号短时间导出大量订单，审计系统应触发告警。

## 深度增强：安全治理图

![开放平台 API 安全链路](../assets/openapi-security.svg)

安全题要从身份、权限、数据、审计和攻击面分层回答。认证只证明是谁，授权判断能做什么；
加密保护机密性，签名保护完整性；审计负责事后追踪，风控负责发现异常行为。

## 深度增强：Java 17 安全策略示例

```java
import java.util.Set;

record SecurityDecision(boolean allowed, String reason) {
}

final class ScopePolicy {

    SecurityDecision check(Set<String> scopes, String requiredScope) {
        if (scopes.contains(requiredScope)) {
            return new SecurityDecision(true, "allowed");
        }
        return new SecurityDecision(false, "missing scope: " + requiredScope);
    }
}
```

这段代码体现最小权限原则。生产系统还要结合租户、资源归属、IP、设备、风险等级和操作审计。

## 深度增强：生产边界

安全不能只靠前端隐藏按钮，也不能只在网关做一次判断。核心资源要在服务层做资源级授权，
敏感数据要脱敏，密钥要支持轮换，失败日志不能泄露 token、secret、身份证号和银行卡号。

## 深度增强：面试高分表达

我会把安全问题拆成认证、授权、防篡改、防重放、数据保护、审计和风控。
每一层解决的问题不同，不能互相替代。这能体现我理解开放平台和电商系统的真实攻击面。

## 专家级完整回答

```text
接口权限审计要记录主体、身份、权限、接口、资源、来源、结果和 trace ID。高危操作还要记录审批单、
变更前后值和业务摘要。

审计日志要可检索、防篡改、权限受控，并用于异常告警、合规证明和事故复盘。审计不是普通业务日志，
它是安全和治理证据。
```

## 回答评分点

高分答案应该覆盖：

- 记录谁访问了什么资源。
- 高危操作记录审批和变更前后值。
- 审计日志要可检索和防篡改。
- 支持告警和合规。
- 审计日志也要权限保护。
