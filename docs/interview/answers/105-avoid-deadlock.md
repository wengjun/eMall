# 105 如何避免死锁？

[返回按分类学习面试题](../README.md)

## 题目

如何避免死锁？

## 先给面试官的短答案

避免死锁的核心是破坏死锁条件，最常用方法是固定加锁顺序、减少锁嵌套、缩小锁粒度、避免锁内 IO、
使用超时锁和事务短小化。数据库场景要统一更新顺序，减少长事务和热点行竞争。

死锁预防应该在设计和代码评审阶段完成，而不是等线上发生后再处理。

## 固定加锁顺序

如果多个线程需要获取多把锁，必须按统一顺序。

例如总是先锁用户，再锁订单。

不要一个流程先锁 A 再锁 B，另一个流程先锁 B 再锁 A。

固定顺序能破坏循环等待。

## 减少锁嵌套

嵌套锁越多，死锁概率越高。

可以把大临界区拆成多个小步骤，减少同时持有多把锁的时间。

如果必须嵌套，要明确顺序并写入规范。

## 锁内不做 IO

锁内不要做：

- HTTP 调用。
- 数据库查询。
- Redis 调用。
- 文件 IO。
- MQ 发送。

IO 时间不可控，会拉长持锁时间，增加死锁和锁竞争风险。

## 使用超时锁

`ReentrantLock.tryLock(timeout)` 可以避免无限等待。

获取不到锁时，可以：

- 返回稍后重试。
- 降级。
- 放入队列。
- 记录告警。

但业务必须处理失败路径。

## 数据库死锁预防

数据库中要：

- 统一表和行更新顺序。
- 缩短事务时间。
- 避免事务中调用外部服务。
- 使用合适索引。
- 控制批量更新范围。
- 降低热点行竞争。

很多线上死锁其实是数据库事务死锁。

## 在 eMall 项目中怎么讲？

订单支付成功后更新订单、库存、账务时，要避免在一个长事务中做所有事情。

更常见的设计是本地事务更新自身状态，发送事件，其他模块异步处理，并通过状态机、幂等和补偿保证最终一致。

这样能减少跨资源锁等待。

## 深度增强：并发治理图

![Java 并发从线程安全到容量保护](../assets/concurrency-governance.svg)

并发题不能只回答 API 用法。生产系统要同时考虑线程安全、资源隔离、超时、拒绝、幂等和分布式多实例。
单机锁只能保护当前 JVM，不能保护整个集群；线程池满也不是小问题，而是容量和可用性风险。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

## 深度增强：面试高分表达

我会把并发问题分成三层：JMM 和锁保证单机正确性，线程池和隔离保证资源不被拖垮，
幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
避免死锁要破坏循环等待和持有并等待。工程上我会固定加锁顺序，减少锁嵌套，缩小锁范围，
避免锁内 IO，必要时用 tryLock 超时。数据库层要统一更新顺序、缩短事务、避免长事务和热点行锁。

在分布式业务中，我倾向于用状态机、事件和补偿拆掉大事务，减少同时持有多个资源的时间。
```

## 回答评分点

高分答案应该覆盖：

- 固定加锁顺序。
- 减少嵌套和锁内 IO。
- 使用超时锁。
- 数据库更新顺序和短事务。
- 用状态机和异步拆复杂流程。
