# 490 为什么只测 happy path 不够？

[返回按分类学习面试题](../README.md)

## 题目

为什么只测 happy path 不够？

## 先给面试官的短答案

happy path 只验证一切正常时系统能工作，但生产故障大多发生在异常输入、并发冲突、依赖超时、重复请求、乱序消息、
资源不足和部分失败时。只测正常路径会让系统在真实复杂场景下缺少防御能力。

## 什么是 happy path

happy path 是最顺利的业务路径。例如用户有库存、价格正常、优惠可用、支付成功、消息投递成功，订单顺利完成。
它应该被测试，但它只能证明系统在理想条件下可用。

## 还必须覆盖什么

异常输入要测，例如金额为负、优惠券过期、SKU 不存在、用户被冻结。

边界条件要测，例如库存为 0、库存为 1、订单金额刚好等于阈值、活动开始和结束瞬间。

并发场景要测，例如多个用户抢同一个 SKU、重复点击支付、重复提交订单。

部分失败要测，例如订单已创建但消息发送失败、支付成功但回调超时、库存预占成功但订单落库失败。

下游异常要测，例如 Redis 超时、数据库唯一键冲突、Kafka 暂不可用、第三方支付返回未知状态。

## 测试的真实目标

测试不是证明代码“在顺利时能跑”，而是证明系统在不顺利时不会造成资金损失、库存错误、状态污染和不可恢复故障。

高质量测试一定包含失败路径、补偿路径和恢复路径。

## 在 eMall 项目中怎么讲？

eMall 的核心链路要测重复下单、库存不足、支付重复回调、优惠失效、Outbox 失败重试和 MQ 重复消费。
这些场景比单纯下单成功更能体现系统是否生产可用。

## 深度增强：现场编码工程化图

![现场编码题的工程化解法](../assets/coding-patterns.svg)

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
只测 happy path 会给团队一种系统已经稳定的错觉。

真实生产环境中，问题往往来自异常输入、并发、重复请求、下游超时、消息乱序和部分失败。
这些场景不覆盖，系统就无法证明自己具备幂等、补偿、容错和恢复能力。

对电商系统来说，我更关注失败时是否不多扣钱、不超卖、不丢消息、不产生错误状态。
所以测试必须覆盖正常路径、异常路径、边界路径和恢复路径。
```

## 回答评分点

高分答案应该覆盖：

- 说明 happy path 只覆盖理想条件。
- 能列出异常输入、边界、并发、部分失败和下游异常。
- 知道测试要证明系统失败时安全。
- 能结合库存、支付、订单、消息举例。
- 能体现容错和恢复意识。
