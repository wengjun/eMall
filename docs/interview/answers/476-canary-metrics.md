# 476 灰度发布观察哪些指标？

[返回按分类学习面试题](../README.md)

## 题目

灰度发布观察哪些指标？

## 先给面试官的短答案

灰度发布要观察技术指标、业务指标、依赖指标和用户体验指标。技术上看错误率、延迟、吞吐、资源、重试和熔断；
业务上看下单成功率、支付成功率、库存预占成功率、转化率和退款异常；依赖上看数据库、缓存、消息和第三方通道；
用户体验上看页面错误、接口超时和客户端崩溃。

## 技术指标

技术指标用于判断服务是否稳定：

- HTTP 5xx、4xx、业务错误码和异常数。
- P50、P95、P99 延迟和慢请求比例。
- QPS、线程池队列、连接池使用率和限流量。
- CPU、内存、GC、容器重启、Pod readiness 变化。
- 重试次数、熔断打开次数、降级命中次数。

这些指标要同时比较新版本和旧版本，不能只看全局平均值。

## 业务指标

电商系统最重要的是业务正确性。灰度期间要看下单成功率、支付成功率、库存扣减成功率、优惠计算失败率、
订单取消率、退款率、履约创建成功率和核心漏斗转化率。

业务指标比技术指标更能反映真实影响。一个版本可能没有 5xx，但因为优惠计算错误导致转化率下降。

## 依赖和数据指标

灰度还要观察数据库慢 SQL、锁等待、连接池耗尽、Redis 命中率、Kafka lag、Outbox 积压和第三方支付通道延迟。
如果新版本改变了访问模式，依赖层可能先出现异常。

同时要关注数据质量，例如重复订单、库存负数、支付单状态不一致、消息重复消费和幂等冲突。

## 在 eMall 项目中怎么讲？

eMall 灰度发布可以把 `release` 模块的发布策略和 `traffic` 模块的流量治理结合起来。放量前设置自动停止条件，
例如新版本订单成功率低于旧版本、支付回调延迟升高、库存失败率超过阈值或 P99 明显劣化。

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
灰度发布不能只看服务有没有报错，而要看新版本是否比旧版本更差。

我会建立四类指标：技术稳定性指标、业务结果指标、依赖健康指标和数据质量指标。
技术指标包括错误率、延迟、资源和熔断；业务指标包括下单、支付、库存和转化；
依赖指标包括数据库、Redis、Kafka 和第三方通道；数据质量指标包括重复单、状态不一致和幂等冲突。

发布系统应该支持自动化判定。如果新版本任一核心指标劣于基线，就暂停放量或自动回滚。
核心交易链路不能依赖人工盯图。
```

## 回答评分点

高分答案应该覆盖：

- 指标要区分技术、业务、依赖和数据质量。
- 强调新旧版本对比，而不是只看全局平均值。
- 能列出订单、支付、库存、优惠等电商业务指标。
- 知道灰度应该有自动停止和回滚条件。
- 能指出没有 5xx 不代表业务没有受损。
