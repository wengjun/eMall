# 020 业务异常和系统异常应该如何区分？

[返回按分类学习面试题](../README.md)

## 题目

业务异常和系统异常应该如何区分？

## 先给面试官的短答案

业务异常表示系统正常运行，但用户请求不满足业务规则；系统异常表示代码、依赖或基础设施出现故障。

例如库存不足、订单状态不允许取消是业务异常；数据库连接失败、Redis 超时、空指针是系统异常。

区分它们的价值是：

- 业务异常返回明确错误码，通常不报警。
- 系统异常要记录日志、打指标、必要时告警。
- 下游异常要考虑重试、熔断、降级和补偿。

## 从零基础理解

用户买一件商品，但库存没有了：

```text
系统正常，只是业务不能满足请求。
```

这是业务异常。

如果数据库宕机，订单无法保存：

```text
系统本身出问题。
```

这是系统异常。

## 业务异常例子

电商系统中常见业务异常：

- 用户不存在。
- 用户被冻结。
- 商品已下架。
- 库存不足。
- 优惠券不可用。
- 订单不存在。
- 已支付订单不能直接取消。
- 支付金额不一致。
- 重复 requestId 对应不同业务参数。

这些异常通常是请求和业务规则冲突，不代表系统故障。

## 系统异常例子

系统异常包括：

- 数据库连接失败。
- SQL 语法错误。
- Redis 不可用。
- Kafka 发送失败。
- HTTP 客户端连接池耗尽。
- 空指针。
- 配置缺失。
- JSON 反序列化失败。

这些需要研发或运维介入。

## 下游异常怎么归类？

下游异常是系统异常的一类，但处理方式更细。

例如库存服务超时：

- 对订单来说，下游不可用。
- 对用户来说，可能返回“系统繁忙”。
- 对系统来说，需要熔断、重试、补偿或降级。

如果库存预占失败是因为库存不足，这是业务异常。
如果库存服务超时，这是下游系统异常。

## 响应处理差异

业务异常：

```json
{
  "success": false,
  "code": "INSUFFICIENT_STOCK",
  "message": "Insufficient stock"
}
```

系统异常：

```json
{
  "success": false,
  "code": "INTERNAL_ERROR",
  "message": "System is temporarily unavailable",
  "traceId": "..."
}
```

系统异常不能把堆栈直接返回给用户。

## 监控处理差异

业务异常通常不需要高优先级报警，但要作为业务指标统计：

- 库存不足率。
- 支付金额不匹配次数。
- 订单状态冲突次数。

系统异常需要技术指标：

- 5xx 错误率。
- 下游超时率。
- 数据库连接失败数。
- Kafka 发送失败数。

## 在 eMall 项目中怎么讲？

在 eMall 中：

- `BusinessException(ErrorCode.NOT_FOUND)` 表示资源不存在。
- `BusinessException(ErrorCode.CONFLICT)` 表示业务状态冲突。
- `DOWNSTREAM_UNAVAILABLE` 表示下游不可用。
- `INTERNAL_ERROR` 表示未预期系统错误。

统一异常处理层将异常转为响应，避免每个 Controller 重复处理。

## 常见追问

### 库存不足要不要报警？

通常不需要作为系统故障报警，但要监控趋势。如果库存不足率异常升高，可能说明库存同步或活动配置有问题。

### 支付金额不一致是业务异常还是安全风险？

它首先是业务冲突，必须拒绝；同时也可能是安全或渠道异常信号，应该记录审计并进入风控或对账分析。

### 系统异常能不能包装成业务异常？

不要。否则监控会误判系统正常，真实故障被隐藏。

## 深度增强：工程化理解图

![Java 工程能力从语法到生产设计](../assets/java-engineering-model.svg)

这类题不能只停留在语法解释。生产系统更关心它如何改善建模、降低误用、保护兼容性、提升可测试性，
以及能否让团队在多人协作中保持稳定边界。回答时要从语言特性落到业务约束和工程治理。

## 深度增强：Java 17 落地示例

```java
import java.util.Objects;

record StableApiField(String name, String type, boolean required) {

    StableApiField {
        Objects.requireNonNull(name);
        Objects.requireNonNull(type);
        if (name.isBlank() || type.isBlank()) {
            throw new IllegalArgumentException("API field metadata must be explicit");
        }
    }
}

final class ApiCompatibilityPolicy {

    boolean canAddField(StableApiField field) {
        return !field.required();
    }
}
```

这段代码体现 Java 17 在工程建模中的价值：用 `record` 表达不可变数据，用构造校验保护边界，
用小的策略类表达兼容规则。面试中要把语法能力和 API 演进、错误预防、团队协作联系起来。

## 深度增强：生产边界

语言特性不是越新越好。核心原则是可读、可测、可维护、可兼容。任何语法选择都要能让代码意图更清晰，
而不是为了炫技。公共 API、金额、时间、状态、异常和 DTO 都要有稳定约束，避免线上数据被随意破坏。

## 深度增强：面试高分表达

我会先回答概念，再说明它在电商系统中的真实作用。例如金额要避免精度错误，状态要可兼容扩展，
DTO 和领域对象要隔离外部契约和内部模型。这样能体现我不是只会写 Java 语法，而是能做工程设计。

## 专家级完整回答

```text
业务异常表示系统按预期拒绝了不合法业务请求，例如库存不足、订单状态冲突、支付金额不一致。
系统异常表示系统自身或依赖出现问题，例如数据库连接失败、Redis 超时、Kafka 发送失败。

区分二者是为了正确响应、监控和恢复。业务异常返回稳定错误码，通常不触发事故告警；
系统异常要记录 traceId、错误日志、技术指标，并根据情况触发告警。
下游异常还要结合超时、重试、熔断、降级和补偿处理。
```

## 回答评分点

高分答案应该覆盖：

- 能清楚定义业务异常和系统异常。
- 能举库存不足、订单状态冲突、数据库失败例子。
- 能说明响应、日志、监控和告警差异。
- 能处理下游异常这个灰度地带。
- 能指出不要把系统异常伪装成业务异常。

## 深度完善：面向 L6 的回答框架

围绕「业务异常和系统异常应该如何区分？」，高分答案不能停在概念定义，而要把「语言特性、建模边界、兼容性和团队编码规范」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「Java 语言和工程基础」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `common、order、inventory、payment 的 DTO、值对象、异常和公共 API` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：代码评审问题数、缺陷逃逸率、兼容性测试结果、静态检查违规数。
- 验证证据：代码规范、单元测试、契约测试、兼容性用例和重构前后缺陷数据。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
