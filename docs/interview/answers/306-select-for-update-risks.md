## 题目

`select for update` 有什么风险？

## 先给面试官的短答案

`select for update` 会对读取到的记录加锁，使用不当会导致锁等待、死锁、间隙锁、吞吐下降和事务阻塞。
如果查询条件没有命中合适索引，锁范围可能远大于预期。

它适合短事务内保护关键行，不适合长事务和大范围查询。

## 主要风险

风险包括：

- 锁持有时间过长。
- 阻塞其他更新。
- 范围查询产生间隙锁。
- 缺少索引导致锁范围扩大。
- 加锁顺序不一致导致死锁。
- 事务内远程调用放大等待。

锁越久，系统吞吐越低。

## 错误用法

危险做法：

- 在事务内调用远程服务。
- 对大范围数据 `for update`。
- 查询条件没有索引。
- 用户交互期间持锁。
- 批量任务长时间持锁。

这些会明显影响在线交易。

## 替代方式

可以考虑：

- 条件更新。
- 乐观锁。
- 唯一约束。
- 状态机。
- 队列串行化。
- 更小粒度锁。

不是所有并发问题都要用 `for update`。

## 在 eMall 项目中怎么讲？

库存扣减可以优先使用条件更新，而不是先 `select for update` 再扣减。

如果确实使用加锁读，事务内只做必要本地修改，不能调用支付或库存外部服务。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
select for update 是当前读，会加锁保护读取到的记录。风险是锁等待、死锁、间隙锁和锁范围扩大，
尤其在缺少索引或范围查询时。

它应只用于短事务和小范围关键行。很多库存和状态更新场景可以用条件更新、乐观锁、状态机和唯一约束替代。
```

## 回答评分点

高分答案应该覆盖：

- `select for update` 会加锁。
- 缺少索引会扩大锁范围。
- 长事务和远程调用很危险。
- 可能导致死锁和锁等待。
- 条件更新和乐观锁可替代部分场景。

## 深度完善：面向 L6 的回答框架

围绕「`select for update` 有什么风险？」，高分答案不能停在概念定义，而要把「索引、事务、锁、MVCC、分库分表、冷热分层、备份恢复和在线变更」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「数据库和数据建模」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `order、payment、inventory、product 的核心表、唯一约束、索引和归档表` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：慢 SQL、锁等待、死锁数、主从延迟、连接池使用率、归档任务耗时。
- 验证证据：EXPLAIN、慢 SQL 日志、死锁日志、迁移方案、备份恢复演练和容量预测。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
