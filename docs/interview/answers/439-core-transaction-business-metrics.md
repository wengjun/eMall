## 题目

核心交易链路要监控哪些业务指标？

## 先给面试官的短答案

核心交易链路除了技术指标，还要监控业务指标，包括下单成功率、支付成功率、库存扣减成功率、订单
取消率、退款率、履约创建成功率、GMV、转化率和各状态积压量。

业务指标能告诉我们用户和交易是否真的受影响。

## 下单指标

下单关注：

- 下单请求数。
- 下单成功率。
- 创建订单失败率。
- 幂等冲突数。
- 库存不足比例。
- 风控拒绝率。
- P99 延迟。

下单是交易入口。

## 支付指标

支付关注：

- 支付单创建成功率。
- 支付回调成功率。
- 支付金额不一致数。
- 支付状态机冲突数。
- 第三方通道错误率。
- 支付超时率。

支付指标直接影响收入。

## 履约和售后指标

关注：

- 履约单创建成功率。
- 发货延迟。
- 退款成功率。
- 售后创建成功率。
- Outbox 积压。
- 状态长时间卡住数量。

交易不是支付成功就结束。

## 在 eMall 项目中怎么讲？

eMall 首页流量正常但 GMV 下降时，技术 QPS 可能看不出问题。

需要看商品详情转化、下单成功率、支付成功率、库存失败率和风控拒绝率，才能定位是哪一段漏斗异常。

## 深度增强：Kubernetes 运维治理图

![Kubernetes 生产运行和故障治理](../assets/kubernetes-operations.svg)

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
核心交易链路要同时监控技术指标和业务指标。业务上要看下单成功率、支付成功率、库存扣减成功率、
履约创建成功率、退款率、GMV、转化率和状态积压。

这些指标能反映真实业务影响。技术指标正常但业务指标异常时，仍然是生产事故。
```

## 回答评分点

高分答案应该覆盖：

- 下单成功率。
- 支付成功率。
- 库存和履约成功率。
- GMV 和转化率。
- 状态积压和异常状态。

## 深度完善：面向 L6 的回答框架

围绕「核心交易链路要监控哪些业务指标？」，高分答案不能停在概念定义，而要把「日志、指标、Trace、SLO、告警、Runbook、事故复盘和故障演练」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「可观测性和 SRE」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `observability、analytics、operations、gateway、order、payment 的核心看板` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：RED/USE 指标、SLO、错误预算、告警准确率、MTTD、MTTR。
- 验证证据：Trace 样例、指标看板、告警规则、Runbook、事故时间线和复盘行动项。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
