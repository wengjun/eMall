# 241 幂等表无限增长怎么办？

[返回按分类学习面试题](../README.md)

## 题目

幂等表无限增长怎么办？

## 先给面试官的短答案

幂等表不能无限保留在线热数据，要按业务风险设计过期、分区、归档和清理策略。
普通幂等记录可以 TTL 清理，高风险资金类记录可以冷热分层，在线保留近期数据，历史进入归档或审计库。

清理前必须确认重复请求和消息重放窗口已经结束。

## 增长风险

风险包括：

- 表越来越大导致索引膨胀。
- 写入和查询变慢。
- 清理任务影响主库。
- 存储成本上升。
- 备份和恢复时间变长。

幂等表本身也需要容量治理。

## 治理方式

常见方式：

- 按时间分区。
- 设置过期时间。
- 定期归档。
- 批量小步删除。
- 冷热分离。
- 按业务线拆表。
- 监控表大小和索引大小。

删除要低峰执行，并控制每批数量。

## 保留策略

保留期要覆盖：

- 客户端重试窗口。
- MQ 最大保留期。
- 死信重放窗口。
- 对账周期。
- 审计要求。

支付和退款记录通常比普通接口保留更久。

## 在 eMall 项目中怎么讲？

订单创建幂等记录可以按月分区，近期分区在线查询，历史分区归档。

支付幂等记录要覆盖渠道回调、对账和审计周期，不能简单一天后删除。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
幂等表增长要通过过期、分区、归档和冷热分离治理。保留期要覆盖客户端重试、MQ 重放、死信恢复和对账周期。
普通业务可以 TTL 清理，高风险资金类记录要长期保留或归档。

清理任务要小批量、低峰执行，并监控表大小、索引大小和查询性能，避免幂等表成为新的瓶颈。
```

## 回答评分点

高分答案应该覆盖：

- 幂等表需要生命周期治理。
- 保留期要覆盖重试和重放窗口。
- 分区、归档、TTL 是常见手段。
- 高风险业务保留更久。
- 清理要避免影响主库。
