## 题目

熔断和限流有什么区别？

## 先给面试官的短答案

限流是控制进入系统或某个资源的请求量，防止超过承载能力。
熔断是当下游已经出现高错误率或高延迟时，临时停止调用下游，防止故障扩散。

限流面向流量入口和容量保护，熔断面向依赖故障隔离。

## 限流关注什么？

限流关注：

- QPS。
- 并发数。
- 用户频率。
- appKey 配额。
- SKU 热点。
- 下游容量。

它通常在请求进入系统前或访问关键资源前生效。

## 熔断关注什么？

熔断关注：

- 下游错误率。
- 下游慢调用。
- 连续失败。
- 调用超时。
- 依赖健康状态。

它通常在调用下游时生效。

## 两者关系

两者可以同时存在：

- 限流防止过量请求进入。
- 熔断防止故障下游拖垮调用方。
- 降级提供兜底结果。

三者共同构成稳定性保护。

## 在 eMall 项目中怎么讲？

秒杀入口要限流，防止超出订单系统承载。

如果优惠服务错误率飙升，订单服务要熔断优惠调用，避免下单线程被优惠服务拖住。

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
限流和熔断保护的对象不同。限流控制请求流量，防止系统或资源超过容量；
熔断根据下游错误率、慢调用和超时判断依赖不可用，临时停止调用以防故障扩散。

在生产系统中，入口限流、依赖熔断和业务降级要组合使用，而不是互相替代。
```

## 回答评分点

高分答案应该覆盖：

- 限流控制流量。
- 熔断隔离故障依赖。
- 触发条件不同。
- 生效位置不同。
- 二者可以组合使用。

## 深度完善：面向 L6 的回答框架

围绕「熔断和限流有什么区别？」，高分答案不能停在概念定义，而要把「限流、熔断、降级、超时、重试、热点隔离、平滑恢复和过载保护」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「高并发和稳定性」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `traffic、reliability、gateway、flash-sale、order、inventory 的稳定性治理` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：限流命中率、熔断状态、降级比例、P99、错误率、队列积压和恢复斜率。
- 验证证据：压测报告、故障演练、灰度曲线、熔断状态日志、Runbook 和大促复盘。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
