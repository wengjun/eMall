# 526 设计商家入驻和商品审核系统

[返回按分类学习面试题](../README.md)

## 题目

设计商家入驻和商品审核系统。

## 先给面试官的短答案

商家入驻和商品审核系统要管理商家资质、合同、店铺、品牌授权、商品发布、内容审核和风控。核心是流程编排、
状态机、材料存证、人工审核、自动审核、审计追踪和违规处置。它影响商品供给质量和平台合规。

## 商家入驻流程

商家提交企业信息、法人信息、营业执照、税务信息、银行账户、品牌授权和联系人。系统先做格式校验和自动风控，
再进入人工审核或第三方资质校验。

审核通过后创建商家、店铺、结算账户和合同记录。审核失败要给出原因并允许补充材料。

## 商品审核流程

商家发布商品时，系统校验类目、品牌、标题、图片、属性、价格、库存和合规词。自动审核可以拦截敏感词、
禁售品、图片违规和类目不匹配。

高风险商品进入人工审核。审核通过后才能上架，审核失败则返回整改原因。

## 状态机和审计

商家状态包括待提交、审核中、通过、拒绝、冻结、清退。商品状态包括草稿、待审核、审核通过、上架、下架、违规冻结。

每次审核动作要记录审核人、时间、原因、前后状态、材料版本和证据，便于争议处理和合规审计。

## 风控和治理

系统要支持黑名单、风险评分、抽检、违规处罚、保证金、类目准入和高危商品二次审核。

商品审核不是一次性动作。上架后也要支持巡检和用户投诉触发复审。

## 在 eMall 项目中怎么讲？

eMall 的 `merchant`、`catalog`、`product` 和 `risk` 可以协同实现商家入驻和商品审核。
`operations` 提供内部审核后台，`openapi` 可提供商家系统接口。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

## 深度增强：面试高分表达

这样能体现架构判断力，而不是简单罗列中间件。

## 专家级完整回答

```text
商家入驻和商品审核系统本质是平台供给侧治理系统。

我会用流程引擎或状态机管理商家资质审核和商品审核。自动审核处理格式、敏感词、资质和风险规则，
人工审核处理高风险和争议场景。

关键是审计和合规。每个材料版本、审核动作、状态变化和拒绝原因都要可追溯。
否则后续投诉、监管和商家争议都无法处理。
```

## 回答评分点

高分答案应该覆盖：

- 覆盖商家资质、合同、店铺、品牌授权和结算账户。
- 覆盖商品标题、图片、类目、品牌和合规审核。
- 能说明自动审核、人工审核和复审。
- 强调状态机、材料版本和审计追踪。
- 能结合风控、运营后台和开放接口说明。

## 深度完善：系统设计高分展开

围绕「设计商家入驻和商品审核系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。

深度完善标记：系统设计题已补容量、数据流、失败路径和验收口径。
