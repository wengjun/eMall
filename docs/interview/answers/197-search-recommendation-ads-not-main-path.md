# 197 搜索、推荐、广告为什么通常不放在交易主链路？

[返回按分类学习面试题](../README.md)

## 题目

搜索、推荐、广告为什么通常不放在交易主链路？

## 先给面试官的短答案

搜索、推荐、广告属于流量和转化链路，核心目标是曝光、召回、排序和点击转化；交易主链路关注下单、库存、支付和履约。
它们 SLA、数据模型、容量特征和故障容忍度不同，放在主链路会增加下单延迟和故障半径。

推荐和广告失败可以降级，支付和库存失败则会影响交易正确性。

## 差异在哪里？

差异包括：

- 交易链路要求正确性和一致性。
- 推荐广告更关注效果和实时特征。
- 搜索需要复杂索引和排序。
- 推荐广告可容忍降级或空结果。
- 交易链路不能被非核心能力拖慢。

把它们混在一起会让系统目标冲突。

## 正确关系

它们应通过事件和数据平台连接：

- 交易服务发布订单、支付、履约事件。
- 推荐系统消费行为数据更新模型。
- 搜索系统消费商品变更更新索引。
- 广告系统消费曝光和点击事件。

在线交易链路不依赖它们完成下单。

## 降级策略

推荐失败可以：

- 返回默认推荐。
- 返回热门商品。
- 返回空模块。
- 使用缓存结果。

广告失败可以隐藏广告位，但不能影响用户结算和支付。

## 在 eMall 项目中怎么讲？

订单提交接口不应同步调用广告计费、推荐模型刷新或搜索索引更新。

订单完成后发布事件，搜索、推荐、广告和数据仓库异步消费，这样交易主链路保持短而稳定。

## 深度增强：可观测与配置治理图

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
搜索、推荐、广告和交易主链路的目标不同。交易链路强调正确性、一致性和低故障半径，
搜索推荐广告强调召回、排序、效果和流量变现，失败时通常可以降级。

我会让交易服务通过事件把行为和订单数据发出去，由搜索、推荐、广告异步消费。主链路不依赖它们完成下单和支付。
```

## 回答评分点

高分答案应该覆盖：

- 交易链路和流量链路目标不同。
- 推荐广告失败可降级。
- 主链路要短而稳定。
- 通过事件连接数据。
- 搜索索引和推荐模型应异步更新。
