# 238 对账发现差异后如何处理？

[返回按分类学习面试题](../README.md)

## 题目

对账发现差异后如何处理？

## 先给面试官的短答案

对账发现差异后，应先分类和定级，再判断数据权威来源，最后通过自动补偿或人工审核修复。
资金、库存、订单状态类差异要高优先级处理，并保留完整审计。

不能看到差异就直接覆盖数据，必须理解业务语义和权威系统。

## 差异分类

常见差异：

- 我方有，对方没有。
- 对方有，我方没有。
- 金额不一致。
- 状态不一致。
- 数量不一致。
- 时间延迟导致暂时不一致。

不同类型处理方式不同。

## 权威来源

要明确：

- 支付渠道对支付结果更权威。
- 订单服务对订单生命周期更权威。
- 库存服务对库存预占和扣减更权威。
- 财务系统对结算账务更权威。

权威系统不是固定一个，而是按数据类型划分。

## 处理流程

流程：

- 生成差异单。
- 分类和定级。
- 判断是否延迟窗口内。
- 自动补偿低风险差异。
- 高风险差异人工审核。
- 修复后复核。
- 记录审计和根因。

对账系统要支持重复运行。

## 在 eMall 项目中怎么讲？

如果支付渠道成功但订单待支付，且已超过正常回调延迟窗口，系统生成高优先级差异单。

补偿任务确认支付单后推进订单状态，并触发库存确认；如果订单无法履约，则进入退款流程。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
对账发现差异后，先分类和定级，再判断权威数据来源和是否处于正常延迟窗口。
低风险差异可以自动补偿，高风险资金和库存差异要人工审核或双人复核。

处理完成后要复核结果、记录审计和分析根因，避免同类差异反复出现。
```

## 回答评分点

高分答案应该覆盖：

- 差异要分类和定级。
- 要判断权威数据来源。
- 延迟窗口内不一定立即修复。
- 低风险自动补偿，高风险人工审核。
- 修复后要复核和追根因。
