# 192 同步调用和异步事件如何取舍？

[返回按分类学习面试题](../README.md)

## 题目

同步调用和异步事件如何取舍？

## 先给面试官的短答案

同步调用适合调用方必须立即知道结果的场景；异步事件适合解耦、削峰、通知多个订阅方和允许最终一致的场景。
核心交易决策通常需要同步确认，后续通知、积分、推荐、数据分析更适合异步。

取舍要看是否阻塞用户、是否必须强一致、失败后能否补偿。

## 同步调用适合什么？

适合：

- 需要立即给用户结果。
- 下游结果决定当前流程是否继续。
- 延迟可控。
- 调用关系简单。

例如下单时检查库存是否可预占，订单服务通常需要同步知道结果。

## 异步事件适合什么？

适合：

- 非核心后置动作。
- 多个系统订阅同一事实。
- 能接受最终一致。
- 需要削峰填谷。
- 失败可重试和补偿。

例如订单支付成功后通知积分、推荐、数据仓库和消息通知。

## 风险对比

同步风险：

- 调用链变长。
- 下游故障向上传播。
- 尾延迟放大。

异步风险：

- 状态延迟。
- 消息重复。
- 消息乱序。
- 排障链路更复杂。

两者都需要治理，不存在绝对更好。

## 在 eMall 项目中怎么讲？

下单主链路中，库存预占可以同步调用，因为结果决定订单是否创建成功。

订单创建后的站内信、优惠推荐刷新、数据分析入仓应通过事件异步处理，避免拖慢用户下单。

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
同步调用适合必须立即知道结果的决策点，异步事件适合后置动作、多订阅方、削峰和最终一致场景。
我会把核心链路的必要判断放在同步调用中，并设置超时、熔断和降级；把通知、积分、推荐和分析等非核心动作放到异步事件中。

异步不是不用管一致性，它仍需要幂等、重试、死信、顺序和对账。
```

## 回答评分点

高分答案应该覆盖：

- 同步适合立即决策。
- 异步适合解耦、削峰和最终一致。
- 同步会放大调用链故障。
- 异步需要治理重复、乱序和失败。
- 核心链路和非核心链路应区别处理。
