# 083 公平锁和非公平锁有什么区别？

[返回按分类学习面试题](../README.md)

## 题目

公平锁和非公平锁有什么区别？

## 先给面试官的短答案

公平锁尽量按线程等待顺序获取锁，减少饥饿；非公平锁允许新来的线程插队竞争锁，吞吐通常更高。
高并发服务默认更常用非公平锁，因为公平调度会增加上下文切换和调度成本。只有在强顺序或饥饿风险明显时才考虑公平锁。

## 公平锁是什么？

公平锁强调先来先服务。

如果多个线程等待同一把锁，等待时间更久的线程更优先获得锁。

在 `ReentrantLock` 中：

```java
Lock lock = new ReentrantLock(true);
```

`true` 表示公平锁。

## 非公平锁是什么？

非公平锁不严格保证等待顺序。

新来的线程如果碰巧遇到锁释放，可能直接抢到锁。

默认：

```java
Lock lock = new ReentrantLock();
```

就是非公平锁。

## 为什么非公平锁吞吐更高？

公平锁需要维护等待顺序，并唤醒队列中的下一个线程。

被唤醒线程真正运行还需要操作系统调度。

非公平锁允许当前正在运行的线程或新线程直接竞争锁，可能减少线程切换。

所以非公平锁吞吐通常更高，但尾部线程可能等待更久。

## 公平锁的代价

公平锁代价：

- 吞吐下降。
- 上下文切换增加。
- 平均延迟可能升高。
- 实现和调度成本更高。

公平不等于更快。

公平锁解决的是饥饿和顺序问题，不是性能问题。

## 非公平锁的风险

非公平锁风险：

- 个别线程可能长期抢不到锁。
- 尾延迟可能不稳定。
- 在强顺序场景下不符合业务预期。

但大多数业务临界区很短，非公平锁的吞吐收益更重要。

## 如何选择？

选择原则：

- 默认非公平锁。
- 有明确饥饿风险再考虑公平锁。
- 有严格业务顺序时不要只靠锁公平性，应设计队列或状态机。
- 高并发核心链路优先关注吞吐和锁粒度。

公平锁不是订单公平性的解决方案，业务公平通常要靠排队模型和规则设计。

## 在 eMall 项目中怎么讲？

秒杀场景中，是否先到先得不能靠 JVM 公平锁保证。

因为请求经过网关、网络、队列、多个实例和数据库，单 JVM 公平锁无法提供全局公平。

如果要实现公平排队，应该使用统一入口排队、令牌、消息队列或库存资格机制。

并发题不能只回答 API 用法。生产系统要同时考虑线程安全、资源隔离、超时、拒绝、幂等和分布式多实例。
单机锁只能保护当前 JVM，不能保护整个集群；线程池满也不是小问题，而是容量和可用性风险。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

## 深度增强：面试高分表达

我会把并发问题分成三层：JMM 和锁保证单机正确性，线程池和隔离保证资源不被拖垮，
幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
公平锁尽量按等待顺序获取锁，能减少饥饿；非公平锁允许插队竞争，通常吞吐更高。
公平锁的代价是更多调度和上下文切换，所以高并发服务默认更常用非公平锁。

如果业务要求全局先到先得，不能依赖单 JVM 公平锁，因为微服务是多实例部署，需要统一排队、
令牌或消息化机制来实现业务公平。
```

## 回答评分点

高分答案应该覆盖：

- 公平锁按等待顺序。
- 非公平锁允许插队。
- 非公平锁通常吞吐更高。
- 公平锁减少饥饿但成本更高。
- 业务公平不能靠单机锁。

## 深度完善：面向 L6 的回答框架

围绕「公平锁和非公平锁有什么区别？」，高分答案不能停在概念定义，而要把「线程安全、锁、CAS、线程池、隔离、超时和并发容量」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「并发和线程治理」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `order 创建、inventory 扣减、payment 回调、outbox relay 和异步补偿任务` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：线程池活跃数、队列长度、拒绝数、锁等待、超时率、重复请求数。
- 验证证据：并发单测、压测曲线、线程 dump、拒绝日志、幂等记录和容量评估。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
