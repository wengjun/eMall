# 204 如何设计内部运维接口？

[返回按分类学习面试题](../README.md)

## 题目

如何设计内部运维接口？

## 先给面试官的短答案

内部运维接口用于诊断、修复、开关控制、缓存刷新和任务重放，必须按高权限入口治理。
设计时要做到鉴权、授权、审计、限流、参数校验、幂等、审批和最小权限。

运维接口不能因为“内部使用”就绕过安全和变更流程。

## 常见运维能力

包括：

- 健康检查。
- 配置刷新。
- 缓存清理。
- 任务重试。
- 消息重放。
- 订单补偿。
- 灰度开关。
- 诊断信息查询。

这些接口通常影响生产状态，风险高于普通查询接口。

## 安全要求

要求包括：

- 只能内网访问。
- 强认证。
- 基于角色授权。
- 高危操作二次确认。
- 操作审计。
- 参数白名单。
- 频率限制。
- 敏感数据脱敏。

高危操作最好通过运维平台统一入口执行。

## 可靠性要求

还要考虑：

- 幂等。
- 可回滚。
- 可暂停。
- 可追踪进度。
- 失败可重试。
- 不能长时间占用业务线程池。

运维任务应使用独立执行器，避免影响在线流量。

## 在 eMall 项目中怎么讲？

订单补偿接口不能裸露为一个无鉴权 HTTP 地址。

它应只允许运维平台调用，要求审批、记录操作者、记录原因、限制订单范围，并在补偿过程中保证幂等和审计。

## 深度增强：可观测与配置治理图

![指标、日志、Trace 和告警平台](../assets/observability-platform.svg)

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
内部运维接口是高权限生产入口，必须做认证、授权、审计、限流、参数校验和最小权限。
高危操作要通过运维平台、审批和二次确认执行。

可靠性上要支持幂等、可回滚、进度追踪和失败重试，并使用独立线程池或任务系统，不能抢占在线业务资源。
```

## 回答评分点

高分答案应该覆盖：

- 运维接口风险很高。
- 必须认证、授权、审计和限流。
- 高危操作要审批和二次确认。
- 运维任务要幂等和可追踪。
- 不能影响在线业务线程池。
