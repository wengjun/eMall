# 247 QPS、TPS、并发数、响应时间之间是什么关系？

[返回按分类学习面试题](../README.md)

## 题目

QPS、TPS、并发数、响应时间之间是什么关系？

## 先给面试官的短答案

它们可以用 Little's Law 粗略理解：并发数约等于吞吐量乘以响应时间。
如果系统 QPS 是 `1000`，平均响应时间是 `200ms`，平均在途请求约为 `1000 * 0.2 = 200`。

响应时间升高时，即使 QPS 不变，并发占用也会升高，线程和连接更容易被打满。

## 概念区别

概念：

- QPS：每秒请求数。
- TPS：每秒事务数。
- 并发数：同时在处理的请求数。
- 响应时间：一次请求从发出到返回的耗时。

QPS 和 TPS 有时接近，但一个交易事务可能包含多个请求。

## 基本关系

公式：

```text
并发数 = 吞吐量 * 响应时间
```

响应时间单位要换成秒。

这个关系适用于稳定系统的粗略估算，不代表所有瞬时峰值。

## 为什么重要？

如果下游变慢：

- 响应时间升高。
- 在途请求增加。
- 线程池被占满。
- 连接池被占满。
- 排队增加。
- 延迟进一步升高。

这就是慢服务拖垮调用方的原因。

## 在 eMall 项目中怎么讲？

下单服务目标 `50000 QPS`，如果平均响应时间 `100ms`，在途请求约 `5000`。

如果库存服务变慢到 `1s`，在途请求会变成约 `50000`，线程池和连接池很容易耗尽。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
QPS/TPS 表示吞吐，并发数表示同时处理的请求数量，响应时间表示单次耗时。
在稳定状态下可以用 Little's Law 估算：并发数约等于吞吐量乘以响应时间。

所以高并发系统不仅要提升 QPS，还要控制响应时间，尤其是 P95 和 P99，否则慢请求会占满线程、连接和队列。
```

## 回答评分点

高分答案应该覆盖：

- QPS、TPS、并发和响应时间定义。
- Little's Law 基本关系。
- 响应时间升高会放大并发占用。
- 平均值不够，还要看 P95/P99。
- TPS 和 QPS 不一定相同。
