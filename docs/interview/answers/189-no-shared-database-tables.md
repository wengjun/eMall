# 189 为什么服务之间不应该共享数据库表？

[返回按分类学习面试题](../README.md)

## 题目

为什么服务之间不应该共享数据库表？

## 先给面试官的短答案

服务之间共享数据库表会破坏服务边界。多个服务直接读写同一张表，会导致业务规则分散、表结构难演进、故障责任不清、
事务边界混乱，最终形成共享数据库的分布式单体。

微服务可以共享数据语义，但不应该共享底层表的写权限。

## 共享表的问题

问题包括：

- 一个服务改表影响多个服务。
- 多个服务绕过彼此业务规则。
- 数据状态可能被非法修改。
- 难以做权限和审计。
- 无法独立发布和回滚。
- 数据库成为所有服务的耦合中心。
- 扩容和迁移困难。

表结构一旦成为公共接口，演进成本会非常高。

## 正确方式

更合理的方式：

- 数据拥有者提供 API。
- 数据拥有者发布领域事件。
- 消费方维护本地只读投影。
- 分析需求进入数仓。
- 跨服务查询通过聚合服务或 CQRS 读模型。

写入必须回到数据拥有者。

## 是否完全不能读？

生产系统也有例外，比如：

- 数仓离线同步。
- 只读副本给报表使用。
- 临时迁移阶段的双读。

但这些应受治理，不应成为业务服务之间的常规调用方式。

## 在 eMall 项目中怎么讲？

支付服务不能直接更新订单表为已支付。它应发布支付成功事件，订单服务消费事件后按订单状态机更新订单。

这样订单状态规则仍由订单服务掌控。

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
服务之间不应该共享数据库表，因为表共享会让服务通过数据库耦合，绕过 API 和业务规则。
结果是表结构无法独立演进、状态流转不可控、发布回滚困难，系统变成共享数据库的分布式单体。

正确方式是数据拥有者提供 API 和领域事件，其他服务通过只读投影、聚合查询或数仓使用数据。写入必须回到拥有者。
```

## 回答评分点

高分答案应该覆盖：

- 共享表破坏服务边界。
- 业务规则会被绕过。
- 表结构会变成隐式公共契约。
- 正确方式是 API、事件和只读投影。
- 写权限必须属于数据拥有者。
