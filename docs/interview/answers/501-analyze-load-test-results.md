# 501 压测结果如何分析？

[返回按分类学习面试题](../README.md)

## 题目

压测结果如何分析？

## 先给面试官的短答案

压测结果要同时看吞吐、延迟、错误率、资源水位、依赖指标和业务正确性。不能只看平均响应时间，要重点看 P95、P99、
错误码分布、瓶颈资源、扩容曲线和是否出现订单、库存、支付等业务异常。

## 关键指标

吞吐指标包括 QPS、TPS、并发数和成功请求数。延迟指标包括 P50、P95、P99 和最大值。
稳定性指标包括错误率、超时率、限流率、重试次数和熔断次数。

资源指标包括 CPU、内存、GC、线程池、连接池、磁盘 IO、网络、数据库 CPU、慢 SQL、Redis 命中率和 Kafka lag。

业务指标包括下单成功率、支付成功率、库存扣减成功率、重复订单、库存负数和消息积压。

## 分析方法

先看系统是否达到目标容量，再看延迟是否满足 SLO。然后观察错误开始出现的拐点，找到吞吐增加但成功数不再增长的位置。

如果增加并发只增加延迟不增加吞吐，说明系统已经达到瓶颈。此时要用指标定位瓶颈是在应用、数据库、缓存、消息、
网络还是第三方依赖。

## 结果输出

压测报告应包括目标、环境、数据规模、场景比例、压测曲线、瓶颈点、容量结论、优化建议和风险。
只给一个“能扛多少 QPS”的数字是不够的。

## 在 eMall 项目中怎么讲？

eMall 压测要重点分析网关、订单、库存、支付、Redis、MySQL、Kafka 和 Outbox。比如 10 万下单 QPS 的目标，
不仅要看接口成功率，还要看库存是否准确、支付单是否唯一、消息是否堆积。

## 深度增强：现场编码工程化图

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
压测分析要从“能不能跑”提升到“容量边界在哪里、瓶颈是什么、业务是否正确”。

我会先看吞吐、P99、错误率和成功率，再看资源水位和下游依赖指标。
当并发继续增加但成功吞吐不再增长，或者 P99 和错误率明显上升，就说明达到瓶颈区。

对电商系统，还必须分析业务正确性。压测期间不能出现库存负数、重复订单、支付单重复或消息丢失。
否则即使 QPS 达标，也不能认为系统通过压测。
```

## 回答评分点

高分答案应该覆盖：

- 覆盖吞吐、延迟、错误率、资源、依赖和业务指标。
- 强调 P95、P99 比平均值更重要。
- 知道观察吞吐拐点和错误拐点。
- 能输出容量结论、瓶颈和优化建议。
- 能结合订单、库存、支付业务正确性。
