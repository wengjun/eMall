# 263 为什么重试可能放大故障？

[返回按分类学习面试题](../README.md)

## 题目

为什么重试可能放大故障？

## 先给面试官的短答案

重试会把一次请求变成多次请求。当下游已经变慢或不可用时，大量上游同时重试会增加下游负载，
让原本短暂故障变成雪崩，这就是重试风暴。

重试必须受超时、次数、退避、抖动、熔断和限流约束。

## 放大机制

例如一次请求最多重试 3 次。

如果入口有 `10000 QPS`，最坏情况下下游可能收到接近 `40000 QPS`。

如果多个服务层层重试，放大倍数会继续叠加。

## 典型风险

风险包括：

- 下游负载更高。
- 线程和连接占用更久。
- 请求排队变长。
- 超时进一步增加。
- 熔断前系统已经过载。
- 写操作可能重复产生副作用。

重试如果没有幂等，还会破坏业务正确性。

## 控制方式

控制方式：

- 限制最大次数。
- 使用指数退避。
- 添加 jitter。
- 设置总超时预算。
- 对下游熔断。
- 只重试可重试错误。
- 客户端和服务端不要层层重复重试。

重试策略要全链路统一治理。

## 在 eMall 项目中怎么讲？

大促时库存服务变慢，如果订单服务、网关和客户端都各自重试三次，库存服务会收到成倍流量。

正确做法是限制重试层级，并在库存服务过载时快速熔断或限流。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
重试会放大请求量。下游故障时，上游大量重试会增加下游负载，导致更慢、更多超时和更多重试，
最终形成重试风暴。多层调用各自重试时，放大倍数会叠加。

所以重试要限制次数、使用退避和 jitter、遵守总超时预算，并和限流、熔断配合。
```

## 回答评分点

高分答案应该覆盖：

- 重试会增加请求量。
- 下游故障时重试会形成正反馈。
- 多层重试会倍数放大。
- 写操作需要幂等。
- 退避、抖动、熔断和限流是控制手段。
