# 511 如何处理依赖漏洞？

[返回按分类学习面试题](../README.md)

## 题目

如何处理依赖漏洞？

## 先给面试官的短答案

处理依赖漏洞要先识别影响范围和可利用性，再选择升级、替换、排除传递依赖、配置缓解或临时隔离。修复后要跑回归测试，
生成 SBOM 和审计记录。高危漏洞要有 SLA 和应急发布流程。

## 处理流程

第一步是发现。通过 Dependabot、OWASP Dependency-Check、Snyk、Trivy 或云安全扫描识别 CVE。

第二步是评估。看漏洞组件是否真的被使用，是否暴露在外部入口，是否有远程代码执行、认证绕过或数据泄露风险。

第三步是修复。优先升级到安全版本；如果无法升级，可以排除传递依赖、替换库、关闭危险功能或加网络隔离。

第四步是验证。跑单元测试、集成测试、Smoke 和关键业务回归，确认升级没有破坏行为。

第五步是记录。记录漏洞编号、影响模块、修复版本、验证结果和发布时间。

## 风险控制

不是所有漏洞都要同等紧急，但高危远程可利用漏洞必须快速处理。对于无法立即修复的漏洞，要给出临时缓解措施和到期时间。

依赖漏洞管理不能只在上线前做一次扫描，应进入 CI 和定期治理。

## 在 eMall 项目中怎么讲？

eMall 的网关、开放平台、身份和支付模块暴露面更大，应优先处理高危依赖漏洞。公共父 POM 集中管理版本后，
修复一个漏洞可以统一覆盖所有模块。

## 深度增强：现场编码工程化图

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
依赖漏洞处理不能只看扫描结果，要结合可利用性和业务暴露面判断优先级。

我的流程是发现、评估、修复、验证和记录。优先升级到安全版本；无法升级时，采用排除传递依赖、
关闭功能、网络隔离或临时 WAF 规则等缓解措施。

对高危漏洞要有 SLA 和应急发布流程。修复后还要跑回归测试，避免安全修复引入业务故障。
```

## 回答评分点

高分答案应该覆盖：

- 覆盖发现、评估、修复、验证和记录。
- 知道要判断可利用性和暴露面。
- 能列出升级、排除、替换、缓解和隔离手段。
- 知道高危漏洞需要 SLA。
- 能结合集中版本管理和 CI 安全扫描。
