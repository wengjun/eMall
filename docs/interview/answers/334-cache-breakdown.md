# 334 缓存击穿是什么？

[返回按分类学习面试题](../README.md)

## 题目

缓存击穿是什么？

## 先给面试官的短答案

缓存击穿是某个热点 key 在过期瞬间，大量并发请求同时未命中缓存，然后一起访问数据库。
它通常发生在单个热点商品、秒杀活动页、热门价格或库存展示数据上。

治理重点是热点 key 不同时失效，并且只允许少量请求回源重建缓存。

## 和穿透的区别

区别：

- 穿透查询的是不存在的数据。
- 击穿查询的是存在的热点数据。
- 穿透会持续绕过缓存。
- 击穿通常发生在热点 key 过期瞬间。

击穿的本质是热点 key 回源并发过高。

## 治理方式

方式：

- 热点 key 逻辑过期。
- 互斥锁重建缓存。
- 单飞请求合并。
- 后台异步刷新。
- 热点 key 永不过期加版本控制。
- 本地缓存兜底。

核心是控制回源并发。

## 逻辑过期

逻辑过期是 value 中保存过期时间，Redis key 本身不过期或长 TTL。

请求发现逻辑过期后，先返回旧值，再由一个后台线程或获得锁的请求异步刷新缓存。这样用户不会
同时打到数据库。

## 在 eMall 项目中怎么讲？

热门商品详情、秒杀活动配置和首页推荐坑位容易击穿。

eMall 可以对热点商品使用逻辑过期和异步刷新。即使缓存过期，也先返回短暂旧数据，同时只让一个
线程重建缓存，避免数据库被热点流量打穿。

## 深度增强：缓存和消息治理图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

缓存和消息题要关注一致性、削峰、延迟、积压和恢复。
Redis 很快，但会遇到穿透、击穿、雪崩、热点 key 和内存淘汰；
MQ 能解耦和削峰，但会带来重复消费、乱序、积压和死信处理。

## 深度增强：Java 17 幂等消费示例

```java
import java.util.Set;
import java.util.concurrent.ConcurrentHashMap;

final class LocalIdempotentConsumer {
    private final Set<String> processedKeys = ConcurrentHashMap.newKeySet();

    boolean tryHandle(String messageKey, Runnable handler) {
        if (!processedKeys.add(messageKey)) {
            return false;
        }
        handler.run();
        return true;
    }
}
```

这个示例只适合解释幂等思想。生产环境不能用本地内存做全局幂等，要使用数据库唯一键、Redis 原子操作或业务状态机。

## 深度增强：生产边界

缓存要有 TTL、容量、降级和回源保护；消息要有重试、死信、延迟队列、消费幂等和积压告警。
缓存不一致要能修复，消息失败要能回放，不能只依赖人工查日志。

## 深度增强：面试高分表达

我会把缓存和消息都看成性能与稳定性工具，而不是正确性事实来源。
正确性由数据库事实、状态机、幂等和对账保证；缓存和 MQ 负责降低延迟、削峰填谷和解耦系统。

## 专家级完整回答

```text
缓存击穿是热点 key 过期后，大量并发请求同时未命中缓存并访问数据库。它和穿透不同，击穿的数据
在数据库中存在，只是热点 key 在某个时间点失效。

治理核心是限制回源并发，可以用互斥锁、单飞、逻辑过期、异步刷新和本地缓存。对电商热点商品，
宁可短时间返回旧值，也不能让数据库被瞬时流量打爆。
```

## 回答评分点

高分答案应该覆盖：

- 击穿发生在热点 key 过期。
- 数据本身存在。
- 大量请求同时回源。
- 互斥锁和逻辑过期。
- 热点商品场景。
