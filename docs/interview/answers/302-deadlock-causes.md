## 题目

死锁如何产生？

## 先给面试官的短答案

死锁是多个事务互相持有对方需要的锁，并且都等待对方释放，形成循环等待。
常见原因是不同事务加锁顺序不一致、范围锁过大、索引不合理、事务太长和并发更新热点数据。

数据库通常会检测死锁并回滚其中一个事务，但业务必须能重试。

## 典型例子

事务 A：

```text
锁订单 1
等待锁订单 2
```

事务 B：

```text
锁订单 2
等待锁订单 1
```

两个事务互相等待，形成死锁。

## 常见原因

原因包括：

- 加锁顺序不一致。
- 范围查询加锁。
- 缺少索引导致锁范围扩大。
- 长事务持锁时间长。
- 热点行并发更新。
- 混合批量更新和单条更新。

死锁不一定代表数据库坏了，通常是并发访问模式有问题。

## 数据库如何处理？

InnoDB 会检测死锁。

检测到后：

- 选择一个事务回滚。
- 返回死锁错误。
- 另一个事务继续执行。

应用层应识别死锁错误并在幂等前提下重试。

## 在 eMall 项目中怎么讲？

两个补偿任务同时处理订单和库存，如果一个先锁订单再锁库存，另一个先锁库存再锁订单，就可能死锁。

应统一加锁顺序，并缩短事务时间。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
死锁来自循环等待。多个事务分别持有锁，并等待对方持有的锁，导致谁也无法继续。
常见诱因是加锁顺序不一致、范围锁、缺少索引、长事务和热点更新。

InnoDB 会检测死锁并回滚一个事务，应用层要能识别并在幂等条件下重试，同时通过统一加锁顺序和缩短事务降低死锁概率。
```

## 回答评分点

高分答案应该覆盖：

- 死锁是循环等待。
- 加锁顺序不一致是常见原因。
- 缺少索引会扩大锁范围。
- 数据库会回滚一个事务。
- 应用层要支持安全重试。
