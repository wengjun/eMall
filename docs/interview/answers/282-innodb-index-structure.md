# 282 MySQL InnoDB 的索引结构是什么？

[返回按分类学习面试题](../README.md)

## 题目

MySQL InnoDB 的索引结构是什么？

## 先给面试官的短答案

InnoDB 主要使用 B+Tree 作为索引结构。主键索引是聚簇索引，叶子节点存放整行数据；
二级索引的叶子节点存放索引列和主键值，需要时再通过主键回表查询整行。

理解聚簇索引和二级索引，是理解 MySQL 查询性能的基础。

## 主键索引

主键索引特点：

- B+Tree 结构。
- 叶子节点存整行数据。
- 数据按主键逻辑有序组织。
- 一个表只有一个聚簇索引。

所以主键选择会影响数据组织和二级索引大小。

## 二级索引

二级索引特点：

- 也是 B+Tree。
- 叶子节点存二级索引列。
- 叶子节点还存主键值。
- 查询非索引列时需要回表。

二级索引不是直接存整行数据。

## 为什么主键重要？

如果主键很长：

- 所有二级索引都会保存较长主键。
- 索引体积增大。
- 缓存命中率下降。
- IO 成本上升。

因此很多业务表会使用稳定、短小、递增趋势较好的主键。

## 在 eMall 项目中怎么讲？

订单表主键可以使用雪花 ID 或趋势递增 ID。

按 `user_id` 查询订单时使用二级索引，二级索引叶子节点拿到订单主键后，再回表读取订单详情。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
InnoDB 索引主要是 B+Tree。主键索引是聚簇索引，叶子节点存整行数据；
二级索引叶子节点存索引列和主键值，查询其他列时需要通过主键回表。

因为二级索引会携带主键，所以主键长度、稳定性和插入顺序会影响索引体积和写入性能。
```

## 回答评分点

高分答案应该覆盖：

- InnoDB 使用 B+Tree 索引。
- 主键索引是聚簇索引。
- 二级索引叶子节点存主键值。
- 回表通过主键完成。
- 主键设计会影响二级索引。
