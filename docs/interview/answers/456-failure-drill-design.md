# 456 如何设计一次故障演练？

[返回按分类学习面试题](../README.md)

## 题目

如何设计一次故障演练？

## 先给面试官的短答案

故障演练要先定义目标和边界，再设计故障场景、影响范围、回滚方案、观察指标、参与人员和成功标准。
演练不是制造混乱，而是验证系统和团队是否能发现、隔离、降级和恢复。

生产演练必须有止损机制。

## 演练准备

准备：

- 明确演练目标。
- 明确影响范围。
- 选择低峰时段或灰度范围。
- 准备回滚方案。
- 通知相关人员。
- 确认监控和告警。
- 准备 Runbook。

没有止损方案不能演练。

## 场景设计

场景：

- Redis 不可用。
- Kafka 延迟。
- 数据库主从延迟。
- 下游支付超时。
- 单实例宕机。
- 网络延迟。
- 线程池耗尽。

场景要贴近真实风险。

## 成功标准

标准：

- 告警是否及时。
- 值班是否按 Runbook 处理。
- 降级是否生效。
- 核心链路是否保护住。
- 恢复是否符合 RTO。
- 数据是否一致。

演练后要复盘。

## 在 eMall 项目中怎么讲？

eMall 可以演练 Redis 商品缓存不可用。目标是验证商品详情是否启用本地旧值、回源是否限流、数据库
是否被保护、Redis 恢复后是否平滑预热。

演练范围可以先限制到灰度环境或小流量租户。

## 深度增强：Kubernetes 运维治理图

![Kubernetes 生产运行和故障治理](../assets/kubernetes-operations.svg)

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
故障演练要定义目标、范围、场景、指标、回滚和成功标准。它验证的不只是系统功能，还包括告警、
Runbook、值班响应、降级和恢复能力。

生产演练要小范围、可观测、可回滚、可止损，并在演练后复盘改进。没有回滚和止损机制的演练是
不负责任的。
```

## 回答评分点

高分答案应该覆盖：

- 明确目标和范围。
- 准备回滚和止损。
- 选择真实故障场景。
- 定义成功标准。
- 演练后复盘改进。
