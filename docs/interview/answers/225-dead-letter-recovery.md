# 225 死信消息如何恢复？

[返回按分类学习面试题](../README.md)

## 题目

死信消息如何恢复？

## 先给面试官的短答案

死信恢复要先分析失败原因，修复数据、配置或代码，再通过受控重放重新处理消息。
重放必须支持幂等、限速、筛选、审批、审计和结果追踪，不能把死信直接一键全量打回主队列。

恢复的目标是修复业务状态，而不是简单清空死信。

## 恢复步骤

步骤：

- 查看死信内容和失败原因。
- 判断是否仍需处理。
- 修复根因。
- 按业务键筛选消息。
- 小批量重放。
- 观察处理结果。
- 扩大重放范围。
- 标记恢复完成。

核心死信需要记录操作人和恢复依据。

## 重放风险

风险包括：

- 重复扣减。
- 重复退款。
- 重复发券。
- 大量重放压垮下游。
- 旧消息和新业务规则不兼容。
- 顺序被打乱。

因此重放前要确认消费者幂等和状态机规则。

## 工具能力

运维平台应支持：

- 按 topic 查询。
- 按业务 ID 查询。
- 按错误类型筛选。
- 单条重放。
- 批量限速重放。
- 跳过或归档。
- 操作审计。

人工直接改 MQ 风险很高。

## 在 eMall 项目中怎么讲？

如果支付成功事件进入死信，不能随便全量重放。

应先确认订单状态和支付状态，再按订单号重放，确保订单只从待支付推进到已支付一次。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
死信恢复要先修复根因，再受控重放。重放要支持筛选、限速、审批、审计和结果追踪，
并依赖消费者幂等和状态机防止重复副作用。

对核心交易消息，恢复目标是让订单、库存、支付等业务状态达到正确状态，而不是简单把死信队列清空。
```

## 回答评分点

高分答案应该覆盖：

- 死信恢复前先定位和修复根因。
- 重放要受控和限速。
- 幂等和状态机是安全前提。
- 核心消息要审批和审计。
- 目标是修复业务状态。
