# 311 分库分表后如何按订单号查订单？

[返回按分类学习面试题](../README.md)

## 题目

分库分表后如何按订单号查订单？

## 先给面试官的短答案

分库分表后按订单号查订单，最好让订单号或订单 ID 中携带分片路由信息，直接定位到目标分片。
如果订单号没有路由信息，就需要订单路由表、全局索引表或搜索读模型，避免广播查询所有分片。

核心目标是把订单号查询变成单分片查询。

## 订单号携带路由

常见做法：

- 订单 ID 中包含时间和分片位。
- 订单号生成时写入分片号。
- 路由规则能从订单号反推出库表。

这样查询时不需要额外查路由表。

## 路由表方案

如果订单号本身不能计算分片，可以维护：

```text
order_route(order_no, shard_id, table_id, user_id, created_at)
```

先查路由表，再查目标分片。

路由表要高可用，并且和订单创建保持一致。

## 全局索引或读模型

也可以维护全局索引：

- 订单号到订单 ID。
- 订单号到用户 ID。
- 订单号到分片位置。

它可以通过本地事务或事件更新，但要处理一致性和补偿。

## 在 eMall 项目中怎么讲？

eMall 订单号可以用趋势递增 ID，并在 ID 中保留分片位。

客服按订单号查询时，系统从订单号解析分片，直接路由到对应订单表，避免扫所有分片。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
分库分表后按订单号查询，最佳方式是订单号或订单 ID 自带分片路由信息，查询时直接计算库表。
如果没有路由信息，就需要订单路由表或全局索引表，先定位分片再查询订单。

不能每次按订单号查询都广播所有分片。路由信息要在订单创建时可靠写入，并有补偿和校验。
```

## 回答评分点

高分答案应该覆盖：

- 订单号查询应尽量单分片定位。
- 订单号可携带分片信息。
- 路由表和全局索引是替代方案。
- 避免广播所有分片。
- 路由数据要保证一致性。
