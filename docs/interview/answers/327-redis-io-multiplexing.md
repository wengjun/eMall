# 327 Redis IO 多路复用是什么？

[返回按分类学习面试题](../README.md)

## 题目

Redis IO 多路复用是什么？

## 先给面试官的短答案

IO 多路复用是指一个线程可以同时监听多个网络连接的读写事件。Redis 不需要为每个连接创建
一个线程，而是通过 epoll、kqueue 或 select 这类机制，在连接可读或可写时再处理。

它解决的是大量连接下的网络事件管理问题。

## 工作过程

过程可以简化为：

- 客户端连接 Redis。
- Redis 将连接注册到事件循环。
- 操作系统通知哪些连接可读或可写。
- Redis 读取请求并解析命令。
- Redis 执行命令。
- Redis 写回响应。

线程不需要阻塞等待单个连接。

## 为什么重要

价值包括：

- 支持大量连接。
- 减少线程数量。
- 降低上下文切换。
- 避免每连接一线程模型的资源浪费。
- 配合内存操作实现低延迟。

这也是 Redis 单线程模型能工作的基础之一。

## 容易误解的点

需要区分：

- IO 多路复用负责连接事件。
- Redis 命令执行仍要占用执行线程。
- 慢命令仍会阻塞后续命令。
- 多路复用不能让一个慢命令变快。

它解决的是网络并发，不解决命令计算复杂度。

## 在 eMall 项目中怎么讲？

eMall 大量商品详情、库存展示和会话校验会访问 Redis。IO 多路复用让 Redis 能用少量线程服务
大量短连接请求。

但如果某个商品热 key 被大量访问，瓶颈可能从连接管理变成单 key 单分片 CPU，因此还要做热点治理。

缓存和消息题要关注一致性、削峰、延迟、积压和恢复。
Redis 很快，但会遇到穿透、击穿、雪崩、热点 key 和内存淘汰；
MQ 能解耦和削峰，但会带来重复消费、乱序、积压和死信处理。

## 深度增强：Java 17 幂等消费示例

```java
import java.util.Set;
import java.util.concurrent.ConcurrentHashMap;

final class LocalIdempotentConsumer {
    private final Set<String> processedKeys = ConcurrentHashMap.newKeySet();

    boolean tryHandle(String messageKey, Runnable handler) {
        if (!processedKeys.add(messageKey)) {
            return false;
        }
        handler.run();
        return true;
    }
}
```

这个示例只适合解释幂等思想。生产环境不能用本地内存做全局幂等，要使用数据库唯一键、Redis 原子操作或业务状态机。

## 深度增强：生产边界

缓存要有 TTL、容量、降级和回源保护；消息要有重试、死信、延迟队列、消费幂等和积压告警。
缓存不一致要能修复，消息失败要能回放，不能只依赖人工查日志。

## 深度增强：面试高分表达

我会把缓存和消息都看成性能与稳定性工具，而不是正确性事实来源。
正确性由数据库事实、状态机、幂等和对账保证；缓存和 MQ 负责降低延迟、削峰填谷和解耦系统。

## 专家级完整回答

```text
Redis 的 IO 多路复用是一个线程监听多个连接的网络事件，只有连接可读可写时才处理它。这样
Redis 不需要为每个客户端连接分配线程，减少了线程切换和资源消耗。

但它只解决网络事件管理问题。真正的命令执行仍然需要占用 Redis 执行线程，所以慢命令和大 key
仍然会造成阻塞。
```

## 回答评分点

高分答案应该覆盖：

- 一个线程监听多个连接。
- 操作系统事件通知机制。
- 降低线程和上下文切换成本。
- 区分网络 IO 和命令执行。
- 能说明慢命令仍然有风险。
