# 409 最小权限原则如何落地？

[返回按分类学习面试题](../README.md)

## 题目

最小权限原则如何落地？

## 先给面试官的短答案

最小权限原则是让人、服务和系统只拥有完成任务所需的最小权限，并且权限可审计、可过期、可回收。
落地要从身份、角色、资源、接口、数据、运维和临时授权几层做。

它不是一句安全口号，而是权限生命周期管理。

## 人员权限

做法：

- 按岗位授权。
- 默认无权限。
- 高危权限单独申请。
- 临时权限自动过期。
- 离职和转岗自动回收。
- 定期权限复核。

权限要和组织流程联动。

## 服务权限

做法：

- 服务账号独立。
- 每个服务只访问需要的数据库和 Topic。
- 内部接口也鉴权。
- Secret 不共享。
- 生产和测试权限隔离。

微服务之间也不能默认互信。

## 数据权限

做法：

- 用户只能看自己的订单。
- 商家只能看自己店铺数据。
- 客服按工单授权。
- 敏感字段脱敏。
- 导出权限单独审批。

数据权限要到资源级。

## 在 eMall 项目中怎么讲？

eMall 商家运营账号可以编辑本店商品，但不能查看其他商家的订单。客服只能在工单范围内查看用户
订单摘要，不能导出全量用户数据。

支付退款、批量改价和高危配置变更需要临时授权或审批，并记录审计日志。

加密保护机密性，签名保护完整性；审计负责事后追踪，风控负责发现异常行为。

## 深度增强：Java 17 安全策略示例

```java
import java.util.Set;

record SecurityDecision(boolean allowed, String reason) {
}

final class ScopePolicy {

    SecurityDecision check(Set<String> scopes, String requiredScope) {
        if (scopes.contains(requiredScope)) {
            return new SecurityDecision(true, "allowed");
        }
        return new SecurityDecision(false, "missing scope: " + requiredScope);
    }
}
```

这段代码体现最小权限原则。生产系统还要结合租户、资源归属、IP、设备、风险等级和操作审计。

## 深度增强：生产边界

安全不能只靠前端隐藏按钮，也不能只在网关做一次判断。核心资源要在服务层做资源级授权，
敏感数据要脱敏，密钥要支持轮换，失败日志不能泄露 token、secret、身份证号和银行卡号。

## 深度增强：面试高分表达

我会把安全问题拆成认证、授权、防篡改、防重放、数据保护、审计和风控。
每一层解决的问题不同，不能互相替代。这能体现我理解开放平台和电商系统的真实攻击面。

## 专家级完整回答

```text
最小权限要覆盖人员、服务和数据。人员按岗位授权，高危权限临时申请并自动过期；服务使用独立身份，
只访问必要资源；数据按租户、商家、用户和工单做资源级隔离。

落地还需要审批、审计、定期复核、离职回收和异常告警。权限必须有生命周期，而不是一次授予永久
有效。
```

## 回答评分点

高分答案应该覆盖：

- 默认最小权限。
- 人员、服务、数据都要控制。
- 临时权限和自动过期。
- 资源级数据权限。
- 审计和定期复核。
