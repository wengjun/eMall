## 题目

设计商品详情系统。

## 先给面试官的短答案

商品详情系统是典型读多写少、高并发、低延迟系统。核心设计是商品基础信息、价格、库存、促销、评价、推荐分层聚合，
通过 CDN、本地缓存、Redis、热点保护、异步刷新和降级保证高可用。实时性强的价格和库存要单独查询或短缓存。

## 数据组成

商品详情页通常包含 SPU、SKU、标题、图片、属性、规格、详情描述、商家、类目、价格、促销、库存、评价、推荐和广告。

这些数据来源不同、实时性不同，不能全部按一个存储模型处理。

## 架构设计

静态内容如图片和详情 HTML 可以放 CDN 和对象存储。商品基础信息可以缓存到 Redis 或本地缓存。

价格、促销和库存实时性更强，可以短 TTL 缓存或请求对应服务。评价、推荐和广告可以异步加载或失败降级。

BFF 或商品详情聚合服务负责组装页面所需数据，避免前端直接调用太多后端服务。

## 缓存和一致性

商品详情要防缓存穿透、击穿和雪崩。不存在商品要缓存空值，热点商品要预热和互斥刷新，TTL 要加随机抖动。

商品信息变更后，通过事件驱动缓存失效或刷新。价格和库存变化要控制缓存时间，避免展示严重旧值。

## 高可用

详情页可以分层降级。推荐、评价、广告失败不影响商品基础信息展示。价格服务失败时可以提示暂不可购买，
不能展示错误价格并允许下单。

## 在 eMall 项目中怎么讲？

eMall 的 `product`、`pricing`、`promotion`、`inventory`、`review`、`recommendation` 和 `advertising` 可以共同支撑详情页。
商品详情聚合层要做缓存、降级、trace 和热点保护。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

## 专家级完整回答

```text
商品详情系统的难点是高并发读和多来源数据聚合。

我会按实时性拆分数据：图片和详情走 CDN，基础商品信息走缓存，价格和库存短缓存或实时查询，
推荐、评价和广告异步加载并可降级。

缓存策略要处理穿透、击穿、雪崩和热点 SKU。商品变更通过事件刷新缓存，价格和库存则在下单前再次强校验。
这样既能保证详情页低延迟，也不会把展示数据当成交易事实。
```

## 回答评分点

高分答案应该覆盖：

- 说明商品详情是读多写少、高并发系统。
- 能按数据实时性拆分缓存策略。
- 覆盖 CDN、Redis、本地缓存、异步聚合和降级。
- 知道缓存穿透、击穿、雪崩和热点保护。
- 能强调下单前价格库存强校验。

## 深度完善：系统设计高分展开

围绕「设计商品详情系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。
