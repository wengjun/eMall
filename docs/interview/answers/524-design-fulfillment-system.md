## 题目

设计履约系统。

## 先给面试官的短答案

履约系统负责支付后的发货执行，包括拆单、仓库分配、库存确认、拣货、打包、出库、物流、签收和异常处理。
它通常通过订单已支付事件触发，采用状态机、幂等、补偿和与仓储物流系统的异步集成保证最终履约。

## 核心职责

履约系统不是订单系统的一部分，但它依赖订单事实。订单支付成功后，履约服务创建履约单，根据商品、仓库、地址、
库存和物流能力决定从哪里发货。

履约还要处理拆单、合单、预售、跨仓、缺货、拦截发货和物流异常。

## 数据模型

核心模型包括履约单、履约明细、仓库任务、物流单、出库单、签收记录和异常记录。

履约状态可以包括待分仓、待拣货、待出库、已发货、运输中、已签收、异常、已取消。

## 数据流

订单已支付事件到达后，履约服务幂等创建履约单。然后调用仓储系统锁定实物库存或生成出库任务。

仓储系统回传拣货、出库结果，物流系统回传运单和轨迹。履约服务更新状态，并通知订单服务展示物流进度。

## 一致性和异常

履约链路长，不能依赖同步事务。要通过事件、状态机、重试、补偿和人工处理保证最终一致。

如果支付成功但履约创建失败，要补偿创建。若仓库缺货，要触发调仓、拆单、退款或客服介入。

## 在 eMall 项目中怎么讲？

eMall 的 `fulfillment` 模块消费订单已支付事件，创建履约单并模拟仓储物流状态。它和 `order`、`inventory`、
`supply-chain`、`customer-service` 通过事件协作。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

```text
履约系统负责把订单承诺变成实际交付。

我会让履约由订单已支付事件触发，幂等创建履约单，再进行分仓、拣货、出库、物流和签收状态流转。
由于仓储和物流是外部异步系统，履约必须用状态机、重试、补偿和人工处理兜底。
```

## 回答评分点

高分答案应该覆盖：

- 覆盖支付后履约、分仓、拣货、出库、物流和签收。
- 能说明履约单、出库单、物流单等模型。
- 知道履约通过事件异步驱动。
- 覆盖幂等、状态机、补偿和人工异常处理。
- 能区分订单事实和履约执行。

## 深度完善：系统设计高分展开

围绕「设计履约系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。
