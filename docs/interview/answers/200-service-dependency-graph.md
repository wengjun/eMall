# 200 如何画服务依赖图？

[返回按分类学习面试题](../README.md)

## 题目

如何画服务依赖图？

## 先给面试官的短答案

服务依赖图要表达谁调用谁、调用类型、强弱依赖、同步异步、数据依赖、流量大小和故障影响。
它不只是架构图，而是容量评估、变更评审、故障演练和应急排障的工具。

好的依赖图应能回答：一个服务故障会影响哪些链路。

## 图上应该有什么？

应包含：

- 服务节点。
- 同步调用边。
- 异步事件边。
- 数据库和缓存依赖。
- 第三方依赖。
- 核心链路标记。
- 强弱依赖标记。
- QPS、延迟和错误率。
- 服务负责人。

不同信息可以分层展示，避免一张图过载。

## 数据来源

依赖图数据可以来自：

- 链路追踪。
- 服务注册中心。
- 网关访问日志。
- MQ 订阅关系。
- 配置仓库。
- CMDB 或服务目录。
- 代码扫描。

人工画图适合设计评审，生产治理应尽量自动生成。

## 如何使用？

使用场景：

- 发布前影响面分析。
- 故障时快速定位上下游。
- 容量规划。
- 识别长链路。
- 识别循环依赖。
- 评估服务下线风险。

依赖图要和监控指标结合才有价值。

## 在 eMall 项目中怎么讲？

下单链路依赖图应展示网关、用户、购物车、订单、库存、价格、优惠、支付和风控。

图上还要标明哪些是同步强依赖，哪些是异步事件，例如推荐和数据仓库只消费订单事件。

## 深度增强：可观测与配置治理图

![指标、日志、Trace 和告警平台](../assets/observability-platform.svg)

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
服务依赖图要画出服务节点、同步调用、异步事件、数据库缓存、第三方依赖、强弱依赖和核心链路。
它的价值是帮助做影响面分析、故障定位、容量规划和依赖治理。

生产环境不能只靠手工图，应从 tracing、注册中心、网关日志、MQ 订阅和配置仓库自动生成，并叠加 QPS、延迟和错误率。
```

## 回答评分点

高分答案应该覆盖：

- 依赖图不只是静态架构图。
- 要区分同步、异步、强依赖和弱依赖。
- 数据来源应自动化。
- 图要服务发布、排障和容量规划。
- 指标叠加后才适合生产治理。
