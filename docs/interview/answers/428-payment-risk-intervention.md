# 428 支付风控应在什么环节介入？

[返回按分类学习面试题](../README.md)

## 题目

支付风控应在什么环节介入？

## 先给面试官的短答案

支付风控应在下单前、支付前、支付中、支付回调后和退款售后环节分层介入。不同环节风险不同，
下单前偏资格和账号风险，支付前偏交易风险，回调后偏结果核验和异常处置。

风控不能只在支付成功后补救。

## 下单前

检查：

- 账号风险。
- 设备风险。
- 地址风险。
- 优惠券滥用。
- 购买频率。
- 黑名单。

这里可以减少无效订单。

## 支付前

检查：

- 金额异常。
- 收货信息异常。
- 支付方式风险。
- 订单状态。
- 价格是否被篡改。
- 用户是否被冻结。

支付前是关键拦截点。

## 支付后

检查：

- 支付回调签名。
- 支付金额是否一致。
- 支付单是否重复。
- 第三方流水是否可信。
- 异常订单是否冻结履约。

支付成功不代表风险结束。

## 在 eMall 项目中怎么讲？

eMall 在创建支付单前调用风控服务，检查账号、设备、金额、优惠和收货地址。

支付回调时还要校验第三方签名、金额和订单状态。如果风险较高，可以将订单置为待人工审核，暂停
履约而不是直接发货。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
支付风控要分层介入。下单前看账号、设备、优惠和频率；支付前看金额、支付方式、订单状态和用户
风险；支付回调后校验签名、金额、流水和重复通知；退款售后也要做风险识别。

高风险订单可以暂停履约或进入人工审核，而不是简单成功或失败。风控要和订单状态机协同。
```

## 回答评分点

高分答案应该覆盖：

- 风控不是只在一个点。
- 下单前、支付前、支付后都要看。
- 回调要校验签名和金额。
- 高风险订单可暂停履约。
- 风控要和状态机协同。
