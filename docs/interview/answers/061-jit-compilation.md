# 061 JIT 编译是什么？

[返回按分类学习面试题](../README.md)

## 题目

JIT 编译是什么？

## 先给面试官的短答案

JIT 是 Just-In-Time Compilation，即即时编译。Java 程序启动后，字节码先由解释器执行，
JVM 会监控哪些代码执行频繁，然后把热点代码编译成本地机器码，提升长期运行性能。

JIT 是 Java 能兼顾跨平台和高性能的重要机制。

## 为什么需要 JIT？

Java 源码先编译成字节码。字节码可以在不同平台的 JVM 上运行，所以 Java 有跨平台能力。

但 CPU 不能直接执行字节码，JVM 需要执行它。

执行方式有两种：

- 解释执行：一条条解释字节码。
- 编译执行：把热点字节码编译成机器码。

解释执行启动快，但长期性能不如机器码。
JIT 会把频繁执行的代码编译成本地机器码，让长期运行服务获得更好性能。

## 热点代码

JIT 不会把所有代码都立即编译。

JVM 会统计：

- 方法调用次数。
- 循环回边次数。
- 分支行为。
- 类型分布。
- 对象分配情况。

当某段代码足够热，就可能触发 JIT 编译。

这就是“热点代码”的含义。

## 分层编译

现代 HotSpot JVM 使用分层编译。

大致可以理解为：

- 解释器先快速启动。
- C1 编译器做轻量优化。
- C2 编译器对真正热点代码做深度优化。

这样既能控制启动时间，又能让长期运行性能更高。

Java 17 中服务端应用通常可以依赖默认分层编译策略。

## JIT 做哪些优化？

常见优化包括：

- 方法内联。
- 逃逸分析。
- 锁消除。
- 标量替换。
- 循环优化。
- 常量折叠。
- 分支预测优化。
- 去虚拟化。

这些优化会让 Java 代码在热点路径上接近本地语言性能。

## 方法内联为什么重要？

方法调用有成本，而且跨方法会限制优化范围。

JIT 可以把小方法直接展开到调用点。

示例：

```java
int totalPrice(OrderLine line) {
    return line.price() * line.quantity();
}
```

如果这个方法很热，JIT 可能把它内联到订单计算主流程中，减少调用开销，并继续做后续优化。

## JIT 和服务预热

Java 服务刚启动时，热点代码还没被充分编译。

所以第一次请求或刚扩容实例可能更慢。

生产中常见做法：

- 启动后做健康检查。
- 使用 readiness 控制接流量。
- 对核心接口做预热请求。
- 灰度引流，逐步放大流量。

这能避免冷实例直接承接高峰流量。

## JIT 的代价

JIT 不是免费的。

它会消耗：

- CPU。
- Code Cache。
- 编译线程资源。
- 启动预热时间。

如果服务生命周期很短，JIT 还没来得及发挥效果，进程就结束了，收益会下降。

所以短任务、函数计算和命令行程序更关注启动速度，长生命周期微服务更能受益于 JIT。

## 在 eMall 项目中怎么讲？

eMall 的订单计算、价格规则、优惠匹配、库存扣减和风控判断都是高频路径。

这些代码在服务运行一段时间后会成为热点，JIT 可能对它们做方法内联、逃逸分析和锁优化。

因此压测时不能只看冷启动后的前几秒，而要区分冷态性能和预热后的稳定态性能。

## 深度增强：JVM 生产运行图

![Java 17 容器内 JVM 内存结构](../assets/jvm-runtime-memory.svg)

JVM 题要从运行时资源解释到业务影响。堆、直接内存、元空间、线程栈和容器 memory limit 共同决定服务稳定性；
GC、CPU throttling、线程池队列和下游超时会一起影响 P99，而不是孤立存在。

## 深度增强：Java 17 诊断模型示例

```java
record RuntimeSignal(
        double heapUsage,
        double containerMemoryUsage,
        long gcPauseMillis,
        int threadCount,
        int queuedTasks) {

    boolean requiresTriage() {
        return heapUsage > 0.85
                || containerMemoryUsage > 0.90
                || gcPauseMillis > 500
                || threadCount > 800
                || queuedTasks > 1_000;
    }
}
```

这个模型强调线上诊断要看组合信号。只看 heap 不够，只看 GC 也不够；
要把 JVM、容器、线程池和业务延迟放到同一条时间线。

## 深度增强：生产边界

JVM 调优不能靠背参数。要先明确服务目标：低延迟、吞吐、容器资源、对象分配速率和 P99 SLO。
然后通过 GC 日志、JFR、指标和压测验证。错误地调大 `-Xmx` 可能挤压堆外内存，导致容器 OOMKilled。

## 深度增强：面试高分表达

我会用证据链回答 JVM 问题：先看业务影响，再看 JVM 指标、GC 日志、线程栈、heap dump、容器事件和最近变更。
结论要能解释现象，并能给出降级、扩容、参数调整或代码优化方案。

## 专家级完整回答

```text
JIT 是 JVM 在运行时把热点字节码编译成本地机器码的机制。Java 先用解释器保证快速启动，
再通过运行时 profiling 找到热点方法和热点循环，之后由 C1/C2 分层编译器做方法内联、
逃逸分析、锁消除和循环优化。

对微服务来说，JIT 解释了为什么服务需要预热，也解释了为什么压测要区分冷态和稳定态。
我会通过 readiness、预热请求和灰度引流，避免刚启动实例直接承接高峰流量。
```

## 回答评分点

高分答案应该覆盖：

- JIT 是运行时编译热点字节码。
- 解释执行启动快，编译执行长期性能好。
- 知道分层编译。
- 能说出方法内联、逃逸分析等优化。
- 能联系服务预热和压测稳定态。
