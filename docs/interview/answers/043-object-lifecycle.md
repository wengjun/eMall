# 043 对象从创建到回收大致经历什么过程？

[返回按分类学习面试题](../README.md)

## 题目

对象从创建到回收大致经历什么过程？

## 先给面试官的短答案

对象通常经历类加载、内存分配、初始化、使用、变为不可达、被 GC 标记、回收几个阶段。
大多数对象在堆上分配，短生命周期对象通常在年轻代被快速回收，长期存活对象会进入老年代。

生产中关注对象生命周期，是为了理解 GC 压力、内存泄漏、大对象分配和缓存增长问题。

## 从零基础理解

这行代码：

```java
Order order = new Order(...);
```

大致发生：

1. JVM 确认 `Order` 类已经加载。
2. 在堆上为对象分配内存。
3. 对字段设置默认值。
4. 执行构造函数。
5. 栈上的变量 `order` 保存对象引用。
6. 代码使用对象。
7. 当没有任何可达引用指向它时，对象可以被 GC 回收。

## 对象分配

大多数对象在堆上分配。JVM 会尽量让分配很快，比如使用线程本地分配缓冲 TLAB。

这就是为什么 Java 中创建短生命周期对象不一定很慢。

## 对象使用

对象被引用时就是可达的。

例如：

```java
List<Order> orders = new ArrayList<>();
orders.add(order);
```

只要 `orders` 还可达，并且里面还引用 order，order 就不能被回收。

## 变为不可达

方法结束后，局部变量消失；如果没有其他对象引用它，它就可能变为不可达。

```java
public void handle() {
    Order order = new Order(...);
}
```

方法返回后，如果 order 没被放进其他地方，就可以回收。

## 年轻代和老年代

很多 GC 会把堆分成年轻代和老年代。

年轻代：

- 存新创建对象。
- 大多数对象很快死亡。
- 回收频繁但通常较快。

老年代：

- 存活较久的对象。
- 回收成本更高。
- 老年代增长需要关注。

电商系统中，请求 DTO、临时计算对象通常短命；本地缓存、连接池、线程池、静态集合通常长命。

## 对象什么时候不能回收？

只要从 GC Roots 可达，就不能回收。

常见原因：

- 静态集合引用。
- 本地缓存未淘汰。
- ThreadLocal 未清理。
- 线程池任务队列堆积。
- 监听器未注销。
- ClassLoader 泄漏。

## 在 eMall 项目中怎么讲？

订单请求对象、价格计算中间对象通常很快回收。
但如果本地缓存无限保存商品详情、Outbox 事件、用户上下文，就会长期占用堆。

MQ 消费者如果把消息先放进无界内存队列，积压时也会导致对象无法回收。

## 深度增强：JVM 生产运行图

![Java 17 容器内 JVM 内存结构](../assets/jvm-runtime-memory.svg)

JVM 题要从运行时资源解释到业务影响。堆、直接内存、元空间、线程栈和容器 memory limit 共同决定服务稳定性；
GC、CPU throttling、线程池队列和下游超时会一起影响 P99，而不是孤立存在。

## 深度增强：Java 17 诊断模型示例

```java
record RuntimeSignal(
        double heapUsage,
        double containerMemoryUsage,
        long gcPauseMillis,
        int threadCount,
        int queuedTasks) {

    boolean requiresTriage() {
        return heapUsage > 0.85
                || containerMemoryUsage > 0.90
                || gcPauseMillis > 500
                || threadCount > 800
                || queuedTasks > 1_000;
    }
}
```

这个模型强调线上诊断要看组合信号。只看 heap 不够，只看 GC 也不够；
要把 JVM、容器、线程池和业务延迟放到同一条时间线。

## 深度增强：生产边界

JVM 调优不能靠背参数。要先明确服务目标：低延迟、吞吐、容器资源、对象分配速率和 P99 SLO。
然后通过 GC 日志、JFR、指标和压测验证。错误地调大 `-Xmx` 可能挤压堆外内存，导致容器 OOMKilled。

## 深度增强：面试高分表达

我会用证据链回答 JVM 问题：先看业务影响，再看 JVM 指标、GC 日志、线程栈、heap dump、容器事件和最近变更。
结论要能解释现象，并能给出降级、扩容、参数调整或代码优化方案。

## 专家级完整回答

```text
对象创建时，JVM 会先确保类已加载，然后在堆上分配内存、设置默认值、执行构造函数。
对象被业务代码使用时，只要从 GC Roots 可达，就不会被回收。
当对象不可达后，GC 会在合适时机标记并回收它。

大多数请求级对象生命周期很短，通常在年轻代回收；缓存、线程池、连接池、静态集合等对象生命周期长，
如果引用不释放就可能造成内存泄漏。生产排查时我会结合 heap dump 看哪些对象被谁引用。
```

## 回答评分点

高分答案应该覆盖：

- 创建、初始化、使用、不可达、GC 回收。
- 年轻代和老年代基本概念。
- 能说明可达性决定是否回收。
- 能联系缓存、ThreadLocal、队列积压。
- 能说明生产中看引用链。
