# 410 内部运维接口如何鉴权？

[返回按分类学习面试题](../README.md)

## 题目

内部运维接口如何鉴权？

## 先给面试官的短答案

内部运维接口必须强鉴权，不能因为在内网就默认安全。应结合 SSO、MFA、RBAC、ABAC、mTLS、IP
白名单、审批、审计和高危操作二次确认。

运维接口往往权限更大，保护等级应高于普通业务接口。

## 基础鉴权

基础能力：

- 统一 SSO 登录。
- 多因素认证。
- 角色权限控制。
- 资源级授权。
- mTLS 服务身份。
- IP 或网络段限制。

内网不是安全边界。

## 高危操作保护

高危操作包括：

- 批量改价。
- 批量退款。
- 修改库存。
- 下架商品。
- 修改风控规则。
- 数据导出。

这些操作要二次确认、审批和审计。

## 审计要求

审计字段：

- 操作人。
- 操作时间。
- 来源 IP。
- 操作对象。
- 请求参数摘要。
- 审批单号。
- 变更前后值。
- trace ID。

审计日志要防篡改。

## 在 eMall 项目中怎么讲？

eMall 平台运营接口不能只靠网关内网访问。批量改价接口需要 SSO 登录、运营角色、商家或类目范围
授权，并要求审批单号。

执行后写审计日志，记录变更前后价格，支持事后追溯和回滚。

安全题要从身份、权限、数据、审计和攻击面分层回答。认证只证明是谁，授权判断能做什么；
加密保护机密性，签名保护完整性；审计负责事后追踪，风控负责发现异常行为。

## 深度增强：Java 17 安全策略示例

```java
import java.util.Set;

record SecurityDecision(boolean allowed, String reason) {
}

final class ScopePolicy {

    SecurityDecision check(Set<String> scopes, String requiredScope) {
        if (scopes.contains(requiredScope)) {
            return new SecurityDecision(true, "allowed");
        }
        return new SecurityDecision(false, "missing scope: " + requiredScope);
    }
}
```

这段代码体现最小权限原则。生产系统还要结合租户、资源归属、IP、设备、风险等级和操作审计。

## 深度增强：生产边界

安全不能只靠前端隐藏按钮，也不能只在网关做一次判断。核心资源要在服务层做资源级授权，
敏感数据要脱敏，密钥要支持轮换，失败日志不能泄露 token、secret、身份证号和银行卡号。

## 深度增强：面试高分表达

我会把安全问题拆成认证、授权、防篡改、防重放、数据保护、审计和风控。
每一层解决的问题不同，不能互相替代。这能体现我理解开放平台和电商系统的真实攻击面。

## 专家级完整回答

```text
内部运维接口要按高危接口保护，不能信任内网。基础上要有 SSO、MFA、RBAC、资源级授权、mTLS、
IP 限制和审计。高危操作还要审批、二次确认和变更记录。

生产事故中很多风险来自内部误操作，所以运维接口的权限、审计和回滚能力必须强于普通业务接口。
```

## 回答评分点

高分答案应该覆盖：

- 内网不等于安全。
- SSO、MFA、RBAC 和 mTLS。
- 高危操作需要审批。
- 审计要记录操作上下文。
- 支持追溯和回滚。
