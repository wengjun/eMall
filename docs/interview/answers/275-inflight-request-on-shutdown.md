# 275 服务关闭时如何处理正在执行的请求？

[返回按分类学习面试题](../README.md)

## 题目

服务关闭时如何处理正在执行的请求？

## 先给面试官的短答案

服务关闭时应停止接收新请求，并给正在执行的请求一个有限时间完成。
如果请求能在关闭窗口内完成，就正常返回；如果超过窗口，要中断、返回可重试错误或让业务进入可恢复状态。

不能直接杀进程，否则容易产生请求丢失、事务中断和消息重复。

## HTTP 请求

处理方式：

- readiness 失败。
- 停止接收新连接。
- 等待在途请求完成。
- 超时后返回 503 或中断连接。
- 客户端按幂等策略重试。

写接口要有幂等键，防止客户端重试造成重复提交。

## 数据库事务

如果请求正在事务中：

- 正常完成则提交。
- 异常中断则回滚。
- 不应在关闭中开启长事务。
- 事务状态要可恢复。

长事务会拖慢发布和扩缩容。

## MQ 消费

消费者关闭时：

- 停止拉取新消息。
- 当前消息处理完成后 ack。
- 如果处理未完成，不 ack，让消息重新投递。
- 消费逻辑必须幂等。

宁愿重复消费，也不要 ack 后丢处理。

## 在 eMall 项目中怎么讲？

订单服务关闭时，已经进入订单创建的请求尽量完成本地事务。

如果实例被迫退出，客户端重试时用幂等键返回同一订单结果，避免重复创建订单。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
服务关闭时先停止新流量，再等待在途请求有限时间完成。HTTP 请求可以正常返回或超时后让客户端重试；
MQ 消费要停止拉新消息，当前消息处理成功后 ack，未完成则不 ack。

关键是写操作具备幂等和可恢复状态，确保发布、扩容和缩容不会造成重复副作用或数据丢失。
```

## 回答评分点

高分答案应该覆盖：

- 停止新请求后等待在途请求。
- 等待要有超时。
- 写接口需要幂等。
- MQ 消费未完成不能 ack。
- 业务状态要可恢复。
