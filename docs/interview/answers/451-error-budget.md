# 451 错误预算是什么？

[返回按分类学习面试题](../README.md)

## 题目

错误预算是什么？

## 先给面试官的短答案

错误预算是 SLO 允许的不可靠额度。例如月可用性目标是 99.9%，那一个月允许约 0.1% 的请求失败或
不可用。错误预算用于平衡稳定性和迭代速度。

预算消耗过快时，应减少高风险发布，把精力转向稳定性修复。

## 基本含义

如果 SLO 是：

```text
99.9% availability
```

则错误预算是：

```text
0.1% unavailable or failed requests
```

这给团队一个可量化的稳定性边界。

## 用途

用途：

- 指导发布节奏。
- 判断是否冻结高风险变更。
- 衡量可靠性投入。
- 避免过度追求 100%。
- 让业务和技术对齐。

错误预算把稳定性讨论量化。

## 消耗过快怎么办

动作：

- 暂停高风险发布。
- 修复高频故障。
- 增加自动化回滚。
- 加强监控和限流。
- 优先处理可靠性债务。

预算不是用完就算了，而是触发管理动作。

## 在 eMall 项目中怎么讲？

eMall 下单链路月 SLO 为 99.95%。如果大促前已经消耗了大部分错误预算，团队应暂停非必要改动，
优先修复库存超时和订单错误率问题。

这样能避免在稳定性风险高时继续叠加变更风险。

## 深度增强：Kubernetes 运维治理图

![Kubernetes 生产运行和故障治理](../assets/kubernetes-operations.svg)

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
错误预算是 SLO 允许的失败额度。它把可靠性目标转化为可消耗的预算，用来平衡新功能发布和系统
稳定性。

当错误预算充足时，可以保持正常迭代；当预算快速消耗时，应降低发布频率、暂停高风险变更，并把
工程资源投入到稳定性修复。
```

## 回答评分点

高分答案应该覆盖：

- 错误预算来自 SLO。
- 是允许失败额度。
- 用于平衡速度和稳定。
- 预算消耗快要限制发布。
- 能结合核心链路举例。
