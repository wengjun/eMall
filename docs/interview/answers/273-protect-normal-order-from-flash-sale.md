# 273 如何保护普通下单不被秒杀拖垮？

[返回按分类学习面试题](../README.md)

## 题目

如何保护普通下单不被秒杀拖垮？

## 先给面试官的短答案

普通下单和秒杀要做流量、服务、线程池、队列、缓存、数据库和容量隔离。
秒杀流量应有独立入口、限流、令牌、队列和资源配额，不能与普通订单共享所有核心资源。

核心目标是秒杀失败只影响秒杀，不影响正常购物。

## 隔离维度

隔离包括：

- 网关路由隔离。
- 服务集群隔离。
- 线程池隔离。
- 数据库连接池隔离。
- MQ topic 隔离。
- 缓存 key 和实例隔离。
- 限流配额隔离。
- 监控告警隔离。

共享资源越多，故障越容易扩散。

## 秒杀链路设计

秒杀应：

- 独立活动入口。
- 静态页面 CDN。
- 令牌控制进入量。
- 异步排队创建订单。
- 独立库存策略。
- 快速售罄标记。

普通下单继续走稳定交易链路。

## 数据库保护

普通订单库不能被秒杀写入打满。

可以：

- 秒杀订单先写专用队列。
- 限制秒杀订单创建速率。
- 使用专用表或分区。
- 设置数据库连接配额。

数据库是最后防线，不能无限承接秒杀洪峰。

## 在 eMall 项目中怎么讲？

秒杀活动使用 `flash-sale` 模块和独立队列处理，普通订单仍由 `order` 模块按常规容量运行。

秒杀队列积压时，只影响秒杀结果，不占满普通订单服务的线程池和数据库连接池。

## 深度增强：数据访问和扩展图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
保护普通下单的关键是隔离秒杀流量。秒杀要有独立入口、限流、令牌、队列、线程池、连接池和资源配额，
不能和普通订单共享所有核心资源。

当秒杀过载时，可以拒绝、排队或售罄，但不能让普通商品下单、支付和履约被拖垮。
```

## 回答评分点

高分答案应该覆盖：

- 秒杀和普通下单要隔离。
- 隔离覆盖入口、线程池、队列和数据库。
- 秒杀要削峰和令牌。
- 普通订单资源要有保底。
- 秒杀故障不能扩散到全站交易。
