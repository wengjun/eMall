# 036 如何判断一段代码是否可测试？

[返回按分类学习面试题](../README.md)

## 题目

如何判断一段代码是否可测试？

## 先给面试官的短答案

可测试代码通常依赖清晰、输入输出明确、业务逻辑和外部 IO 分离、时间和随机数可替换、
错误分支可触发、没有隐藏全局状态。

如果一段代码必须启动全套环境、连真实数据库、调真实下游才能验证一个业务分支，
通常说明它的设计可测试性不够。

## 可测试代码的特征

### 构造函数注入依赖

```java
public OrderService(OrderRepository repository,
                    InventoryClient inventoryClient,
                    SnowflakeIdGenerator idGenerator) {
}
```

测试时可以替换成 fake 或 mock。

### 业务逻辑和 IO 分离

库存是否足够是业务规则；查询数据库是 IO。
两者不要全部混在 Controller 或工具方法里。

### 时间可控制

不要在核心逻辑里到处写：

```java
Instant.now()
```

可以注入 `Clock`，测试中过固定时间。

### 随机数和 ID 可替换

ID 生成器、随机策略都应该可替换，否则测试难断言。

### 错误分支可触发

例如库存服务超时、支付金额不一致、MQ 发送失败，都要能在测试中模拟。

## 不可测试代码的坏味道

- 方法里直接 new 下游客户端。
- 静态全局状态太多。
- 业务逻辑写在私有大方法里，无法独立验证。
- 依赖当前时间、随机数、线程 sleep。
- catch 后吞异常。
- 只通过日志表达结果。
- 一个方法同时做参数解析、业务校验、数据库、HTTP、MQ。

## 在 eMall 项目中怎么讲？

订单创建可测试，是因为：

- `OrderService` 通过构造函数注入 Repository、Client 和 ID 生成器。
- 可以用内存 Repository 测试业务规则。
- 可以模拟库存预占成功或失败。
- 可以断言订单状态和 Outbox 事件。

测试重点：

- 重复 requestId 只创建一个订单。
- 库存失败进入 `PENDING_RETRY`。
- 支付后状态变为 `PAID`。
- 非法状态抛业务异常。

## 专家级完整回答

```text
我判断代码是否可测试，主要看依赖是否可替换、业务逻辑是否和 IO 分离、
时间和随机数是否可控制、错误分支是否可触发。

可测试性是设计质量的结果。构造函数注入、清晰分层、领域方法和接口隔离，
都能让核心业务在不启动完整环境的情况下验证。
对于数据库、Kafka、Redis 这类真实行为，再用集成测试覆盖。
```

## 回答评分点

高分答案应该覆盖：

- 依赖注入。
- IO 和业务逻辑分离。
- 时间、随机数、ID 可替换。
- 错误分支可测试。
- 能区分单元测试和集成测试。
