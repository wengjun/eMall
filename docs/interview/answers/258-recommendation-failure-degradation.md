# 258 推荐服务故障如何降级？

[返回按分类学习面试题](../README.md)

## 题目

推荐服务故障如何降级？

## 先给面试官的短答案

推荐服务故障通常可以降级，因为它影响转化和体验，但不应影响下单正确性。
常见降级方式是返回热门商品、类目热销、缓存结果、人工配置榜单，或者隐藏推荐模块。

降级目标是保住主链路，同时尽量减少用户体验和转化损失。

## 降级层级

可以分层：

- 个性化推荐。
- 用户相似推荐。
- 类目热销。
- 全站热销。
- 运营配置。
- 隐藏模块。

从个性化逐步退化到稳定兜底。

## 技术实现

实现方式：

- 熔断推荐接口。
- 设置短超时。
- 使用本地缓存或 Redis 缓存。
- 预计算热门榜单。
- BFF 层隐藏模块。
- 记录降级指标。

推荐不能拖慢商品详情页和首页主渲染。

## 注意事项

注意：

- 缓存结果要有过期时间。
- 热门榜单要提前预热。
- 降级内容要符合合规和风控。
- 恢复时要小流量放开。

推荐降级也可能影响收入，要监控点击率和转化率。

## 在 eMall 项目中怎么讲？

商品详情页调用推荐服务超过 `50ms` 时，可以返回同类目热销商品。

如果推荐服务持续失败，BFF 熔断推荐调用，首页保留核心商品和搜索入口。

## 深度增强：数据访问和扩展图

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
推荐服务适合分层降级。优先返回个性化推荐，失败后使用缓存结果、类目热销、全站热销或运营配置，
最差隐藏推荐模块。推荐调用要短超时和可熔断，不能拖慢商品详情、首页和下单链路。

降级期间要监控错误率、延迟、点击率和转化率，恢复时小流量探测。
```

## 回答评分点

高分答案应该覆盖：

- 推荐服务通常可降级。
- 可用缓存、热门榜单和隐藏模块。
- 推荐不能阻塞核心链路。
- 降级也要监控业务指标。
- 恢复要小流量探测。
