# 203 核心链路和非核心链路如何隔离？

[返回按分类学习面试题](../README.md)

## 题目

核心链路和非核心链路如何隔离？

## 先给面试官的短答案

核心链路隔离的目标是让推荐、广告、通知、报表等非核心能力故障时，不影响下单、支付、库存等核心交易。
常用手段包括服务分级、线程池隔离、连接池隔离、限流、熔断、降级、异步化、数据隔离和资源配额。

隔离不是只在代码里判断异常，而是从运行资源到业务流程都要隔离。

## 隔离维度

维度包括：

- 服务依赖隔离。
- 线程池隔离。
- 连接池隔离。
- 缓存和数据库资源隔离。
- MQ topic 和消费组隔离。
- 部署集群隔离。
- 配置和开关隔离。
- 告警和值班隔离。

共享资源越多，故障越容易传播。

## 主链路治理

核心链路要：

- 减少同步依赖。
- 明确强依赖和弱依赖。
- 弱依赖必须可降级。
- 下游调用设置超时预算。
- 失败快速返回或使用兜底。
- 禁止大查询和慢任务抢占资源。

核心资源应优先保障交易成功。

## 非核心处理方式

非核心能力可以：

- 异步消费事件。
- 使用独立线程池。
- 使用独立队列。
- 使用缓存默认值。
- 失败后补偿。
- 高峰期主动降级。

非核心链路要接受延迟和降级。

## 在 eMall 项目中怎么讲？

下单接口可以同步依赖库存和价格，但不应同步依赖短信、推荐刷新和数据仓库写入。

短信失败不能导致下单失败，推荐失败可以隐藏模块，数据仓库写入失败可以通过 MQ 重试。

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
核心链路和非核心链路要从调用、资源和业务语义上隔离。核心链路只保留必要强依赖，并设置超时、熔断和降级；
非核心能力通过事件异步处理，使用独立线程池、队列、连接池和资源配额。

目标是非核心服务故障、积压或慢查询时，不会抢占核心交易资源，也不会让用户下单和支付失败。
```

## 回答评分点

高分答案应该覆盖：

- 隔离要覆盖调用和资源。
- 核心链路要控制同步依赖。
- 弱依赖必须可降级。
- 非核心能力应异步化。
- 线程池、连接池、队列和数据资源都要隔离。
