# 110 分布式锁有哪些风险？

[返回按分类学习面试题](../README.md)

## 题目

分布式锁有哪些风险？

## 先给面试官的短答案

分布式锁风险包括锁过期导致并发执行、业务执行时间超过 TTL、客户端 GC 暂停、网络抖动、锁误删、
续约失败、时钟问题、脑裂、锁服务不可用和锁粒度过大导致性能瓶颈。使用分布式锁必须配合唯一锁值、
原子解锁、合理 TTL、续约、超时、幂等和 fencing token。

不能因为用了分布式锁就认为业务一定安全。

## 锁过期风险

如果业务还没执行完，锁 TTL 先过期。

其他实例可能获得锁并开始执行。

结果两个实例同时操作共享资源。

这在 GC 暂停、下游慢、网络抖动时都可能发生。

## 锁误删风险

实例 A 获得锁，锁过期后实例 B 获得锁。

实例 A 执行完后直接删除 key，可能误删 B 的锁。

所以解锁必须校验唯一 value。

Redis 中通常使用 Lua：

```text
if get(key) == value then del(key)
```

## 续约风险

看门狗续约可以降低锁过期风险。

但续约也可能失败：

- 进程卡顿。
- 网络异常。
- Redis 抖动。
- GC 暂停。

续约不是绝对安全，只是降低概率。

## 网络分区和脑裂

分布式系统中，客户端和锁服务之间可能网络异常。

客户端以为自己持有锁，但锁服务可能已经过期或切换。

如果业务资源不校验版本，仍可能并发写入。

## Fencing Token

更强的方式是使用 fencing token。

每次获取锁时生成递增 token，下游资源只接受更大的 token。

这样即使旧客户端恢复后继续写，也会被资源端拒绝。

这比单纯依赖锁 TTL 更可靠。

## 性能风险

锁粒度过大会导致：

- 请求串行化。
- 吞吐下降。
- P99 升高。
- 锁服务热点。

高频场景要优先拆分热点，而不是加一把全局锁。

## 可用性风险

锁服务不可用时，业务怎么办？

必须明确：

- 快速失败。
- 降级。
- 走本地保护。
- 进入队列。
- 禁止执行。

不同业务选择不同策略。

## 在 eMall 项目中怎么讲？

如果用分布式锁保护优惠券领取，仍然要有数据库唯一键防重复领取。

因为锁可能过期、误删或续约失败。

更稳妥设计是锁降低并发冲突，数据库唯一键和幂等表兜底保证正确性。

## 深度增强：并发治理图

![Java 并发从线程安全到容量保护](../assets/concurrency-governance.svg)

并发题不能只回答 API 用法。生产系统要同时考虑线程安全、资源隔离、超时、拒绝、幂等和分布式多实例。
单机锁只能保护当前 JVM，不能保护整个集群；线程池满也不是小问题，而是容量和可用性风险。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

## 深度增强：面试高分表达

我会把并发问题分成三层：JMM 和锁保证单机正确性，线程池和隔离保证资源不被拖垮，
幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
分布式锁的风险包括 TTL 过期、业务执行超时、GC 暂停、网络分区、误删他人锁、续约失败、
锁服务不可用和锁粒度过大。正确实现至少要有唯一锁值、原子解锁、合理 TTL、超时、续约和监控。
更强场景要用 fencing token，让资源端拒绝旧 token 写入。

业务正确性不能只依赖锁。优惠券、订单、支付这类场景还要用唯一键、幂等表、状态机和补偿兜底。
```

## 回答评分点

高分答案应该覆盖：

- TTL 过期和 GC 暂停风险。
- 唯一锁值和原子解锁。
- 续约不是绝对安全。
- fencing token。
- 业务要用唯一键和幂等兜底。
