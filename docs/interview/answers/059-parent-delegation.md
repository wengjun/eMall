# 059 双亲委派模型解决什么问题？

[返回按分类学习面试题](../README.md)

## 题目

双亲委派模型解决什么问题？

## 先给面试官的短答案

双亲委派模型的核心是：类加载器加载类时，先委托父加载器尝试加载，父加载器加载不了，
子加载器才自己加载。它主要解决核心类安全和类加载一致性问题，避免应用自定义一个
`java.lang.String` 这种类来替换 JDK 核心类。

它不是绝对不可打破的规则。插件、容器、SPI 和热部署场景可能需要改变委派顺序。

## 什么是双亲委派？

类加载器通常形成层级：

```text
Bootstrap ClassLoader
        ^
Platform ClassLoader
        ^
Application ClassLoader
        ^
Custom ClassLoader
```

当子加载器收到加载请求时，不是马上自己加载，而是先交给父加载器。

只有父加载器找不到这个类时，子加载器才尝试加载。

## 解决问题一：保护核心类

如果没有双亲委派，应用可以自己写一个：

```text
java.lang.String
```

然后让应用加载器优先加载它。这样 JDK 核心类就可能被替换，安全性和稳定性都会崩溃。

双亲委派保证 JDK 核心类优先由 Bootstrap ClassLoader 加载。

这让核心 API 在 JVM 内保持一致。

## 解决问题二：避免类重复加载

如果每个加载器都优先自己加载，常用类可能被重复加载多份。

双亲委派能让公共类尽量由上层加载器统一加载，减少重复和冲突。

例如 JDK 类由 Bootstrap 加载，平台类由 Platform 加载，应用类由 Application 加载。

## 解决问题三：保证类型一致性

JVM 中类的身份是：

```text
类全限定名 + 定义它的 ClassLoader
```

如果同名类由不同加载器加载，它们不是同一个类型。

这可能导致：

```text
ClassCastException: A cannot be cast to A
```

看起来类名一样，但 ClassLoader 不同。

双亲委派减少这种混乱。

## 加载流程

典型流程：

1. 子加载器收到加载请求。
2. 检查自己是否已经加载过。
3. 委托父加载器加载。
4. 父加载器继续向上委托。
5. 上层加载失败后，子加载器自己查找并加载。

面试中可以说：这是“向上委托，向下尝试加载”。

## 双亲委派能不能被打破？

可以。

一些场景需要打破或调整委派模型：

- SPI。
- 应用服务器。
- 插件系统。
- 热部署。
- OSGi。
- 某些框架的类隔离。

例如 SPI 中，JDK 核心类需要加载应用 classpath 下的实现类，这时会使用线程上下文类加载器。

## 线程上下文类加载器

线程上下文类加载器用于解决父加载器需要访问子加载器类的问题。

典型场景是：

- JDBC Driver。
- JNDI。
- ServiceLoader。
- 日志框架适配。

父加载器负责 API，子加载器负责实现，线程上下文类加载器帮助找到实现类。

## 打破双亲委派的风险

风险包括：

- 核心类被错误覆盖。
- 同名类冲突。
- `ClassCastException`。
- 类加载器泄漏。
- 依赖版本不一致。
- 安全边界被破坏。

所以自定义类加载策略要非常谨慎。

## 在 eMall 项目中怎么讲？

eMall 普通服务没有必要打破双亲委派。

但如果营销平台允许商家上传插件，或者风控平台支持动态策略，就可能需要插件 ClassLoader。

这时建议：

- 公共 API 由父加载器加载。
- 插件实现由子加载器加载。
- 插件依赖做隔离。
- 禁止覆盖 JDK 和平台核心类。
- 插件卸载时清理线程、缓存和 ThreadLocal。

这能兼顾扩展能力和系统安全。

## 深度增强：JVM 生产运行图

![Java 17 容器内 JVM 内存结构](../assets/jvm-runtime-memory.svg)

JVM 题要从运行时资源解释到业务影响。堆、直接内存、元空间、线程栈和容器 memory limit 共同决定服务稳定性；
GC、CPU throttling、线程池队列和下游超时会一起影响 P99，而不是孤立存在。

## 深度增强：Java 17 诊断模型示例

```java
record RuntimeSignal(
        double heapUsage,
        double containerMemoryUsage,
        long gcPauseMillis,
        int threadCount,
        int queuedTasks) {

    boolean requiresTriage() {
        return heapUsage > 0.85
                || containerMemoryUsage > 0.90
                || gcPauseMillis > 500
                || threadCount > 800
                || queuedTasks > 1_000;
    }
}
```

这个模型强调线上诊断要看组合信号。只看 heap 不够，只看 GC 也不够；
要把 JVM、容器、线程池和业务延迟放到同一条时间线。

## 深度增强：生产边界

JVM 调优不能靠背参数。要先明确服务目标：低延迟、吞吐、容器资源、对象分配速率和 P99 SLO。
然后通过 GC 日志、JFR、指标和压测验证。错误地调大 `-Xmx` 可能挤压堆外内存，导致容器 OOMKilled。

## 深度增强：面试高分表达

我会用证据链回答 JVM 问题：先看业务影响，再看 JVM 指标、GC 日志、线程栈、heap dump、容器事件和最近变更。
结论要能解释现象，并能给出降级、扩容、参数调整或代码优化方案。

## 专家级完整回答

```text
双亲委派模型的核心是加载类时先委托父加载器，父加载器找不到才由子加载器加载。
它主要解决 JDK 核心类安全、公共类一致性和重复加载问题。因为 JVM 中类身份由类名和
ClassLoader 共同决定，如果随意加载同名类，会出现类型不一致和 ClassCastException。

但双亲委派不是绝对不能打破。SPI、插件、应用服务器和热部署可能需要调整委派顺序，
这时要明确父子加载器边界，避免核心类被覆盖、依赖冲突和类加载器泄漏。
```

## 回答评分点

高分答案应该覆盖：

- 先委托父加载器。
- 保护 JDK 核心类。
- 避免重复加载和类型混乱。
- 知道类名加 ClassLoader 决定类身份。
- 知道 SPI 和插件会打破或绕过委派。
- 知道打破委派的风险。
