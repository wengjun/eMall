# 525 设计售后退款系统

[返回按分类学习面试题](../README.md)

## 题目

设计售后退款系统。

## 先给面试官的短答案

售后退款系统负责退货、退款、换货、仅退款、审核、逆向物流和资金退回。核心难点是订单状态、履约状态、支付状态、
优惠分摊、退款金额、库存回补和财务对账。退款必须幂等、可审计、可对账，不能简单按用户输入金额退款。

## 核心流程

用户提交售后申请，系统校验订单、商品、签收时间、售后政策和可退金额。需要审核的进入商家或平台审核。

审核通过后，退货场景生成逆向物流；仓库收货质检后触发退款。仅退款场景可直接进入退款。

支付服务执行原路退款或其他退款方式，退款成功后通知订单和财务系统。

## 金额计算

退款金额要基于订单价格快照，而不是当前商品价格。还要考虑优惠券、满减、积分、运费、税费和多商品优惠分摊。

部分退款时，要按规则分摊优惠。不能让用户通过拆分退款套利。

## 状态和幂等

售后状态包括申请中、审核中、待退货、待收货、退款中、退款成功、拒绝、关闭。每次状态流转要校验旧状态。

退款请求必须幂等。支付渠道重复通知时，不能重复退款或重复更新财务流水。

## 库存和财务

退货入库后是否恢复可售库存，取决于商品状态和质检结果。退款流水要参与支付渠道对账和财务结算。

高风险退款要接入风控和人工审核。

## 在 eMall 项目中怎么讲？

eMall 的 `after-sales` 模块负责售后申请、审核、退款状态和逆向流程，`payment` 执行退款，`inventory` 处理回补，
`finance` 做对账，`customer-service` 处理人工介入。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

## 专家级完整回答

```text
售后退款是交易的逆向链路，复杂度不低于下单。

我会先基于订单快照计算可退金额，再通过售后状态机管理审核、退货、收货、退款和关闭。
退款由支付系统执行，必须幂等，并参与渠道对账。

关键风险是金额算错、重复退款、库存错误回补和状态不一致。解决方式是价格快照、优惠分摊、
状态机、幂等、风控、对账和人工处理入口。
```

## 回答评分点

高分答案应该覆盖：

- 覆盖退货、退款、审核、逆向物流和质检。
- 知道退款金额基于订单快照和优惠分摊。
- 能说明售后状态机和退款幂等。
- 覆盖库存回补、财务对账和风控。
- 能结合支付、库存、财务和客服协作。

## 深度完善：系统设计高分展开

围绕「设计售后退款系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。

深度完善标记：系统设计题已补容量、数据流、失败路径和验收口径。
