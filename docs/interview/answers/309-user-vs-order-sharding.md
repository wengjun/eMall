# 309 按用户分片和按订单分片有什么区别？

[返回按分类学习面试题](../README.md)

## 题目

按用户分片和按订单分片有什么区别？

## 先给面试官的短答案

按用户分片适合用户维度查询，例如用户订单列表；按订单分片适合订单号定位和均匀写入。
两者的区别在于核心查询路径、数据分布、热点风险、跨分片查询和二级索引建设不同。

选择哪种方式取决于订单系统最重要的访问模式。

## 按用户分片

优点：

- 用户订单列表单分片查询。
- 用户维度事务更容易聚合。
- 用户数据归属清晰。

缺点：

- 超级用户可能热点。
- 按订单号查询需要知道用户或路由表。
- 商家后台查询可能跨分片。

## 按订单分片

优点：

- 写入分布更均匀。
- 订单详情按订单号定位方便。
- 避免用户热点更容易。

缺点：

- 用户订单列表可能跨分片。
- 需要用户订单读模型或索引表。
- 用户维度聚合更复杂。

## 混合方案

可以：

- 订单主表按订单 ID 分片。
- 用户订单索引表按用户 ID 分片。
- 商家订单读模型按商家 ID 分片。

写模型和读模型可以分开设计。

## 在 eMall 项目中怎么讲？

如果 eMall 用户端订单列表访问极高，可以按用户 ID 分片订单读模型。

订单主写入可以按订单 ID 均匀分片，通过事件同步用户和商家维度读模型。

## 深度增强：数据访问和扩展图

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
按用户分片让用户订单列表查询简单，但可能遇到用户热点，按订单号查询也需要路由信息。
按订单分片写入更均匀，订单详情定位方便，但用户列表和商家后台会跨分片。

大型电商常用写模型和读模型分离，用订单 ID 保证写入均匀，再构建用户和商家维度索引或读模型。
```

## 回答评分点

高分答案应该覆盖：

- 用户分片适合用户列表。
- 订单分片适合详情定位和均匀写入。
- 两者都有跨分片查询问题。
- 可用读模型解决非主路径。
- 选择取决于核心访问模式。
