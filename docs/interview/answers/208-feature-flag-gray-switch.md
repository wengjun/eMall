# 208 如何做灰度开关和功能开关？

[返回按分类学习面试题](../README.md)

## 题目

如何做灰度开关和功能开关？

## 先给面试官的短答案

灰度开关用于按用户、商家、地区、流量比例或版本逐步放量；功能开关用于快速开启、关闭或降级某个能力。
生产级开关要支持规则配置、实时生效、审计、回滚、默认值、过期清理和监控。

开关不是 if 判断那么简单，它是发布风险控制工具。

## 常见规则

规则包括：

- 按用户 ID。
- 按商家 ID。
- 按地区。
- 按客户端版本。
- 按流量百分比。
- 按白名单和黑名单。
- 按实验分组。

规则要稳定命中，不能每次请求随机变化。

## 开关设计要点

要点包括：

- 默认值安全。
- 配置变更可审计。
- 支持快速回滚。
- 规则可测试。
- 命中结果可观测。
- 开关有负责人。
- 开关有过期时间。
- 移除过期代码。

长期不清理的开关会让代码复杂度上升。

## 风险控制

高风险功能发布流程：

```text
内部白名单
-> 小比例用户
-> 单地区
-> 扩大流量
-> 全量
```

每一步观察错误率、延迟、业务指标和用户反馈。

## 在 eMall 项目中怎么讲？

新优惠计算逻辑可以先对内部员工和少量商家开启。

如果订单金额异常、错误率升高或支付转化下降，立刻关闭开关回到旧逻辑。

## 深度增强：可观测与配置治理图

![指标、日志、Trace 和告警平台](../assets/observability-platform.svg)

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
灰度开关用于逐步放量，功能开关用于快速启停和降级。生产级开关需要规则引擎、稳定分流、审计、回滚、监控、
默认值和过期清理。

我会按用户、商家、地区、版本或流量比例灰度，并观察技术指标和业务指标。开关使用后要清理，避免长期堆积技术债。
```

## 回答评分点

高分答案应该覆盖：

- 灰度开关和功能开关目标不同。
- 规则要支持用户、商家、地区和比例。
- 默认值、审计、回滚和监控重要。
- 分流要稳定。
- 过期开关要清理。
