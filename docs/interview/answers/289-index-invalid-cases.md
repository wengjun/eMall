# 289 什么情况下索引会失效？

[返回按分类学习面试题](../README.md)

## 题目

什么情况下索引会失效？

## 先给面试官的短答案

索引失效通常是因为查询条件无法按索引有序结构定位数据，或者优化器判断走索引成本更高。
常见情况包括对索引列做函数计算、隐式类型转换、前置模糊匹配、违反最左前缀、低选择性和返回大量数据。

索引是否使用最终要看执行计划，而不是只看有没有建索引。

## 常见失效场景

包括：

- `WHERE DATE(created_at) = ?`。
- 字符串列和数字比较导致隐式转换。
- `LIKE '%keyword'`。
- 联合索引跳过最左列。
- `OR` 条件无法都用索引。
- 不等于和低选择性条件。
- 返回表中大量数据。

这些都可能让优化器放弃索引。

## 函数和计算

错误写法：

```sql
WHERE DATE(created_at) = '2026-04-30'
```

更好写法：

```sql
WHERE created_at >= '2026-04-30 00:00:00'
  AND created_at < '2026-05-01 00:00:00'
```

让索引列保持原样更容易使用索引。

## 优化器成本判断

即使有索引，优化器也可能全表扫描。

例如某个状态值覆盖 90% 的记录，走索引再回表可能比全表扫描更慢。

## 在 eMall 项目中怎么讲？

订单表按 `created_at` 建索引后，如果后台查询使用 `DATE(created_at)`，可能无法有效使用索引。

应改成时间范围查询，并限制查询时间窗口。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
索引失效通常因为查询条件无法利用 B+Tree 的有序性，或优化器判断走索引成本更高。
常见原因是函数计算、隐式类型转换、前置模糊匹配、违反最左前缀、低选择性和返回大量数据。

排查时要看 explain 和实际执行情况，不能只看表上是否有索引。
```

## 回答评分点

高分答案应该覆盖：

- 函数计算会影响索引使用。
- 隐式类型转换很常见。
- 前置模糊匹配难用 B+Tree。
- 违反最左前缀会影响联合索引。
- 优化器可能因成本选择全表扫描。
