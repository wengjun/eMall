## 题目

分片键如何选择？

## 先给面试官的短答案

分片键要优先服务最高频、最高价值的查询和写入路径，同时保证数据分布均匀、热点少、路由简单、扩容可控。
常见候选包括用户 ID、订单 ID、商家 ID、租户 ID 和时间，但每个都有取舍。

分片键一旦选错，后续查询和扩容成本会很高。

## 选择标准

标准包括：

- 高频查询是否带该字段。
- 数据分布是否均匀。
- 是否会产生热点。
- 是否支持单分片路由。
- 是否影响事务边界。
- 是否方便扩容迁移。
- 是否符合数据所有权。

最重要的是让核心查询尽量命中单分片。

## 用户 ID

优点：

- 用户订单列表查询方便。
- 用户维度数据聚合方便。

缺点：

- 按订单号查询需要路由表或订单号包含分片信息。
- 大客户或高活跃用户可能热点。

## 订单 ID

优点：

- 订单详情定位方便。
- 数据分布容易均匀。

缺点：

- 用户订单列表可能跨分片。
- 需要额外索引或读模型支持用户查询。

## 在 eMall 项目中怎么讲？

如果用户查订单列表是核心场景，可以按用户 ID 分片，并让订单 ID 中携带分片信息。

如果订单号定位和写入均匀更重要，可以按订单 ID 分片，再为用户订单列表构建读模型。

分片键不是技术偏好，而是业务访问模式的固化。选错分片键后，后续每个高频查询都可能变成跨库查询。

## 深度增强：选择矩阵

| 分片键 | 适合场景 | 主要风险 |
| --- | --- | --- |
| `user_id` | 用户订单列表、用户维度售后 | 大客户或高活跃用户热点 |
| `order_id` | 订单详情、写入均匀、按订单号定位 | 用户列表需要读模型 |
| `merchant_id` | 商家后台、商家维度结算 | 大商家热点明显 |
| `tenant_id` | SaaS 或多租户隔离 | 大租户和小租户差异大 |
| 时间 | 归档、历史查询、账单 | 当前时间写入热点 |

## 深度增强：Java 17 订单号携带路由信息

如果按用户分片，可以让内部订单 ID 携带分片信息，方便订单号定位时少一次路由表查询：

```java
public final class RoutedOrderId {

    private static final long SHARD_MASK = 0x3FFL;

    public long compose(long timestampPart, int shardId, long sequence) {
        return (timestampPart << 22)
                | ((long) shardId << 12)
                | sequence;
    }

    public int extractShardId(long orderId) {
        return (int) ((orderId >> 12) & SHARD_MASK);
    }
}
```

对外展示订单号不一定要直接暴露这个内部 ID，避免泄露业务规模和分片规则。

热点可控，并且支持后续扩容。非主路径不要强行要求一个分片键全解决，可以用读模型、路由表或数仓补齐。

## 专家级完整回答

```text
分片键选择要看核心查询路径和数据分布。好的分片键能让高频查询单分片完成，并且数据均匀、热点少、扩容可控。
用户 ID 适合用户订单列表，订单 ID 适合订单详情定位和均匀写入，商家 ID 适合商家后台。

分片键没有绝对正确，要用业务访问模式决定，并为非主路径准备读模型或路由表。
```

## 回答评分点

高分答案应该覆盖：

- 分片键服务核心查询。
- 数据要分布均匀。
- 要避免热点。
- 用户 ID 和订单 ID 取舍不同。
- 非主查询路径需要读模型或路由。
