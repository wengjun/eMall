# 354 Redis 故障时系统如何降级？

[返回按分类学习面试题](../README.md)

## 题目

Redis 故障时系统如何降级？

## 先给面试官的短答案

Redis 故障时要防止所有请求直接回源数据库。系统应按业务重要性采取本地缓存、限流、熔断、返回
旧值、默认值、只读模式、排队或拒绝非核心请求。

降级目标不是保持所有功能完整，而是保护核心交易链路和数据库。

## 故障影响

可能影响：

- 商品详情缓存失效。
- 会话校验变慢。
- 限流计数失效。
- 分布式锁不可用。
- 购物车缓存不可用。
- 热点数据回源数据库。

不同业务要有不同预案。

## 降级策略

策略：

- 本地缓存返回旧值。
- 非核心推荐降级为空。
- 商品详情限制回源并发。
- 下单接口保留核心校验。
- Redis 锁不可用时拒绝高风险操作。
- 网关启用本地限流。
- 数据库熔断保护。

不能让缓存故障演变成数据库故障。

## 恢复策略

恢复时要注意：

- 不要立刻放开全部流量。
- 缓存逐步预热。
- 回源限速。
- 观察命中率和数据库压力。
- 确认 Redis 延迟恢复正常。

恢复也需要平滑，避免二次雪崩。

## 在 eMall 项目中怎么讲？

eMall Redis 故障时，商品详情可以返回本地旧缓存，推荐和热榜可以降级为空。

下单链路不能直接跳过库存和价格校验。如果 Redis 限流不可用，网关启用本地限流；如果库存热点
令牌不可用，秒杀接口应该限流或暂停，而不是把请求全部打到库存库。

## 深度增强：缓存和消息治理图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

缓存和消息题要关注一致性、削峰、延迟、积压和恢复。
Redis 很快，但会遇到穿透、击穿、雪崩、热点 key 和内存淘汰；
MQ 能解耦和削峰，但会带来重复消费、乱序、积压和死信处理。

## 深度增强：Java 17 幂等消费示例

```java
import java.util.Set;
import java.util.concurrent.ConcurrentHashMap;

final class LocalIdempotentConsumer {
    private final Set<String> processedKeys = ConcurrentHashMap.newKeySet();

    boolean tryHandle(String messageKey, Runnable handler) {
        if (!processedKeys.add(messageKey)) {
            return false;
        }
        handler.run();
        return true;
    }
}
```

这个示例只适合解释幂等思想。生产环境不能用本地内存做全局幂等，要使用数据库唯一键、Redis 原子操作或业务状态机。

## 深度增强：生产边界

缓存要有 TTL、容量、降级和回源保护；消息要有重试、死信、延迟队列、消费幂等和积压告警。
缓存不一致要能修复，消息失败要能回放，不能只依赖人工查日志。

## 深度增强：面试高分表达

我会把缓存和消息都看成性能与稳定性工具，而不是正确性事实来源。
正确性由数据库事实、状态机、幂等和对账保证；缓存和 MQ 负责降低延迟、削峰填谷和解耦系统。

## 专家级完整回答

```text
Redis 故障降级的核心是保护数据库和核心链路。不能因为缓存不可用就让所有流量回源。展示类业务
可以返回旧值或默认值，推荐类可以降级为空，交易类要保留最终校验，高风险操作可以拒绝。

同时要启用本地限流、熔断和回源并发控制。Redis 恢复后也要逐步预热和平滑放量，避免再次雪崩。
```

## 回答评分点

高分答案应该覆盖：

- Redis 故障不能直接全量回源。
- 区分核心和非核心业务。
- 本地缓存、限流、熔断、降级。
- 交易链路不能跳过最终校验。
- 恢复要平滑预热。
