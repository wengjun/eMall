# 280 如何判断系统已经进入过载？

[返回按分类学习面试题](../README.md)

## 题目

如何判断系统已经进入过载？

## 先给面试官的短答案

系统进入过载通常表现为吞吐不再增长、延迟快速升高、错误率增加、队列堆积、线程池和连接池耗尽。
判断时不能只看 CPU，要同时看资源指标、排队指标、延迟分位数和业务成功率。

过载的典型信号是系统花更多时间排队和重试，而不是处理有效请求。

## 技术指标

指标包括：

- CPU 长时间高位。
- GC 时间增加。
- 线程池活跃数满。
- 队列长度持续增长。
- 数据库连接池耗尽。
- HTTP 连接池等待增加。
- P95/P99 延迟升高。
- 超时和 5xx 增加。

队列持续增长是强信号。

## 业务指标

还要看：

- 下单成功率下降。
- 支付成功率下降。
- 库存扣减失败率升高。
- 用户取消率增加。
- 客服投诉增加。

技术指标正常但业务成功率下降，也说明系统可能有局部过载。

## 吞吐曲线

压测时如果继续增加并发：

- QPS 不再上升。
- 延迟明显升高。
- 错误率上升。

这说明系统已经越过稳定容量点。

## 在 eMall 项目中怎么讲？

秒杀期间如果订单服务 QPS 不再增加，但 P99 从 `200ms` 升到 `3s`，线程池队列持续增长，
说明系统已过载，应立即限流或降级。

## 深度增强：数据访问和扩展图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
判断过载要看多维指标。典型表现是吞吐不再增长，P95/P99 延迟升高，错误率增加，
线程池和连接池耗尽，队列持续堆积，业务成功率下降。

CPU 不高也可能过载，例如系统阻塞在数据库连接或下游超时上。因此要结合资源、队列、延迟和业务指标判断。
```

## 回答评分点

高分答案应该覆盖：

- 吞吐不增长但延迟升高是典型信号。
- 队列持续增长很关键。
- CPU 不是唯一指标。
- 要看 P95/P99 和错误率。
- 业务成功率也要纳入判断。
