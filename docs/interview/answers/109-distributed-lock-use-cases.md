# 109 分布式锁适合哪些场景？

[返回按分类学习面试题](../README.md)

## 题目

分布式锁适合哪些场景？

## 先给面试官的短答案

分布式锁适合多实例之间需要短时间互斥访问共享资源的场景，例如防止重复执行定时任务、保护低频共享资源、
控制某个全局操作同一时间只有一个执行者。它不适合长事务、高频热点库存扣减和强一致金融记账这类场景。

优先考虑数据库约束、幂等和状态机，分布式锁只是工具之一。

## 适合场景

适合：

- 多实例定时任务只执行一次。
- 防止缓存重建并发击穿。
- 低频全局配置刷新。
- 低频资源迁移。
- 管理后台操作互斥。
- 非核心任务调度抢占。

这些场景共同特点是低频、短时间、可重试、失败可恢复。

## 不适合场景

不适合：

- 高频库存扣减。
- 支付扣款强一致。
- 长时间业务流程。
- 锁内调用多个下游。
- 不可重试操作。
- 无法容忍锁失效的场景。

分布式锁不是分布式事务。

## 为什么库存不一定适合锁？

秒杀库存是高频热点。

如果所有请求都抢同一把分布式锁，吞吐会很低。

更常见方案：

- Redis 原子预扣。
- 库存桶。
- MQ 削峰。
- 数据库条件更新。
- 售罄标记。

锁会把高并发串行化，不一定是好方案。

## 分布式锁基本要求

一个可用的分布式锁至少要有：

- 原子加锁。
- 唯一锁值。
- 过期时间。
- 原子解锁。
- 超时控制。
- 续约机制。
- 异常释放。
- 可观测性。

Redis 锁通常要使用 `SET key value NX PX ttl`，解锁用 Lua 校验 value 后删除。

## 在 eMall 项目中怎么讲？

缓存击穿时，可以用分布式锁让一个实例重建热点商品缓存，其他实例短暂等待或返回旧值。

但下单扣库存不应简单用一把分布式锁包住全流程。

库存扣减更适合用 Redis Lua、数据库条件更新、库存桶和 MQ 削峰。

## 深度增强：并发治理图

![Java 并发从线程安全到容量保护](../assets/concurrency-governance.svg)

并发题不能只回答 API 用法。生产系统要同时考虑线程安全、资源隔离、超时、拒绝、幂等和分布式多实例。
单机锁只能保护当前 JVM，不能保护整个集群；线程池满也不是小问题，而是容量和可用性风险。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

## 深度增强：面试高分表达

我会把并发问题分成三层：JMM 和锁保证单机正确性，线程池和隔离保证资源不被拖垮，
幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
分布式锁适合多实例之间短时间、低频、可重试的互斥操作，比如定时任务去重、缓存重建、配置刷新和后台管理操作。
它不适合长事务、高频热点库存和强一致资金操作，因为锁失效、续约失败、网络分区和串行化都会带来风险。

我会优先考虑数据库唯一键、条件更新、幂等表和状态机。只有确实需要跨实例短时互斥时才使用分布式锁。
```

## 回答评分点

高分答案应该覆盖：

- 适合短时低频互斥。
- 不适合长事务和高频热点。
- 不是分布式事务。
- Redis 锁要唯一值和原子解锁。
- 库存场景通常有更好方案。
