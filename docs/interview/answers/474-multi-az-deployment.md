# 474 多可用区部署要考虑什么？

[返回按分类学习面试题](../README.md)

## 题目

多可用区部署要考虑什么？

## 先给面试官的短答案

多可用区部署要考虑容量分布、故障隔离、流量调度、数据复制、跨区延迟、依赖同城容灾、发布策略和演练验证。
目标不是简单把 Pod 分散到多个区，而是在任意一个可用区故障时，核心链路仍能承接流量并保持数据一致性。

## 需要考虑的维度

首先是计算容量。服务实例要通过拓扑分布约束分散到不同可用区，避免一次节点或机房故障打掉全部副本。
如果系统要求任一可用区故障后仍可运行，剩余可用区必须预留足够容量。

其次是流量路由。入口负载均衡要支持跨区摘除和就近访问，内部调用最好优先访问同区实例，降低延迟和跨区流量成本。
但同区不可用时，要能自动切到其他可用区。

第三是数据层。数据库、缓存、消息队列和搜索集群要明确主从、复制、选主、故障转移和一致性策略。
多可用区下，数据复制延迟和网络抖动会直接影响交易正确性。

## 常见风险

多可用区最常见的问题是“应用多区了，依赖没有多区”。例如订单服务部署在三个区，但 Redis 主节点只在一个区，
这个区故障时订单链路仍然不可用。

另一个风险是没有演练。架构图上看起来多区可用，不代表真实故障时连接池、DNS、缓存、消息消费和数据库连接都能恢复。

## 在 eMall 项目中怎么讲？

eMall 核心交易链路应该至少覆盖网关、身份、商品、促销、库存、订单、支付、消息和风控。多可用区设计要确保这些服务、
Redis、MySQL、Kafka、搜索和配置中心都有故障转移方案，并通过故障演练证明恢复时间满足要求。

## 深度增强：Kubernetes 运维治理图

![Kubernetes 生产运行和故障治理](../assets/kubernetes-operations.svg)

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
多可用区部署不是把服务副本平均放到多个区这么简单，而是要让一个区不可用时系统仍能工作。

我会从四个层面设计：计算层通过 topology spread 和反亲和分散实例；流量层支持健康摘除、
同区优先和跨区兜底；数据层明确复制、一致性和故障切换；运维层通过演练验证 RTO 和 RPO。

对电商系统来说，订单和支付链路不能只看服务副本，还要看 Redis、数据库、Kafka、配置中心、
支付通道和风控依赖是否具备同等容灾能力。否则一个单点依赖就会让多可用区失效。
```

## 回答评分点

高分答案应该覆盖：

- 说明多可用区目标是故障隔离和持续可用。
- 覆盖容量、路由、数据复制、依赖容灾和演练。
- 能指出同区优先和跨区兜底的权衡。
- 知道应用多区但依赖单区是常见缺陷。
- 能结合 RTO、RPO 和故障演练回答。
