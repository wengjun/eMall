# 520 设计商品搜索系统

[返回按分类学习面试题](../README.md)

## 题目

设计商品搜索系统。

## 先给面试官的短答案

商品搜索系统要支持关键词检索、类目筛选、排序、召回、相关性、个性化和高可用。典型架构是商品主数据通过事件或 CDC
同步到搜索索引，搜索服务基于 Elasticsearch 或 OpenSearch 查询，结合价格、库存、促销和排序策略返回结果。

## 核心需求

搜索要支持关键词、类目、品牌、属性筛选、价格区间、销量排序、综合排序、纠错、联想词和个性化。

它还要处理商品上下架、库存状态、价格变化、敏感词、商家风控和广告混排。

## 索引设计

索引中通常保存商品 ID、标题、类目、品牌、属性、分词字段、销量、评分、商家、上架状态和部分价格信息。

不是所有数据都适合放索引。价格和库存实时性强，可以在索引中保存近似值用于排序和过滤，但最终展示和下单前要二次确认。

## 数据同步

商品主数据变更后，通过 Kafka 事件、Outbox 或 CDC 同步到搜索索引。同步要保证幂等、可重放和可监控。

如果索引更新失败，要进入重试或重建流程。搜索结果缺商品或出现下架商品时，要能根据同步 lag 和失败事件排查。

## 查询链路

用户请求进入搜索服务后，先做 query 分析、分词、纠错和意图识别，再查询搜索引擎，得到候选商品 ID。

然后聚合价格、促销、库存和广告信息，最后排序和返回。对热点词可以缓存结果页或候选 ID。

## 高可用和降级

搜索引擎要分片副本、多可用区部署和限流保护。搜索失败时可以返回热门商品、类目兜底或提示稍后重试。
但不能把下架或违规商品作为兜底结果。

## 在 eMall 项目中怎么讲？

eMall 的 `search` 模块负责查询，`product` 和 `catalog` 负责商品主数据，`event-platform` 负责同步事件，
`advertising` 和 `recommendation` 可参与排序和混排。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

## 回答评分点

高分答案应该覆盖：

- 覆盖关键词、筛选、排序、纠错、联想和个性化。
- 能说明搜索索引字段和实时性取舍。
- 知道事件、Outbox 或 CDC 同步索引。
- 覆盖同步失败、重试、重放和监控。
- 能强调价格库存和下架状态的二次确认。

## 深度完善：系统设计高分展开

围绕「设计商品搜索系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。

深度完善标记：系统设计题已补容量、数据流、失败路径和验收口径。
