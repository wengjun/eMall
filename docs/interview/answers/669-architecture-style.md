# 669 当前系统是一体化架构还是微服务架构？

[返回按分类学习面试题](../README.md)

## 题目

当前系统是一体化架构还是微服务架构？

## 先给面试官的短答案

反问面试官用于判断团队工程成熟度、业务复杂度、岗位影响力和成长空间。

## 核心拆解

- 了解业务指标、系统规模和岗位职责。
- 判断 on-call、复盘、灰度和稳定性文化。
- 了解技术栈、部署平台和技术债。
- 判断 Senior/L6 岗位的真实影响力。

```text
我想了解这个问题，是为了判断团队的工程成熟度、业务复杂度和岗位影响力。
如果团队有稳定的指标、复盘和灰度机制，我能更快投入性能、稳定性和架构演进。
```

## 生产边界和常见坑

这个问题的关键不是“能不能做”，而是能否在高并发、灰度发布、故障恢复和数据修复场景下安全运行。
如果方案缺少监控、限流、幂等、回滚、审计或补偿，就只能算 demo，不能算生产级方案。

## 在 eMall 项目中怎么讲？

可以结合 eMall 的 `gateway`、`order`、`inventory`、`payment`、`risk`、`traffic`、
`reliability`、`release`、`operations` 和 `analytics` 模块说明。核心表达是：
先保护交易主链路，再保证数据可追踪，最后通过观测、补偿和复盘把风险沉淀为平台能力。

## 专家级完整回答

```text
我会先明确这个问题影响的是容量、可用性、一致性、安全还是工程效率。
然后拆解核心链路和失败场景，给出当前规模下最务实的方案。
生产系统里我会同时设计指标、告警、灰度、回滚、审计和补偿，避免方案只在正常路径成立。
如果规模继续增长，我会再从分片、异步化、多区域、自动化治理和成本优化上演进。
```

## 回答评分点

- 能先讲业务目标和生产影响。
- 能拆解核心链路、数据流和失败场景。
- 能给出 Java 17、SQL 或工程实现示例。
- 能说明监控、告警、回滚、补偿和审计。
- 能结合 eMall 项目说明落地方式。

## 深度完善：反问题使用策略

围绕「当前系统是一体化架构还是微服务架构？」，反问不是闲聊，而是判断岗位真实复杂度、团队成熟度和自己能否产生影响。
好的反问要礼貌、具体、能引出系统规模、工程文化和成长空间。

### 你真正想判断什么

- 团队是否有清晰业务指标，而不是只按需求堆功能。
- 核心系统是否有 SLO、on-call、复盘、灰度和自动化治理。
- Senior/L6 是否能影响架构方向、跨团队协作和长期质量。
- 技术债是否可被正视，团队是否愿意用数据推动改进。

### 如何追问

```text
我想了解这个问题，是为了判断我加入后能在哪些关键指标上产生影响。
如果方便的话，能否举一个最近团队通过工程手段改善业务或稳定性的例子？
```

### 如何解读回答

- 如果对方能讲清指标、事故、复盘和改进，说明团队工程成熟度较高。
- 如果只强调加班、救火和临时需求，说明稳定性和长期治理可能不足。
- 如果能说明新人前三个月的责任边界，说明团队 onboarding 和岗位期望更清晰。

## 二轮完善：反问题风险识别

围绕「当前系统是一体化架构还是微服务架构？」，反问的目的不是展示聪明，而是判断这个岗位是否真的有复杂问题、工程空间和成长空间。

### 好信号

- 对方能清楚说出核心业务指标、SLO、系统规模和近期技术挑战。
- 对方能讲出最近一次事故复盘后的机制改进，而不只是救火。
- 对方能说明 Senior/L6 工程师如何影响跨团队方案、平台能力和长期质量。
- 对方愿意讨论技术债、取舍和优先级，而不是只说“来了再看”。

### 风险信号

- 团队没有明确 owner、没有 on-call 机制、没有复盘文化。
- 所有问题都靠人肉处理，没有自动化、监控和 Runbook。
- 技术决策只看短期交付，不允许讨论稳定性、成本和长期演进。
- 面试官无法说明岗位前三个月的目标和衡量标准。

### 收束表达

`	ext
我问这个问题，是想判断我加入后能在哪些指标和系统问题上产生实际影响。
如果团队当前有明确的稳定性、性能或平台化挑战，我可以更快把项目经验转化为交付结果。
`
