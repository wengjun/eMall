# 062 热点代码和解释执行有什么区别？

[返回按分类学习面试题](../README.md)

## 题目

热点代码和解释执行有什么区别？

## 先给面试官的短答案

解释执行是 JVM 一条条解释字节码，启动快但执行效率相对低。热点代码是运行中被频繁执行的代码，
JVM 会把它识别出来，并通过 JIT 编译成本地机器码。热点代码经过优化后执行效率更高。

区别本质是：解释执行偏启动速度，热点编译偏长期性能。

## 解释执行是什么？

解释执行可以理解为 JVM 边读字节码边执行。

优点：

- 启动快。
- 不需要等待编译。
- 适合只执行一次或很少执行的代码。

缺点：

- 每次都要解释字节码。
- 很难做深度优化。
- 高频路径性能不如机器码。

应用启动阶段大量代码通常先解释执行。

## 热点代码是什么？

热点代码是执行频率很高、对整体性能影响大的代码。

常见热点：

- 高频接口的核心方法。
- 大循环。
- 序列化和反序列化。
- 价格计算。
- 规则匹配。
- 缓存读写。
- 加密签名。

JVM 会根据运行时计数和 profiling 数据识别热点。

## JIT 为什么只编译热点？

编译所有代码成本太高，也没有必要。

很多代码只在启动时执行一次，或者异常场景才执行。

如果把所有代码都编译，会带来：

- 启动变慢。
- CPU 消耗增加。
- Code Cache 占用增加。
- 编译收益很低。

所以 JVM 选择“先解释，后编译热点”。

## 热点代码会被怎样优化？

热点代码可能被 JIT 优化：

- 方法内联。
- 消除不必要锁。
- 消除不必要对象分配。
- 优化循环。
- 优化虚方法调用。
- 基于运行时类型做推测优化。

这些优化依赖运行时信息，所以 Java 的 JIT 有时能做出非常激进的优化。

## 推测优化和去优化

JIT 会根据当前运行情况做推测。

例如某个接口调用点一直只有一个实现类，JIT 可能把它优化成直接调用。

但如果运行后出现新的实现类，原来的假设不成立，JVM 可能触发 deoptimization，回退到解释执行或重新编译。

这说明 Java 性能是动态变化的。

## 为什么压测需要预热？

刚启动时，很多核心代码还在解释执行。

一段时间后，热点代码被识别并编译，性能才进入稳定状态。

压测如果不预热，可能把冷启动成本算进稳定吞吐，结果不准确。

更合理的压测阶段：

- 冷启动观察。
- 预热。
- 稳定压测。
- 阶梯加压。
- 长稳压测。

## 在 eMall 项目中怎么讲？

订单创建接口中的校验、价格计算、优惠匹配、库存预占会被高频调用。

服务刚启动时这些代码可能还在解释执行。大促前如果新扩容实例没有预热，第一批用户请求可能更慢。

所以 eMall 这类系统需要预热策略，让热点路径先被加载、执行和 JIT 优化。

## 深度增强：JVM 生产运行图

![Java 17 容器内 JVM 内存结构](../assets/jvm-runtime-memory.svg)

JVM 题要从运行时资源解释到业务影响。堆、直接内存、元空间、线程栈和容器 memory limit 共同决定服务稳定性；
GC、CPU throttling、线程池队列和下游超时会一起影响 P99，而不是孤立存在。

## 深度增强：Java 17 诊断模型示例

```java
record RuntimeSignal(
        double heapUsage,
        double containerMemoryUsage,
        long gcPauseMillis,
        int threadCount,
        int queuedTasks) {

    boolean requiresTriage() {
        return heapUsage > 0.85
                || containerMemoryUsage > 0.90
                || gcPauseMillis > 500
                || threadCount > 800
                || queuedTasks > 1_000;
    }
}
```

这个模型强调线上诊断要看组合信号。只看 heap 不够，只看 GC 也不够；
要把 JVM、容器、线程池和业务延迟放到同一条时间线。

## 深度增强：生产边界

JVM 调优不能靠背参数。要先明确服务目标：低延迟、吞吐、容器资源、对象分配速率和 P99 SLO。
然后通过 GC 日志、JFR、指标和压测验证。错误地调大 `-Xmx` 可能挤压堆外内存，导致容器 OOMKilled。

## 深度增强：面试高分表达

我会用证据链回答 JVM 问题：先看业务影响，再看 JVM 指标、GC 日志、线程栈、heap dump、容器事件和最近变更。
结论要能解释现象，并能给出降级、扩容、参数调整或代码优化方案。

## 专家级完整回答

```text
解释执行是 JVM 逐条解释字节码，优势是启动快；热点代码是运行时被频繁执行的方法或循环，
JVM 会根据计数和 profiling 数据识别它们，并通过 JIT 编译成本地机器码。热点代码经过方法内联、
循环优化、逃逸分析和去虚拟化后，长期执行性能更好。

这也是为什么 Java 服务压测要预热。冷启动阶段很多代码还没有 JIT 编译，不能代表稳定态性能。
```

## 回答评分点

高分答案应该覆盖：

- 解释执行启动快。
- 热点代码会被 JIT 编译。
- JVM 不会编译所有代码。
- 知道推测优化和去优化。
- 能联系服务预热和压测。
