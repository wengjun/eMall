# 401 商品详情缓存和搜索索引是什么关系？

[返回按分类学习面试题](../README.md)

## 题目

商品详情缓存和搜索索引是什么关系？

## 先给面试官的短答案

商品详情缓存和搜索索引都是读模型，但服务不同场景。商品详情缓存服务单商品详情页的快速读取，
搜索索引服务关键词搜索、过滤、排序和召回。

它们都不是商品事实来源，最终事实来源仍是商品、价格、库存等主数据服务。

## 商品详情缓存

特点：

- 面向单商品读取。
- 通常按 `skuId` 或 `productId` 查询。
- 可以聚合商品基础信息。
- 适合 Redis 和本地缓存。
- 关注低延迟和热点保护。

详情缓存更接近页面数据。

## 搜索索引

特点：

- 面向多商品召回。
- 支持分词和倒排索引。
- 支持过滤、排序和聚合。
- 适合 OpenSearch 或 Elasticsearch。
- 关注相关性和查询能力。

搜索索引更接近导购数据。

## 一致性关系

二者可能都来自商品变更事件，但更新链路不同。

商品变更后：

- 删除或刷新商品详情缓存。
- 更新搜索索引文档。
- 详情页和下单仍回源校验关键状态。

二者短暂不一致是可能的。

## 在 eMall 项目中怎么讲？

eMall 用户搜索商品时先查搜索索引，点击后进入商品详情页读取商品详情缓存。

如果搜索索引还展示旧价格，详情页和下单链路仍要通过 `pricing` 和 `inventory` 服务校验，避免
搜索读模型滞后造成交易错误。

## 深度增强：缓存和消息治理图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

缓存和消息题要关注一致性、削峰、延迟、积压和恢复。
Redis 很快，但会遇到穿透、击穿、雪崩、热点 key 和内存淘汰；
MQ 能解耦和削峰，但会带来重复消费、乱序、积压和死信处理。

## 深度增强：Java 17 幂等消费示例

```java
import java.util.Set;
import java.util.concurrent.ConcurrentHashMap;

final class LocalIdempotentConsumer {
    private final Set<String> processedKeys = ConcurrentHashMap.newKeySet();

    boolean tryHandle(String messageKey, Runnable handler) {
        if (!processedKeys.add(messageKey)) {
            return false;
        }
        handler.run();
        return true;
    }
}
```

这个示例只适合解释幂等思想。生产环境不能用本地内存做全局幂等，要使用数据库唯一键、Redis 原子操作或业务状态机。

## 深度增强：生产边界

缓存要有 TTL、容量、降级和回源保护；消息要有重试、死信、延迟队列、消费幂等和积压告警。
缓存不一致要能修复，消息失败要能回放，不能只依赖人工查日志。

## 深度增强：面试高分表达

我会把缓存和消息都看成性能与稳定性工具，而不是正确性事实来源。
正确性由数据库事实、状态机、幂等和对账保证；缓存和 MQ 负责降低延迟、削峰填谷和解耦系统。

## 专家级完整回答

```text
商品详情缓存和搜索索引都是为读优化的模型。详情缓存面向单商品详情页低延迟读取，搜索索引面向
关键词召回、过滤、排序和聚合。

它们可以都由商品变更事件驱动更新，但不是事实来源。商品主库、价格和库存服务才是最终校验依据。
设计上要允许短暂不一致，并通过事件、重试、对账和回源校验控制影响。
```

## 回答评分点

高分答案应该覆盖：

- 二者都是读模型。
- 详情缓存服务单商品读取。
- 搜索索引服务搜索召回和排序。
- 最终事实来源不是它们。
- 下单要回源校验价格库存。
