# 249 如何估算线程池、连接池和数据库连接数？

[返回按分类学习面试题](../README.md)

## 题目

如何估算线程池、连接池和数据库连接数？

## 先给面试官的短答案

线程池、连接池和数据库连接数要按 QPS、响应时间、阻塞比例、下游容量和实例数量估算。
不能只把池子调大，因为池子过大会增加排队、上下文切换和数据库压力。

池大小要让调用方受控排队，而不是把压力无限传给下游。

## 线程池估算

可用思路：

- CPU 密集型接近 CPU 核数。
- IO 密集型可大于 CPU 核数。
- 根据在途请求数估算线程需求。
- 设置有界队列。
- 设置拒绝策略。

线程池要按业务隔离，核心和非核心不要共用。

## 连接池估算

连接池要看：

- 下游最大承载。
- 调用超时。
- 单实例 QPS。
- 平均和 P99 延迟。
- 实例数量。

所有应用实例的总连接数不能超过数据库或下游可承受范围。

## 数据库连接数

数据库连接不是越多越好。

连接过多会导致：

- 数据库上下文切换增加。
- 锁竞争加剧。
- 慢查询堆积。
- 故障时雪崩。

应按数据库能力分配到各服务，并给核心服务优先级。

## 在 eMall 项目中怎么讲？

订单服务如果有 100 个实例，每个实例数据库池配置 100，总连接就是 10000。

如果订单库只能稳定承受 3000 连接，这种配置会在高峰时直接拖垮数据库。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
池大小要基于吞吐、响应时间和下游容量估算。线程池看任务类型和在途请求，连接池看单实例流量、下游延迟和总实例数。
所有实例连接数之和不能超过数据库承载能力。

我会使用有界队列、超时、拒绝策略和核心非核心隔离，让压力在调用方被控制，而不是无限堆到数据库。
```

## 回答评分点

高分答案应该覆盖：

- 池大小不是越大越好。
- 要考虑实例总连接数。
- 数据库连接数受数据库能力限制。
- 线程池要有界队列和拒绝策略。
- 核心和非核心线程池要隔离。
