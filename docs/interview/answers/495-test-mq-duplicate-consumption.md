# 495 如何测试 MQ 重复消费？

[返回按分类学习面试题](../README.md)

## 题目

如何测试 MQ 重复消费？

## 先给面试官的短答案

测试 MQ 重复消费要验证同一消息被消费多次时，业务副作用只发生一次。要覆盖相同消息 ID 重投、消费者处理成功但提交 offset
失败、消费者重启、并发消费和乱序消息。断言重点是业务状态、幂等记录、消息处理记录和下游副作用。

## 为什么会重复消费

消息系统通常提供至少一次投递。消费者处理完业务后，可能在提交 offset 或 ack 前宕机。消息代理不知道业务是否成功，
就会重新投递。同一消息也可能因为重平衡、网络抖动或人工重放再次被消费。

所以消费者必须按业务唯一键或消息 ID 做幂等。

## 测试场景

相同消息重复投递：同一消息发送两次，消费者只能更新业务一次。

处理后 ack 失败：模拟业务已成功但 offset 未提交，重新消费时不能重复执行。

并发重复：两个消费者或两个线程同时处理相同业务事件，只允许一个成功。

乱序消息：先收到后续状态，再收到旧状态时，旧消息不能覆盖新状态。

失败重试：业务失败时不能错误写入幂等成功记录，否则后续重试会被跳过。

## 断言方式

不要只断言消费者方法被调用。要检查业务表、幂等表、处理日志和下游事件。例如订单已支付消息重复消费后，
履约单只能创建一条，积分只能发一次，处理记录只能有一个成功状态。

## 在 eMall 项目中怎么讲？

eMall 的履约、积分、优惠券、通知和数据同步都可能消费订单事件。消费者必须基于 eventId、orderId 和业务动作建立幂等键，
并通过集成测试验证重复投递不会重复履约。

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;

## 专家级完整回答

```text
MQ 重复消费测试的核心是证明消费者是幂等的。

因为大多数消息系统提供至少一次投递，消费者处理成功后仍可能因为 ack 或 offset 提交失败而再次收到消息。
所以测试要模拟重复投递、消费者重启、并发消费和乱序消息。

断言不能停留在方法调用次数，而要看业务副作用。例如履约单、积分流水、优惠券发放和处理记录是否只产生一次。
```

## 回答评分点

高分答案应该覆盖：

- 说明至少一次投递导致重复消费。
- 覆盖重复投递、ack 失败、重启、并发和乱序。
- 知道消费者要基于业务键或消息 ID 幂等。
- 能强调断言业务副作用，而不是方法调用。
- 能结合履约、积分、优惠券和通知举例。
