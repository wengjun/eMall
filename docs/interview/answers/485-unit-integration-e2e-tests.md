# 485 单元测试、集成测试、端到端测试有什么区别？

[返回按分类学习面试题](../README.md)

## 题目

单元测试、集成测试、端到端测试有什么区别？

## 先给面试官的短答案

单元测试验证一个类或一个函数的逻辑，运行快、定位准；集成测试验证多个组件和真实依赖之间是否协作正确；
端到端测试从用户或外部系统视角验证完整业务链路。三者不是互相替代，而是用不同成本覆盖不同风险。

## 单元测试

单元测试关注小范围逻辑，例如金额计算、状态机流转、幂等键生成、规则匹配和参数校验。它通常不依赖真实数据库、
消息队列或外部系统，失败时能快速定位到某个类或方法。

单元测试适合覆盖边界条件和复杂分支。它不适合证明数据库事务、消息投递或服务间调用真实可用。

## 集成测试

集成测试关注组件协作，例如 MyBatis 映射、事务提交、唯一索引幂等、Redis Lua、Kafka 消费、Outbox 重试和
HTTP 客户端超时。它可以使用 Testcontainers 启动真实 MySQL、Redis 或 Kafka。

集成测试成本高于单元测试，但能发现配置、序列化、事务、SQL 和中间件行为问题。

## 端到端测试

端到端测试从入口开始走完整链路。例如用户登录、加购、下单、库存预占、支付回调、订单状态更新和消息发送。
它最接近真实用户体验，但运行慢、定位难、维护成本高。

端到端测试应该覆盖少量核心链路，而不是替代所有单元和集成测试。

## 在 eMall 项目中怎么讲？

eMall 的金额计算和状态机适合单元测试；库存预占事务、支付回调幂等和 Outbox 适合集成测试；
完整下单支付链路适合 Smoke 或端到端测试。测试金字塔应该底层多、上层少。

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
我会按风险和成本划分测试层次。

单元测试验证局部逻辑，速度快、定位准，适合覆盖复杂分支和边界条件。
集成测试验证代码和真实依赖协作，适合发现数据库事务、消息、缓存和 HTTP 配置问题。
端到端测试从用户视角验证完整链路，价值高但成本也高，所以只覆盖核心路径。

高质量工程不是只追求某一种测试，而是让每类缺陷在最便宜的层次被发现。
```

## 回答评分点

高分答案应该覆盖：

- 能区分测试范围、速度、成本和定位能力。
- 知道单元测试适合纯逻辑。
- 知道集成测试适合真实依赖协作。
- 知道端到端测试适合核心业务链路。
- 能结合测试金字塔说明分层价值。

## 深度完善：面向 L6 的回答框架

围绕「单元测试、集成测试、端到端测试有什么区别？」，高分答案不能停在概念定义，而要把「单元测试、集成测试、契约测试、压测、CI、代码评审、覆盖率和依赖治理」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「测试、工程治理和质量」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `common、smoke、chaos、loadtest、各服务的单元和集成测试` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：测试通过率、覆盖关键分支数、 flaky 率、构建耗时、漏洞数量。
- 验证证据：CI 记录、测试报告、覆盖率报告、依赖扫描、契约测试和评审清单。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
