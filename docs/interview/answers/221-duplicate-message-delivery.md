# 221 消息重复投递怎么办？

[返回按分类学习面试题](../README.md)

## 题目

消息重复投递怎么办？

## 先给面试官的短答案

消息系统通常只能保证至少一次投递，所以重复消息是正常现象。消费者必须做幂等，不能假设一条消息只处理一次。
常用方式是业务唯一键、消息去重表、状态机校验、数据库唯一约束和幂等结果记录。

消息重复不是 MQ 的异常，而是消费者设计必须承担的责任。

## 为什么会重复？

原因包括：

- 生产者重试发送。
- Broker 重试投递。
- 消费者处理成功但 ack 失败。
- 消费者超时后被重新投递。
- 投递器发送成功但标记失败。

这些场景在分布式系统中都可能发生。

## 幂等方式

常用方式：

- 使用消息 ID 去重。
- 使用业务 ID 去重。
- 数据库唯一索引。
- 状态机防止重复跳转。
- 处理结果缓存。
- 去重表设置过期时间。

业务 ID 通常比纯消息 ID 更可靠，因为同一业务事件可能被重新生成。

## 注意事项

要注意：

- 去重记录要和业务处理在同一事务。
- 去重表要定期归档。
- 不同事件类型不能共用裸 ID。
- 幂等不是简单忽略所有重复请求。

如果重复消息携带更新后的版本，也要按版本判断是否处理。

## 在 eMall 项目中怎么讲？

库存服务消费 `OrderPaid` 事件确认扣减库存时，应以 `orderId + eventType` 做幂等键。

如果同一订单支付成功事件重复到达，库存服务只能确认扣减一次。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
消息重复投递是至少一次语义下的正常现象。消费者必须幂等，可以用业务唯一键、消息去重表、唯一索引和状态机校验。
去重记录最好和业务处理在同一个本地事务中提交，避免业务成功但去重失败。

在电商里，订单支付、库存扣减、优惠券核销都必须按业务 ID 做幂等，不能依赖 MQ 只投递一次。
```

## 回答评分点

高分答案应该覆盖：

- 重复消息是正常现象。
- 消费者必须幂等。
- 业务唯一键通常优于纯消息 ID。
- 去重和业务处理要保持一致。
- 去重记录要归档或设置有效期。
