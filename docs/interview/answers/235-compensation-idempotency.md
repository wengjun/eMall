# 235 补偿任务如何避免重复执行？

[返回按分类学习面试题](../README.md)

## 题目

补偿任务如何避免重复执行？

## 先给面试官的短答案

补偿任务要通过业务幂等键、补偿记录表、状态机、唯一索引和执行结果记录避免重复副作用。
任务可以重复触发，但同一业务对象的同一补偿动作只能生效一次。

补偿系统的原则是“允许重复调度，不允许重复产生业务影响”。

## 幂等键

幂等键通常由以下字段组成：

- 业务 ID。
- 补偿类型。
- 补偿目标状态。
- 版本号。

例如 `orderId + RELEASE_STOCK + reserveVersion`。

## 补偿记录

记录内容包括：

- 补偿 ID。
- 业务 ID。
- 补偿类型。
- 当前状态。
- 执行次数。
- 最后错误。
- 执行结果。
- 创建和更新时间。

补偿记录和业务变更最好在同一事务内更新。

## 状态机保护

状态机要拒绝非法重复操作。

例如库存预占已经释放，再次释放应返回成功但不增加库存。

这种返回成功的幂等处理比抛异常更适合补偿重试。

## 在 eMall 项目中怎么讲？

订单取消后的库存释放补偿，可以用 `orderId + skuId + reservationId` 做幂等键。

库存服务发现预占已释放时，直接返回成功，不再修改可售库存。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
补偿任务避免重复执行的关键是业务幂等。任务可以被重复调度，但业务效果只能发生一次。
我会使用补偿记录表、业务唯一键、数据库唯一索引和状态机校验，记录每次补偿的状态和结果。

例如释放库存时，如果预占记录已释放，再次执行应幂等返回成功，不能重复增加库存。
```

## 回答评分点

高分答案应该覆盖：

- 补偿任务可能重复调度。
- 业务效果必须幂等。
- 幂等键要包含业务 ID 和补偿类型。
- 补偿记录表记录状态和结果。
- 状态机防止重复副作用。
