# 304 悲观锁和乐观锁怎么选？

[返回按分类学习面试题](../README.md)

## 题目

悲观锁和乐观锁怎么选？

## 先给面试官的短答案

悲观锁适合冲突概率高、必须串行处理、失败重试成本高的场景；乐观锁适合冲突概率低、读多写少、允许失败后重试的场景。
悲观锁用锁阻塞竞争者，乐观锁用版本号或条件更新检测冲突。

选择关键是冲突概率、锁持有时间、吞吐要求和重试成本。

## 悲观锁

常见方式：

```sql
SELECT * FROM inventory
WHERE sku_id = ?
FOR UPDATE;
```

特点：

- 先锁住资源。
- 冲突事务等待。
- 一致性强。
- 并发性能可能下降。

适合关键资源强竞争。

## 乐观锁

常见方式：

```sql
UPDATE inventory
SET available = available - 1, version = version + 1
WHERE sku_id = ?
  AND version = ?
  AND available > 0;
```

特点：

- 不先阻塞。
- 更新时检测版本。
- 冲突时失败重试。
- 适合冲突少场景。

## 如何选择

看：

- 冲突是否频繁。
- 重试是否便宜。
- 是否能接受失败。
- 事务是否很短。
- 是否热点资源。

热点秒杀库存不一定适合简单乐观锁反复重试，因为重试会放大数据库压力。

## 在 eMall 项目中怎么讲？

普通订单状态更新可以使用乐观锁版本号。

热点库存扣减可使用条件更新、库存桶和令牌前置拦截，避免大量乐观锁失败重试打爆数据库。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
悲观锁适合冲突高、必须串行和失败成本高的场景；乐观锁适合冲突低、读多写少和可重试场景。
悲观锁通过加锁等待保证互斥，乐观锁通过版本号或条件更新在提交时检测冲突。

在电商中，订单状态更新常用乐观锁；热点库存要结合条件更新、限流和库存桶，而不是简单无限重试。
```

## 回答评分点

高分答案应该覆盖：

- 悲观锁先锁资源。
- 乐观锁提交时检测冲突。
- 冲突高更偏悲观或前置削峰。
- 冲突低适合乐观锁。
- 选择要考虑重试成本。
