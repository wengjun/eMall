# 224 重试多次失败如何进入死信？

[返回按分类学习面试题](../README.md)

## 题目

重试多次失败如何进入死信？

## 先给面试官的短答案

消息超过最大重试次数仍失败时，应进入死信队列或失败表，并记录消息内容、失败原因、重试次数、消费者、时间和 traceId。
死信不是垃圾桶，而是需要告警、分析、修复和可控重放的异常处理通道。

没有死信治理，失败消息会在系统里无限循环或静默丢失。

## 进入死信的条件

常见条件：

- 超过最大重试次数。
- 发现不可重试错误。
- 消息格式不兼容。
- 业务状态长期不满足。
- 消费者主动拒绝处理。

条件要明确，避免所有异常都无限重试。

## 死信记录内容

应记录：

- 原始消息。
- topic 和 partition。
- 消息 key。
- 消息 ID。
- 失败原因。
- 异常栈摘要。
- 重试次数。
- 首次失败时间。
- 最后失败时间。
- traceId。

这些信息决定后续能否恢复。

## 告警和分级

不是所有死信都同等紧急。

支付、库存、订单状态类死信要高优先级告警。日志分析或推荐类死信可以较低优先级处理。

死信告警也要服务分级。

## 在 eMall 项目中怎么讲？

库存扣减消息重试多次失败后进入死信，必须告警并展示在运维平台。

运维人员修复库存数据或代码问题后，可以按订单维度重放死信消息。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
消息多次重试失败后应进入死信队列或失败表，并记录原始消息、失败原因、重试次数、traceId 和消费上下文。
死信要告警、分级、可查询、可修复和可重放。

死信不是丢弃消息，而是把自动处理不了的异常交给受控恢复流程。核心交易类死信要高优先级处理。
```

## 回答评分点

高分答案应该覆盖：

- 死信有明确进入条件。
- 死信要保留完整上下文。
- 死信要告警和分级。
- 死信需要可查询和可重放。
- 核心业务死信不能静默堆积。
