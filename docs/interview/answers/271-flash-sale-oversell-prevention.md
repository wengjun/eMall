# 271 秒杀如何防止超卖？

[返回按分类学习面试题](../README.md)

## 题目

秒杀如何防止超卖？

## 先给面试官的短答案

秒杀防超卖要在多层控制库存：入口令牌控制进入量，缓存或队列削峰，数据库条件扣减兜底。
最终扣减必须使用原子条件更新、库存预占记录、幂等键和对账，保证成功订单数不超过可售库存。

前置拦截提升性能，数据库条件更新保证正确性底线。

## 多层防线

防线包括：

- 活动库存预热。
- 秒杀令牌限量发放。
- 用户购买资格校验。
- Redis 原子扣减。
- MQ 排队削峰。
- 数据库条件更新。
- 订单幂等。
- 售罄标记。

不能只依赖单层缓存扣减。

## 数据库兜底

典型 SQL：

```sql
UPDATE sku_stock
SET available = available - 1
WHERE sku_id = ?
  AND available > 0;
```

只有更新成功才说明扣减成功。

并发下不能先查库存再扣减，因为查询和扣减之间会有竞争窗口。

## 幂等和回滚

还需要：

- 同一用户同一活动只能成功一次。
- 同一订单只能扣减一次。
- 订单取消要释放预占。
- 支付超时要释放库存。
- 对账发现差异要修复。

防超卖不只是扣减瞬间，还包括订单生命周期。

## 在 eMall 项目中怎么讲？

秒杀服务先按 SKU 发放有限令牌，拿到令牌的请求进入订单队列。

库存服务最终用条件更新或库存预占确认扣减，即使令牌或缓存异常，也不能突破数据库正确性边界。

## 深度增强：数据访问和扩展图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
秒杀防超卖要用多层防线。入口令牌和限流减少进入量，Redis 原子扣减和队列削峰提升吞吐，
数据库条件更新作为最终正确性兜底。

同时要有订单幂等、用户购买限制、库存预占、取消释放、支付超时释放和对账，保证整个生命周期不超卖。
```

## 回答评分点

高分答案应该覆盖：

- 防超卖需要多层控制。
- 令牌和缓存是前置拦截。
- 数据库条件更新是正确性兜底。
- 不能先查再扣。
- 订单取消和支付超时也要释放库存。
