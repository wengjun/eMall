# 513 如何保证文档和代码一致？

[返回按分类学习面试题](../README.md)

## 题目

如何保证文档和代码一致？

## 先给面试官的短答案

保证文档和代码一致要把文档纳入开发流程和 CI，而不是靠事后补。接口文档、架构决策、配置说明、Runbook 和部署文档
应随代码一起评审、测试和发布。能自动生成的尽量自动生成，不能自动生成的要有文档 owner 和变更门禁。

## 常见问题

文档失效通常来自三个原因：代码变了文档没改，文档没有 owner，文档没有被使用和验证。
如果文档只在项目初期写一次，后续一定会和代码脱节。

生产系统中，过期文档会直接影响排障、发布和新人理解。

## 具体做法

第一，文档和代码同仓库管理，同一个 PR 修改代码和相关文档。

第二，接口文档尽量从 OpenAPI、注解或测试样例生成，减少手写不一致。

第三，CI 检查 Markdown 链接、格式、示例命令和配置文件是否存在。

第四，重要设计使用 ADR 记录背景、决策和取舍，避免只留下结论。

第五，Runbook 要在演练和事故中验证，发现不准确就立即更新。

## 在 eMall 项目中怎么讲？

eMall 已经把学习文档、架构文档、路线图、测试标记和模块说明放在 `docs` 中。后续可以让接口示例、配置说明、
Smoke 命令和故障 Runbook 进入 CI 校验，确保文档不是摆设。

## 深度增强：现场编码工程化图

![现场编码题的工程化解法](../assets/coding-patterns.svg)

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
文档一致性不是靠提醒，而是靠流程和自动化。

我会让文档和代码同仓库、同评审、同发布。接口文档尽量自动生成，架构决策用 ADR 记录，
Runbook 通过演练和事故复盘验证。

CI 可以检查链接、格式、示例文件和命令。更重要的是团队约定：如果一个变更影响接口、配置、部署或排障方式，
PR 就必须同时更新文档，否则不能合并。
```

## 回答评分点

高分答案应该覆盖：

- 说明文档要进入开发流程和 CI。
- 覆盖同仓库、同评审、自动生成和 owner。
- 知道接口文档、ADR、Runbook 的不同作用。
- 能指出过期文档会影响排障和发布。
- 能结合 eMall 的 docs 目录说明。
