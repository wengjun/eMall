# 432 为什么三者都需要？

[返回按分类学习面试题](../README.md)

## 题目

为什么三者都需要？

## 先给面试官的短答案

只看指标不知道具体请求发生了什么，只看日志很难快速发现整体趋势，只看 Trace 不一定知道业务影响
有多大。日志、指标和 Trace 分别覆盖宏观监控、链路定位和细节确认。

生产系统需要三者组合才能做到快速发现、快速定位和快速修复。

## 只有指标的问题

指标能看到：

- 错误率升高。
- P99 升高。
- QPS 突增。
- CPU 升高。

但指标通常无法告诉你某个订单为什么失败。

## 只有日志的问题

日志能看到细节，但问题是：

- 数据量大。
- 聚合成本高。
- 很难直接做实时告警。
- 跨服务串联困难。

日志适合确认原因，不适合单独承担监控体系。

## 只有 Trace 的问题

Trace 能看链路，但问题是：

- 采样可能漏掉请求。
- 不一定有完整业务上下文。
- 不适合作为长期趋势统计。
- 不能替代业务指标。

Trace 适合定位链路，不适合衡量全局健康度。

## 在 eMall 项目中怎么讲？

eMall 支付成功率下降时，指标能告诉我们影响范围，Trace 能定位支付回调链路是否变慢，日志能确认
是签名失败、状态机冲突还是第三方返回异常。

三者缺一项，排障都会变慢。

## 深度增强：Kubernetes 运维治理图

![Kubernetes 生产运行和故障治理](../assets/kubernetes-operations.svg)

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
三者都需要是因为它们解决的问题不同。指标负责发现异常和衡量影响范围，Trace 负责跨服务链路定位，
日志负责还原业务细节和异常原因。

一个生产级系统不能只靠日志排障，也不能只靠指标告警。最有效的方式是用统一 trace ID 把三者关联，
形成从告警到定位再到根因确认的闭环。
```

## 回答评分点

高分答案应该覆盖：

- 指标看整体。
- Trace 看链路。
- 日志看细节。
- 单独使用都有盲区。
- 三者通过 trace ID 关联。
