# 491 如何测试幂等？

[返回按分类学习面试题](../README.md)

## 题目

如何测试幂等？

## 先给面试官的短答案

测试幂等要验证同一个业务请求重复执行时，最终结果只生效一次，并且重复请求返回可预期结果。要覆盖串行重复、
并发重复、异常后重试、消息重复、回调重复和幂等记录过期等场景。

## 幂等测试目标

幂等不是简单“不报错”，而是要保证副作用只发生一次。例如支付回调重复 10 次，订单只能从待支付变为已支付一次，
资金流水只能生成一条，库存不能重复扣减，Outbox 事件不能重复产生业务影响。

测试要同时断言返回值、数据库状态、消息数量和业务副作用。

## 关键测试场景

串行重复：同一个请求连续调用多次，第一次成功，后续返回成功或已处理，但状态不重复变化。

并发重复：多个线程同时提交相同幂等键，只允许一个线程获得执行权，其余线程读取已有结果或被安全拒绝。

失败后重试：第一次执行到一半失败，第二次重试能继续完成或安全回滚，不产生半成品。

重复消息：同一消息被消费多次，消费者根据业务唯一键或消息 ID 去重。

幂等过期：幂等记录过期策略不能让仍可能重试的请求重复生效。

## 实现验证点

如果幂等依赖数据库唯一索引，要在集成测试中真实验证唯一约束冲突。如果依赖 Redis `SET NX`，
要验证锁过期、执行超时和释放安全。如果依赖状态机，要验证非法重复状态不会再次执行业务动作。

## 在 eMall 项目中怎么讲？

eMall 可以测试支付回调、订单提交、库存预占、退款申请和 Outbox 消费的幂等。比如支付回调重复到达时，
订单状态只能更新一次，支付流水唯一，后续回调返回已处理。

## 深度增强：现场编码工程化图

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
幂等测试的核心是验证重复触发不会重复产生业务副作用。

我会覆盖串行重复、并发重复、失败后重试、消息重复和幂等记录过期。断言时不只看接口返回，
还要检查数据库状态、流水数量、库存数量和消息副作用。

对关键幂等逻辑，我倾向用集成测试验证真实唯一索引、事务和锁行为。
因为幂等问题经常出在并发和基础设施语义上，单纯 mock 很难证明正确。
```

## 回答评分点

高分答案应该覆盖：

- 说明幂等测试验证副作用只发生一次。
- 覆盖串行、并发、失败重试、重复消息和过期场景。
- 知道要断言数据库、消息和业务状态。
- 能说明唯一索引、Redis 锁和状态机的测试重点。
- 能结合支付回调、库存、订单举例。
