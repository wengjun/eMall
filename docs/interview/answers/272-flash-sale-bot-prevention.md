# 272 秒杀如何防止黄牛？

[返回按分类学习面试题](../README.md)

## 题目

秒杀如何防止黄牛？

## 先给面试官的短答案

秒杀防黄牛要从账号、设备、行为、频率、资格、风控和履约规则多层治理。
常见手段包括登录态校验、验证码、设备指纹、限购、黑名单、行为风控、令牌签名、接口混淆和异常订单审核。

防黄牛不是靠一个验证码解决，而是风控体系。

## 风控维度

维度包括：

- 用户账号。
- 手机号。
- 设备指纹。
- IP 和代理。
- 收货地址。
- 支付账户。
- 行为轨迹。
- 请求频率。
- 历史风险。

单一维度容易被绕过。

## 技术手段

手段包括：

- 登录和实名认证。
- 动态验证码。
- 活动资格预约。
- 一人一单限购。
- 令牌防伪和过期。
- 请求签名。
- 频率限制。
- 黑灰名单。
- 机器行为识别。

高风险用户可以进入排队或人工审核。

## 业务手段

业务上可以：

- 限制同地址购买数量。
- 限制同支付账户。
- 延迟发货审核。
- 取消异常订单。
- 预约抽签。
- 分批放货。

技术和业务规则要结合。

## 在 eMall 项目中怎么讲？

秒杀请求要校验登录用户、设备、活动资格和令牌签名。

同一用户、设备、地址和支付账户的异常频率要进入风控，命中高风险时拒绝或进入审核。

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
秒杀防黄牛是多层风控问题。要从账号、设备、IP、行为、地址、支付账户和历史风险识别异常，
并结合限购、预约、验证码、令牌签名、频率限制、黑名单和异常订单审核。

不能只依赖验证码或限流。黄牛治理要和业务履约规则结合，例如同地址限购、抽签和延迟发货审核。
```

## 回答评分点

高分答案应该覆盖：

- 防黄牛是多维风控。
- 账号、设备、IP、地址和支付账户都要考虑。
- 令牌和限购是基础。
- 验证码不是唯一手段。
- 业务规则和技术风控要结合。
