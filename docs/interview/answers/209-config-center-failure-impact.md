# 209 配置中心故障会带来什么影响？

[返回按分类学习面试题](../README.md)

## 题目

配置中心故障会带来什么影响？

## 先给面试官的短答案

配置中心故障可能导致服务启动失败、配置无法刷新、灰度和降级开关失效、错误配置无法回滚。
如果服务运行时强依赖配置中心，还可能引发大面积不可用。

生产系统应使用本地快照、默认值、缓存、超时和降级，避免配置中心成为强依赖单点。

## 故障影响

影响包括：

- 新实例无法启动。
- 配置变更无法发布。
- 降级开关无法生效。
- 灰度规则无法调整。
- 密钥轮换受阻。
- 服务读取配置超时。
- 配置不一致导致行为分裂。

最危险的是启动和请求路径都强依赖配置中心。

## 防护设计

防护包括：

- 本地配置快照。
- 安全默认值。
- 配置读取超时。
- 配置缓存。
- 配置中心多副本。
- 变更失败告警。
- 客户端容错。
- 启动时允许使用最近一次有效配置。

服务不能因为配置中心短时不可用就停止处理所有请求。

## 一致性问题

配置中心恢复后，还要处理：

- 不同实例配置版本不一致。
- 部分实例刷新失败。
- 回滚配置未完全传播。
- 旧配置和新代码不兼容。

因此配置版本和实例配置状态要可观测。

## 在 eMall 项目中怎么讲？

如果限流阈值和降级开关都依赖配置中心，配置中心故障时不能让下单服务无法启动。

下单服务应使用本地最后有效配置，并告警提示配置刷新失败。

## 深度增强：可观测与配置治理图

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
配置中心故障会影响服务启动、配置刷新、灰度、降级和回滚。如果运行时强依赖配置中心，可能扩大成全站故障。
我会让客户端缓存最近一次有效配置，提供安全默认值和本地快照，并设置读取超时和告警。

配置中心恢复后，还要观察实例配置版本，防止部分实例配置不一致。
```

## 回答评分点

高分答案应该覆盖：

- 配置中心故障会影响启动和动态变更。
- 服务不能强依赖实时读取配置中心。
- 本地快照和默认值很重要。
- 配置版本和传播状态要可观测。
- 配置中心本身也要高可用。
