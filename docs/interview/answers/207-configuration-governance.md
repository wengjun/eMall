# 207 如何做配置治理？

[返回按分类学习面试题](../README.md)

## 题目

如何做配置治理？

## 先给面试官的短答案

配置治理要解决配置是谁改的、为什么改、改了什么、影响哪些服务、如何灰度、如何回滚、如何审计。
生产配置不能像普通文本一样随意修改，必须有权限、审批、校验、版本、灰度、回滚和监控。

配置错误造成的事故不比代码错误少。

## 配置分类

可以分为：

- 环境配置。
- 业务开关。
- 限流阈值。
- 熔断降级策略。
- 路由和灰度规则。
- 密钥和证书。
- 第三方接入参数。

密钥不应和普通配置放在同一治理方式里。

## 治理能力

需要：

- 权限控制。
- 变更审批。
- 格式校验。
- 语义校验。
- 版本历史。
- 灰度发布。
- 快速回滚。
- 变更通知。
- 审计日志。
- 变更后监控。

关键配置还要有双人复核。

## 工程实践

实践包括：

- 配置命名规范。
- 默认值和边界值。
- 启动时校验关键配置。
- 配置变更事件记录。
- 配置与代码版本兼容检查。
- 配置中心故障时使用本地快照。

配置要作为生产变更管理。

## 在 eMall 项目中怎么讲？

秒杀限流阈值、支付渠道开关、库存降级策略都属于高风险配置。

这些配置变更要审批、灰度和监控，不能让任何人直接在生产配置中心随意改。

## 深度增强：可观测与配置治理图

![指标、日志、Trace 和告警平台](../assets/observability-platform.svg)

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
配置治理要覆盖权限、审批、校验、版本、灰度、回滚、通知、审计和监控。
关键配置变更和代码发布一样有风险，尤其是限流、熔断、路由、支付渠道和库存策略。

我会对配置分类管理，密钥进入专门密钥系统，普通配置进入配置中心，并要求关键配置有默认值、本地快照和变更后监控。
```

## 回答评分点

高分答案应该覆盖：

- 配置变更是生产变更。
- 权限、审批、版本、灰度和回滚不可缺少。
- 配置要做格式和语义校验。
- 密钥要专门治理。
- 配置中心故障要有本地快照。
