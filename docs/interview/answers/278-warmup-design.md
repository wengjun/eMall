# 278 如何做预热？

[返回按分类学习面试题](../README.md)

## 题目

如何做预热？

## 先给面试官的短答案

预热是在服务承接真实高流量前，提前加载配置、缓存、连接、规则、模型和热点数据，并用小流量让运行时状态稳定。
它用于降低冷启动延迟、缓存击穿和恢复后瞬时过载风险。

预热不是启动成功，而是具备承载流量的准备过程。

## 预热内容

包括：

- 配置和规则。
- 数据库连接池。
- HTTP 连接池。
- Redis 连接。
- 本地缓存。
- 热点商品数据。
- 价格和库存快照。
- JVM JIT 和类加载。
- 推荐模型或风控规则。

不同服务预热内容不同。

## 预热方式

方式包括：

- 启动后主动加载热点数据。
- 小流量灰度。
- 影子请求。
- 定时刷新缓存。
- 连接池最小连接初始化。
- 大促前预计算。

预热要可观测，不能只打印日志表示完成。

## 风险控制

预热时要注意：

- 不要同时让所有实例回源。
- 控制预热并发。
- 失败可降级启动。
- 热点数据要有版本。
- 预热不能阻塞太久。

预热本身也可能打垮数据库。

## 在 eMall 项目中怎么讲？

大促前，商品、价格、库存和秒杀令牌要提前加载到缓存。

新实例上线后先只接少量流量，等连接池、缓存命中率和 P99 延迟稳定后再扩大流量。

## 深度增强：数据访问和扩展图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
预热是在服务承接流量前准备运行时状态，包括配置、连接池、缓存、热点数据、规则、模型和 JVM 运行状态。
它可以降低冷启动延迟和缓存击穿风险。

我会用主动加载、小流量灰度、影子请求和大促前预计算实现预热，同时控制预热并发，避免预热流量打垮后端。
```

## 回答评分点

高分答案应该覆盖：

- 预热不是简单启动成功。
- 预热包括缓存、连接、规则和热点数据。
- 小流量灰度是常见方式。
- 预热过程要可观测。
- 预热本身要限速。
