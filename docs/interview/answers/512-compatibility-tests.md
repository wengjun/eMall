# 512 如何设计兼容性测试？

[返回按分类学习面试题](../README.md)

## 题目

如何设计兼容性测试？

## 先给面试官的短答案

兼容性测试要验证新旧版本、客户端、API、消息、数据库 schema 和配置能在灰度与回滚期间共存。核心是确认旧客户端能调用新服务，
新服务能读取旧数据，旧服务也能在新 schema 上运行，并且消息消费者能处理新增字段或未知枚举。

## 需要测试的兼容性

API 兼容性：新增字段不能破坏旧客户端；删除或改名字段要有过渡期；错误码和语义不能随意改变。

消息兼容性：事件新增字段要有默认值，消费者要能忽略未知字段，枚举扩展不能让旧消费者崩溃。

数据库兼容性：新旧应用要能在同一个 schema 上运行，字段删除和类型修改要遵循 expand-contract。

配置兼容性：旧版本服务读取新配置时不能启动失败，新版本缺少新配置时要有默认值。

客户端兼容性：App、小程序、商家系统和开放平台调用方可能不会同步升级。

## 测试方法

可以保留旧版本契约样本、旧消息样本和旧数据库快照。测试新版本处理旧数据，旧版本处理新 schema，
并模拟灰度期间新旧版本同时运行。

对开放平台 API，要使用契约测试和示例请求响应验证兼容性。

## 在 eMall 项目中怎么讲？

eMall 的开放平台、订单事件、支付状态和商品 API 都需要兼容性测试。比如订单事件新增字段时，
履约、积分、数据仓库等旧消费者不能因为未知字段失败。

## 深度增强：现场编码工程化图

![现场编码题的工程化解法](../assets/coding-patterns.svg)

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
兼容性测试的目标是证明系统可以安全灰度和回滚。

我会覆盖 API、消息、数据库、配置和客户端。重点验证旧客户端能调用新服务，新服务能读旧数据，
旧服务能在新 schema 上运行，旧消费者能处理新消息中的未知字段。

对电商系统，开放平台和异步事件尤其重要。外部客户和异步消费者不可能和服务端同时升级，
所以兼容性必须通过测试锁定。
```

## 回答评分点

高分答案应该覆盖：

- 说明兼容性测试服务灰度和回滚。
- 覆盖 API、消息、数据库、配置和客户端。
- 知道新旧版本共存是常态。
- 能提到契约测试、旧样本和旧快照。
- 能结合开放平台和订单事件举例。
