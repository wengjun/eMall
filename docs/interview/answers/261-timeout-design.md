# 261 如何设置超时？

[返回按分类学习面试题](../README.md)

## 题目

如何设置超时？

## 先给面试官的短答案

超时要按端到端 SLA 和下游延迟分位数设置，而不是随便写一个很大的值。
通常要分别设置连接超时、读取超时、请求总超时和业务 deadline，并保证下游超时小于上游剩余预算。

超时的目标是快速释放资源，避免慢请求拖垮线程池和连接池。

## 超时类型

常见类型：

- 连接超时。
- TLS 握手超时。
- 连接池获取超时。
- 读取超时。
- 写入超时。
- 整体请求超时。
- 业务 deadline。

只设置读取超时通常不够。

## 设置依据

依据包括：

- 用户接口 SLA。
- 下游 P95 和 P99 延迟。
- 网络环境。
- 重试次数。
- 调用链长度。
- 是否核心链路。

下游越不关键，超时越应保守，避免拖慢主流程。

## 常见错误

错误包括：

- 不设置超时。
- 超时设置过大。
- 所有下游使用同一个超时。
- 重试总耗时超过上游 SLA。
- 数据库慢查询没有超时控制。

没有超时的远程调用等同于无限等待风险。

## 在 eMall 项目中怎么讲？

下单接口总预算如果是 `500ms`，订单服务调用库存、价格、优惠时要按剩余预算分配。

推荐和广告这类非核心调用应使用更短超时，超过就降级。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
超时要基于端到端 SLA、下游延迟分位数和调用链预算设置。要区分连接超时、连接池获取超时、读取超时和总超时。
下游超时必须小于上游剩余时间，并与重试策略一起计算总耗时。

设置超时的本质是释放资源和控制尾延迟，防止慢下游拖垮调用方。
```

## 回答评分点

高分答案应该覆盖：

- 超时要基于 SLA 和延迟分位数。
- 区分连接、读取、总超时和 deadline。
- 下游超时要小于上游剩余预算。
- 重试也要计入总耗时。
- 不设置超时是严重风险。
