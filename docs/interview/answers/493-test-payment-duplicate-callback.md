# 493 如何测试支付重复回调？

[返回按分类学习面试题](../README.md)

## 题目

如何测试支付重复回调？

## 先给面试官的短答案

测试支付重复回调要验证同一支付单的相同回调重复到达时，订单状态、支付流水、资金状态和消息副作用只生效一次。
还要覆盖并发重复、乱序回调、签名错误、金额不一致和已关闭订单收到成功回调等异常场景。

## 基础场景

准备一笔待支付订单和支付单，模拟支付渠道连续发送多次成功回调。测试断言：

- 支付单只从待支付变成成功一次。
- 订单只从待支付变成已支付一次。
- 支付流水唯一。
- Outbox 事件或订单已支付消息只产生一次有效业务影响。
- 后续重复回调返回渠道期望的成功确认，避免渠道继续重试。

## 并发重复

支付渠道可能因为网络重试导致多条回调同时到达。测试要用多个线程同时提交相同回调，验证唯一索引、状态机或分布式锁
能保证只有一个线程执行业务更新。

只测串行重复不够，因为很多幂等漏洞只在并发下暴露。

## 异常场景

签名错误时不能更新支付状态。金额不一致时要拒绝并记录风险事件。订单已取消但收到支付成功回调时，
不能简单忽略，需要进入退款或人工处理流程。支付失败回调晚于成功回调时，不能把成功状态回退为失败。

## 在 eMall 项目中怎么讲？

eMall 的 `payment` 模块应测试支付回调验签、金额校验、状态机、幂等流水和 Outbox。对资金链路，测试断言不能只看接口返回，
必须检查订单、支付单、流水和事件表。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
支付重复回调测试的核心是验证资金状态只被确认一次。

我会覆盖串行重复和并发重复。断言包括支付单状态、订单状态、支付流水、Outbox 事件和返回给渠道的确认结果。
同时要覆盖签名错误、金额不一致、乱序回调、订单已取消后收到成功回调等异常路径。

支付系统不能只保证“不报错”，还要保证重复、乱序和异常回调不会导致重复入账、状态回退或消息重复生效。
```

## 回答评分点

高分答案应该覆盖：

- 说明重复回调要验证副作用只发生一次。
- 覆盖串行重复和并发重复。
- 能断言支付单、订单、流水和消息副作用。
- 知道签名、金额、乱序和已取消订单场景。
- 能体现资金链路不能只看接口返回。
