# 236 补偿任务如何避免多实例并发执行？

[返回按分类学习面试题](../README.md)

## 题目

补偿任务如何避免多实例并发执行？

## 先给面试官的短答案

多实例补偿要通过任务分片、数据库抢占、分布式锁、乐观锁、状态机和幂等控制并发。
关键不是只靠锁，而是即使锁失效，业务处理也不会重复产生副作用。

生产系统通常使用“抢任务 + 状态 CAS + 幂等执行”的组合。

## 常见方案

方案包括：

- 按业务 ID 分片。
- 使用任务状态抢占。
- 使用数据库乐观锁。
- 使用分布式锁。
- 使用唯一约束。
- 使用状态机幂等。

不同方案可以叠加。

## 数据库抢占示例

可以执行：

```sql
UPDATE compensation_task
SET status = 'RUNNING', worker_id = ?, updated_at = now()
WHERE id = ?
  AND status = 'PENDING';
```

只有更新成功的实例获得执行权。

## 锁失效怎么办？

即使锁超时、实例暂停或网络抖动，也可能出现并发。

所以业务层必须：

- 检查当前状态。
- 使用版本号更新。
- 保证重复执行无副作用。
- 记录执行结果。

锁是优化，不是唯一正确性来源。

## 在 eMall 项目中怎么讲？

多个库存补偿实例扫描过期预占时，用预占记录状态从 `RESERVED` CAS 更新为 `RELEASING`。

只有抢占成功的实例执行释放，释放本身仍按 `reservationId` 幂等。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
补偿任务避免多实例并发，要用分片、任务抢占、乐观锁、分布式锁和幂等组合。
我更依赖数据库状态 CAS 和业务幂等，而不是单纯依赖分布式锁。

即使两个实例同时执行，状态机和唯一约束也要保证同一业务补偿只生效一次。
```

## 回答评分点

高分答案应该覆盖：

- 多实例并发是补偿系统常态。
- 任务抢占和乐观锁常用。
- 分布式锁不能作为唯一保障。
- 业务幂等是最终防线。
- 状态 CAS 能控制执行权。
