# 216 Saga 适合哪些场景？

[返回按分类学习面试题](../README.md)

## 题目

Saga 适合哪些场景？

## 先给面试官的短答案

Saga 适合长流程、跨多个服务、每一步都有本地事务且可以补偿的业务，比如订单履约、退款、售后、旅行预订。
它把大事务拆成一系列本地事务，失败时按相反顺序执行补偿动作。

Saga 适合最终一致，不适合无法补偿或必须强原子提交的场景。

## Saga 模式

两种常见形式：

- 编排式：由一个协调器控制流程。
- 协同式：服务之间通过事件推进流程。

编排式流程清晰，协同式耦合低但排障更复杂。

## 适合场景

适合：

- 订单履约。
- 退款流程。
- 售后流程。
- 跨系统审批。
- 多步骤业务流程。

这些流程通常持续时间长，无法用数据库事务包住。

## 补偿不是回滚

补偿是业务反向操作，不等于数据库回滚。

例如发货失败后退款，不是把支付记录删除，而是创建退款记录并更新状态。

因此补偿要符合业务语义和审计要求。

## 在 eMall 项目中怎么讲？

售后退款可以使用 Saga。

流程可能包括创建售后单、审核通过、通知仓库退货、确认收货、发起退款、更新订单状态。
任一步失败都进入可恢复状态，由补偿或人工介入处理。

## 深度增强：可观测与配置治理图

![指标、日志、Trace 和告警平台](../assets/observability-platform.svg)

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
Saga 适合长流程和最终一致场景。它把跨服务大事务拆成多个本地事务，每一步提交后发布事件或通知编排器，
如果后续失败，就执行业务补偿动作。

Saga 的关键是状态机、幂等、补偿、重试和审计。补偿不是数据库回滚，而是符合业务语义的反向操作。
```

## 回答评分点

高分答案应该覆盖：

- Saga 适合长流程最终一致。
- 分为编排式和协同式。
- 每一步是本地事务。
- 失败后做业务补偿。
- 补偿不是简单数据库回滚。
