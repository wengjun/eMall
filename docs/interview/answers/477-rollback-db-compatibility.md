# 477 回滚前为什么要考虑数据库兼容？

[返回按分类学习面试题](../README.md)

## 题目

回滚前为什么要考虑数据库兼容？

## 先给面试官的短答案

因为应用回滚只是代码回到旧版本，但数据库中的 schema 和数据可能已经被新版本改变。如果旧版本不能读取新字段、
不能处理新枚举、依赖已删除字段，或者无法理解新数据格式，回滚后仍然会失败，甚至造成数据损坏。

## 常见不兼容场景

数据库兼容问题通常来自：

- 新版本删除了旧版本仍在读取的字段。
- 新版本修改了字段含义、类型或枚举值。
- 新版本写入了旧版本无法解析的 JSON 结构。
- 新版本改变了唯一索引、状态机或约束规则。
- 新版本迁移了数据，但旧版本仍按旧模型查询。

这些问题会让“代码回滚”不等于“系统恢复”。

## 为什么核心交易链路更敏感

订单、支付、库存和账务数据都有强状态语义。比如新版本新增了支付状态 `PARTIAL_CAPTURED`，
旧版本只认识 `PAID` 和 `FAILED`，回滚后可能把订单判断成异常状态。再比如新版本把优惠明细拆表，
旧版本仍从旧字段读取，会导致订单金额展示或退款计算错误。

## 生产原则

数据库变更要遵循向前兼容和向后兼容。发布新版本前，旧版本应该能在新 schema 上运行；回滚时，
旧版本应该能读取新版本写入的数据，至少能安全忽略未知字段。

如果无法做到兼容，就不能依赖快速回滚，而要设计数据修复、双写校验、只读降级或人工审批。

## 在 eMall 项目中怎么讲？

eMall 的订单、支付、库存和促销发布前，要评估数据库变更是否支持旧代码继续运行。状态枚举要兼容扩展，
新增字段要允许为空或有默认值，删除字段必须经过多阶段迁移，金额和状态类字段不能随意改变语义。

## 深度增强：Java 17 兼容读取示例

```java
import java.math.BigDecimal;
import java.util.Map;

enum PaymentStatus {
    INIT,
    PAID,
    FAILED,
    UNKNOWN
}

record PaymentSnapshot(String orderId, PaymentStatus status, BigDecimal amount) {
}

final class PaymentStatusReader {

    PaymentSnapshot read(Map<String, String> row) {
        String rawStatus = row.getOrDefault("payment_status", "UNKNOWN");
        PaymentStatus status = switch (rawStatus) {
            case "INIT" -> PaymentStatus.INIT;
            case "PAID" -> PaymentStatus.PAID;
            case "FAILED" -> PaymentStatus.FAILED;
            default -> PaymentStatus.UNKNOWN;
        };
        BigDecimal amount = new BigDecimal(row.getOrDefault("pay_amount", "0"));
        return new PaymentSnapshot(row.get("order_id"), status, amount);
    }
}
```

这个示例强调“未知状态安全处理”。旧版本如果遇到新枚举，不应该直接抛异常导致订单页不可用；
更好的做法是展示保守状态、禁止危险操作、触发告警，并等待兼容版本或数据修复。

## 深度增强：生产边界

数据库兼容包含 schema 兼容和数据语义兼容。新增 nullable 字段通常风险较低，删除字段、
修改字段类型、复用枚举含义、改变金额计算语义都是高风险。上线前要做兼容性测试：
旧代码跑新 schema，新代码读旧数据，回滚版本读灰度期间写入的新数据。

如果无法做到兼容，就不能承诺秒级回滚。此时要准备只读降级、数据修复脚本、人工审批和回滚窗口。
尤其是支付、账务、库存这种不可随意覆盖的数据，宁可多阶段发布，也不要一次性破坏 schema。

## 深度增强：面试高分表达

我会把回滚拆成代码回滚和状态回滚。代码可以切回旧版本，但数据库、消息和缓存已经被新版本写过。
所以生产发布前必须验证旧版本在新数据上是否安全运行。我的默认方案是 expand-contract：
先兼容扩展，再双写迁移和灰度，最后确认旧依赖消失后再收缩。

## 专家级完整回答

```text
回滚的本质是把运行逻辑切回旧版本，但数据库是共享状态，不会自动回到旧版本。

如果新版本已经修改了 schema 或写入了旧版本不能理解的数据，代码回滚可能无法恢复服务，
甚至会产生更严重的数据错误。所以生产发布必须把数据库兼容作为回滚预案的一部分。

我会要求数据库变更遵循 expand-contract：先扩展兼容 schema，让新旧版本都能运行；
再发布应用并完成数据迁移；最后确认没有旧版本依赖后再收缩删除旧字段。
核心交易系统宁可多阶段发布，也不能把数据库变更和代码变更绑成一次不可逆操作。
```

## 回答评分点

高分答案应该覆盖：

- 说明代码可回滚但数据库状态不会自动回滚。
- 能举出字段删除、枚举新增、JSON 格式变化等风险。
- 能结合订单、支付状态说明业务损坏风险。
- 知道向前兼容、向后兼容和 expand-contract。
- 能指出数据库不兼容会让回滚失效。
