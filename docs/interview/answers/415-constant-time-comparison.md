# 415 常量时间比较解决什么问题？

[返回按分类学习面试题](../README.md)

## 题目

常量时间比较解决什么问题？

## 先给面试官的短答案

常量时间比较用于防止时间侧信道攻击。普通字符串比较通常在发现第一个不同字符时提前返回，攻击者
可以通过响应时间推测签名或密钥的前缀是否正确。

验证 HMAC、token、验证码等敏感值时应使用常量时间比较。

## 普通比较的问题

普通比较可能：

- 第一个字符不同很快返回。
- 前几个字符相同耗时更长。
- 攻击者多次试探响应时间。
- 逐步猜出正确前缀。

这就是 timing attack 的基础。

## 常量时间比较

常量时间比较会遍历完整输入，并把差异累积到结果中。

无论前缀匹配多少，耗时尽量保持一致，从而减少攻击者通过时间推断内容的机会。

## 适用场景

适用：

- HMAC 签名比较。
- API token 比较。
- webhook secret 校验。
- 密码 hash 结果比较。
- 一次性验证码比较。

业务普通字符串不需要过度使用。

## 在 eMall 项目中怎么讲？

eMall 开放平台验证商家 HMAC 签名时，不能用普通字符串比较签名。

应使用安全库提供的常量时间比较，避免攻击者通过大量请求推测签名内容。

## 深度增强：Java 17 安全策略示例

```java
import java.util.Set;

record SecurityDecision(boolean allowed, String reason) {
}

final class ScopePolicy {

    SecurityDecision check(Set<String> scopes, String requiredScope) {
        if (scopes.contains(requiredScope)) {
            return new SecurityDecision(true, "allowed");
        }
        return new SecurityDecision(false, "missing scope: " + requiredScope);
    }
}
```

这段代码体现最小权限原则。生产系统还要结合租户、资源归属、IP、设备、风险等级和操作审计。

## 深度增强：生产边界

安全不能只靠前端隐藏按钮，也不能只在网关做一次判断。核心资源要在服务层做资源级授权，
敏感数据要脱敏，密钥要支持轮换，失败日志不能泄露 token、secret、身份证号和银行卡号。

## 专家级完整回答

```text
常量时间比较解决的是时间侧信道问题。普通比较会在第一个不同字符处提前返回，攻击者可能通过
响应时间差异猜测签名或 token 的正确前缀。

验证 HMAC、token、验证码和密码 hash 等敏感值时，应使用安全库提供的常量时间比较，减少 timing
attack 风险。
```

## 回答评分点

高分答案应该覆盖：

- 普通比较可能提前返回。
- 响应时间可能泄露前缀信息。
- 常量时间比较遍历完整输入。
- 适用于签名和 token。
- 使用安全库实现。
