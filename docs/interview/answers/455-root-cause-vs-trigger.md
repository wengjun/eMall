# 455 如何区分根因和触发因素？

[返回按分类学习面试题](../README.md)

## 题目

如何区分根因和触发因素？

## 先给面试官的短答案

触发因素是让故障暴露出来的直接事件，根因是系统为什么会因为这个事件而失败的深层原因。比如流量
突增是触发因素，缺少限流、容量不足和缓存预热失败才可能是根因。

复盘时只停留在触发因素，会导致问题再次发生。

## 触发因素

触发因素可能是：

- 发布新版本。
- 流量突增。
- 第三方超时。
- 某个节点宕机。
- 运维误操作。
- 大促开始。

触发因素回答“这次为什么现在发生”。

## 根因

根因可能是：

- 缺少灰度。
- 缺少限流。
- 缺少降级。
- 容量规划不足。
- 监控缺失。
- 数据模型不合理。
- 自动恢复能力不足。

根因回答“系统为什么扛不住”。

## 分析方法

方法：

- 5 Whys。
- 故障树分析。
- 对照时间线。
- 看防线为何失效。
- 看哪些机制本应阻止故障扩大。

重点是系统性改进。

## 在 eMall 项目中怎么讲？

eMall 秒杀开始导致库存服务超时，秒杀流量是触发因素。

根因可能是热点 SKU 未分桶、令牌发放过多、库存接口没有隔离、数据库行锁竞争和没有过载保护。

## 深度增强：Kubernetes 运维治理图

![Kubernetes 生产运行和故障治理](../assets/kubernetes-operations.svg)

Kubernetes 题不能只背 Deployment、Service 和 Ingress。生产稳定性还取决于资源 requests/limits、探针、HPA、PDB、
灰度发布、配置回滚、日志指标 Trace 和故障 Runbook。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
触发因素是直接引发故障的事件，根因是系统设计或流程中让故障发生或扩大的深层问题。流量突增、
发布和第三方抖动常常只是触发因素。

复盘要追问为什么系统没有防住，比如限流、降级、隔离、容量、监控和回滚为什么没有发挥作用。
只修触发点，故障会换一种方式再次发生。
```

## 回答评分点

高分答案应该覆盖：

- 触发因素是直接事件。
- 根因是深层系统问题。
- 不能把流量突增简单当根因。
- 用 5 Whys 等方法。
- 根因分析服务于系统改进。
