# 246 高并发系统的瓶颈通常在哪里？

[返回按分类学习面试题](../README.md)

## 题目

高并发系统的瓶颈通常在哪里？

## 先给面试官的短答案

高并发系统瓶颈通常出现在数据库、缓存热点、下游服务、线程池、连接池、锁竞争、网络带宽、序列化和单点资源。
定位瓶颈要看 CPU、内存、IO、连接数、队列长度、响应时间分位数和业务成功率。

不要只盯 QPS，尾延迟和错误率更能说明系统是否健康。

## 常见瓶颈

包括：

- 数据库慢查询和锁。
- 热点 key。
- 缓存击穿。
- MQ 积压。
- 下游超时。
- 线程池打满。
- HTTP 连接池耗尽。
- 分布式锁竞争。
- 大对象序列化。
- 网关和负载均衡瓶颈。

瓶颈往往出现在共享资源。

## 定位指标

指标包括：

- CPU 使用率。
- GC 暂停。
- 数据库连接数。
- 慢 SQL。
- 缓存命中率。
- 线程池队列长度。
- P95 和 P99 延迟。
- 错误率。
- 限流和熔断次数。

业务指标也要同时看，例如下单成功率。

## 解决思路

思路：

- 缓存和预计算。
- 分库分表。
- 读写分离。
- 限流削峰。
- 异步化。
- 热点打散。
- 降级非核心能力。
- 扩容和隔离。

先定位，再优化，避免盲目扩容。

## 在 eMall 项目中怎么讲？

秒杀场景瓶颈可能不是 Java 代码，而是库存热点行、Redis 热点 key、MQ 积压或数据库连接池。

要通过压测和监控定位具体瓶颈，再做热点库存预扣、令牌限流和异步削峰。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
高并发瓶颈通常在共享资源：数据库、缓存热点、连接池、线程池、下游服务、锁、网络和 MQ。
定位时要看资源指标、队列长度、P95/P99 延迟、错误率和业务成功率。

优化要基于证据，常用手段是缓存、异步削峰、限流、分片、隔离、降级和容量扩展。
```

## 回答评分点

高分答案应该覆盖：

- 瓶颈常在共享资源。
- 数据库和缓存热点很常见。
- P99、错误率和队列长度重要。
- 优化前先定位。
- 解决手段包括缓存、异步、限流和隔离。
