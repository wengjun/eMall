# 321 如何做数据库备份和恢复演练？

[返回按分类学习面试题](../README.md)

## 题目

如何做数据库备份和恢复演练？

## 先给面试官的短答案

数据库备份不是把数据文件拷贝出来就结束，真正关键是能在目标时间内恢复，并且恢复后的数据可信。

生产级做法要同时设计全量备份、增量备份、binlog 归档、异地保存、加密、校验和恢复演练。
面试时要明确 RPO 和 RTO。RPO 表示最多能丢多少数据，RTO 表示多久能恢复服务。

## 备份体系

常见备份包括：

- 全量备份。
- 增量备份。
- binlog 归档。
- 快照备份。
- 跨可用区或跨地域备份。
- 逻辑备份。
- 物理备份。

全量备份用于建立恢复基线，binlog 用于恢复到指定时间点。

## 恢复演练

恢复演练要覆盖：

- 从备份介质拉取数据。
- 恢复到临时实例。
- 回放 binlog 到指定时间。
- 校验表结构和行数。
- 抽样校验核心订单、支付、库存数据。
- 验证应用能否连接恢复实例。
- 记录实际恢复耗时。

只做备份不做恢复演练，等于不知道备份是否有效。

## 关键指标

指标包括：

- 备份成功率。
- 备份耗时。
- 备份文件大小。
- 备份校验结果。
- RPO。
- RTO。
- 恢复成功率。
- 最近一次恢复演练时间。

这些指标要进入监控和审计。

## 在 eMall 项目中怎么讲？

eMall 的订单、支付、库存库属于核心数据。订单库可以每天做全量备份，持续归档 binlog。

恢复演练时先恢复订单库，再恢复支付库，通过 `order_no`、`payment_no` 和金额做一致性校验。
如果恢复后的订单支付状态不一致，要通过对账和补偿任务修复，而不是直接对外恢复流量。

## 深度增强：数据访问和扩展图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
数据库备份恢复要从业务目标倒推，先定义 RPO 和 RTO，再设计全量备份、增量备份、binlog
归档、异地存储和加密校验。

更重要的是定期做恢复演练。演练要恢复到独立实例，回放日志到指定时间点，然后校验订单、
支付、库存等核心数据，并记录实际恢复耗时。只有演练通过，才能证明备份真正可用。
```

## 回答评分点

高分答案应该覆盖：

- 区分备份和恢复能力。
- 解释 RPO 和 RTO。
- 说明全量、增量、binlog 的组合。
- 强调恢复演练和数据校验。
- 结合订单、支付、库存一致性说明。
