# 516 设计库存系统

[返回按分类学习面试题](../README.md)

## 题目

设计库存系统。

## 先给面试官的短答案

库存系统要管理可售库存、预占库存、已售库存和释放库存，核心目标是不超卖、可高并发扣减、可补偿恢复。
普通商品可以用数据库条件更新和预占记录保证正确性，秒杀和热点 SKU 可以用 Redis 原子预扣加数据库最终确认。

## 库存模型

库存通常至少包含总库存、可售库存、预占库存、已售库存和冻结库存。下单时先预占，支付成功后确认扣减，
支付超时或订单取消后释放预占。

预占记录必须有业务唯一键，例如 orderId 和 skuId，防止重复预占。

## 扣减方案

数据库方案可以使用条件更新：只有当可售库存大于等于扣减数量时才更新成功。它正确性强，适合普通商品。

Redis 方案适合热点和秒杀。通过 Lua 脚本原子扣减缓存库存，成功后异步或同步写预占记录。最终仍要有数据库兜底和对账。

队列方案适合极端热点，把请求排队串行化处理，牺牲部分实时性换取稳定性。

## 一致性和补偿

库存预占成功但订单创建失败，要释放预占。订单超时未支付，要定时释放。支付成功后，要确认库存从预占转已售。

补偿任务要扫描过期预占、异常状态和 Redis 与数据库差异。所有释放和确认操作都必须幂等，不能多释放。

## 热点保护

热点 SKU 会造成单行热点、Redis 热点 key 和接口突发流量。可以使用限流、排队、库存分片、热点缓存、本地预热和异步削峰。

但库存分片会增加汇总和补偿复杂度，必须权衡。

## 在 eMall 项目中怎么讲？

eMall 的 `inventory` 模块可以提供预占、确认、释放、查询和补偿接口。秒杀由 `flash-sale` 先做流量削峰和资格校验，
再调用库存服务或 Redis 预扣。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

## 专家级完整回答

```text
库存系统最重要的是不超卖，其次才是吞吐。

我会把库存分成可售、预占和已售。下单时创建预占记录并减少可售；支付成功后确认预占；
取消或超时后释放预占。每一步都要有幂等键和状态机。

```

## 回答评分点

高分答案应该覆盖：

- 能解释可售、预占、已售和释放。
- 知道预占记录和幂等键的重要性。
- 能比较数据库、Redis 和队列方案。
- 覆盖超时释放、支付确认和补偿对账。
- 能说明热点 SKU 保护和库存分片取舍。

## 深度完善：系统设计高分展开

围绕「设计库存系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。
