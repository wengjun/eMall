# 222 消费成功但 ack 失败怎么办？

[返回按分类学习面试题](../README.md)

## 题目

消费成功但 ack 失败怎么办？

## 先给面试官的短答案

消费成功但 ack 失败会导致 Broker 认为消息没有完成，后续可能重新投递。
解决方式不是追求 ack 永远成功，而是让消费者处理逻辑幂等，重复收到同一消息时不会产生重复副作用。

消费者必须按“处理可能成功，但确认可能丢失”的模型设计。

## 典型场景

流程可能是：

```text
消费者收到消息
-> 本地业务处理成功
-> 准备 ack
-> 网络抖动或消费者宕机
-> Broker 未收到 ack
-> 消息重新投递
```

此时业务已经成功，下一次消费应识别为重复。

## 处理策略

策略包括：

- 业务处理幂等。
- 消费状态记录。
- 数据库唯一索引。
- 状态机校验。
- ack 失败后允许重新投递。
- 监控重复消费率。

不要在 ack 失败后盲目回滚已经提交的业务操作。

## 事务边界

通常顺序是：

- 处理业务并提交本地事务。
- 本地事务提交成功后 ack。

如果先 ack 再处理业务，消费者宕机时消息会丢失。

所以宁愿重复，也不要丢失。

## 在 eMall 项目中怎么讲？

积分服务消费 `OrderCompleted` 后给用户加积分。如果加积分成功但 ack 失败，消息会重投。

积分服务应通过 `orderId + pointsRuleId` 保证同一订单只加一次积分。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
消费成功但 ack 失败会造成消息重新投递，这是至少一次语义下的正常结果。
正确做法是业务先提交，再 ack；如果 ack 失败导致重投，消费者通过幂等记录、唯一索引和状态机识别重复。

系统设计上宁愿重复投递，也不能先 ack 后处理导致消息丢失。
```

## 回答评分点

高分答案应该覆盖：

- ack 失败会导致重新投递。
- 消费者必须幂等。
- 通常先提交业务再 ack。
- 不应为了避免重复而先 ack。
- 重复消费率要监控。
