# 290 慢 SQL 如何分析？

[返回按分类学习面试题](../README.md)

## 题目

慢 SQL 如何分析？

## 先给面试官的短答案

慢 SQL 分析要先确认慢在哪里，再看执行计划、索引、扫描行数、回表、排序、锁等待和返回数据量。
常用工具是慢查询日志、`EXPLAIN`、执行耗时分布、数据库监控和业务调用链。

不要一上来就加索引，要先确认瓶颈原因。

## 分析步骤

步骤：

- 从慢查询日志找到 SQL。
- 确认执行频率和影响面。
- 使用 `EXPLAIN` 看执行计划。
- 看是否走合适索引。
- 看扫描行数和返回行数。
- 看是否 filesort 或临时表。
- 看锁等待和事务。
- 看是否返回过多字段。

最后再决定优化方式。

## 常见原因

包括：

- 没有合适索引。
- 索引选择不合理。
- 回表太多。
- 深分页。
- 大范围扫描。
- 排序无法用索引。
- 锁等待。
- 查询字段过多。

不同原因对应不同优化手段。

## 优化方式

方式：

- 添加或调整联合索引。
- 改写查询条件。
- 使用覆盖索引。
- 限制时间范围。
- 游标分页。
- 拆分大查询。
- 异步导出。
- 降低事务持锁时间。

优化后要验证执行计划和线上指标。

## 在 eMall 项目中怎么讲？

后台订单查询慢，不能直接给每个筛选字段都建索引。

应先看最常用查询路径、扫描行数、排序方式和是否深分页，再设计少量高价值联合索引或改为异步导出。

## 深度增强：数据访问和扩展图

![数据库、缓存和消息一致性链路](../assets/data-cache-mq.svg)

数据库题要从访问路径、索引、锁、事务和容量出发。电商系统的数据层既要支撑高并发读写，
又要保证订单、库存、支付等事实数据可追踪。缓存和消息可以提升性能，但不能替代数据库事实来源。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
慢 SQL 分析要从慢查询日志和业务链路定位 SQL，再用 explain 看访问类型、索引、扫描行数、排序和临时表。
还要结合锁等待、事务时间、返回数据量和执行频率判断影响。

优化要基于原因选择索引、改写 SQL、覆盖索引、游标分页或异步导出，并通过执行计划和监控验证效果。
```

## 回答评分点

高分答案应该覆盖：

- 先定位慢 SQL 和影响面。
- `EXPLAIN` 是基础工具。
- 看索引、扫描行数、排序和锁等待。
- 不要盲目加索引。
- 优化后要验证。
