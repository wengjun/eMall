# 034 如何设计一个可扩展的插件机制？

[返回按分类学习面试题](../README.md)

## 题目

如何设计一个可扩展的插件机制？

## 先给面试官的短答案

插件机制的核心不是动态加载类，而是设计稳定、受控、可观测、可回滚的扩展点。
我会先定义最小接口和输入输出，再设计注册发现、版本兼容、超时隔离、权限边界、指标监控和测试契约。

## 设计步骤

### 1. 明确扩展点

先判断哪里需要扩展。

例如支付渠道：

```text
核心流程稳定：创建支付单、发起支付、接收回调、入账、对账。
变化部分：不同渠道的请求签名、下单接口、回调验签。
```

扩展点应该只覆盖变化部分，不要把整个支付流程交给插件。

### 2. 定义最小接口

```java
public interface PaymentChannelPlugin {
    String channelCode();

    PaymentCreateResult createPayment(PaymentCreateCommand command);

    PaymentCallbackResult verifyCallback(PaymentCallbackCommand command);
}
```

接口要小，避免插件获得过多能力。

### 3. 定义稳定输入输出

命令对象和结果对象要稳定、可版本化。

不要用：

```java
Map<String, Object>
```

除非确实是开放扩展字段，否则会损失类型安全。

### 4. 注册和发现

方式：

- Java `ServiceLoader`。
- Spring 自动注入 `List<Plugin>`。
- 数据库配置启用某些插件。
- 配置中心动态开关。

Spring 项目通常用 Bean 更简单。

### 5. 隔离和超时

插件不能拖垮核心链路。

需要：

- 超时。
- 熔断。
- 线程池隔离。
- 失败降级策略。
- 指标监控。

### 6. 权限边界

插件不应该随意：

- 改订单状态。
- 改支付流水。
- 访问所有数据库表。
- 发任意 MQ。
- 读取敏感配置。

核心状态变化仍由核心服务控制。

### 7. 测试契约

每个插件必须通过契约测试：

- 正常请求。
- 超时。
- 签名失败。
- 重复回调。
- 金额不一致。
- 渠道异常。

## 插件机制的风险

- 接口设计过大，插件绕过核心规则。
- 插件异常影响主流程。
- 插件版本不兼容。
- 插件权限过大。
- 插件缺少指标，故障难定位。
- 动态加载带来安全风险。

## 在 eMall 项目中怎么讲？

适合插件化：

- 支付渠道。
- 风控规则。
- 物流承运商。
- 报表导出格式。
- 开放平台签名算法。

不适合插件化：

- 订单状态核心迁移。
- 库存扣减不变量。
- 支付流水入账规则。

这些必须由核心服务掌控。

## 专家级完整回答

```text
我设计插件机制时会先确认扩展点是否稳定，然后定义最小接口、稳定命令对象和结果对象。
插件通过 Spring Bean 或 ServiceLoader 注册，核心服务按类型选择插件。
同时必须有超时、熔断、线程池隔离、指标和契约测试。

插件不能直接修改订单状态、库存和支付流水。核心业务不变量必须留在核心服务里。
否则插件机制会变成破坏边界的后门。
```

## 回答评分点

高分答案应该覆盖：

- 插件机制核心是扩展点治理。
- 能定义接口、输入输出、注册发现。
- 能说明超时、隔离、指标和测试。
- 能指出权限边界。
- 能区分适合插件化和不适合插件化的业务。
