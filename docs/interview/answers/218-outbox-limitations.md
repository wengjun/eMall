# 218 Outbox 不能解决什么问题？

[返回按分类学习面试题](../README.md)

## 题目

Outbox 不能解决什么问题？

## 先给面试官的短答案

Outbox 只能保证业务事务提交后消息最终可投递，不能保证消费者一定成功处理、不能消除重复消息、不能自动保证全局强一致。
它也不能替代业务幂等、状态机、死信恢复、顺序治理和对账。

Outbox 是可靠发布事件的基础设施，不是完整一致性方案。

## 不能保证消费者成功

消息发到 MQ 后：

- 消费者可能失败。
- 消费者可能超时。
- 消费者可能重复消费。
- 下游数据库可能不可用。

这些需要消费者重试、幂等、死信和补偿解决。

## 不能避免重复

投递器可能在发送 MQ 成功后，标记 outbox 状态前宕机。

恢复后它会再次发送同一条消息。因此消费者必须按消息 ID 或业务 ID 幂等处理。

## 不能保证强一致读取

业务表提交后，消息投递和消费者处理需要时间。

这段时间内其他服务看到的数据可能还是旧的。业务必须接受并治理最终一致。

## 在 eMall 项目中怎么讲？

订单服务用 Outbox 发布支付成功事件，不代表库存服务一定已经确认扣减。

库存服务消费失败时还要重试、进入死信、触发补偿或对账，不能只依赖订单服务的 Outbox。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
Outbox 解决的是业务数据写入和消息发布之间的可靠性问题。它不能保证消费者成功处理，
也不能避免重复消息，更不能提供跨服务强一致。

因此使用 Outbox 时，消费者必须幂等，系统要有重试、死信、补偿、顺序治理和对账。Outbox 是可靠事件发布的一环。
```

## 回答评分点

高分答案应该覆盖：

- Outbox 只保证消息最终投递。
- 消费者仍可能失败。
- 重复消息仍会发生。
- 最终一致有延迟窗口。
- 仍需要幂等、死信、补偿和对账。
