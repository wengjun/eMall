# 087 CAS 是什么？

[返回按分类学习面试题](../README.md)

## 题目

CAS 是什么？

## 先给面试官的短答案

CAS 是 Compare-And-Swap，比较并交换。它会比较内存中的当前值是否等于预期值，如果相等就更新为新值，
否则更新失败。CAS 是很多原子类和无锁算法的基础，可以在不使用互斥锁的情况下完成并发更新。

CAS 的问题包括 ABA、循环重试 CPU 消耗和只能直接更新单个变量。

## CAS 三个参数

CAS 通常包含：

- 内存位置。
- 预期旧值。
- 新值。

逻辑是：

```text
if current == expected:
    current = newValue
else:
    fail
```

这个比较和交换必须是硬件级原子操作。

## AtomicInteger 示例

```java
AtomicInteger stock = new AtomicInteger(10);

boolean success = stock.compareAndSet(10, 9);
```

如果当前值是 10，就更新为 9。

如果当前值已经被其他线程改了，更新失败。

## 自旋重试

CAS 更新失败时，常见做法是循环重试。

示例：

```java
int current;
int next;
do {
    current = stock.get();
    next = current - 1;
} while (!stock.compareAndSet(current, next));
```

这避免了锁阻塞，但竞争激烈时会不断重试，消耗 CPU。

## CAS 的优点

优点：

- 不需要阻塞线程。
- 避免上下文切换。
- 适合低到中等竞争。
- 是原子类和并发容器基础。

在计数、状态切换、无锁队列中很常见。

## CAS 的问题

问题：

- ABA 问题。
- 自旋消耗 CPU。
- 多变量一致性困难。
- 竞争激烈时性能下降。
- 代码复杂度高。

CAS 不是锁的绝对替代品。

## 和锁的区别

锁通常是悲观并发：先获取互斥权限，再修改。

CAS 更像乐观并发：先尝试修改，失败再重试。

低竞争场景 CAS 效率高；高竞争或复杂临界区，锁或队列化可能更合适。

## 在 eMall 项目中怎么讲？

单 JVM 内的本地计数器可以用 CAS 或原子类。

但分布式库存扣减不能只靠 JVM CAS，因为多个实例各自有内存。

跨实例一致性要依赖数据库条件更新、Redis Lua、消息串行化或库存中心。

## 深度增强：并发治理图

![Java 并发从线程安全到容量保护](../assets/concurrency-governance.svg)

并发题不能只回答 API 用法。生产系统要同时考虑线程安全、资源隔离、超时、拒绝、幂等和分布式多实例。
单机锁只能保护当前 JVM，不能保护整个集群；线程池满也不是小问题，而是容量和可用性风险。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

## 深度增强：面试高分表达

我会把并发问题分成三层：JMM 和锁保证单机正确性，线程池和隔离保证资源不被拖垮，
幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
CAS 是比较并交换，它以原子方式判断当前值是否等于预期值，相等才更新为新值。它是 AtomicInteger、
AtomicReference 和很多无锁结构的基础，低竞争时能避免线程阻塞和上下文切换。

但 CAS 有 ABA、自旋 CPU 消耗和多变量一致性困难等问题。工程上我会把它用于单进程内简单状态更新，
跨实例库存这类问题不能靠 JVM CAS，需要数据库、Redis 或消息化保证一致性。
```

## 回答评分点

高分答案应该覆盖：

- CAS 是 Compare-And-Swap。
- 比较和交换是原子操作。
- 原子类基于 CAS。
- 知道 ABA 和自旋问题。
- 知道 CAS 不解决分布式一致性。
