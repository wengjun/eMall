## 题目

如何做服务依赖治理？

## 先给面试官的短答案

服务依赖治理要先看清依赖，再控制依赖。核心是建立服务目录、依赖拓扑、调用指标、分级管理、变更评审和故障演练。
关键链路要限制同步依赖数量，并为每个依赖定义超时、熔断、降级和负责人。

没有依赖治理，微服务数量越多，系统越不可控。

## 治理对象

需要治理：

- 服务调用关系。
- 数据库和缓存依赖。
- MQ topic 依赖。
- 第三方依赖。
- 配置和开关依赖。
- 部署和发布依赖。

依赖不仅是 HTTP 或 RPC 调用。

## 依赖信息

服务目录应记录：

- 服务负责人。
- 业务能力。
- 上游和下游。
- SLA。
- 限流规则。
- 降级策略。
- 变更窗口。
- 应急联系人。

这些信息要能从监控、注册中心和配置中自动补全。

## 治理动作

常见动作：

- 识别循环依赖。
- 控制核心链路依赖数量。
- 下线无用依赖。
- 为强依赖设置熔断。
- 为弱依赖设计降级。
- 变更前评估影响面。
- 定期做故障演练。

治理目标是减少未知依赖和隐式耦合。

## 在 eMall 项目中怎么讲？

下单链路依赖库存、价格、优惠和风控，这些依赖都应标注为核心或准核心。

短信、推荐和数据分析属于弱依赖，不能同步阻塞下单。

## 深度增强：可观测与配置治理图

![指标、日志、Trace 和告警平台](../assets/observability-platform.svg)

配置、日志、指标和 Trace 不是附属能力，而是生产系统定位问题和控制变更风险的基础。
没有可观测性，限流、熔断、回滚和补偿都很难判断是否有效。

## 深度增强：Java 17 观测信号示例

```java
import java.time.Instant;
import java.util.Map;

record ObservabilityEvent(
        Instant time,
        String traceId,
        String service,
        String eventType,
        Map<String, String> tags) {
}

final class TraceTagPolicy {

    boolean shouldKeep(String key) {
        return !key.equalsIgnoreCase("password")
                && !key.equalsIgnoreCase("secret")
                && !key.equalsIgnoreCase("token");
    }
}
```

这段代码体现生产观测的两个重点：所有关键事件要能关联 traceId，敏感信息不能进入日志和标签。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
服务依赖治理要先建立服务目录和依赖拓扑，明确每个服务依赖谁、被谁依赖、SLA、负责人和降级策略。
然后治理循环依赖、长链路、强弱依赖和无用依赖。

核心链路要限制同步依赖数量，强依赖设置超时和熔断，弱依赖必须可降级。变更前要评估影响面并做故障演练。
```

## 回答评分点

高分答案应该覆盖：

- 依赖治理先要可观测。
- 依赖包括服务、数据、MQ、配置和第三方。
- 服务目录和拓扑很重要。
- 强弱依赖要区别处理。
- 变更评审和故障演练是治理手段。

## 深度完善：面向 L6 的回答框架

围绕「如何做服务依赖治理？」，高分答案不能停在概念定义，而要把「服务拆分、数据归属、依赖图、调用链、团队边界和演进策略」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「微服务边界和架构治理」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `user、product、order、inventory、payment、fulfillment、search、risk 的边界` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：跨服务调用深度、依赖数、变更耦合度、故障半径、跨团队交付周期。
- 验证证据：服务依赖图、ADR、契约测试、发布记录、事故复盘和边界评审结论。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
