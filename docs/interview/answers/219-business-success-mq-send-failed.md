# 219 业务数据写成功但 MQ 发送失败怎么办？

[返回按分类学习面试题](../README.md)

## 题目

业务数据写成功但 MQ 发送失败怎么办？

## 先给面试官的短答案

正确做法是避免直接在业务事务提交后同步发送 MQ 作为唯一手段。
应使用 Outbox 或事务消息，让业务数据和待发送消息在同一可靠边界内提交，再由后台任务重试投递。

如果已经发生发送失败，要通过补偿扫描业务表或 outbox 表重新构造并发送事件。

## 为什么直接发送有风险？

常见风险：

- 数据库提交成功后应用宕机。
- MQ 短时不可用。
- 网络超时但消息实际已发送。
- 发送成功后状态标记失败。

这些都会导致业务数据和消息状态不一致。

## 推荐方案

使用 Outbox：

```text
本地事务：
写业务表
写 outbox 表
提交事务

异步投递：
扫描 outbox
发送 MQ
标记已发送
```

投递失败时持续重试，超过阈值告警。

## 补偿扫描

如果老系统没有 Outbox，可以用补偿任务：

- 扫描已成功但未发事件的业务记录。
- 构造事件。
- 幂等发送。
- 标记补偿结果。

这属于兜底方案，不如 Outbox 可靠。

## 在 eMall 项目中怎么讲？

订单创建成功后必须发布 `OrderCreated` 事件。

订单服务应在创建订单的同一事务中写 outbox 事件，不能只在事务提交后调用 MQ 客户端发送一次。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
业务数据写成功但 MQ 发送失败，本质是数据库和 MQ 之间缺少原子性。生产系统应使用 Outbox 或事务消息，
让业务数据和待发送消息在同一个可靠边界提交。

之后由投递器异步重试发送。对于历史数据，可以用补偿扫描找出已成功但未发事件的记录，幂等补发。
```

## 回答评分点

高分答案应该覆盖：

- 直接事务后发送 MQ 有丢消息风险。
- Outbox 是常见方案。
- 投递器要重试和告警。
- 历史问题可用补偿扫描。
- 发送和消费都要幂等。
