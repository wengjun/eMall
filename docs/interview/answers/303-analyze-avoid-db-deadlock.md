# 303 如何分析和避免数据库死锁？

[返回按分类学习面试题](../README.md)

## 题目

如何分析和避免数据库死锁？

## 先给面试官的短答案

分析死锁要看数据库死锁日志、事务 SQL、加锁顺序、索引使用、锁范围和事务时长。
避免死锁的核心是统一加锁顺序、使用合适索引、缩短事务、减少范围锁、拆小批量操作，并让应用支持幂等重试。

死锁无法百分百避免，但可以降低概率并自动恢复。

## 如何分析

步骤：

- 查看 InnoDB deadlock 日志。
- 找到两个或多个事务的 SQL。
- 看它们分别持有什么锁。
- 看它们等待什么锁。
- 检查执行计划和索引。
- 检查事务是否过长。
- 复现并验证修复。

不要只看报错 SQL，要看整个事务内的 SQL 顺序。

## 避免方式

方式包括：

- 统一资源加锁顺序。
- 为查询条件建立合适索引。
- 避免大范围当前读。
- 缩短事务时间。
- 分批处理大更新。
- 减少事务内远程调用。
- 避免用户交互中持锁。

事务越短，死锁概率越低。

## 重试策略

死锁被数据库回滚后，应用可以重试。

重试要求：

- 操作幂等。
- 有最大次数。
- 有退避。
- 记录日志和指标。

不能无限重试。

## 在 eMall 项目中怎么讲？

订单补偿和库存释放同时执行时，要统一先锁订单再锁库存，或通过事件避免跨表长事务。

批量处理订单时按主键排序分批更新，减少交叉锁等待。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
分析死锁要看死锁日志中的事务、SQL、持有锁和等待锁，并结合 explain 判断锁范围是否被索引放大。
避免死锁要统一加锁顺序、建立合适索引、缩短事务、减少范围锁和拆小批量操作。

死锁在高并发下不能完全杜绝，所以应用层必须能识别死锁错误，并在幂等前提下有限重试。
```

## 回答评分点

高分答案应该覆盖：

- 看死锁日志和完整事务。
- 关注持有锁和等待锁。
- 索引影响锁范围。
- 统一加锁顺序和缩短事务。
- 应用层要有限重试。

## 深度完善：面向 L6 的回答框架

围绕「如何分析和避免数据库死锁？」，高分答案不能停在概念定义，而要把「索引、事务、锁、MVCC、分库分表、冷热分层、备份恢复和在线变更」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「数据库和数据建模」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `order、payment、inventory、product 的核心表、唯一约束、索引和归档表` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：慢 SQL、锁等待、死锁数、主从延迟、连接池使用率、归档任务耗时。
- 验证证据：EXPLAIN、慢 SQL 日志、死锁日志、迁移方案、备份恢复演练和容量预测。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
