## 题目

推荐服务故障如何降级？

## 先给面试官的短答案

推荐服务故障通常可以降级，因为它影响转化和体验，但不应影响下单正确性。
常见降级方式是返回热门商品、类目热销、缓存结果、人工配置榜单，或者隐藏推荐模块。

降级目标是保住主链路，同时尽量减少用户体验和转化损失。

## 降级层级

可以分层：

- 个性化推荐。
- 用户相似推荐。
- 类目热销。
- 全站热销。
- 运营配置。
- 隐藏模块。

从个性化逐步退化到稳定兜底。

## 技术实现

实现方式：

- 熔断推荐接口。
- 设置短超时。
- 使用本地缓存或 Redis 缓存。
- 预计算热门榜单。
- BFF 层隐藏模块。
- 记录降级指标。

推荐不能拖慢商品详情页和首页主渲染。

## 注意事项

注意：

- 缓存结果要有过期时间。
- 热门榜单要提前预热。
- 降级内容要符合合规和风控。
- 恢复时要小流量放开。

推荐降级也可能影响收入，要监控点击率和转化率。

## 在 eMall 项目中怎么讲？

商品详情页调用推荐服务超过 `50ms` 时，可以返回同类目热销商品。

如果推荐服务持续失败，BFF 熔断推荐调用，首页保留核心商品和搜索入口。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。


```java
enum CircuitState {
    CLOSED, OPEN, HALF_OPEN
}

final class CircuitBreaker {
    private CircuitState state = CircuitState.CLOSED;
    private int failures;

    synchronized boolean allowRequest() {
        return state != CircuitState.OPEN;
    }

    synchronized void record(boolean success) {
        if (success) {
            failures = 0;
            state = CircuitState.CLOSED;
        } else if (++failures >= 5) {
            state = CircuitState.OPEN;
        }
    }
}
```

### 可选方案对比

- 入口限流：保护整体容量，但可能误伤部分真实用户。
- 线程池隔离：保护调用方资源，但需要额外容量规划和拒绝策略。
- 熔断降级：能快速止血，但需要半开探测和平滑恢复避免二次雪崩。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果影响系统稳定性，优先保护核心链路和整体可用性，而不是追求单次请求成功。

### 决策证据

- 成本指标

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
