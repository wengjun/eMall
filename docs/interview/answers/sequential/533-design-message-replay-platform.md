## 题目

设计消息重放平台。

## 先给面试官的短答案

消息重放平台用于在消费失败、数据修复或业务补偿时重新投递历史消息。核心要求是可筛选、可审批、可限速、
可幂等、可审计和可停止。重放不是简单把 Kafka offset 往回拨，必须控制影响范围和业务副作用。

## 使用场景

常见场景包括消费者 bug 修复后重放失败消息、数据仓库漏消费补数据、履约事件未处理、搜索索引缺数据、
以及 Outbox 事件需要重新发送。

不同场景的风险不同。交易类消息重放必须非常谨慎，分析类消息重放风险较低。

## 平台能力

平台要支持按 topic、时间、key、eventId、订单号、状态和错误类型筛选消息。重放前展示影响范围和样本。

重放任务要支持审批、限速、暂停、恢复、停止、进度查看和失败重试。

所有操作要记录操作者、原因、筛选条件、消息数量、开始结束时间和结果。

## 幂等和安全

消费者必须幂等。重放平台要提醒或校验目标消费者是否支持幂等，尤其是履约、发券、退款和库存类消息。

对高风险 topic，可以要求双人审批、只允许重放到隔离 topic，或者先 dry run。

## 在 eMall 项目中怎么讲？

eMall 的 `event-platform` 可以提供消息检索和重放，`operations` 提供审批后台。
订单、支付、库存、履约的消息重放要受严格权限和限速控制。

## 专家级完整回答

```text
消息重放平台的价值是把消息修复从手工脚本变成受控流程。

我会支持按条件筛选消息、创建重放任务、审批、限速、暂停、恢复和审计。
重放前必须评估消费者幂等性和业务副作用，不能简单回拨 offset。

对核心交易消息，重放要有 dry run、双人审批和严格限速。否则一次错误重放可能造成重复发货、
重复发券或错误状态写入。
```

## 回答评分点

高分答案应该覆盖：

- 说明重放用于失败修复和数据补偿。
- 覆盖筛选、审批、限速、暂停、恢复和审计。
- 知道不能简单回拨 offset。
- 强调消费者幂等和业务副作用。
- 能结合订单、履约、搜索索引等场景。

## 深度完善：系统设计高分展开

围绕「设计消息重放平台」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。
