## 题目

如何测试库存防超卖？

## 先给面试官的短答案

测试库存防超卖要验证高并发扣减时，成功预占数量不能超过可售库存，库存不能变成负数，失败请求要明确返回库存不足。
需要覆盖数据库乐观锁、行锁、Redis 预扣、唯一预占记录、重复请求和补偿释放。

## 基本测试思路

先准备一个库存很小的 SKU，例如可售库存为 10。然后并发发起 100 个预占请求，等待全部完成后断言：

- 成功请求数等于 10。
- 失败请求数等于 90。
- 数据库库存不小于 0。
- 预占记录数量等于成功请求数。
- 同一个订单或用户重复请求不会重复预占。

这类测试必须使用真实数据库或 Redis，不能只靠 mock。

## 需要覆盖的场景

边界库存：库存为 0、库存为 1、库存刚好等于请求数。

并发扣减：多个线程同时扣同一个 SKU，验证锁或条件更新是否有效。

重复预占：同一个订单重复请求，验证幂等键或唯一索引。

释放补偿：订单取消或支付超时释放库存，不能多释放或少释放。

回滚场景：库存扣减成功但订单创建失败时，要验证补偿或事务一致性。

## SQL 级验证点

如果使用数据库条件更新，关键是类似 `where available_quantity >= ?` 的原子条件。测试要证明并发下只有满足条件的请求成功。
如果使用乐观锁，要验证版本号冲突处理。如果使用 Redis 预扣，要验证最终数据库落库和 Redis 数量能对齐。

## 在 eMall 项目中怎么讲？

eMall 的 `inventory` 模块应有并发集成测试。秒杀 SKU 可以使用 Redis 预扣提升吞吐，但最终必须有数据库库存和预占记录兜底，
并通过对账修正 Redis 和数据库差异。

### 可选方案对比

- 数据库条件扣减：正确性强，热点 SKU 下可能出现锁等待。
- Redis 预扣库存：吞吐高，但必须有最终落库、回补和对账。
- 库存分桶：缓解热点，但会增加库存聚合、迁移和补偿复杂度。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果影响交易正确性或资金安全，优先选择可审计、可补偿、可对账的方案。

- 对账差异趋势
