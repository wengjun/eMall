## 题目

设计开放平台 API。

## 先给面试官的短答案

开放平台 API 面向商家、ISV 和第三方系统，核心是安全、稳定、可治理和可审计。架构上通过 API Gateway 统一接入，
使用 appKey、secret、HMAC 签名、timestamp、nonce、权限范围、限流、审计、版本管理和回调机制管理外部调用。

## 核心能力

开放平台要提供应用注册、密钥管理、接口授权、API 文档、SDK、签名验签、限流、审计、沙箱环境、版本管理和工单支持。

API 范围可以包括商品管理、订单查询、库存同步、发货回传、售后处理和消息订阅。

## 安全设计

每个应用有 appKey 和 secret。请求必须包含 timestamp、nonce、签名和 body hash，防止篡改和重放。

服务端验证签名、时间窗口、nonce 是否使用过、应用状态、接口权限和 IP 白名单。secret 要加密存储并支持轮换。

高风险接口要做二次授权、双人审批或更严格限流。

## 流量治理

开放 API 要按应用、商家、接口和租户限流。错误请求、签名失败和高频调用要进入风控。

外部调用不稳定，所以服务端要有幂等、异步任务、回调重试和死信处理。不能让第三方慢请求拖垮内部服务。

## 版本和兼容

开放平台调用方升级慢，所以 API 需要版本管理和兼容策略。字段新增要向后兼容，字段删除要公告和过渡期。

文档、示例、错误码和 SDK 要和接口版本保持一致。

## 在 eMall 项目中怎么讲？

eMall 的 `openapi` 模块负责开放接口，`gateway` 做统一入口，`identity` 管应用身份，`risk` 做防刷和风险控制，
`operations` 做审计和管理后台。

## 深度增强：开放平台 API 安全图

![开放平台 API 安全链路](../../assets/openapi-security.svg)

开放平台的架构不能等同于“把内部接口包一层”。它要把外部调用方和内部服务隔离开：
外部只看到稳定 API contract，内部服务可以继续演进。网关负责验签、限流、审计和路由，
开放平台服务负责应用、权限、版本、文档、SDK、工单和回调治理。

## 深度增强：Java 17 请求校验链路示例

```java
import java.util.Set;

record OpenApiCall(
        String appKey,
        String apiName,
        String method,
        String path,
        String signature,
        String nonce,
        String timestamp,
        String clientIp) {
}

record AppProfile(
        String appKey,
        boolean enabled,
        Set<String> scopes,
        Set<String> allowedIps,
        int qpsLimit) {
}

final class OpenApiGuard {

    boolean allow(OpenApiCall call, AppProfile app) {
        if (!app.enabled()) {
            return false;
        }
        if (!app.scopes().contains(call.apiName())) {
            return false;
        }
        if (!app.allowedIps().isEmpty() && !app.allowedIps().contains(call.clientIp())) {
            return false;
        }
        return true;
    }
}
```

这段代码只展示授权前置判断，真实生产还要串上 HMAC 验签、nonce 去重、限流、风控、审计和 traceId。
面试时可以强调：每一步都要失败快速返回，不能让无效请求进入核心订单或支付链路。

## 深度增强：生产边界

开放平台最难的是“外部调用方不可控”。调用方可能 SDK 版本旧、网络不稳定、重试策略错误、
请求量突增或密钥泄露。因此关键接口必须幂等，错误码必须稳定，回调要支持重试和死信，
高风险操作要有审计和人工处置入口。

版本管理要采用兼容优先策略。新增字段应让老客户端能忽略；删除字段和改变字段语义要公告、
提供迁移期和调用方观测。外部 API 的破坏性变更比内部服务更危险，因为调用方不受你控制。

## 深度增强：面试高分表达

我会把开放平台设计成“安全接入层加生态治理平台”。核心能力包括应用注册、密钥、签名、
授权、限流、审计、沙箱、SDK、文档、版本兼容和回调重试。这样既保护内部服务，
又能支撑商家和 ISV 长期接入，而不是简单暴露内部接口。

## 专家级完整回答

```text
开放平台 API 的重点不是把内部接口暴露出去，而是建立一个可控的外部生态入口。

我会通过 API Gateway 统一接入，使用 appKey、secret、HMAC 签名、timestamp、nonce 和权限范围保障安全。
同时按应用和接口限流，记录审计日志，并提供沙箱、文档、SDK 和版本管理。

开放 API 必须高度重视兼容性和幂等。外部调用方不会和平台同步升级，网络也不可控，
所以接口要稳定、错误码清晰、回调可重试，关键操作可审计和可追踪。
```
