## 题目

蓝绿发布和金丝雀发布有什么区别？

## 先给面试官的短答案

蓝绿发布是同时保留蓝、绿两套完整环境，切流时把流量从旧环境整体切到新环境。金丝雀发布是先让少量用户或少量流量
进入新版本，观察稳定后逐步放量。蓝绿切换快、回滚简单但成本高；金丝雀风险更低、观察更细，但需要流量分层和指标治理。

## 蓝绿发布

蓝绿发布适合环境可复制、状态依赖少、需要快速切换的系统。它的优势是新旧环境隔离清晰，发现问题可以快速切回旧环境。
缺点是需要两套资源，数据库和外部依赖仍然可能共享，真正回滚时必须考虑数据兼容。

蓝绿发布不是万能回滚。如果新版本已经写入旧版本不能理解的数据，切回旧版本可能仍然失败。

## 金丝雀发布

金丝雀发布适合用户量大、风险敏感、需要逐步验证的业务。可以按用户 ID、地区、客户端版本、商家、租户或流量比例放量。
它能在影响面很小时发现错误率、延迟、订单转化率和支付成功率异常。

金丝雀的难点在于观测和一致性。新旧版本同时存在时，请求路由、缓存 key、消息 schema、数据库字段和幂等逻辑都要兼容。

## 如何选择

如果系统入口简单、环境成本可接受、切换窗口明确，可以选择蓝绿。若业务复杂、用户量大、要控制爆炸半径，
金丝雀通常更适合核心交易链路。大厂实践中，两者经常结合使用：先部署新环境，再逐步放量，最后完成整体切换。

## 在 eMall 项目中怎么讲？

eMall 的订单、支付、库存和促销更适合金丝雀，因为这些链路需要按用户或流量比例逐步验证。后台管理或低风险内部服务
可以使用蓝绿。无论哪种发布，都必须和数据库 expand-contract、配置回滚、监控告警和自动止损配套。

蓝绿和金丝雀解决的是流量如何进入新版本，不能单独解决数据库、消息和缓存兼容问题。
如果新版本写入旧版本不能理解的数据，即使把流量切回蓝色旧环境，旧代码仍然可能报错。

## 深度增强：Java 17 金丝雀分流示例

```java
import java.util.Set;

record ReleaseRule(
        String version,
        int percent,
        Set<String> enabledCities,
        Set<Long> allowUsers) {
}

record RequestContext(long userId, String city, String clientVersion) {
}

final class CanaryRouter {

    boolean routeToNewVersion(RequestContext context, ReleaseRule rule) {
        if (rule.allowUsers().contains(context.userId())) {
            return true;
        }
        if (!rule.enabledCities().isEmpty() && !rule.enabledCities().contains(context.city())) {
            return false;
        }
        int bucket = Math.floorMod(Long.hashCode(context.userId()), 100);
        return bucket < rule.percent();
    }
}
```

分流要稳定，通常按 userId、merchantId 或 tenantId 做 hash，而不是每次请求随机。这样用户体验稳定，
也方便比较新旧版本指标。支付、订单和库存这类链路，还要让同一个业务事务尽量落到同一版本处理。

## 深度增强：生产边界

蓝绿发布的回滚路径短，但资源成本高，且共享数据库仍可能成为不可逆风险。金丝雀影响面小，
但要求指标、链路追踪、自动止损和流量标签成熟。生产中常见做法是两者结合：
新环境先部署好，再通过金丝雀逐步引流。

发布门禁不能只看技术指标。电商核心链路还要看下单成功率、支付成功率、库存扣减失败率、
退款异常率、GMV 波动和用户投诉。只看错误率可能发现不了业务逻辑错误。

## 深度增强：面试高分表达

我会说蓝绿是环境级切换，金丝雀是流量级渐进验证。核心交易链路更适合金丝雀，
因为它能把爆炸半径控制在小流量内。但任何发布方式都必须配合数据库兼容、指标门禁和自动回滚，
否则只是把代码发布做了分流，系统风险仍然没有闭环。

## 专家级完整回答

```text
蓝绿发布强调环境级切换，金丝雀发布强调流量级渐进验证。

蓝绿的优点是新旧环境隔离明确，切换和回滚路径简单；缺点是资源成本高，而且数据库兼容仍然是关键风险。
金丝雀的优点是能把风险控制在少量用户或少量流量内；缺点是要求网关、服务治理、指标和数据兼容能力更成熟。

在电商核心链路里，我更倾向金丝雀，因为订单成功率、支付成功率、库存扣减失败率和 P99 延迟都需要逐步观察。
如果指标异常，应自动暂停放量或回滚，而不是等用户投诉。
```

- 知道数据库兼容会影响回滚。
- 能结合电商核心链路选择金丝雀。
