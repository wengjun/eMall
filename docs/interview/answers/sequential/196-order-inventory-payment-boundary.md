## 题目

如何设计订单、库存、支付的边界？

## 先给面试官的短答案

订单服务负责交易状态和订单生命周期，库存服务负责库存数量、预占和释放，支付服务负责支付单、渠道交互和支付状态。
三者通过 API 和事件协作，不能互相直接写表。

订单是交易编排中心，但不应该拥有库存和支付的内部数据。

## 订单服务边界

订单服务负责：

- 创建订单。
- 维护订单状态机。
- 记录订单明细。
- 管理取消、超时、完成。
- 处理支付结果对订单状态的影响。
- 发布订单领域事件。

订单服务不直接扣库存余额，也不直接调用第三方支付渠道。

## 库存服务边界

库存服务负责：

- 可售库存。
- 库存预占。
- 库存释放。
- 库存扣减确认。
- 仓库和 SKU 库存规则。
- 库存流水。

它对外提供预占、确认、释放等能力。

## 支付服务边界

支付服务负责：

- 创建支付单。
- 调用支付渠道。
- 处理支付回调。
- 维护支付状态。
- 支付幂等。
- 支付对账。

支付服务发布支付成功或失败事件，订单服务据此更新订单状态。

## 在 eMall 项目中怎么讲？

下单流程可以是：

```text
订单服务创建待确认订单
-> 调用库存服务预占库存
-> 创建待支付订单
-> 支付服务创建支付单
-> 支付成功事件
-> 订单服务更新为已支付
-> 库存服务确认扣减
```

失败时通过状态机和补偿任务恢复。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
订单、库存、支付应按数据所有权和业务职责划分。订单服务负责订单生命周期和状态机，库存服务负责库存预占、
释放和扣减，支付服务负责支付单、渠道回调和对账。

订单可以编排交易流程，但不能直接写库存表或支付表。三者通过 API、事件、幂等、状态机和补偿协作，实现最终一致。
```

- 支付负责支付单、回调和对账。
- 服务之间通过 API 和事件协作。

## 架构取舍与反驳补强

架构取舍补强标记：已完成

### 先给立场

- 回答「如何设计订单、库存、支付的边界？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 同步强校验：资金风险低，但会拉长交易链路并放大下游抖动。
- 异步对账补偿：主链路更稳，但需要对账、补偿、审计和人工兜底。
- 第三方结果为准：实现简单，但必须处理通知延迟、重复通知和账务差异。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果影响交易正确性或资金安全，优先选择可审计、可补偿、可对账的方案。

### 决策证据

- 状态机和幂等记录
- 对账差异趋势
- 补偿任务成功率
- 核心链路错误率

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

### 上线前检查

- 针对「如何设计订单、库存、支付的边界？」，先确认它影响的是正确性、稳定性、性能、安全还是成本。

### 灰度和回滚

- 先在测试环境和影子流量中验证，再做 1%、5%、25%、50%、100% 分阶段灰度。
- 每个阶段都设置自动暂停条件和人工回滚负责人。
- 回滚不是只回代码，还要确认配置、数据、缓存、消息和任务状态能一起回到安全状态。

### 监控和验收证据

- 幂等测试通过
- 补偿任务可重复执行
- 对账差异可解释
- 灰度期间核心链路无新增错误

## 规模化与成本治理补强

规模成本补强标记：已完成

### 规模化视角

- 回答「如何设计订单、库存、支付的边界？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
- 按订单峰值、支付通知峰值、对账批量窗口分别估算容量。
- 资金链路优先保证正确性和可审计，不用低成本换不可解释风险。

### 成本治理

- 控制索引数量、历史数据在线保留周期、慢查询和跨分片扫描。
- 冷热分离、归档和汇总表能显著降低在线库成本。

### 自动化和 owner

- 为关键指标建立看板、告警、owner 和 Runbook。
- 把经验沉淀成自动化检查、流水线门禁或平台能力。
