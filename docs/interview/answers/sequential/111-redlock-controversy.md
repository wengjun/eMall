## 题目

Redlock 争议是什么？

## 先给面试官的短答案

Redlock 是 Redis 作者提出的基于多个独立 Redis 节点的分布式锁算法，试图避免单 Redis 实例故障导致锁不可靠。
争议点在于网络分区、时钟漂移、客户端暂停和锁过期情况下，它是否能提供足够强的安全性。

工程上不要把 Redlock 当成强一致分布式事务。关键业务要用 fencing token、数据库约束、幂等和状态机兜底。

## Redlock 想解决什么？

单 Redis 锁如果 Redis 故障或主从切换，可能出现锁丢失。

Redlock 思路是：

- 部署多个独立 Redis 节点。
- 客户端向多数节点加锁。
- 多数成功且耗时小于 TTL 才认为加锁成功。
- 释放时删除多个节点上的锁。

它试图用多数派提高锁可用性和可靠性。

## 争议点一：时间假设

Redlock 依赖 TTL 和时间计算。

分布式环境中存在：

- 时钟漂移。
- GC 暂停。
- 网络延迟。
- 进程被挂起。

客户端以为锁还有效，不代表资源侧也应该接受它的写入。

## 争议点二：网络分区

网络分区时，不同客户端可能看到不同节点集合。

如果系统对时钟和网络延迟假设不成立，可能出现多个客户端都认为自己持有锁的情况。

这对强一致资源更新很危险。

## 争议点三：缺少 fencing token

即使锁过期后旧客户端恢复，它仍可能继续写共享资源。

更强方案是 fencing token：

- 每次获取锁得到递增 token。
- 资源端只接受更大的 token。
- 旧 token 写入被拒绝。

这样资源端能防止过期持锁者继续写。

## 如何正确看待 Redlock？

Redlock 可以用于低风险、可重试、短时间互斥场景。

但不要用于：

- 资金扣款。
- 强一致库存。
- 无法补偿的写入。
- 长事务。

关键业务正确性要落在资源端约束和幂等上。

## 在 eMall 项目中怎么讲？

缓存重建可以用 Redlock 或普通 Redis 锁降低并发重建。

优惠券领取不能只依赖 Redlock，仍要有数据库唯一键：

```text
user_id + coupon_id unique
```

即使锁异常，也不会重复领取。

并发题不能只回答 API 用法。生产系统要同时考虑线程安全、资源隔离、超时、拒绝、幂等和分布式多实例。
单机锁只能保护当前 JVM，不能保护整个集群；线程池满也不是小问题，而是容量和可用性风险。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

## 深度增强：面试高分表达

我会把并发问题分成三层：JMM 和锁保证单机正确性，线程池和隔离保证资源不被拖垮，
幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
Redlock 的争议在于它依赖多数 Redis 节点、TTL 和时间假设，但网络分区、GC 暂停、客户端挂起和时钟漂移
可能让旧客户端在锁失效后继续写。没有 fencing token 时，资源端无法识别过期持锁者。

所以我不会把 Redlock 当成强一致保证。低风险短时互斥可以用，关键业务必须用数据库约束、版本、
幂等、状态机或 fencing token 兜底。
```

## 回答评分点

高分答案应该覆盖：

- Redlock 基于多个 Redis 节点多数派。
- 争议来自时间、网络分区和暂停。
- fencing token 很重要。
- 不能当强一致事务。
- 关键业务要有资源端约束。

## 深度完善：面向 L6 的回答框架

围绕「Redlock 争议是什么？」，高分答案不能停在概念定义，而要把「线程安全、锁、CAS、线程池、隔离、超时和并发容量」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「并发和线程治理」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `order 创建、inventory 扣减、payment 回调、outbox relay 和异步补偿任务` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：线程池活跃数、队列长度、拒绝数、锁等待、超时率、重复请求数。
- 验证证据：并发单测、压测曲线、线程 dump、拒绝日志、幂等记录和容量评估。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
