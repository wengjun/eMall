## 题目

公共库升级如何保证向后兼容？

## 先给面试官的短答案

公共库升级要同时考虑源码兼容、二进制兼容和行为兼容。不能只看编译通过，
还要确认默认行为、序列化格式、错误码、超时、配置项没有破坏调用方。

安全做法是新增不删除、先 deprecated、保留适配层、增加契约测试、分批升级和灰度验证。

## 兼容类型

### 源码兼容

调用方重新编译仍然通过。

### 二进制兼容

调用方不重新编译，运行时仍然能链接旧方法。

### 行为兼容

代码还能运行不代表行为一样。比如默认超时从 1 秒改成 10 秒，编译没问题，但生产行为变了。

## 常见破坏性变更

- 删除 public 方法。
- 修改方法签名。
- 修改返回字段名。
- 修改 JSON 序列化结构。
- 删除枚举值。
- 改变默认配置。
- 改变异常类型。
- 改变重试、超时、熔断策略。

## 安全升级策略

### 新增 API，不直接改旧 API

旧方法保留：

```java
@Deprecated
OperationResult replayOutbox(String status, int limit);
```

新增：

```java
OperationResult replayOutbox(ReplayOutboxCommand command);
```

### deprecated 周期

先标记废弃，给调用方迁移时间，再删除。

### 适配层

旧 API 内部调用新 API，保持行为一致。

### 契约测试

对公共 API 的输入输出、错误码、序列化格式建立测试。

### 分批升级

先升级少量模块，观察通过后再全量升级。

## 在 eMall 项目中怎么讲？

如果升级 `common` 里的 `ApiResponse`：

- 不能随便改字段名。
- 不能删除 `code`、`message`。
- 新增 `traceId` 要保证老客户端能忽略。
- 错误码枚举不能随便重命名。

如果升级 Outbox 基础设施：

- 不能改变状态语义。
- 不能改变重试默认策略而不通知调用方。
- 不能让老模块发布失败。

## 深度增强：Java 17 落地示例

```java
import java.util.Objects;

record StableApiField(String name, String type, boolean required) {

    StableApiField {
        Objects.requireNonNull(name);
        Objects.requireNonNull(type);
        if (name.isBlank() || type.isBlank()) {
            throw new IllegalArgumentException("API field metadata must be explicit");
        }
    }

final class ApiCompatibilityPolicy {

    boolean canAddField(StableApiField field) {
        return !field.required();
```
