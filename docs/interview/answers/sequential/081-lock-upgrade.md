## 题目

偏向锁、轻量级锁、重量级锁是什么？

## 先给面试官的短答案

它们是 HotSpot 对 `synchronized` 的不同实现和优化形态，目标是在不同竞争程度下减少锁成本。
偏向锁面向几乎无竞争且总是同一线程进入的场景；轻量级锁面向短时间少量竞争；重量级锁面向竞争激烈、
线程需要阻塞唤醒的场景。

但 Java 17 中偏向锁已经废弃并默认不可用，面试时要说明版本差异，不能把旧版本细节当成当前默认行为。

## 为什么有这些锁形态？

`synchronized` 语义简单，但 JVM 不希望每次进入同步块都直接使用操作系统重量级互斥。

因为很多锁：

- 没有竞争。
- 只被同一个线程反复进入。
- 临界区很短。
- 竞争只持续很短时间。

所以 JVM 会根据竞争情况使用更轻量的方式。

## 偏向锁

偏向锁的思想是：如果一把锁总是被同一个线程使用，就偏向这个线程。

之后这个线程再次进入同步块时，可以减少 CAS 等同步成本。

适合：

- 单线程反复进入同步块。
- 几乎没有竞争。

风险：

- 一旦出现竞争，需要撤销偏向。
- 撤销本身有成本。

Java 17 中偏向锁已经废弃并默认不可用，所以生产讨论要以当前版本为准。

## 轻量级锁

轻量级锁适合竞争不激烈、临界区很短的场景。

它通常通过 CAS 尝试获取锁。

如果竞争线程很快释放锁，等待线程可以通过自旋等方式避免阻塞和唤醒。

优点：

- 避免频繁进入内核态阻塞。
- 短竞争场景效率高。

缺点：

- 竞争激烈时自旋会浪费 CPU。

## 重量级锁

重量级锁通常意味着竞争激烈，线程需要阻塞和唤醒。

当锁膨胀为重量级锁，等待线程会进入阻塞状态，由操作系统调度。

优点：

- 避免长时间自旋浪费 CPU。

缺点：

- 阻塞和唤醒成本较高。
- 上下文切换增加。
- P99 可能变差。

## 锁升级

旧版本常见说法是：

```text
无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁
```

锁通常会随着竞争加剧逐步膨胀。

现在面试要补一句：Java 17 中偏向锁不再是默认优化路径，理解思路即可，具体行为要看 JVM 版本。

## 工程重点

工程实践中不应该依赖某种锁优化来掩盖设计问题。

更重要的是：

- 控制锁粒度。
- 避免锁内 IO。
- 避免热点全局锁。
- 减少锁嵌套。
- 使用并发集合或分段设计。
- 用 JFR 和 `jstack` 验证锁竞争。

锁优化是 JVM 的事，锁设计是工程师的事。

## 在 eMall 项目中怎么讲？

如果库存扣减使用一个全局锁，即使 JVM 有锁优化，大促下也会膨胀成严重竞争。

更合理的是按商品、库存桶或分片拆分锁，并把数据库事务和远程调用移出本地锁。

面试中这样讲能体现你不会停留在 JVM 概念，而是知道如何落到业务设计。

## 深度增强：并发治理图

![Java 并发从线程安全到容量保护](../../assets/concurrency-governance.svg)

并发题不能只回答 API 用法。生产系统要同时考虑线程安全、资源隔离、超时、拒绝、幂等和分布式多实例。
单机锁只能保护当前 JVM，不能保护整个集群；线程池满也不是小问题，而是容量和可用性风险。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

## 深度增强：面试高分表达

我会把并发问题分成三层：JMM 和锁保证单机正确性，线程池和隔离保证资源不被拖垮，
幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
偏向锁、轻量级锁和重量级锁是 HotSpot 为 synchronized 在不同竞争程度下做的优化形态。
偏向锁面向无竞争且同一线程反复进入，轻量级锁通过 CAS 和短暂自旋优化低竞争场景，
重量级锁在竞争激烈时让线程阻塞等待，避免长期自旋浪费 CPU。

但 Java 17 中偏向锁已经废弃并默认不可用，所以回答时要说明版本差异。工程上更重要的是
减少锁竞争，而不是依赖 JVM 锁升级机制拯救不合理的全局锁设计。
```

## 回答评分点

高分答案应该覆盖：

- 三种锁面向不同竞争程度。
- 轻量级锁常与 CAS、自旋相关。
- 重量级锁涉及阻塞和唤醒。
- Java 17 中偏向锁已废弃。
- 能联系锁粒度和热点业务。

## 深度完善：面向 L6 的回答框架

围绕「偏向锁、轻量级锁、重量级锁是什么？」，高分答案不能停在概念定义，而要把「线程安全、锁、CAS、线程池、隔离、超时和并发容量」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「并发和线程治理」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `order 创建、inventory 扣减、payment 回调、outbox relay 和异步补偿任务` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：线程池活跃数、队列长度、拒绝数、锁等待、超时率、重复请求数。
- 验证证据：并发单测、压测曲线、线程 dump、拒绝日志、幂等记录和容量评估。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
