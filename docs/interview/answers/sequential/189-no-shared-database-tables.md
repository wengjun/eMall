## 题目

为什么服务之间不应该共享数据库表？

## 先给面试官的短答案

服务之间共享数据库表会破坏服务边界。多个服务直接读写同一张表，会导致业务规则分散、表结构难演进、故障责任不清、
事务边界混乱，最终形成共享数据库的分布式单体。

微服务可以共享数据语义，但不应该共享底层表的写权限。

## 共享表的问题

问题包括：

- 一个服务改表影响多个服务。
- 多个服务绕过彼此业务规则。
- 数据状态可能被非法修改。
- 难以做权限和审计。
- 无法独立发布和回滚。
- 数据库成为所有服务的耦合中心。
- 扩容和迁移困难。

表结构一旦成为公共接口，演进成本会非常高。

## 正确方式

更合理的方式：

- 数据拥有者提供 API。
- 数据拥有者发布领域事件。
- 消费方维护本地只读投影。
- 分析需求进入数仓。
- 跨服务查询通过聚合服务或 CQRS 读模型。

写入必须回到数据拥有者。

## 是否完全不能读？

生产系统也有例外，比如：

- 数仓离线同步。
- 只读副本给报表使用。
- 临时迁移阶段的双读。

但这些应受治理，不应成为业务服务之间的常规调用方式。

## 在 eMall 项目中怎么讲？

支付服务不能直接更新订单表为已支付。它应发布支付成功事件，订单服务消费事件后按订单状态机更新订单。

这样订单状态规则仍由订单服务掌控。

## 深度增强：生产边界

日志越多不代表越好。核心链路要控制日志成本、采样率、脱敏和索引字段。告警也不能只看机器指标，
还要看下单成功率、支付成功率、库存失败率、Outbox 积压和用户投诉。

## 深度增强：面试高分表达

我会把可观测性讲成故障闭环：指标发现异常，Trace 定位慢在哪里，日志解释发生了什么，
告警和 Runbook 指导恢复。配置变更也要有版本、审批、灰度、审计和回滚，避免配置事故变成全站事故。

## 专家级完整回答

```text
服务之间不应该共享数据库表，因为表共享会让服务通过数据库耦合，绕过 API 和业务规则。
结果是表结构无法独立演进、状态流转不可控、发布回滚困难，系统变成共享数据库的分布式单体。

正确方式是数据拥有者提供 API 和领域事件，其他服务通过只读投影、聚合查询或数仓使用数据。写入必须回到拥有者。
```

## 回答评分点

高分答案应该覆盖：

- 共享表破坏服务边界。
- 业务规则会被绕过。
- 表结构会变成隐式公共契约。
- 正确方式是 API、事件和只读投影。
- 写权限必须属于数据拥有者。

## 架构取舍与反驳补强

### 先给立场

- 回答「为什么服务之间不应该共享数据库表？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 简单方案：上线快、成本低，但容量和故障边界有限。
- 平台化方案：复用强、治理强，但建设成本和组织协调更高。
- 外部托管方案：交付快，但可控性、成本和供应商风险需要评估。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果约束不明确，先补齐规模、延迟、可用性、一致性、成本和团队能力，再做选择。

### 决策证据

- 业务指标
- 稳定性指标
- 成本指标
- 灰度和回滚记录

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

### 上线前检查

- 针对「为什么服务之间不应该共享数据库表？」，先确认它影响的是正确性、稳定性、性能、安全还是成本。
- 确认执行计划、索引命中、锁等待、回滚脚本和数据备份。
- 灰度期间观察慢 SQL、连接池等待、主从延迟和分片热点。

### 灰度和回滚

- 先在测试环境和影子流量中验证，再做 1%、5%、25%、50%、100% 分阶段灰度。
- 每个阶段都设置自动暂停条件和人工回滚负责人。
- 回滚不是只回代码，还要确认配置、数据、缓存、消息和任务状态能一起回到安全状态。

## 规模化与成本治理补强

规模成本补强标记：已完成

### 规模化视角

- 回答「为什么服务之间不应该共享数据库表？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
- 按读写比例、数据增长、热点分布、索引大小和归档周期估算数据库容量。
- 数据库扩展要优先减少热点和慢查询，再考虑分片。

### 成本治理

- 控制索引数量、历史数据在线保留周期、慢查询和跨分片扫描。
- 冷热分离、归档和汇总表能显著降低在线库成本。

### 自动化和 owner

- 为关键指标建立看板、告警、owner 和 Runbook。
- 把经验沉淀成自动化检查、流水线门禁或平台能力。

### 面试表达

我会补一句：方案能跑只是第一步，大规模下还要回答容量怎么估、成本怎么控、故障谁负责。
