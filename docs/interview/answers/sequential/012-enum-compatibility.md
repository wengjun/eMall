## 题目

枚举状态扩展时如何保证兼容？

## 先给面试官的短答案

枚举一旦进入数据库、API、MQ 或日志指标，就不再只是代码内部实现，而是系统契约。
扩展枚举时要保证老代码、老客户端、老消费者、报表和监控都能处理新值。

我的做法是：

- 新增不删除，避免重命名已有值。
- 读方先兼容未知值，写方再灰度写入新值。
- API 和 MQ 事件增加契约测试。
- 报表、监控、告警和补偿任务同步适配。
- 必要时提供 `UNKNOWN` 或默认处理分支。

## 从零基础理解：为什么新增枚举也危险？

假设订单原来只有：

```text
CREATED
PAID
CANCELLED
```

后来新增：

```text
PENDING_RETRY
```

如果老代码中有：

```java
switch (status) {
    case CREATED -> ...
    case PAID -> ...
    case CANCELLED -> ...
}
```

老服务遇到 `PENDING_RETRY` 可能无法处理。前端也可能显示空白，报表也可能漏统计。

所以“新增枚举”是一个兼容性变更。

## 兼容性风险有哪些？

### API 客户端不认识新状态

移动端、Web 前端、第三方开放平台可能写死状态列表。
收到新状态后可能：

- 解析失败。
- 页面显示异常。
- 按默认失败处理。
- 无法继续操作。

### MQ 消费者不认识新事件状态

如果消息中有新枚举值，老消费者可能反序列化失败或进入异常分支。

### 数据库和报表漏统计

报表 SQL 可能只统计旧状态：

```sql
where status in ('CREATED', 'PAID', 'CANCELLED')
```

新增状态后，数据被漏掉。

### 运维和补偿任务不处理新状态

例如新增 `PENDING_RETRY`，但补偿任务没扫描它，订单就会永久卡住。

## 推荐发布流程

### 第一步：读方先兼容

所有读取状态的地方先能识别或容忍新状态。

包括：

- 前端。
- 下游服务。
- MQ 消费者。
- 报表。
- 监控。
- 补偿任务。
- 运维后台。

### 第二步：写方再产生新状态

等读方兼容后，再让核心服务开始写入新状态。

例如订单服务先发布能处理 `PENDING_RETRY` 的版本，然后再打开写入该状态的开关。

### 第三步：灰度和观测

不要全量立刻写新状态。先小流量观察：

- 新状态数量是否符合预期。
- 老错误是否上升。
- 消费者是否反序列化失败。
- 报表和监控是否正常。
- 补偿任务是否能处理。

## 技术手段

### 提供 `UNKNOWN`

对外反序列化可以兜底：

```java
public enum ExternalOrderStatus {
    CREATED,
    PAID,
    CANCELLED,
    UNKNOWN
}
```

当客户端不认识新值时，映射到 `UNKNOWN`，至少不崩溃。

### switch 保留 default

内部核心状态机不一定要 default，因为希望编译器帮忙发现遗漏。
但对外兼容解析可以有 default。

原则：

- 内部强约束：尽量穷尽处理。
- 外部输入：必须容忍未知值。

### 契约测试

对 API 和 MQ 事件建立契约测试，确保新增状态不会破坏消费者。

### 版本化事件

MQ 事件可以带 schemaVersion：

```json
{
  "eventType": "ORDER_STATUS_CHANGED",
  "schemaVersion": 2,
  "status": "PENDING_RETRY"
}
```

消费者按版本处理。

## 在 eMall 项目中怎么讲？

例如给订单新增 `PENDING_RETRY`：

1. 订单查询 API 先能返回该状态。
2. 前端展示为“处理中”。
3. 补偿任务扫描该状态。
4. 运营后台能筛选该状态。
5. 指标增加该状态数量。
6. MQ 消费者能识别事件 payload。
7. 最后订单服务在库存预占失败时写入 `PENDING_RETRY`。

专家级表达：

```text
我会把枚举扩展当成契约变更处理，而不是简单改 enum。
尤其订单、支付、库存这些状态会进入数据库、API、MQ、报表和运维系统，
必须先让读方兼容，再让写方灰度产生。
```

## 常见追问

### 能不能重命名枚举？

不要直接重命名已持久化或已对外暴露的枚举。更安全的方式：

- 新增新值。
- 同时兼容旧值和新值。
- 数据迁移。
- 观察稳定后废弃旧值。

### 删除枚举值怎么做？

删除前先确认：

- 数据库是否还有旧值。
- 老消息是否还会重放。
- 老客户端是否还会传旧值。
- 报表是否依赖旧值。

一般先 deprecated，再长期兼容，最后清理。

## 深度增强：Java 17 落地示例

```java
import java.util.Objects;

record StableApiField(String name, String type, boolean required) {

    StableApiField {
        Objects.requireNonNull(name);
        Objects.requireNonNull(type);
        if (name.isBlank() || type.isBlank()) {
            throw new IllegalArgumentException("API field metadata must be explicit");
        }
    }
}

final class ApiCompatibilityPolicy {

    boolean canAddField(StableApiField field) {
        return !field.required();
    }
}
```

这段代码体现 Java 17 在工程建模中的价值：用 `record` 表达不可变数据，用构造校验保护边界，
用小的策略类表达兼容规则。面试中要把语法能力和 API 演进、错误预防、团队协作联系起来。

## 深度增强：生产边界

语言特性不是越新越好。核心原则是可读、可测、可维护、可兼容。任何语法选择都要能让代码意图更清晰，
而不是为了炫技。公共 API、金额、时间、状态、异常和 DTO 都要有稳定约束，避免线上数据被随意破坏。

## 回答评分点

高分答案应该覆盖：

- 能意识到枚举是系统契约。
- 能说出 API、MQ、DB、报表、监控风险。
- 能提出读方先兼容、写方后灰度。
- 能说明 `UNKNOWN`、default、契约测试、事件版本。

围绕「枚举状态扩展时如何保证兼容？」，高分答案不能停在概念定义，而要把「语言特性、建模边界、兼容性和团队编码规范」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「Java 语言和工程基础」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `common、order、inventory、payment 的 DTO、值对象、异常和公共 API` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：代码评审问题数、缺陷逃逸率、兼容性测试结果、静态检查违规数。
- 验证证据：代码规范、单元测试、契约测试、兼容性用例和重构前后缺陷数据。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
