## 题目

如何做容量估算？

## 先给面试官的短答案

容量估算要从业务目标倒推技术资源：用户规模、DAU、峰值系数、请求路径、读写比例、响应时间、数据库容量、缓存容量、
MQ 吞吐、机器资源和冗余系数。

估算只是起点，必须通过压测和线上监控校准。

## 估算步骤

步骤：

- 明确 DAU 和峰值并发。
- 估算核心接口 QPS。
- 拆解每个接口的下游调用。
- 估算数据库读写 QPS。
- 估算缓存命中率和容量。
- 估算 MQ 消息量。
- 估算机器 CPU、内存和网络。
- 加上冗余和故障切换容量。

核心链路要单独估算。

## 峰值系数

电商流量有明显峰值：

- 大促。
- 秒杀。
- 发券。
- 晚高峰。
- 推送触达。

不能用日均流量直接设计容量。要按峰值小时、峰值分钟甚至峰值秒估算。

## 冗余设计

容量要考虑：

- N+1 冗余。
- 单可用区故障。
- 下游抖动。
- 发布期间双版本。
- 缓存失效。
- MQ 堆积追赶。

生产容量不能刚好够用。

## 在 eMall 项目中怎么讲？

如果 DAU 一亿，大促峰值下单 QPS 可能远高于日均。

要估算下单一次会访问订单库、库存库、价格服务、优惠服务和 MQ，并分别计算容量和冗余。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
容量估算要从业务流量倒推系统资源。先估算 DAU、峰值系数和核心接口 QPS，再拆解每个接口的数据库、缓存、
MQ、下游调用和机器资源消耗，最后加上冗余、故障切换和发布容量。

估算必须用压测验证，并通过线上指标持续校准，不能只停留在纸面计算。
```

## 回答评分点

高分答案应该覆盖：

- 从业务目标倒推技术容量。
- 日均流量不能代表峰值。
- 要拆解到数据库、缓存、MQ 和下游。
- 要考虑冗余和故障场景。
- 压测和监控用于校准。

## 深度完善：面向 L6 的回答框架

围绕「如何做容量估算？」，高分答案不能停在概念定义，而要把「限流、熔断、降级、超时、重试、热点隔离、平滑恢复和过载保护」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「高并发和稳定性」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `traffic、reliability、gateway、flash-sale、order、inventory 的稳定性治理` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：限流命中率、熔断状态、降级比例、P99、错误率、队列积压和恢复斜率。
- 验证证据：压测报告、故障演练、灰度曲线、熔断状态日志、Runbook 和大促复盘。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
