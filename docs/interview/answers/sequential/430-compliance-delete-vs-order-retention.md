## 题目

如何处理数据合规删除和历史订单保留？

## 先给面试官的短答案

数据合规删除和历史订单保留要区分个人可删除信息与法定或业务必须保留的交易记录。用户可以要求
删除或匿名化个人信息，但历史订单可能因财务、税务、审计、售后和风控需要保留。

常见做法是去标识化、匿名化、字段清除和访问限制，而不是简单物理删除所有订单。

## 数据分类

分类：

- 账号资料。
- 联系方式。
- 地址。
- 订单交易记录。
- 支付流水。
- 发票和税务记录。
- 客服和售后记录。

不同数据有不同保留要求。

## 删除策略

策略：

- 可删除数据直接删除。
- 敏感字段匿名化。
- 历史订单保留必要字段。
- 地址和手机号脱敏或清除。
- 支付和税务记录按法规保留。
- 删除请求全链路审计。

要和法务、合规要求对齐。

## 系统难点

难点：

- 数据分布在多个服务。
- 缓存和搜索索引也要清理。
- 数据仓库和备份存在延迟。
- 删除后仍要支持审计。
- 不能破坏财务对账。

删除是流程，不是单表 SQL。

## 在 eMall 项目中怎么讲？

eMall 用户注销后，可以清除或匿名化手机号、地址、昵称等个人信息。

历史订单保留订单号、金额、商品摘要和财务必要字段，但把收货人、手机号和详细地址脱敏或删除。
搜索、缓存、数据仓库也要接收删除事件。

record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
合规删除要先做数据分类。个人资料和联系方式可以删除或匿名化，但历史订单、支付流水、发票和审计
记录可能需要按法规和业务要求保留。

生产上通常采用去标识化、字段清除、访问限制和删除事件传播，覆盖业务库、缓存、搜索、数据仓库和
备份。不能简单删除订单，否则会破坏对账、售后、税务和审计。
```

## 回答评分点

高分答案应该覆盖：

- 区分个人信息和交易记录。
- 历史订单可能必须保留。
- 删除、匿名化和去标识化。
- 覆盖缓存、搜索和数仓。
- 不能破坏对账和审计。

## 深度完善：面向 L6 的回答框架

围绕「如何处理数据合规删除和历史订单保留？」，高分答案不能停在概念定义，而要把「认证、授权、签名、防重放、加密、脱敏、审计、风控和数据保留」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「安全、身份和合规」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `identity、risk、openapi、operations、payment 的身份、权限、签名和审计` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：签名失败率、权限拒绝数、风控命中率、敏感日志数、密钥轮换成功率。
- 验证证据：审计日志、权限矩阵、密钥轮换记录、脱敏测试、安全扫描和合规策略。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
