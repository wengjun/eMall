## 题目

账号冻结对下单链路有什么影响？

## 先给面试官的短答案

账号冻结会影响登录、下单、支付、优惠券使用、售后和履约变更。下单链路必须在关键节点校验账号
状态，不能只在登录时校验一次。

账号状态变更还要考虑缓存失效、正在进行的订单和用户提示。

## 校验点

校验点：

- 登录时。
- 创建订单前。
- 使用优惠券前。
- 发起支付前。
- 支付回调处理时。
- 申请退款和售后时。

冻结后不应允许继续发起高风险交易。

## 缓存问题

账号状态常被缓存。

冻结账号后要：

- 删除用户状态缓存。
- 提高 token version。
- 使旧 token 失效。
- 通知相关服务。
- 对关键接口回源校验。

否则冻结可能延迟生效。

## 已有订单

要区分：

- 已支付订单是否继续履约。
- 未支付订单是否关闭。
- 售后是否允许。
- 风控冻结是否需要人工审核。

不同冻结原因策略不同。

## 在 eMall 项目中怎么讲？

eMall 用户因盗号风险被冻结后，下单接口和支付接口都应拒绝新交易，并返回明确提示。

如果用户已有已支付订单，履约是否暂停应由风控策略决定，不能简单删除订单或破坏支付状态。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
账号冻结不是只影响登录，它会影响下单、支付、优惠券、售后和履约。系统要在关键交易节点校验账号
状态，并让冻结状态及时失效缓存和 token。

已有订单要按冻结原因区分处理。未支付订单可以关闭，已支付订单是否继续履约或人工审核要有风控
策略，不能破坏订单状态机。
```

## 回答评分点

高分答案应该覆盖：

- 冻结影响多个交易节点。
- 不能只登录时校验。
- 缓存和 token 要失效。
- 已有订单要按状态处理。
- 风控策略要可解释。

## 深度完善：面向 L6 的回答框架

围绕「账号冻结对下单链路有什么影响？」，高分答案不能停在概念定义，而要把「认证、授权、签名、防重放、加密、脱敏、审计、风控和数据保留」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「安全、身份和合规」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `identity、risk、openapi、operations、payment 的身份、权限、签名和审计` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：签名失败率、权限拒绝数、风控命中率、敏感日志数、密钥轮换成功率。
- 验证证据：审计日志、权限矩阵、密钥轮换记录、脱敏测试、安全扫描和合规策略。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
