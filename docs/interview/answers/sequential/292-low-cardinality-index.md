## 题目

选择性低的字段是否适合建索引？

## 先给面试官的短答案

选择性低的字段单独建索引通常价值不高，因为过滤不掉多少数据，优化器可能直接全表扫描。
但它可以作为联合索引的一部分，配合高选择性字段、排序字段或覆盖索引服务特定查询。

是否适合建索引要看查询模式和数据分布，不是绝对不能建。

## 什么是选择性？

选择性可以理解为字段区分数据的能力。

例如：

- 用户 ID 选择性通常高。
- 性别选择性通常低。
- 订单状态选择性中等，取决于分布。

选择性越高，索引过滤效果越好。

## 单列低选择性索引问题

例如 `status` 只有少量枚举值。

如果 `PAID` 占 80% 数据，走 `status` 索引再回表可能比全表扫描更慢。

因此优化器可能不用该索引。

## 什么时候有用？

可能有用：

- 和高选择性字段组成联合索引。
- 用于覆盖索引。
- 某些值非常稀有。
- 配合排序和分页。
- 分区或归档后数据量变小。

关键是看实际查询收益。

## 在 eMall 项目中怎么讲？

订单 `status` 单独建索引不一定有效。

但 `(merchant_id, status, created_at)` 对商家后台按状态和时间查询订单很有价值。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 深度增强：面试高分表达

我会从访问模式回答数据题：谁查、按什么条件查、QPS 多少、数据量多大、是否强一致、是否需要分页和排序。
然后再决定索引、分片、缓存、读写分离和归档策略。

## 专家级完整回答

```text
选择性低的字段单独建索引通常收益有限，因为过滤比例低，可能产生大量回表。
但它可以作为联合索引的一部分，配合商家、用户、时间等字段，服务明确查询场景。

所以判断索引价值要看数据分布、查询频率、返回行数和是否覆盖查询，而不是简单按字段是否枚举决定。
```

## 回答评分点

高分答案应该覆盖：

- 选择性是字段区分数据能力。
- 低选择性单列索引通常收益低。
- 联合索引中仍可能有价值。
- 数据分布会影响判断。
- 要结合执行计划验证。

## 深度完善：面向 L6 的回答框架

围绕「选择性低的字段是否适合建索引？」，高分答案不能停在概念定义，而要把「索引、事务、锁、MVCC、分库分表、冷热分层、备份恢复和在线变更」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「数据库和数据建模」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `order、payment、inventory、product 的核心表、唯一约束、索引和归档表` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：慢 SQL、锁等待、死锁数、主从延迟、连接池使用率、归档任务耗时。
- 验证证据：EXPLAIN、慢 SQL 日志、死锁日志、迁移方案、备份恢复演练和容量预测。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
