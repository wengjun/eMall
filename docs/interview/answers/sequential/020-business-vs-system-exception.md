## 题目

业务异常和系统异常应该如何区分？

## 先给面试官的短答案

业务异常表示系统正常运行，但用户请求不满足业务规则；系统异常表示代码、依赖或基础设施出现故障。

例如库存不足、订单状态不允许取消是业务异常；数据库连接失败、Redis 超时、空指针是系统异常。

区分它们的价值是：

- 业务异常返回明确错误码，通常不报警。
- 系统异常要记录日志、打指标、必要时告警。
- 下游异常要考虑重试、熔断、降级和补偿。

## 从零基础理解

用户买一件商品，但库存没有了：

```text
系统正常，只是业务不能满足请求。
```

这是业务异常。

如果数据库宕机，订单无法保存：

```text
系统本身出问题。
```

这是系统异常。

## 业务异常例子

电商系统中常见业务异常：

- 用户不存在。
- 用户被冻结。
- 商品已下架。
- 库存不足。
- 优惠券不可用。
- 订单不存在。
- 已支付订单不能直接取消。
- 支付金额不一致。
- 重复 requestId 对应不同业务参数。

这些异常通常是请求和业务规则冲突，不代表系统故障。

## 系统异常例子

系统异常包括：

- 数据库连接失败。
- SQL 语法错误。
- Redis 不可用。
- Kafka 发送失败。
- HTTP 客户端连接池耗尽。
- 空指针。
- 配置缺失。
- JSON 反序列化失败。

这些需要研发或运维介入。

## 下游异常怎么归类？

下游异常是系统异常的一类，但处理方式更细。

例如库存服务超时：

- 对订单来说，下游不可用。
- 对用户来说，可能返回“系统繁忙”。
- 对系统来说，需要熔断、重试、补偿或降级。

如果库存预占失败是因为库存不足，这是业务异常。
如果库存服务超时，这是下游系统异常。

## 响应处理差异

业务异常：

```json
{
  "success": false,
  "code": "INSUFFICIENT_STOCK",
  "message": "Insufficient stock"
}
```

系统异常：

```json
{
  "success": false,
  "code": "INTERNAL_ERROR",
  "message": "System is temporarily unavailable",
  "traceId": "..."
}
```

系统异常不能把堆栈直接返回给用户。

## 监控处理差异

业务异常通常不需要高优先级报警，但要作为业务指标统计：

- 库存不足率。
- 支付金额不匹配次数。
- 订单状态冲突次数。

系统异常需要技术指标：

- 5xx 错误率。
- 下游超时率。
- 数据库连接失败数。
- Kafka 发送失败数。

## 在 eMall 项目中怎么讲？

在 eMall 中：

- `BusinessException(ErrorCode.NOT_FOUND)` 表示资源不存在。
- `BusinessException(ErrorCode.CONFLICT)` 表示业务状态冲突。
- `DOWNSTREAM_UNAVAILABLE` 表示下游不可用。
- `INTERNAL_ERROR` 表示未预期系统错误。

统一异常处理层将异常转为响应，避免每个 Controller 重复处理。

## 常见追问

### 库存不足要不要报警？

通常不需要作为系统故障报警，但要监控趋势。如果库存不足率异常升高，可能说明库存同步或活动配置有问题。

### 支付金额不一致是业务异常还是安全风险？

它首先是业务冲突，必须拒绝；同时也可能是安全或渠道异常信号，应该记录审计并进入风控或对账分析。

### 系统异常能不能包装成业务异常？

不要。否则监控会误判系统正常，真实故障被隐藏。

import java.util.Objects;

    StableApiField {

## 专家级完整回答

```text
业务异常表示系统按预期拒绝了不合法业务请求，例如库存不足、订单状态冲突、支付金额不一致。
系统异常表示系统自身或依赖出现问题，例如数据库连接失败、Redis 超时、Kafka 发送失败。

区分二者是为了正确响应、监控和恢复。业务异常返回稳定错误码，通常不触发事故告警；
系统异常要记录 traceId、错误日志、技术指标，并根据情况触发告警。
下游异常还要结合超时、重试、熔断、降级和补偿处理。
```

- 能处理下游异常这个灰度地带。

### 可选方案对比

- 保守写法：团队理解成本低，但模板代码和误用风险可能更高。
- 现代 Java17 写法：表达力强，但需要规范约束避免炫技。

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果约束不明确，先补齐规模、延迟、可用性、一致性、成本和团队能力，再做选择。

- 回答「业务异常和系统异常应该如何区分？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。

- 用单位成本看问题，例如单请求成本、单订单成本、单消息成本和单 GB 存储成本。
- 先优化浪费最高的环节，而不是平均用力。

- 为关键指标建立看板、告警、owner 和 Runbook。
- 把经验沉淀成自动化检查、流水线门禁或平台能力。
