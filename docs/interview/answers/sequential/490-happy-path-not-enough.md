## 题目

为什么只测 happy path 不够？

## 先给面试官的短答案

happy path 只验证一切正常时系统能工作，但生产故障大多发生在异常输入、并发冲突、依赖超时、重复请求、乱序消息、
资源不足和部分失败时。只测正常路径会让系统在真实复杂场景下缺少防御能力。

## 什么是 happy path

happy path 是最顺利的业务路径。例如用户有库存、价格正常、优惠可用、支付成功、消息投递成功，订单顺利完成。
它应该被测试，但它只能证明系统在理想条件下可用。

## 还必须覆盖什么

异常输入要测，例如金额为负、优惠券过期、SKU 不存在、用户被冻结。

边界条件要测，例如库存为 0、库存为 1、订单金额刚好等于阈值、活动开始和结束瞬间。

并发场景要测，例如多个用户抢同一个 SKU、重复点击支付、重复提交订单。

部分失败要测，例如订单已创建但消息发送失败、支付成功但回调超时、库存预占成功但订单落库失败。

下游异常要测，例如 Redis 超时、数据库唯一键冲突、Kafka 暂不可用、第三方支付返回未知状态。

## 测试的真实目标

测试不是证明代码“在顺利时能跑”，而是证明系统在不顺利时不会造成资金损失、库存错误、状态污染和不可恢复故障。

高质量测试一定包含失败路径、补偿路径和恢复路径。

## 在 eMall 项目中怎么讲？

eMall 的核心链路要测重复下单、库存不足、支付重复回调、优惠失效、Outbox 失败重试和 MQ 重复消费。
这些场景比单纯下单成功更能体现系统是否生产可用。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 专家级完整回答

```text
只测 happy path 会给团队一种系统已经稳定的错觉。

真实生产环境中，问题往往来自异常输入、并发、重复请求、下游超时、消息乱序和部分失败。
这些场景不覆盖，系统就无法证明自己具备幂等、补偿、容错和恢复能力。

对电商系统来说，我更关注失败时是否不多扣钱、不超卖、不丢消息、不产生错误状态。
所以测试必须覆盖正常路径、异常路径、边界路径和恢复路径。
```

## 架构取舍与反驳补强

架构取舍补强标记：已完成

### 先给立场

- 回答「为什么只测 happy path 不够？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 简单方案：上线快、成本低，但容量和故障边界有限。
- 平台化方案：复用强、治理强，但建设成本和组织协调更高。
- 外部托管方案：交付快，但可控性、成本和供应商风险需要评估。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果约束不明确，先补齐规模、延迟、可用性、一致性、成本和团队能力，再做选择。

### 决策证据

- 业务指标
- 稳定性指标
- 成本指标
- 灰度和回滚记录

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

- 针对「为什么只测 happy path 不够？」，先确认它影响的是正确性、稳定性、性能、安全还是成本。

### 灰度和回滚

- 先在测试环境和影子流量中验证，再做 1%、5%、25%、50%、100% 分阶段灰度。
- 每个阶段都设置自动暂停条件和人工回滚负责人。
- 回滚不是只回代码，还要确认配置、数据、缓存、消息和任务状态能一起回到安全状态。

## 规模化与成本治理补强

规模成本补强标记：已完成

### 规模化视角

- 回答「为什么只测 happy path 不够？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
- 按模块数量、流水线耗时、测试稳定性和环境成本估算工程效率。
- 测试越多越好是误区，关键是风险覆盖和反馈速度。

### 成本治理

- 用单位成本看问题，例如单请求成本、单订单成本、单消息成本和单 GB 存储成本。
- 先优化浪费最高的环节，而不是平均用力。

### 自动化和 owner

- 为关键指标建立看板、告警、owner 和 Runbook。
- 把经验沉淀成自动化检查、流水线门禁或平台能力。

### 面试表达

我会补一句：方案能跑只是第一步，大规模下还要回答容量怎么估、成本怎么控、故障谁负责。
这能体现我不是只会实现单点功能，而是能长期运营一个高并发业务系统。
