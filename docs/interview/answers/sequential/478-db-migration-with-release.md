## 题目

数据库迁移如何配合应用发布？

## 先给面试官的短答案

数据库迁移要和应用发布解耦，采用多阶段、可回滚、可观测的方式。一般先做兼容性 schema 扩展，再发布支持新旧结构的应用，
然后灰度数据迁移和双写校验，确认稳定后切读，最后删除旧结构。不能在核心链路里一次性完成不可逆变更。

## 推荐阶段

第一阶段是扩展。新增表、字段、索引或兼容枚举，不删除旧字段，不修改旧字段语义，保证旧版本应用仍能运行。

第二阶段是应用兼容。新版本应用能同时读取旧数据和新数据，必要时双写旧字段和新字段，并记录校验指标。

第三阶段是数据迁移。通过后台任务分批迁移历史数据，控制批量大小、锁等待、慢 SQL 和复制延迟。

第四阶段是切读。确认新结构数据完整后，让应用优先读取新结构，同时保留旧结构回滚能力。

第五阶段是收缩。等待所有旧版本下线、回滚窗口结束、数据校验通过后，再删除旧字段、旧索引或旧表。

## 迁移风险控制

迁移任务要支持断点续跑、幂等、限速、暂停和回滚。对大表加索引要使用在线 DDL 或影子表方案，
避免长事务和全表锁影响交易链路。

同时要有数据校验。比如迁移订单金额明细时，要校验订单总金额、优惠金额、支付金额和退款金额是否一致。

## 在 eMall 项目中怎么讲？

eMall 可以把数据库迁移作为发布计划的一部分，由 `release`、`operations` 和业务服务共同控制。
例如订单表拆分金额明细时，先新增明细表，新旧版本双写，再迁移历史订单，校验通过后切读，最后删除旧字段。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
数据库迁移和应用发布必须按兼容性设计，而不是一次性强绑定。

我的原则是先扩展再切换，先兼容再删除。具体流程是：新增兼容 schema，发布能读写新旧结构的应用，
分批迁移历史数据，做双写和对账校验，灰度切换读路径，最后在回滚窗口结束后删除旧结构。

迁移过程中要控制数据库影响，包括锁、慢 SQL、主从延迟、连接池占用和批处理速度。
对订单、支付、库存这类核心表，迁移任务必须幂等、可暂停、可恢复，并且有业务对账。
```

## 架构取舍与反驳补强

架构取舍补强标记：已完成

### 先给立场

- 回答「数据库迁移如何配合应用发布？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 快速回滚：止血快，但可能掩盖数据兼容和状态迁移问题。
- 灰度发布：风险可控，但需要指标、分流、回滚和版本兼容能力。
- 自动恢复：降低人工成本，但错误策略可能扩大事故影响。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果约束不明确，先补齐规模、延迟、可用性、一致性、成本和团队能力，再做选择。

### 决策证据

- 业务指标
- 稳定性指标
- 成本指标
- 灰度和回滚记录

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

- 回滚不是只回代码，还要确认配置、数据、缓存、消息和任务状态能一起回到安全状态。

## 规模化与成本治理补强

规模成本补强标记：已完成

### 规模化视角

- 回答「数据库迁移如何配合应用发布？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
- 按读写比例、数据增长、热点分布、索引大小和归档周期估算数据库容量。
- 数据库扩展要优先减少热点和慢查询，再考虑分片。

### 成本治理

- 控制索引数量、历史数据在线保留周期、慢查询和跨分片扫描。
- 冷热分离、归档和汇总表能显著降低在线库成本。

### 自动化和 owner

- 必须有自动告警、自动摘流、自动回滚或一键 Runbook。
- 每个告警要有 owner、SLO、升级路径和复盘模板。

### 面试表达

我会补一句：方案能跑只是第一步，大规模下还要回答容量怎么估、成本怎么控、故障谁负责。
这能体现我不是只会实现单点功能，而是能长期运营一个高并发业务系统。
