## 题目

如何分析接口 P99 突然升高？

## 先给面试官的短答案

P99 突然升高说明最慢的 1% 请求变慢了，不一定是整体都慢。排查要沿着请求链路分层看：
入口流量、应用线程池、GC、数据库、缓存、消息队列、下游 HTTP、锁竞争、发布变更和机器资源。

最有效的方法是用指标发现范围，用链路追踪定位慢在哪一段，用日志和线程 dump/JFR 确认根因。

## 先理解 P99

P99 表示 99% 的请求耗时小于等于这个值，剩下 1% 比它更慢。

平均耗时可能正常，但 P99 很高。这说明少量请求遇到了排队、锁等待、慢 SQL、GC、下游慢或资源竞争。

电商系统非常关注 P99，因为用户感知的是慢请求，不是平均值。

## 第一步：确认影响范围

先回答几个问题：

- 是一个接口慢，还是所有接口慢？
- 是一个实例慢，还是所有实例慢？
- 是一个机房慢，还是所有机房慢？
- 是某类用户慢，还是所有用户慢？
- 是新版本实例慢，还是旧版本也慢？

范围判断很重要。

如果只有一个实例慢，可能是机器、GC、线程池或本地缓存问题。
如果所有实例都慢，可能是下游、数据库、流量或发布问题。

## 第二步：看入口流量和错误率

要看：

- QPS 是否上涨。
- 请求体是否变大。
- 错误率是否上涨。
- 超时是否上涨。
- 重试是否上涨。
- 限流是否触发。

如果流量突然上涨，P99 可能来自排队。
如果错误率同步上涨，可能是下游故障或超时重试。

## 第三步：看链路追踪

链路追踪能把一次请求拆成多个 span。

例如订单创建链路：

```text
gateway -> order -> inventory -> pricing -> promotion -> payment
```

如果慢 span 在 inventory，优先查库存。
如果每一段都不慢，但 order 总耗时高，要查线程池排队、序列化、锁等待或本地逻辑。

链路追踪的价值是把“接口慢”变成“哪一段慢”。

## 第四步：看应用内部资源

应用内部常见原因：

- 线程池队列堆积。
- 数据库连接池耗尽。
- HTTP 连接池耗尽。
- 本地锁竞争。
- 缓存击穿。
- 大对象分配。
- GC 暂停。
- 日志同步写入变慢。

要看这些指标：

- active threads。
- queue size。
- rejected count。
- connection pool active。
- GC pause。
- allocation rate。
- CPU load。
- heap usage。

P99 本质上经常是“排队时间”变长，而不只是“执行时间”变长。

## 第五步：看数据库和缓存

数据库问题常见表现：

- 慢 SQL 增多。
- 锁等待增多。
- 连接池获取慢。
- 索引失效。
- 热点行竞争。
- 主从延迟。

缓存问题常见表现：

- 命中率下降。
- 大 key。
- 热 key。
- Redis 慢查询。
- 缓存击穿。
- 缓存雪崩。

电商核心链路中，库存热点行和优惠券热点 key 很容易放大 P99。

## 第六步：看 GC 和 JVM

GC 会直接影响 P99。

要看：

- GC pause 是否与 P99 时间点重合。
- old gen 是否上涨。
- allocation rate 是否异常。
- direct memory 是否异常。
- 线程数是否异常。
- JFR 是否显示大量对象分配或锁等待。

如果 P99 峰值和 GC pause 对齐，根因可能在内存分配或堆配置。

## 第七步：看发布和配置变更

P99 突然升高经常和变更相关。

要查：

- 最近是否发布新版本。
- 是否调整 JVM 参数。
- 是否调整线程池参数。
- 是否调整超时和重试。
- 是否上线新营销规则。
- 是否修改 SQL 或索引。
- 是否切换下游地址。

没有变更不代表没问题，但有变更时应该优先验证。

## 在 eMall 项目中怎么讲？

如果订单创建 P99 从 300 ms 升到 3 秒，可以这样排查：

- 先看是否只影响订单创建接口。
- 再看 trace 中慢 span 是库存、优惠、支付还是订单自身。
- 如果慢在库存，看库存热点行和数据库锁。
- 如果慢在优惠，看规则计算和缓存命中率。
- 如果慢在订单自身，看线程池排队、GC 和锁竞争。
- 如果慢在多个下游，看网关重试、下游连接池和网络。

修复不是简单扩容，而是根据根因选择限流、隔离、缓存、索引、降级或参数调整。

## 深度增强：JVM 生产运行图

![Java 17 容器内 JVM 内存结构](../../assets/jvm-runtime-memory.svg)

JVM 题要从运行时资源解释到业务影响。堆、直接内存、元空间、线程栈和容器 memory limit 共同决定服务稳定性；
GC、CPU throttling、线程池队列和下游超时会一起影响 P99，而不是孤立存在。

## 深度增强：Java 17 诊断模型示例

```java
record RuntimeSignal(
        double heapUsage,
        double containerMemoryUsage,
        long gcPauseMillis,
        int threadCount,
        int queuedTasks) {

    boolean requiresTriage() {
        return heapUsage > 0.85
                || containerMemoryUsage > 0.90
                || gcPauseMillis > 500
                || threadCount > 800
                || queuedTasks > 1_000;
    }
}
```

这个模型强调线上诊断要看组合信号。只看 heap 不够，只看 GC 也不够；
要把 JVM、容器、线程池和业务延迟放到同一条时间线。

## 深度增强：生产边界

JVM 调优不能靠背参数。要先明确服务目标：低延迟、吞吐、容器资源、对象分配速率和 P99 SLO。
然后通过 GC 日志、JFR、指标和压测验证。错误地调大 `-Xmx` 可能挤压堆外内存，导致容器 OOMKilled。

结论要能解释现象，并能给出降级、扩容、参数调整或代码优化方案。

- 用链路追踪定位慢段。
- 要看线程池、连接池、GC、数据库、缓存和下游。

### Java17 延迟样本建模示例

```java
import java.time.Duration;

public record LatencySample(Duration p99, long heapUsedBytes, long gcPauseMillis) {

    boolean violatesSlo(Duration targetP99) {
        return p99.compareTo(targetP99) > 0 || gcPauseMillis > 200;
    }
}

final class RuntimeTriage {

    String classify(LatencySample sample) {
        if (sample.gcPauseMillis() > 200) {
            return "Investigate allocation rate, heap sizing, and GC logs.";
        }
        if (sample.violatesSlo(Duration.ofMillis(300))) {
            return "Check downstream latency, thread pools, and lock contention.";
        }
        return "Runtime is within the current service objective.";
    }
}
```
