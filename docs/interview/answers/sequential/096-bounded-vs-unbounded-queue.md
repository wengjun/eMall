## 题目

线程池队列应该用有界还是无界？

## 先给面试官的短答案

生产线程池应该优先使用有界队列。无界队列看似不会拒绝任务，但在下游慢或流量突增时会无限堆积，
导致请求等待、内存上涨、GC 抖动和 OOM。有界队列能让系统在过载时及时触发拒绝、限流或降级。

队列容量应该根据可接受排队时间和处理能力设置。

## 无界队列的问题

无界队列的问题是没有明确过载边界。

当消费速度小于生产速度：

- 队列持续增长。
- 任务持有对象无法释放。
- heap 上涨。
- 排队时间变长。
- 用户已经超时但任务仍在队列中。
- 最终可能 OOM。

无界队列不是稳定性设计，而是把失败延后。

## 有界队列的价值

有界队列能明确系统容量边界。

当超过容量时：

- 触发拒绝策略。
- 上游可以快速失败。
- 可以降级非核心能力。
- 避免内存被撑爆。
- 保护核心线程继续处理已接任务。

生产系统需要可控失败，而不是无限等待。

## 队列容量怎么定？

可以从可接受等待时间估算。

假设线程池每秒稳定处理 1000 个任务，希望排队不超过 200 ms：

```text
queueSize ≈ 1000 * 0.2 = 200
```

真实容量还要结合任务耗时分布、峰值流量和下游容量压测验证。

## 队列太小的问题

队列太小也可能问题：

- 短暂抖动就拒绝。
- 资源利用率不足。
- 上游重试增多。

所以队列要吸收合理毛刺，但不能承接长期过载。

## 队列太大的问题

队列太大意味着：

- 排队时间长。
- 超时任务继续执行。
- 内存占用高。
- 故障恢复慢。

大队列经常让系统“看起来还活着”，但用户体验已经失败。

## 在 eMall 项目中怎么讲？

订单创建线程池如果使用无界队列，库存服务慢时订单任务会持续堆积。

用户端已经超时，但订单服务还在排队处理旧请求，可能造成重复下单、库存不一致和内存上涨。

更合理的是有界队列、快速失败、幂等、限流和熔断。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

幂等和唯一键保证分布式正确性。这样能体现我理解 Java 并发，也理解微服务生产稳定性。

## 专家级完整回答

```text
生产线程池应该使用有界队列。无界队列在流量大于处理能力时会无限堆积任务，延长对象生命周期，
导致 heap 上涨、GC 抖动、P99 升高甚至 OOM。有界队列能让系统在过载时触发拒绝、限流或降级。

```

## 回答评分点

高分答案应该覆盖：

- 生产优先有界队列。
- 无界队列会导致内存和延迟问题。
- 有界队列触发可控失败。
- 容量按等待时间估算。
- 队列不是越大越好。

### Java17 有界异步执行示例

```java
import java.util.concurrent.Callable;
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Semaphore;

final class BoundedAsyncExecutor implements AutoCloseable {
    private final Semaphore permits = new Semaphore(128);
    private final ExecutorService pool = Executors.newFixedThreadPool(32);

    <T> CompletableFuture<T> submit(Callable<T> task) {
        if (!permits.tryAcquire()) {
            return CompletableFuture.failedFuture(new IllegalStateException("Too many requests."));
        }
        return CompletableFuture.supplyAsync(() -> {
            try {
                return task.call();
            } catch (Exception ex) {
                throw new IllegalStateException(ex);
            } finally {
                permits.release();
            }
        }, pool);
    }

    @Override
    public void close() {
        pool.shutdown();
    }
}
```
