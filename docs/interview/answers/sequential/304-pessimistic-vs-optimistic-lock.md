## 题目

悲观锁和乐观锁怎么选？

## 先给面试官的短答案

悲观锁适合冲突概率高、必须串行处理、失败重试成本高的场景；乐观锁适合冲突概率低、读多写少、允许失败后重试的场景。
悲观锁用锁阻塞竞争者，乐观锁用版本号或条件更新检测冲突。

选择关键是冲突概率、锁持有时间、吞吐要求和重试成本。

## 悲观锁

常见方式：

```sql
SELECT * FROM inventory
WHERE sku_id = ?
FOR UPDATE;
```

特点：

- 先锁住资源。
- 冲突事务等待。
- 一致性强。
- 并发性能可能下降。

适合关键资源强竞争。

## 乐观锁

常见方式：

```sql
UPDATE inventory
SET available = available - 1, version = version + 1
WHERE sku_id = ?
  AND version = ?
  AND available > 0;
```

特点：

- 不先阻塞。
- 更新时检测版本。
- 冲突时失败重试。
- 适合冲突少场景。

## 如何选择

看：

- 冲突是否频繁。
- 重试是否便宜。
- 是否能接受失败。
- 事务是否很短。
- 是否热点资源。

热点秒杀库存不一定适合简单乐观锁反复重试，因为重试会放大数据库压力。

## 在 eMall 项目中怎么讲？

普通订单状态更新可以使用乐观锁版本号。

热点库存扣减可使用条件更新、库存桶和令牌前置拦截，避免大量乐观锁失败重试打爆数据库。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

## 专家级完整回答

```text
悲观锁适合冲突高、必须串行和失败成本高的场景；乐观锁适合冲突低、读多写少和可重试场景。
悲观锁通过加锁等待保证互斥，乐观锁通过版本号或条件更新在提交时检测冲突。

在电商中，订单状态更新常用乐观锁；热点库存要结合条件更新、限流和库存桶，而不是简单无限重试。
```

## 回答评分点

高分答案应该覆盖：

- 悲观锁先锁资源。
- 乐观锁提交时检测冲突。
- 冲突高更偏悲观或前置削峰。
- 冲突低适合乐观锁。
- 选择要考虑重试成本。

## 深度完善：面向 L6 的回答框架

围绕「悲观锁和乐观锁怎么选？」，高分答案不能停在概念定义，而要把「索引、事务、锁、MVCC、分库分表、冷热分层、备份恢复和在线变更」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「数据库和数据建模」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `order、payment、inventory、product 的核心表、唯一约束、索引和归档表` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：慢 SQL、锁等待、死锁数、主从延迟、连接池使用率、归档任务耗时。
- 验证证据：EXPLAIN、慢 SQL 日志、死锁日志、迁移方案、备份恢复演练和容量预测。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
