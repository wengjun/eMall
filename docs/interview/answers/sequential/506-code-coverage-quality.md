## 题目

代码覆盖率高是否代表质量高？

## 先给面试官的短答案

代码覆盖率高不等于质量高。覆盖率只能说明代码被执行过，不能说明断言是否有效、边界是否覆盖、业务是否正确。
高质量测试要看关键风险是否覆盖、断言是否有意义、失败路径是否覆盖，以及测试是否稳定可维护。

## 覆盖率的价值

覆盖率可以发现完全没有测试的代码区域，也能防止核心模块测试缺失。它适合作为最低门槛和趋势指标。

例如订单金额计算只有 20% 覆盖率，说明风险很高；覆盖率从 80% 突然降到 60%，也需要关注。

## 覆盖率的局限

测试可以执行代码但没有有效断言，这种覆盖率没有质量意义。测试也可能只覆盖 happy path，没有覆盖边界、异常和并发。

覆盖 getter、setter 或简单 DTO 可以提高数字，但对系统质量帮助有限。相反，一个覆盖率不高但精准覆盖状态机、
金额计算和幂等的测试，价值可能更大。

## 正确使用方式

覆盖率应按模块和风险设置目标，而不是全项目一刀切。核心业务规则、公共安全组件和资金链路要更高，
纯配置类、启动类和简单适配层可以合理排除。

评估质量时要结合 mutation testing、缺陷逃逸率、评审质量、集成测试和生产事故数据。

## 在 eMall 项目中怎么讲？

eMall 应把覆盖率重点放在 `order`、`payment`、`inventory`、`pricing`、`promotion` 和 `common` 的核心规则。
不能为了数字去覆盖无意义代码，而要优先覆盖金额、状态机、幂等和库存并发。

现场编码题不只是写出算法，还要说明输入输出、边界条件、复杂度、线程安全和可测试性。
面试官通常更看重思考过程、代码结构和验证意识，而不是只看最终代码。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
覆盖率是质量信号，但不是质量本身。

它告诉我们哪些代码被测试执行过，但不告诉我们断言是否正确，也不告诉我们异常、并发和业务边界是否覆盖。
所以我会把覆盖率作为门槛和趋势，而不是唯一目标。

真正高质量的测试应该覆盖高风险逻辑，有清晰断言，能在代码错误时失败，并且稳定可维护。
```

## 回答评分点

高分答案应该覆盖：

- 说明覆盖率不等于质量。
- 知道覆盖率能发现未测试区域。
- 能指出无断言、只测 happy path、覆盖无意义代码的问题。
- 知道按风险设置覆盖率目标。
- 能结合核心交易模块说明。

## 深度完善：面向 L6 的回答框架

围绕「代码覆盖率高是否代表质量高？」，高分答案不能停在概念定义，而要把「单元测试、集成测试、契约测试、压测、CI、代码评审、覆盖率和依赖治理」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「测试、工程治理和质量」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `common、smoke、chaos、loadtest、各服务的单元和集成测试` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：测试通过率、覆盖关键分支数、 flaky 率、构建耗时、漏洞数量。
- 验证证据：CI 记录、测试报告、覆盖率报告、依赖扫描、契约测试和评审清单。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
