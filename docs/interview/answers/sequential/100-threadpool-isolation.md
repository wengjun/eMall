## 题目

什么是线程池隔离？

## 先给面试官的短答案

线程池隔离是把不同业务、不同下游或不同优先级任务放到不同线程池中执行，限制单个故障源对其他能力的影响。
它是舱壁隔离的一种实现方式，常用于防止慢下游占满所有工作线程。

线程池隔离要配合有界队列、超时、拒绝、熔断、降级和监控。

## 为什么需要隔离？

如果所有任务共享一个线程池，一个慢任务可能拖垮所有任务。

例如：

```text
inventory slow -> shared pool full -> payment query blocked -> order page slow
```

线程池隔离让库存慢只影响库存相关能力，不影响支付或订单基础查询。

## 隔离什么？

可以按这些维度隔离：

- 核心业务和非核心业务。
- 不同下游服务。
- 读请求和写请求。
- 实时请求和后台任务。
- 高优先级和低优先级任务。
- 快任务和慢任务。

隔离粒度要适中。

## 隔离的效果

好处：

- 限制故障范围。
- 避免线程被单个下游占满。
- 更容易设置容量。
- 更容易监控和告警。
- 支持不同拒绝和降级策略。

隔离让系统在局部故障时仍能提供核心能力。

## 代价

代价：

- 线程总数增加。
- 内存占用增加。
- 参数配置复杂。
- 资源利用率下降。
- 运维监控成本增加。

所以隔离要基于风险，而不是无限拆分。

## 和信号量隔离的区别

线程池隔离使用独立线程池执行任务。

信号量隔离是在当前线程中执行，但限制并发数。

线程池隔离适合阻塞调用和强隔离。

信号量隔离开销更小，适合非阻塞或短耗时调用。

## 生产设计要点

线程池隔离必须配合：

- 有界队列。
- 明确拒绝策略。
- 调用超时。
- 熔断器。
- 降级逻辑。
- 指标监控。
- 线程名。

只建多个线程池但没有这些控制，仍然不算生产级隔离。

## 在 eMall 项目中怎么讲？

订单服务可以拆分：

- orderCreateExecutor。
- inventoryClientExecutor。
- paymentClientExecutor。
- riskClientExecutor。
- orderQueryExecutor。
- asyncEventExecutor。

库存慢时，只打满库存池，订单查询和支付查询仍然可用。

非核心推荐失败时直接降级，不占用核心交易线程。

## 深度增强：Java 17 有界并发示例

```java
import java.util.concurrent.Semaphore;
import java.util.function.Supplier;

final class BulkheadGuard {
    private final Semaphore permits;

    BulkheadGuard(int maxConcurrentCalls) {
        this.permits = new Semaphore(maxConcurrentCalls);
    }

    <T> T execute(Supplier<T> supplier) {
        if (!permits.tryAcquire()) {
            throw new IllegalStateException("Bulkhead rejected the call");
        }
        try {
            return supplier.get();
        } finally {
            permits.release();
        }
    }
}
```

这段代码展示了并发控制的生产思路：不是让所有请求无限进入系统，而是在入口保护共享资源。
真实项目还要加超时、指标、降级和按下游隔离。

## 深度增强：生产边界

线程安全不等于系统安全。`ConcurrentHashMap` 只能保护当前进程内的数据结构，
不能替代数据库唯一键、幂等表或分布式一致性。线程池也不能使用无界队列，
否则会把过载转化成内存上涨和 P99 恶化。

### Java17 有界异步执行示例

```java
import java.util.concurrent.Callable;
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Semaphore;

final class BoundedAsyncExecutor implements AutoCloseable {
    private final Semaphore permits = new Semaphore(128);
    private final ExecutorService pool = Executors.newFixedThreadPool(32);

    <T> CompletableFuture<T> submit(Callable<T> task) {
        if (!permits.tryAcquire()) {
            return CompletableFuture.failedFuture(new IllegalStateException("Too many requests."));
        }
        return CompletableFuture.supplyAsync(() -> {
            try {
                return task.call();
            } catch (Exception ex) {
                throw new IllegalStateException(ex);
            } finally {
                permits.release();
            }
        }, pool);
    }

    @Override
    public void close() {
        pool.shutdown();
    }
}
```
