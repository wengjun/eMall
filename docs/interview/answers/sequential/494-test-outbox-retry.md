## 题目

如何测试 Outbox 重试？

## 先给面试官的短答案

测试 Outbox 重试要验证业务数据和事件在同一事务中落库，消息发送失败后事件保持待发送状态，重试任务能按次数、
退避和状态机重新发送，最终成功后标记已发送，重复发送不会让消费者重复产生业务副作用。

## 基础验证点

Outbox 的核心不是“发送一条消息”，而是保证业务提交和事件记录一致。测试应先验证订单创建成功时，
订单表和 outbox_event 表在同一事务中提交。如果事务回滚，两个表都不能留下半成品。

然后模拟消息代理不可用或发送失败，断言 outbox_event 仍处于待发送或失败待重试状态，并记录重试次数和下次重试时间。

## 重试场景

重试任务要覆盖：

- 首次发送失败后能被扫描到。
- 达到下次重试时间前不会提前重试。
- 重试成功后状态变为已发送。
- 多个重试任务并发扫描时，同一事件只被一个任务处理。
- 达到最大重试次数后进入死信或人工处理状态。

## 消费端幂等

Outbox 通常保证至少一次投递，不保证只投递一次。因此测试还要验证消费者幂等。即使同一事件被发送两次，
消费者也只能让业务状态变化一次。

生产级测试应覆盖生产端重试和消费端去重两个方向。

## 在 eMall 项目中怎么讲？

eMall 的订单创建、支付成功、库存预占和退款完成都适合使用 Outbox。测试要验证订单落库和事件落库同事务，
Kafka 暂不可用时事件不丢，恢复后自动发送，并且重复消费不会重复履约或重复发券。

### 贴合本题的实现示例

```java
import java.util.List;

interface OutboxRepository {
    List<OutboxMessage> lockPending(int limit);

    void markSent(long id);
}

record OutboxMessage(long id, String topic, String payload) {
}

final class OutboxRelay {

    void relay(OutboxRepository repository, MessagePublisher publisher) {
        for (OutboxMessage message : repository.lockPending(100)) {
            publisher.publish(message.topic(), message.payload());
            repository.markSent(message.id());
        }
    }
}

interface MessagePublisher {
    void publish(String topic, String payload);
}
```

### 先给立场

- 回答「如何测试 Outbox 重试？」时，不能只给单一方案，要先说明约束、目标和失败边界。

### 可选方案对比

- 本地事务加 Outbox：可靠性高，延迟和存储成本会增加。
- 直接发 MQ：实现简单，但本地事务和消息发送之间存在不一致窗口。
- TCC 或 Saga：业务可控性强，但侵入性和状态管理复杂度更高。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果约束不明确，先补齐规模、延迟、可用性、一致性、成本和团队能力，再做选择。

- 回答「如何测试 Outbox 重试？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
