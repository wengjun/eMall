## 题目

如何测试 Outbox 重试？

## 先给面试官的短答案

测试 Outbox 重试要验证业务数据和事件在同一事务中落库，消息发送失败后事件保持待发送状态，重试任务能按次数、
退避和状态机重新发送，最终成功后标记已发送，重复发送不会让消费者重复产生业务副作用。

## 基础验证点

Outbox 的核心不是“发送一条消息”，而是保证业务提交和事件记录一致。测试应先验证订单创建成功时，
订单表和 outbox_event 表在同一事务中提交。如果事务回滚，两个表都不能留下半成品。

然后模拟消息代理不可用或发送失败，断言 outbox_event 仍处于待发送或失败待重试状态，并记录重试次数和下次重试时间。

## 重试场景

重试任务要覆盖：

- 首次发送失败后能被扫描到。
- 达到下次重试时间前不会提前重试。
- 重试成功后状态变为已发送。
- 多个重试任务并发扫描时，同一事件只被一个任务处理。
- 达到最大重试次数后进入死信或人工处理状态。

## 消费端幂等

Outbox 通常保证至少一次投递，不保证只投递一次。因此测试还要验证消费者幂等。即使同一事件被发送两次，
消费者也只能让业务状态变化一次。

生产级测试应覆盖生产端重试和消费端去重两个方向。

## 在 eMall 项目中怎么讲？

eMall 的订单创建、支付成功、库存预占和退款完成都适合使用 Outbox。测试要验证订单落库和事件落库同事务，
Kafka 暂不可用时事件不丢，恢复后自动发送，并且重复消费不会重复履约或重复发券。

## 深度增强：Java 17 编码模板示例

```java
import java.util.LinkedHashMap;
import java.util.Map;

final class LruCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    LruCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

这段代码展示现场编码的表达方式：先选合适数据结构，再说明复杂度和边界。
若用于生产，还要考虑并发、监控、容量和淘汰策略。

## 深度增强：生产边界

面试中的简化实现通常不是生产实现。生产需要线程安全、容量限制、指标、异常处理、单元测试和压测验证。
如果题目涉及分布式场景，还要说明单机实现和多实例实现的差异。

## 深度增强：面试高分表达

我会先澄清需求和边界，再写最小正确实现，最后补充复杂度、测试用例和生产化改造。
这样即使代码题不复杂，也能体现工程成熟度。

## 专家级完整回答

```text
Outbox 重试测试要同时验证可靠投递和消费幂等。

我会先验证业务表和 outbox 表在同一数据库事务内提交，避免业务成功但事件丢失。
然后模拟消息发送失败，检查事件状态、重试次数、下次重试时间和最大重试后的死信状态。

最后还要验证消费者幂等，因为 Outbox 通常是至少一次投递。生产上只测试发送端是不够的，
必须证明重复事件不会导致重复发货、重复发券或重复状态变更。
```

## 回答评分点

高分答案应该覆盖：

- 说明 Outbox 关键是业务数据和事件同事务。
- 覆盖发送失败、延迟重试、成功标记、最大次数和死信。
- 知道并发扫描要避免重复处理同一事件。
- 知道至少一次投递要求消费者幂等。
- 能结合订单、支付、库存、退款举例。

## 深度完善：面向 L6 的回答框架

围绕「如何测试 Outbox 重试？」，高分答案不能停在概念定义，而要把「单元测试、集成测试、契约测试、压测、CI、代码评审、覆盖率和依赖治理」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「测试、工程治理和质量」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `common、smoke、chaos、loadtest、各服务的单元和集成测试` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：测试通过率、覆盖关键分支数、 flaky 率、构建耗时、漏洞数量。
- 验证证据：CI 记录、测试报告、覆盖率报告、依赖扫描、契约测试和评审清单。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
