## 题目

如何设计可恢复的业务状态？

## 先给面试官的短答案

可恢复的业务状态要求每个中间状态都有明确含义、进入条件、退出条件、超时策略、补偿动作和审计记录。
系统故障后，任务可以根据状态继续推进、重试、取消、退款或人工介入。

不要只记录成功和失败，复杂流程必须记录可恢复的中间态。

## 状态设计

应设计：

- 初始状态。
- 处理中状态。
- 成功状态。
- 失败状态。
- 取消状态。
- 异常待处理状态。

处理中状态要有超时时间和补偿路径。

## 恢复信息

需要记录：

- 当前状态。
- 业务 ID。
- 关联资源 ID。
- 最近事件。
- 重试次数。
- 上次失败原因。
- 下一次执行时间。
- 操作审计。

恢复任务要能靠这些信息继续执行。

## 失败分类

不同失败要进入不同状态：

- 可重试失败。
- 不可重试业务失败。
- 需要人工处理。
- 等待外部回调。
- 等待对账确认。

一个 `FAILED` 状态通常不够表达生产问题。

## 在 eMall 项目中怎么讲？

退款流程应包含 `REFUND_REQUESTED`、`REFUNDING`、`REFUND_SUCCEEDED`、`REFUND_FAILED`、
`MANUAL_REVIEW_REQUIRED` 等状态。

支付渠道超时时，退款单停在可重试状态，由补偿任务继续推进。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
可恢复状态要让系统在宕机、超时或下游故障后知道下一步做什么。每个中间状态都要有语义、超时、补偿和退出路径。
状态记录中要包含业务 ID、关联资源、重试次数、失败原因和下一次执行时间。

复杂流程不要只用成功和失败两个状态，应把等待回调、处理中、需人工审核等状态显式建模。
```

## 回答评分点

高分答案应该覆盖：

- 中间状态要显式建模。
- 每个状态要有退出路径。
- 恢复任务需要足够上下文。
- 失败要分类。
- 状态变化要审计。

### Java17 Outbox 事件示例

```java
import java.time.Instant;
import java.util.UUID;

public record OutboxMessage(
        UUID id,
        String aggregateId,
        String eventType,
        String payload,
        Instant createdAt

}

final class OutboxFactory {

    OutboxMessage orderEvent(long orderId, String eventType, String payload) {
```

## 架构取舍与反驳补强

架构取舍补强标记：已完成

### 先给立场

- 回答「如何设计可恢复的业务状态？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 入口限流：保护整体容量，但可能误伤部分真实用户。
- 线程池隔离：保护调用方资源，但需要额外容量规划和拒绝策略。
- 熔断降级：能快速止血，但需要半开探测和平滑恢复避免二次雪崩。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果影响系统稳定性，优先保护核心链路和整体可用性，而不是追求单次请求成功。

### 决策证据

- 业务指标
- 稳定性指标
- 成本指标
- 灰度和回滚记录

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

## 规模化与成本治理补强

规模成本补强标记：已完成

### 规模化视角

- 回答「如何设计可恢复的业务状态？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
- 按入口峰值、下游容量、线程池容量和排队长度共同设置保护阈值。
- 容量策略要支持按用户、商家、SKU、接口和机房分层保护。

### 成本治理

- 用单位成本看问题，例如单请求成本、单订单成本、单消息成本和单 GB 存储成本。
- 先优化浪费最高的环节，而不是平均用力。

### 自动化和 owner

- 必须有自动告警、自动摘流、自动回滚或一键 Runbook。
- 每个告警要有 owner、SLO、升级路径和复盘模板。

### 面试表达

我会补一句：方案能跑只是第一步，大规模下还要回答容量怎么估、成本怎么控、故障谁负责。
这能体现我不是只会实现单点功能，而是能长期运营一个高并发业务系统。
