## 题目

设计下单系统。

## 先给面试官的短答案

下单系统的核心是把用户购买意图转换成可信订单，并保证金额正确、库存不超卖、重复请求幂等、订单状态可追踪、
支付和履约可衔接。架构上订单服务编排用户、商品、价格、促销、库存和支付，通过本地事务写订单与 Outbox，
再用事件驱动后续支付、履约和通知。

## 核心职责

下单系统负责提交订单、确认价格、校验优惠、预占库存、生成订单号、创建订单明细、创建支付单、记录状态机和发送订单事件。

它不应该承担所有业务逻辑。商品归商品服务，库存归库存服务，价格和促销归各自服务，订单服务负责交易编排和状态。

## 下单数据流

用户提交订单请求，网关完成身份认证和限流。订单服务校验请求幂等键，读取商品快照、价格、促销和收货地址。

订单服务调用库存服务预占库存。库存预占成功后，订单服务在本地事务中写订单主表、订单明细、支付单引用和 Outbox 事件。

事务提交后，支付服务生成支付参数，用户跳转支付。后续支付成功通过支付回调和事件更新订单状态。

## 一致性设计

库存预占和订单创建之间可能失败。常见设计是先预占库存，再创建订单；如果订单创建失败，触发库存释放补偿。
也可以把库存服务设计成预占记录有过期时间，由补偿任务释放。

订单创建和事件发送使用本地事务加 Outbox，避免订单创建成功但消息丢失。消费者必须幂等。

## 幂等和状态机

下单接口要基于用户、购物车项、请求号或客户端幂等键防重复提交。同一个幂等键只能创建一个订单。

订单状态机要清晰，例如待支付、已支付、已取消、待发货、已发货、已完成、退款中、已退款。状态流转必须校验旧状态。

## 高并发和可用性

热点 SKU 下单要结合库存服务限流和排队。普通下单要有线程池隔离、下游超时、支付通道隔离和降级策略。
价格、促销、库存下游异常时，不能默认创建错误订单，应返回明确失败或进入待确认流程。

## 在 eMall 项目中怎么讲？

eMall 的 `order` 模块可以作为交易编排中心，配合 `inventory`、`pricing`、`promotion`、`payment` 和 `event-platform`。
核心亮点是订单状态机、幂等创建、库存预占、Outbox 事件和补偿任务。

## 深度增强：生产边界

生产级系统要考虑容量、数据一致性、灰度发布、容灾、可观测性、安全、成本和团队运维能力。
如果某个方案无法监控、无法回滚、无法对账或无法补偿，就不能算完整设计。

## 专家级完整回答

```text
下单系统设计要围绕正确性和可恢复性展开。

我会让订单服务负责交易编排：校验用户和商品，确认价格和优惠，调用库存预占，
然后在本地事务中写订单和 Outbox 事件。支付和履约通过事件解耦。

关键风险有三个：重复下单、库存超卖和部分失败。重复下单通过幂等键和唯一约束解决；
库存超卖通过库存服务原子预占解决；部分失败通过 Outbox、补偿任务和状态机兜底。

下单链路不能为了可用性牺牲正确性。推荐和评价可以降级，但金额、库存和支付状态必须准确。
```

## 回答评分点

高分答案应该覆盖：

- 明确下单系统职责和边界。
- 能讲清提交订单到支付的核心数据流。
- 覆盖幂等、状态机、库存预占和 Outbox。
- 能处理订单创建失败后的库存释放。
- 知道核心交易链路不能随意降级为成功。

## 深度完善：系统设计高分展开

围绕「设计下单系统」，系统设计题要按目标、规模、架构、数据、失败路径、演进六层展开。
不要只画组件图，必须讲清每个组件为什么存在，以及它如何在大促和故障场景下保持可控。

### 容量和约束

- 先给假设：10 亿注册用户、1 亿 DAU、100 万峰值并发、核心链路优先保证可用性。
- 区分读多写少、写强一致、异步最终一致、离线分析和运维控制面。
- 明确 SLO，例如核心接口成功率、P99、恢复时间、数据可恢复窗口和成本上限。

### 白板架构

- 入口层：Gateway 做认证、签名、限流、灰度、请求体大小控制和 Trace 注入。
- 应用层：按领域拆服务，核心写链路尽量短，非核心能力通过事件异步解耦。
- 数据层：主库负责交易事实，缓存和搜索承担读流量，Outbox 保障事件可靠投递。
- 治理层：配置、发布、熔断、降级、告警、Runbook 和补偿平台形成闭环。

### 数据模型和数据流

核心表要能回答三个问题：当前状态是什么、为什么变成这样、失败后如何恢复。
通常至少需要业务主表、状态流水表、幂等表、Outbox 表、补偿任务表和审计表。

```sql
-- 示例：生产系统中不要只存当前状态，还要保留可追溯的事实和恢复入口。
create table operation_journal (
    id bigint primary key,
    business_key varchar(64) not null,
    operation_type varchar(64) not null,
    before_state varchar(64),
    after_state varchar(64),
    created_at timestamp not null
);
```

### 失败路径

- 下游慢：先超时、隔离和熔断，再用降级结果保护主链路。
- 消息失败：用 Outbox、重试、DLQ 和重放平台恢复。
- 数据不一致：用状态机、对账和补偿任务修复。
- 发布异常：用金丝雀、自动暂停、配置回滚和数据兼容策略控制影响面。

### 验收指标

- 用户侧：成功率、P95/P99、错误码分布、投诉量。
- 系统侧：QPS、CPU、连接池、队列积压、缓存命中率、MQ lag。
- 数据侧：对账差异、补偿积压、重复请求、非法状态迁移。
- 成本侧：单请求成本、存储增长、日志和 Trace 采样成本。

深度完善标记：系统设计题已补容量、数据流、失败路径和验收口径。
