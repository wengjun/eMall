## 题目

`jstack` 可以定位哪些问题？

## 先给面试官的短答案

`jstack` 用来抓取 Java 进程的线程快照。它可以帮助定位死锁、锁竞争、线程池耗尽、
CPU 飙高对应的热点线程、请求卡住、下游调用阻塞和大量线程等待等问题。

但 `jstack` 是瞬时快照，不能只看一次。生产排查通常会连续抓多次，再结合 CPU、GC、日志、
链路追踪和线程池指标判断。

## jstack 输出是什么？

`jstack <pid>` 会输出目标 JVM 中所有 Java 线程的状态和调用栈。

你能看到：

- 线程名。
- 线程 ID。
- native thread id。
- 线程状态。
- 当前执行到哪一行代码。
- 是否持有锁。
- 是否正在等待锁。
- 是否发现 Java-level deadlock。

简单理解，`jstack` 是给 JVM 拍一张“线程当时在干什么”的照片。

## 可以定位死锁

死锁是两个或多个线程互相等待对方释放锁，最终都无法继续执行。

`jstack` 对典型 Java monitor 死锁有自动识别能力，输出中会出现类似：

```text
Found one Java-level deadlock
```

排查时要看：

- 哪些线程参与死锁。
- 每个线程持有什么锁。
- 每个线程又在等待什么锁。
- 锁对应的业务代码位置。

面试时不要只说“jstack 能看死锁”，还要说明死锁修复方向：

- 固定加锁顺序。
- 减少锁嵌套。
- 使用超时锁。
- 缩小锁粒度。
- 避免锁内调用外部系统。

## 可以定位锁竞争

如果大量线程处于 `BLOCKED` 状态，说明它们在等待进入某个 synchronized 临界区。

常见原因：

- 全局 synchronized 方法。
- 热点库存扣减锁粒度过大。
- 本地缓存刷新持有大锁。
- 日志、序列号、规则计算中存在单点同步。

如果大量线程栈都卡在同一行代码，就说明这个位置可能是锁竞争热点。

`jstack` 只能告诉你“卡在哪里”，还需要结合业务含义判断“为什么卡”。

## 可以定位线程池耗尽

线程池耗尽时，`jstack` 可能看到：

- 大量业务线程卡在下游 HTTP 调用。
- 大量线程卡在数据库连接获取。
- 大量线程卡在 MQ 消费处理。
- 线程池工作线程都在执行慢任务。
- 新任务无法被及时执行。

这时要结合线程池指标：

- active count。
- pool size。
- queue size。
- completed task count。
- rejected count。
- task execution latency。

线程池问题不能只看线程 dump，还要看队列和拒绝数。

## 可以辅助分析 CPU 飙高

CPU 飙高时，常见流程是：

1. 用 `top` 或系统监控找到 Java 进程。
2. 找到占用 CPU 高的线程 ID。
3. 把线程 ID 转成十六进制。
4. 在 `jstack` 中查找对应的 `nid`。
5. 看该线程的 Java 调用栈。

如果多次抓取都看到同一个线程在同一段代码中运行，可能是热点循环、JSON 序列化、正则、
加密、压缩、复杂规则计算或死循环。

## 可以定位请求卡住

请求卡住时，`jstack` 能看到线程停在哪类操作：

- 数据库查询。
- Redis 调用。
- HTTP 调用。
- 文件 IO。
- 锁等待。
- 线程睡眠。
- 队列获取任务。

如果大量线程卡在同一个下游客户端，说明可能是下游慢、连接池耗尽或超时设置过长。

## 常见线程状态怎么理解？

常见状态：

- `RUNNABLE`：正在运行或等待 CPU，也可能在 native IO 中。
- `BLOCKED`：等待进入 synchronized。
- `WAITING`：无限期等待某个条件。
- `TIMED_WAITING`：带超时时间的等待。
- `NEW`：创建但未启动。
- `TERMINATED`：已经结束。

注意：`RUNNABLE` 不一定真的在消耗 CPU，也可能阻塞在 socket read 等 native 调用中。

## 生产排查注意点

不要只抓一次 `jstack`。

更可靠的方式是连续抓 3 到 5 次，每次间隔几秒，观察线程栈是否变化：

- 栈一直不变，可能卡住。
- 栈持续变化，可能只是正常忙。
- 某类线程数量持续增加，可能有泄漏或堆积。

排查还要结合：

- 应用日志。
- GC 日志。
- 线程池指标。
- 下游依赖延迟。
- 链路追踪。
- CPU profile。

## 在 eMall 项目中怎么讲？

订单创建接口 P99 升高时，可以抓订单服务 `jstack`。

如果看到大量线程卡在库存扣减客户端，说明库存服务慢或连接池耗尽。
如果卡在 `synchronized` 的本地库存预占逻辑，说明锁粒度不合理。
如果卡在数据库连接池获取，说明数据库慢或连接池容量不足。

`jstack` 能帮助把“接口慢”落到具体代码栈上。

## 深度增强：JVM 生产运行图

![Java 17 容器内 JVM 内存结构](../../assets/jvm-runtime-memory.svg)

JVM 题要从运行时资源解释到业务影响。堆、直接内存、元空间、线程栈和容器 memory limit 共同决定服务稳定性；
GC、CPU throttling、线程池队列和下游超时会一起影响 P99，而不是孤立存在。

## 深度增强：Java 17 诊断模型示例

```java
record RuntimeSignal(
        double heapUsage,
        double containerMemoryUsage,
        long gcPauseMillis,
        int threadCount,
        int queuedTasks) {

    boolean requiresTriage() {
        return heapUsage > 0.85
                || containerMemoryUsage > 0.90
                || gcPauseMillis > 500
                || threadCount > 800
                || queuedTasks > 1_000;
    }
}
```

这个模型强调线上诊断要看组合信号。只看 heap 不够，只看 GC 也不够；
要把 JVM、容器、线程池和业务延迟放到同一条时间线。

## 深度增强：生产边界

JVM 调优不能靠背参数。要先明确服务目标：低延迟、吞吐、容器资源、对象分配速率和 P99 SLO。
然后通过 GC 日志、JFR、指标和压测验证。错误地调大 `-Xmx` 可能挤压堆外内存，导致容器 OOMKilled。

## 深度增强：面试高分表达

我会用证据链回答 JVM 问题：先看业务影响，再看 JVM 指标、GC 日志、线程栈、heap dump、容器事件和最近变更。
结论要能解释现象，并能给出降级、扩容、参数调整或代码优化方案。

```text
jstack 的价值是把 JVM 当前线程状态和调用栈暴露出来。它可以定位死锁、锁竞争、
线程池耗尽、请求阻塞、下游慢调用，以及 CPU 飙高时具体哪个 Java 栈在运行。

```

高分答案应该覆盖：

- `jstack` 是线程快照。
- 能定位死锁和锁竞争。
- 能辅助定位 CPU 高线程。
- 能看到请求卡在哪个调用栈。

## 深度完善：面向 L6 的回答框架

围绕「jstack 可以定位哪些问题？」，高分答案不能停在概念定义，而要把「内存模型、GC、线程、JIT、诊断命令和容器资源边界」讲成一条可验证的工程链路。
面试官真正关注的是：你是否知道它解决什么问题、什么时候会失效、如何在生产系统中验证。

### 1. 先界定边界

- 本题属于「JVM 和性能诊断」，先说明它影响的是正确性、稳定性、性能、安全还是协作效率。
- 不要直接背结论，要先说清业务约束、数据规模、调用链位置和失败后果。
- 如果存在多种方案，要说明默认选择、替代方案、迁移成本和放弃条件。

### 2. 结合 eMall 落地

- 可以从 `gateway、order、payment、search 在高峰期的 P99、GC、线程池和容器资源` 切入，说明它在真实电商链路中的入口、状态、数据和依赖。
- 回答时至少补一个失败路径，例如超时、重复请求、状态不一致、热点流量或配置误发。
- 再说明如何通过代码规范、测试、灰度、回滚、监控或补偿把风险收敛。

### 3. 生产级验证

- 关键指标：P50/P95/P99、GC pause、allocation rate、线程数、CPU throttle、RSS。
- 验证证据：JFR、GC log、jstack、jcmd、heap dump、压测报告和发布对比曲线。
- 如果没有这些证据，只能说明方案在理论上成立，不能证明它能长期稳定运行。

### 4. 追问防守

- 被问“为什么不用更简单方案”时，回答当前规模、团队能力和风险收益是否匹配。
- 被问“为什么不用更复杂方案”时，回答复杂方案的运维成本、故障面和迁移成本。
- 最后用一句话收束：先用简单可靠方案闭环，再用指标驱动演进，而不是提前复杂化。
