## 题目

消费失败后如何重试？

## 先给面试官的短答案

消费失败后要区分临时失败和永久失败。临时失败可以重试，永久失败应直接进入失败记录或死信。
重试要有最大次数、退避策略、错误分类、告警和幂等保护，不能无限快速重试。

重试是恢复手段，也可能变成流量放大器。

## 错误分类

可重试：

- 下游超时。
- 临时网络错误。
- 数据库连接短暂失败。
- 限流后稍后可恢复。

不可重试：

- 参数格式错误。
- 业务资源不存在且不会恢复。
- 权限错误。
- Schema 不兼容。

错误分类决定处理路径。

## 重试策略

策略包括：

- 固定间隔重试。
- 指数退避。
- 延迟队列。
- 分级重试 topic。
- 最大重试次数。
- 超限进入死信。

高并发系统通常要使用退避和抖动，避免同时重试冲击下游。

## 工程要点

要点包括：

- 消费处理幂等。
- 记录失败原因。
- 记录重试次数。
- 对同一业务键避免并发重试。
- 可观测重试堆积。
- 重试失败后告警。

无限重试会隐藏问题并拖垮系统。

## 在 eMall 项目中怎么讲？

库存服务消费订单支付事件时，如果数据库短暂不可用，可以延迟重试。

如果事件缺少必要字段或订单状态非法，应进入死信并告警，而不是一直重试。

## 深度增强：一致性和补偿图

![交易一致性、对账和补偿闭环](../../assets/consistency-compensation-loop.svg)

分布式一致性题要先区分事实来源、状态流转和补偿责任。
订单、库存、支付、优惠和消息不可能总靠一个本地事务完成，
所以要用幂等、状态机、Outbox、重试、对账和补偿形成闭环。

## 深度增强：Java 17 状态机示例

```java
enum TradeState {
    INIT,
    RESERVED,
    PAID,
    CLOSED
}

record TradeTransition(TradeState from, TradeState to, String reason) {

    boolean valid() {
        return switch (from) {
            case INIT -> to == TradeState.RESERVED || to == TradeState.CLOSED;
            case RESERVED -> to == TradeState.PAID || to == TradeState.CLOSED;
            case PAID, CLOSED -> false;
        };
    }
}
```

状态机的价值是防止非法跳转。生产事故中很多错误不是技术异常，而是状态被重复推进、逆向推进或越级推进。

## 深度增强：生产边界

最终一致不是“最终随便一致”。每个异步环节都要有唯一业务键、幂等处理、重试策略、死信、补偿任务和对账报表。
涉及资金和库存时，宁可慢一点，也要保证事实可追踪、可审计、可修复。

## 深度增强：面试高分表达

我会先承认分布式系统无法用一个本地事务覆盖所有服务，再说明如何把不确定性收敛：
本地事务写事实和 Outbox，消费者幂等处理，失败进入重试和死信，后台对账发现差异并补偿。

## 专家级完整回答

```text
消费失败后先分类。临时性错误可以重试，永久性业务错误不应无限重试。
重试要设置最大次数、退避、抖动和死信路径，并记录失败原因和重试次数。

消费者必须幂等，因为每次重试都可能在上一次部分成功后再次执行。重试堆积和失败率也要监控告警。
```

## 回答评分点

高分答案应该覆盖：

- 先区分可重试和不可重试错误。
- 重试要有最大次数。
- 退避和抖动防止冲击下游。
- 超限进入死信。
- 消费者必须幂等。

- 简单方案：上线快、成本低，但容量和故障边界有限。

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
