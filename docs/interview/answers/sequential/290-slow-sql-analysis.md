## 题目

慢 SQL 如何分析？

## 先给面试官的短答案

慢 SQL 分析要先确认慢在哪里，再看执行计划、索引、扫描行数、回表、排序、锁等待和返回数据量。
常用工具是慢查询日志、`EXPLAIN`、执行耗时分布、数据库监控和业务调用链。

不要一上来就加索引，要先确认瓶颈原因。

## 分析步骤

步骤：

- 从慢查询日志找到 SQL。
- 确认执行频率和影响面。
- 使用 `EXPLAIN` 看执行计划。
- 看是否走合适索引。
- 看扫描行数和返回行数。
- 看是否 filesort 或临时表。
- 看锁等待和事务。
- 看是否返回过多字段。

最后再决定优化方式。

## 常见原因

包括：

- 没有合适索引。
- 索引选择不合理。
- 回表太多。
- 深分页。
- 大范围扫描。
- 排序无法用索引。
- 锁等待。
- 查询字段过多。

不同原因对应不同优化手段。

## 优化方式

方式：

- 添加或调整联合索引。
- 改写查询条件。
- 使用覆盖索引。
- 限制时间范围。
- 游标分页。
- 拆分大查询。
- 异步导出。
- 降低事务持锁时间。

优化后要验证执行计划和线上指标。

## 在 eMall 项目中怎么讲？

后台订单查询慢，不能直接给每个筛选字段都建索引。

应先看最常用查询路径、扫描行数、排序方式和是否深分页，再设计少量高价值联合索引或改为异步导出。

## 深度增强：Java 17 数据访问策略示例

```java
record QueryPlan(String accessPath, boolean usesIndex, boolean requiresPagination) {

    boolean safeForOnlineTraffic() {
        return usesIndex && requiresPagination;
    }
}

final class OnlineQueryPolicy {

    void verify(QueryPlan plan) {
        if (!plan.safeForOnlineTraffic()) {
            throw new IllegalArgumentException("Online query must use index and pagination");
        }
    }
}
```

这段代码体现线上查询治理：不是 SQL 能跑就可以上线，而是要确认走索引、可分页、可限流、可观测。

## 深度增强：生产边界

核心表设计要从典型查询倒推索引，避免全表扫描、深分页和大事务。分库分表要先选好分片键，
避免跨分片事务和热点分片。任何数据迁移都要支持灰度、校验、回滚或修复。

### 先给立场

- 回答「慢 SQL 如何分析？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 单库优化：成本低、复杂度低，但容量天花板明确。
- 读写分离：提升读吞吐，但会引入主从延迟和读一致性问题。
- 分库分表：容量高，但跨分片查询、事务、扩容和运维复杂度上升。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果约束不明确，先补齐规模、延迟、可用性、一致性、成本和团队能力，再做选择。

### 决策证据

- 业务指标
- 稳定性指标
- 成本指标
- 灰度和回滚记录

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

## 规模化与成本治理补强

### 规模化视角

- 回答「慢 SQL 如何分析？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
- 按读写比例、数据增长、热点分布、索引大小和归档周期估算数据库容量。
- 数据库扩展要优先减少热点和慢查询，再考虑分片。

### 成本治理

- 控制索引数量、历史数据在线保留周期、慢查询和跨分片扫描。
- 冷热分离、归档和汇总表能显著降低在线库成本。

- 为关键指标建立看板、告警、owner 和 Runbook。
