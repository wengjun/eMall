## 题目

线程池队列堆积和 JVM 内存上涨有什么关系？

## 先给面试官的短答案

线程池队列堆积会让任务对象、请求上下文、参数、日志上下文和回调对象长时间留在内存中，
导致 heap 使用上涨。队列越大，排队时间越长，对象越容易跨过多次 young GC 后晋升到 old gen，
最终可能引发频繁 GC、P99 升高甚至 OOM。

所以生产线程池必须使用有界队列，并监控 queue size、active count 和 rejected count。

## 队列里存的是什么？

线程池队列中存放的是等待执行的任务。

这些任务可能引用：

- 请求参数。
- 用户信息。
- 商品列表。
- trace context。
- 日志 MDC。
- 回调对象。
- 数据库查询条件。
- 大集合或大 JSON。

只要任务还在队列中，这些对象就不会被 GC。

## 为什么会导致 heap 上涨？

队列堆积意味着对象生命周期变长。

原本请求结束就能回收的对象，现在要等任务执行完才可回收。

如果堆积持续发生：

- young gen 对象变多。
- 存活对象跨过多次 GC。
- 对象晋升到 old gen。
- old gen 回收压力增加。
- GC pause 变长。

这就是队列堆积和 JVM 内存上涨的直接关系。

## 无界队列的风险

无界队列看似不会拒绝请求，但风险很大。

它会把压力从“拒绝请求”转移成“消耗内存”。

结果可能是：

- 请求排队时间越来越长。
- 用户早已超时，但任务还在执行。
- heap 持续上涨。
- old gen 被撑满。
- 服务 OOM。

生产中，无界队列通常不是保护系统，而是延迟失败。

## 下游慢时更危险

如果线程池任务依赖下游服务，下游变慢会让工作线程长时间占用。

新任务进入队列后无法及时执行，形成堆积。

如果调用方还有重试，队列会更快增长。

这就是慢依赖拖垮调用方的常见路径。

## 如何治理？

治理手段：

- 使用有界队列。
- 设置合理拒绝策略。
- 设置任务超时。
- 下游调用设置超时。
- 使用熔断和降级。
- 对不同下游做线程池隔离。
- 监控队列长度和等待时间。
- 在入口做限流。

线程池队列不是越大越好，队列大小要和可接受等待时间匹配。

## 如何估算队列大小？

可以从用户可接受等待时间出发。

如果线程池每秒处理 1000 个任务，最多允许排队 200 ms，那么队列长度最好不要超过约 200 个。

这是简化估算，真实还要考虑任务耗时分布和峰值流量。

核心思想是：队列长度代表排队时间，不只是容量。

## 在 eMall 项目中怎么讲？

订单服务调用库存服务时，如果库存慢，订单线程池队列会堆积。

队列中的创建订单任务会持有用户、商品、优惠、trace 等对象，heap 会随队列增长。

如果不及时限流和熔断，订单服务可能因为库存慢而自身 OOM。

正确做法是对库存调用设置独立隔离、超时、熔断和降级。

线程池队列堆积会先表现为排队时间变长，然后表现为对象生命周期变长，最后可能变成 old gen 上涨和 GC 暂停。

## 深度增强：Java 17 有界线程池示例

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;

final class BoundedExecutorFactory {

    ThreadPoolExecutor createInventoryExecutor() {
        return new ThreadPoolExecutor(
                32,
                32,
                30,
                TimeUnit.SECONDS,
                new ArrayBlockingQueue<>(500),
                new ThreadPoolExecutor.AbortPolicy());
    }
}
```

这个示例的重点是有界队列和明确拒绝策略。无界队列看似保护请求不失败，实际是把过载转移到内存里，
让请求排队、对象滞留、GC 变慢，最终可能 OOM。

## 深度增强：生产边界

队列大小应该从可接受等待时间倒推，而不是拍脑袋。假设服务每秒处理 1000 个任务，
最大允许排队 200 ms，那么队列容量约 200 更合理；设置成 100000 只会隐藏过载。

对不同下游要做线程池隔离。库存慢不应该占满支付线程池，推荐慢不应该影响订单创建。
如果使用虚拟线程，也仍然需要限流、超时和背压，因为数据库连接、HTTP 连接和下游容量不是无限的。

## 深度增强：面试高分表达

我会说队列不是免费缓冲区。队列中的任务会持有请求上下文、参数、大对象和 trace 信息，
堆积会延长对象生命周期并推高 old gen。生产线程池要有界、可拒绝、可监控，并和超时、熔断、
限流和下游隔离一起设计。

```text
线程池队列堆积会把请求任务和它引用的上下文对象长期保留在 heap 中，让本该短生命周期的对象
跨过多次 young GC 并晋升到 old gen。无界队列会把过载从拒绝请求变成内存上涨，最终导致

生产上我会使用有界队列、明确拒绝策略、超时、熔断、限流和下游隔离，并监控 active count、
queue size、rejected count 和任务排队时间。
```

## 回答评分点

高分答案应该覆盖：

- 队列任务持有对象引用。
- 堆积会延长对象生命周期。
- 对象可能晋升 old gen。
- 无界队列有 OOM 风险。
- 要有界队列、拒绝、超时、隔离和限流。
