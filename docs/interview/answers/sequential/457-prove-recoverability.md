## 题目

如何证明系统具备可恢复能力？

## 先给面试官的短答案

证明可恢复能力不能靠口头说明，要靠演练记录、恢复指标、备份恢复验证、故障注入结果、Runbook、
RTO/RPO 达成情况和真实事故复盘证据。

能恢复的系统，必须经过验证。

## 证据类型

证据：

- 定期恢复演练报告。
- 备份恢复校验结果。
- 故障演练记录。
- RTO 和 RPO 实测数据。
- 自动化回滚记录。
- 事故复盘行动项。
- 监控和告警覆盖率。

这些比架构图更有说服力。

## 恢复能力维度

维度：

- 服务实例恢复。
- 数据库恢复。
- 缓存恢复。
- MQ 积压恢复。
- 下游恢复后平滑放量。
- 数据一致性修复。
- 人员响应恢复。

恢复不是只重启服务。

## 验证方式

方式：

- 定期灾备演练。
- Chaos 工程。
- 灰度故障注入。
- 备份恢复抽检。
- Runbook 演练。
- 指标回放和复盘。

验证要周期化。

## 在 eMall 项目中怎么讲？

eMall 可以用订单库备份恢复演练证明 RPO/RTO，用 Redis 故障演练证明缓存降级，用 Kafka 积压演练
证明补发和幂等能力。

每次演练记录恢复时间、数据校验结果和改进项。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

### 贴合本题的实现示例

```java
enum CircuitState {
    CLOSED, OPEN, HALF_OPEN
}

final class CircuitBreaker {
    private CircuitState state = CircuitState.CLOSED;
    private int failures;

    synchronized boolean allowRequest() {
        return state != CircuitState.OPEN;
    }

    synchronized void record(boolean success) {
        if (success) {
            failures = 0;
            state = CircuitState.CLOSED;
        } else if (++failures >= 5) {
            state = CircuitState.OPEN;
        }
    }
}
```

## 架构取舍与反驳补强

架构取舍补强标记：已完成

### 先给立场

- 回答「如何证明系统具备可恢复能力？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 入口限流：保护整体容量，但可能误伤部分真实用户。
- 线程池隔离：保护调用方资源，但需要额外容量规划和拒绝策略。
- 熔断降级：能快速止血，但需要半开探测和平滑恢复避免二次雪崩。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果影响系统稳定性，优先保护核心链路和整体可用性，而不是追求单次请求成功。

### 决策证据

- 业务指标
- 稳定性指标
- 成本指标
- 灰度和回滚记录

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

### 灰度和回滚

- 先止血再定位，先保护用户影响最大的核心链路。
- 恢复后逐步放量，并观察是否出现二次抖动。
- 回滚不是只回代码，还要确认配置、数据、缓存、消息和任务状态能一起回到安全状态。

### 监控和验收证据

- 测试报告
- 灰度看板
- 告警规则
- 回滚记录

### 面试表达

## 规模化与成本治理补强

规模成本补强标记：已完成

### 规模化视角

- 回答「如何证明系统具备可恢复能力？」时，要主动放到 10 亿用户、1 亿 DAU、100W 峰值并发的背景下思考。
- 按入口峰值、下游容量、线程池容量和排队长度共同设置保护阈值。
- 容量策略要支持按用户、商家、SKU、接口和机房分层保护。

### 成本治理

- 用单位成本看问题，例如单请求成本、单订单成本、单消息成本和单 GB 存储成本。
- 先优化浪费最高的环节，而不是平均用力。

### 自动化和 owner

- 必须有自动告警、自动摘流、自动回滚或一键 Runbook。
- 每个告警要有 owner、SLO、升级路径和复盘模板。

### 面试表达

我会补一句：方案能跑只是第一步，大规模下还要回答容量怎么估、成本怎么控、故障谁负责。
这能体现我不是只会实现单点功能，而是能长期运营一个高并发业务系统。
