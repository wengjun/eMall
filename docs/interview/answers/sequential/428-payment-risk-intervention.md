## 题目

支付风控应在什么环节介入？

## 先给面试官的短答案

支付风控应在下单前、支付前、支付中、支付回调后和退款售后环节分层介入。不同环节风险不同，
下单前偏资格和账号风险，支付前偏交易风险，回调后偏结果核验和异常处置。

风控不能只在支付成功后补救。

## 下单前

检查：

- 账号风险。
- 设备风险。
- 地址风险。
- 优惠券滥用。
- 购买频率。
- 黑名单。

这里可以减少无效订单。

## 支付前

检查：

- 金额异常。
- 收货信息异常。
- 支付方式风险。
- 订单状态。
- 价格是否被篡改。
- 用户是否被冻结。

支付前是关键拦截点。

## 支付后

检查：

- 支付回调签名。
- 支付金额是否一致。
- 支付单是否重复。
- 第三方流水是否可信。
- 异常订单是否冻结履约。

支付成功不代表风险结束。

## 在 eMall 项目中怎么讲？

eMall 在创建支付单前调用风控服务，检查账号、设备、金额、优惠和收货地址。

支付回调时还要校验第三方签名、金额和订单状态。如果风险较高，可以将订单置为待人工审核，暂停
履约而不是直接发货。

## 深度增强：Java 17 发布门禁示例

```java
record ReleaseSignal(double errorRate, long p99Millis, double cpuThrottleRate, boolean rollbackSafe) {

    boolean canContinue() {
        return errorRate < 0.001
                && p99Millis < 300
                && cpuThrottleRate < 0.05
                && rollbackSafe;
    }
}
```

这段代码表达发布平台的核心：放量不是人工拍脑袋，而是由错误率、延迟、资源和回滚安全共同决定。

## 深度增强：生产边界

K8s 会重启失败容器，但不保证业务一定恢复。错误的 liveness probe 可能造成重启风暴；
过低的 CPU limit 会造成 throttling；不兼容数据库变更会让回滚失效。平台能力要和应用设计配合。

## 深度增强：面试高分表达

我会把 K8s 视为运行平台，而不是稳定性的全部答案。真正生产级要有容量规划、灰度门禁、配置治理、可观测性、
自动回滚和数据库兼容检查，才能支撑核心交易链路。

## 专家级完整回答

```text
支付风控要分层介入。下单前看账号、设备、优惠和频率；支付前看金额、支付方式、订单状态和用户
风险；支付回调后校验签名、金额、流水和重复通知；退款售后也要做风险识别。

高风险订单可以暂停履约或进入人工审核，而不是简单成功或失败。风控要和订单状态机协同。
```

### 贴合本题的实现示例

```sql
select p.payment_id,
       p.order_id,
       p.amount as paid_amount,
       o.payable_amount
  from payments p
  join orders o on o.order_id = p.order_id
 where p.status = 'SUCCESS'
   and p.amount <> o.payable_amount;
```

## 架构取舍与反驳补强

架构取舍补强标记：已完成

### 先给立场

- 回答「支付风控应在什么环节介入？」时，不能只给单一方案，要先说明约束、目标和失败边界。
- 高分回答要让面试官看到你能在正确性、可用性、成本、复杂度和团队能力之间做判断。

### 可选方案对比

- 同步强校验：资金风险低，但会拉长交易链路并放大下游抖动。
- 异步对账补偿：主链路更稳，但需要对账、补偿、审计和人工兜底。
- 第三方结果为准：实现简单，但必须处理通知延迟、重复通知和账务差异。

### 反驳和防守

- 如果面试官问为什么不直接上最复杂方案，可以回答：复杂方案只有在规模和风险证明必要时才值得引入。
- 如果面试官问为什么不用最简单方案，可以回答：简单方案可以做第一期，但必须提前设计观测和迁移边界。
- 我的判断原则是：如果影响交易正确性或资金安全，优先选择可审计、可补偿、可对账的方案。

### 决策证据

- 状态机和幂等记录
- 对账差异趋势
- 补偿任务成功率
- 核心链路错误率

### 一句话总结

我会先用简单可靠的方案解决当前确定性问题，同时保留观测、灰度和迁移能力。
当指标证明瓶颈存在，再演进到更复杂的架构，而不是为了显得高级提前复杂化。

- 先在测试环境和影子流量中验证，再做 1%、5%、25%、50%、100% 分阶段灰度。

- 回滚不是只回代码，还要确认配置、数据、缓存、消息和任务状态能一起回到安全状态。

- 对账差异可解释
