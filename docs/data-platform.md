# 数据平台

本文说明分片、归档、冷热分层、商品到搜索同步和恢复操作。

## 分库分表基线

当前工程主要表达服务级数据所有权，每个服务拥有自己的 schema 和 Flyway 历史。真实高规模生产还需要：

- 按用户、订单、商家或区域选择分片键。
- 避免跨分片事务。
- 为订单、支付、库存等核心表设计分片路由。
- 为全局查询准备搜索、数仓或异步聚合表。
- 为热点 SKU 和热点商家准备单独保护策略。

## 在线归档窗口

归档目标：

- 减少核心交易表体积。
- 降低索引和备份成本。
- 保留历史查询和审计能力。
- 避免归档任务影响白天高峰。

建议在低峰窗口按时间、状态和业务域分批归档。

## 归档规则

- 已完成订单可以按时间归档。
- 支付和退款流水必须满足财务保留要求。
- 审计、安全和合规记录需要单独保留周期。
- 归档必须可追踪、可恢复、可校验。
- 归档前后需要比对数量和金额。

## 冷存储生命周期

数据可以分为：

- 热数据：核心交易库和缓存，支持在线访问。
- 温数据：近期历史数据，支持低频查询。
- 冷数据：对象存储或数仓，用于审计、分析和合规。

冷热分层要考虑查询成本、恢复时间和合规保留周期。

## 商品到搜索同步

商品写入不直接依赖搜索服务：

1. 商品服务先写本地数据库。
2. 商品服务写 Outbox 事件。
3. Outbox relay 发布到 Kafka。
4. 搜索服务消费事件并幂等更新索引。
5. 搜索查询读取 Elasticsearch 投影。

这种方案牺牲短暂实时性，换取商品写入的稳定性和可恢复性。

## 恢复操作

常见恢复场景：

- Outbox 发布失败：通过内部运维接口重放。
- MQ 消费失败：依赖重试、死信和人工修复。
- 搜索索引不一致：按商品库重建投影。
- 支付对账差异：对比渠道账单和本地流水。
- 归档差异：使用数量、金额、校验和做核对。

## 扩展方向

- CDC 同步。
- 数据质量平台。
- 血缘追踪。
- 实时数仓。
- 多活数据冲突检测。
- 分布式查询和离线修复工具。
