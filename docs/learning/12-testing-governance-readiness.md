## 测试体系

### 单元测试

单元测试验证小范围业务规则。

适合测试：

- 订单状态转换。
- 金额计算。
- 库存桶选择。
- 风控规则判断。
- 参数校验。

特点：

- 快。
- 不依赖外部服务。
- 可以频繁执行。

### 集成测试

集成测试验证多个组件一起工作。

适合测试：

- Flyway 迁移。
- Repository SQL。
- Redis 操作。
- Kafka Outbox。
- 服务启动。
- 配置清单。

### Smoke 测试

Smoke 测试验证系统启动后核心链路是否可用。

典型链路：

1. 创建用户。
2. 初始化商品和库存。
3. 创建订单。
4. 支付。
5. 查询订单状态。

### Mock 和真实环境

Mock 适合：

- 单元测试。
- 下游行为可控。
- 错误分支验证。

真实环境适合：

- SQL 验证。
- Redis/Kafka 行为。
- 启动配置验证。
- E2E 验证。

不要只依赖 Mock，因为 Mock 不能发现 SQL、配置、序列化和真实中间件问题。


## 工程治理

### 代码规范

需要统一：

- 缩进。
- 行长。
- 命名。
- 包结构。
- 异常处理。
- 日志格式。
- 测试命名。

工具：

- `.editorconfig`
- Checkstyle
- Maven validate

### 公共能力抽取

适合放到 `common`：

- 统一响应。
- 错误码。
- 业务异常。
- ID 生成。
- 幂等基础设施。
- Outbox 基础设施。
- 审计工具。
- 安全工具。

不适合放到 `common`：

- 具体订单业务。
- 具体支付规则。
- 具体库存扣减策略。
- 会让所有服务互相耦合的业务代码。

### 文档治理

项目文档需要回答：

- 系统是什么。
- 模块怎么分。
- 核心链路怎么走。
- 失败后怎么恢复。
- 怎么启动。
- 怎么测试。
- 怎么部署。
- 当前限制是什么。

好的文档不是越多越好，而是入口清晰、内容不重复、和代码保持一致。
