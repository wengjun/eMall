## 概念补强：从知道到会实现

这一章专门补齐前文中仍然容易停留在概念层的内容。学习时不要只记“是什么”，
要能回答“为什么这样做、代码放在哪里、失败后怎么办、线上如何验证”。

### Java 并发不是只会 synchronized

本工程的 `OrderService.create`、`PaymentService.callback`、`InventoryService.reserve` 里可以看到
`synchronized` 和 `@Transactional` 同时出现。它们解决的是两个不同层面的问题：

- `synchronized` 只保护当前 JVM 进程内的并发。
- `@Transactional` 保护当前数据库本地事务内的写入原子性。
- 多实例部署后，请求可能打到不同 Pod，`synchronized` 不能跨实例生效。
- 真正的生产兜底必须依靠数据库唯一约束、条件更新、分布式锁或消息幂等。

所以面试时不能说“我用了 synchronized，所以并发安全”。更好的表达是：

```text
当前代码为了让内存仓储和本地测试行为稳定，服务方法加了 synchronized。
生产部署时，核心并发安全不会依赖 JVM 锁，而会用数据库唯一键、条件更新和幂等记录兜底。
例如下单 request_id 建唯一索引，库存扣减使用 where available >= quantity 的条件更新。
```

库存并发的正确兜底应该像下面这样：

```sql
update inventory_item
set available = available - :quantity,
    reserved = reserved + :quantity,
    updated_at = current_timestamp
where sku_id = :skuId
  and available >= :quantity;
```

Java 层要检查更新行数：

```java
int updated = jdbcTemplate.update(sql, quantity, quantity, skuId, quantity);
if (updated == 0) {
    throw new BusinessException(ErrorCode.CONFLICT, "insufficient stock");
}
```

这样即使有 100 个实例同时扣同一行，数据库也能保证不会把 `available` 扣成负数。

并发还包括线程池和资源隔离。生产服务不能把所有下游调用都丢到一个默认线程池里。
如果支付渠道变慢，占满线程池，订单查询也被拖死，这就是故障扩散。正确做法是：

- 订单主链路、搜索链路、推荐链路使用不同线程池或不同服务实例池。
- 慢下游设置独立超时、重试和熔断。
- 线程池队列长度要有限，队列满时快速失败。
- 监控线程池活跃线程、队列长度、拒绝次数和任务耗时。

L6 追问常见问题：

```text
Q: synchronized 和数据库锁有什么区别？
A: synchronized 是进程内互斥，数据库锁是数据层互斥。
   多实例部署时，只有数据库约束或分布式协调才能保护共享数据。

Q: 为什么不所有地方都加分布式锁？
A: 分布式锁会增加延迟和可用性风险。能用唯一键、条件更新和状态机解决的，
   优先用数据库原子能力。锁只用于必须串行化且数据库约束表达不出来的场景。
```

### Spring 生命周期和事务代理要真正理解

Spring Boot 常见注解看起来简单，但 L6 面试会追问注解背后的机制。

`@Service`、`@Repository`、`@Configuration` 这些注解会让类成为 Spring Bean。
Spring 容器启动时会扫描这些 Bean，创建对象，解析构造函数依赖，并把依赖注入进去。
因此本工程大量使用构造函数注入，例如 `OrderService` 依赖 `OrderRepository`、
`OutboxRepository`、`InventoryClient`、`PricingClient` 和 `MarketingClient`。

构造函数注入的好处：

- 依赖是显式的，读构造函数就知道这个服务需要什么。
- 依赖不可变，便于测试。
- 启动时就能发现缺少 Bean，而不是运行到一半才空指针。
- 不需要反射修改私有字段，测试替换依赖更清晰。

`@Transactional` 的关键不是“加上就有事务”，而是 Spring 通过代理拦截公开方法调用。
需要特别注意：

- 同一个类内部 `this.someTransactionalMethod()` 调用通常不会触发事务代理。
- 事务只覆盖数据库操作，不覆盖远程服务的原子性。
- 不要在事务中做长时间远程调用，否则数据库锁会被持有更久。
- 事务边界应放在应用服务层，而不是 Controller 层。

本工程的 `OrderService.create` 是一个值得讨论的例子。它在事务方法里调用价格、营销、
库存客户端，然后保存订单和 Outbox。这个设计适合教学展示完整链路，但真实生产可以进一步优化：

- 价格和营销可以在事务前完成，因为它们不持有订单库锁。
- 库存预占是跨服务调用，不能被订单库事务回滚。
- 订单落库后应记录库存预占 ID 和失败状态，靠补偿恢复。
- Outbox 必须和订单状态写入同一个订单库本地事务。

面试表达：

```text
我会把事务边界控制在本服务数据库写入范围内。
跨服务调用不放进“全局事务”的幻想里，而是通过状态机、幂等、Outbox 和补偿来收敛。
```

### 数据库隔离级别、锁和索引不能只会名词

事务有四个常见隔离级别：读未提交、读已提交、可重复读、串行化。
生产中最常见的是读已提交或可重复读。选择隔离级别时要理解它和锁竞争、性能的关系。

库存扣减不应该依赖“先查再改”：

```java
InventoryItem item = repository.findItem(skuId).orElseThrow();
if (item.available() >= quantity) {
    repository.saveItem(item.reserve(quantity));
}
```

这个写法在并发下会出问题，因为两个事务可能都读到 `available = 1`，然后都认为可以扣。
正确方式是把判断和更新合并成一个 SQL 条件更新，或者使用版本号乐观锁。

乐观锁示例：

```sql
update inventory_item
set available = available - :quantity,
    reserved = reserved + :quantity,
    version = version + 1
where sku_id = :skuId
  and version = :version
  and available >= :quantity;
```

如果更新行数为 0，说明版本冲突或库存不足，业务层可以短暂重试。重试必须有上限。

索引设计也不能只说“加索引”。要从查询模式出发：

- `orders(user_id, created_at)`：支持用户订单列表按时间分页。
- `orders(request_id)`：支持下单幂等查重，必须唯一。
- `payment_order(channel_trade_no)`：支持支付回调幂等，必须唯一。
- `outbox_event(status, next_retry_at)`：支持 Relay 扫描待发送事件。
- `operation_audit(trace_id)`：支持按一次操作追踪审计记录。

联合索引要遵守最左前缀。比如 `(user_id, created_at)` 可以支持按 `user_id` 查询，
也可以支持按 `user_id + created_at` 查询，但不能高效支持只按 `created_at` 查询。

大表还要考虑：

- 冷热分离：近期订单在在线库，历史订单进归档库。
- 分库分表：按 `user_id` 或 `order_id` 路由，避免单表无限增长。
- 二级索引代价：索引越多，写入越慢，存储越大。
- 在线 DDL：生产改表要避免长时间锁表。

L6 回答重点：

```text
我会先从核心查询路径反推索引，而不是机械地给每个字段建索引。
写多的表会控制索引数量，读多的读模型可以接受更多索引或转到搜索引擎。
```

### 缓存一致性要讲清楚失败窗口

缓存不是“加 Redis 就快”。缓存设计首先要明确数据源是谁。
在电商系统中，商品详情、类目、品牌、部分价格读模型适合缓存；
订单状态、支付状态、库存扣减结果不能只依赖缓存作为最终真相。

常见的 Cache Aside 读流程：

```text
1. 先查 Redis。
2. 命中则返回。
3. 未命中查数据库。
4. 数据库结果写入 Redis，并设置 TTL。
5. 返回结果。
```

更新流程通常是：

```text
1. 更新数据库。
2. 删除缓存。
3. 下次读取时重新加载。
```

为什么不是“先删缓存再更新数据库”？因为这会出现一个窗口：

```text
线程 A 删除缓存。
线程 B 读缓存未命中，读到数据库旧值，并把旧值写回缓存。
线程 A 更新数据库为新值。
结果：缓存里长期保留旧值。
```

即使“先更新数据库再删除缓存”，也仍然有短暂不一致窗口。所以要配合：

- TTL：缓存最终会过期。
- Binlog 或 Outbox 事件：异步刷新读模型。
- 延迟双删：降低并发读写导致旧值回填的概率。
- 版本号：缓存值带版本，旧版本不能覆盖新版本。
- 本地缓存失效广播：多实例本地缓存要同步失效。

缓存穿透、击穿、雪崩要能区分：

- 穿透：请求不存在的数据，缓存和数据库都没有。解决：空值缓存、布隆过滤器、参数校验。
- 击穿：热点 key 过期，大量请求同时打到数据库。解决：互斥重建、逻辑过期、热点预热。
- 雪崩：大量 key 同时过期或 Redis 故障。解决：TTL 加随机抖动、多级缓存、限流降级。

面试时要主动说明哪些数据不能强依赖缓存：

```text
库存可以用 Redis 做预热、令牌或秒杀前置过滤，但最终扣减仍然要落到数据库或库存中心。
支付状态不能只相信缓存，因为支付回调和对账需要持久化流水作为审计依据。
```

### MQ 和 Outbox 要讲到顺序、重复和积压

Kafka 的核心语义是至少一次投递，不是业务 exactly-once。
所以消费端必须能处理重复消息。本工程 `search`、`fulfillment` 等模块有
`ProcessedMessageRepository`，用于记录已处理消息，这就是消费端幂等表的思路。

消息设计要考虑 partition key：

- 订单事件可以用 `orderId`，保证同一订单的事件进入同一分区。
- 商品事件可以用 `skuId` 或 `productId`，保证同一商品变更顺序。
- 支付事件可以用 `paymentId`，保证同一支付单事件顺序。

如果 key 选错，会出现两个问题：

- 同一业务对象的事件乱序，例如 `ORDER_PAID` 先于 `ORDER_CREATED` 被消费。
- 热点 key 集中到一个分区，导致单分区积压。

Outbox Relay 的生产细节：

```text
1. 扫描 status in (PENDING, FAILED) and next_retry_at <= now 的事件。
2. 用 claim 状态把事件从 READY 改成 PUBLISHING，防止多个 Relay 重复抢同一批。
3. 发送 Kafka。
4. 成功后标记 PUBLISHED。
5. 失败后增加 retry_count，计算 next_retry_at。
6. 超过最大重试进入 DEAD，等待人工重放或修复。
```

消费者处理流程：

```text
1. 开启本地事务。
2. 尝试插入 processed_message(message_id)。
3. 如果唯一键冲突，说明处理过，直接提交并 ack。
4. 执行业务写入。
5. 提交事务。
6. ack Kafka offset。
```

关键点是“业务写入”和“processed_message 插入”要在同一个本地事务中。
否则可能出现业务成功但去重记录失败，下一次重复消费又执行一次业务。

积压处理不能只说“扩容消费者”：

- 先看是所有分区积压，还是个别热点分区积压。
- 如果是所有分区，增加 consumer 实例可能有效，但不能超过分区数。
- 如果是单分区热点，扩容消费者无效，需要拆 key、拆 topic 或优化单条处理耗时。
- 如果下游数据库慢，盲目扩容消费者会把数据库打垮。
- 积压恢复期间要保护核心交易链路，必要时暂停非核心消费者。

### 微服务边界要从数据所有权出发

服务拆分不是“模块越多越高级”。拆分的核心是数据所有权和变化原因。

订单服务拥有订单状态和订单快照；库存服务拥有库存数量和预占记录；
支付服务拥有支付单、支付流水和对账结果。其他服务不能直接改这些表。

跨服务访问要通过 API 或事件，而不是共享数据库：

```text
订单服务需要库存结果 -> 调库存 API。
库存状态变化需要通知搜索或履约 -> 发库存事件。
支付成功需要订单变更 -> 支付服务调订单 API 或发支付事件，由订单服务消费。
```

共享数据库的短期好处是简单，长期问题是：

- 数据所有权不清晰。
- 任一服务改表都可能影响其他服务。
- 难以独立扩缩容。
- 难以做权限隔离。
- 故障半径扩大。

服务边界也不是一开始就固定。早期可以用模块化单体或少量服务，
等团队规模、数据量和发布频率上来后再拆。L6 面试更看重你能解释演进路径：

```text
我不会为了微服务而微服务。早期可以保留核心交易模块在一个部署单元中，
但代码按领域分包、数据库表按领域设计，后续拆服务时边界清晰。
当订单和商品读流量、发布节奏、团队 ownership 明显不同，再拆成独立服务。
```

### 安全不是只加登录

电商系统的安全要覆盖认证、授权、数据保护、接口签名、审计和风控。

敏感数据保护要分两类：

- 可逆加密：例如手机号展示给客服时需要解密，可以用 AES-GCM。
- 不可逆查询：例如按手机号查重，可以用 HMAC hash。

本工程 `common.crypto.AesGcmFieldEncryptor` 同时提供加密和查询 hash 思路。
为什么不能只加密后直接查密文？因为 AES-GCM 每次加密都有随机 IV，
同一个手机号两次加密结果不同，无法直接等值查询。所以需要额外保存 HMAC hash。

开放平台签名要防两类攻击：

- 篡改：攻击者改了请求体金额或订单号。
- 重放：攻击者拿旧请求重复提交。

签名校验应覆盖：

```text
canonicalString = method + path + timestamp + nonce + bodyHash
signature = HMAC_SHA256(secret, canonicalString)
```

服务端检查：

- `timestamp` 是否在允许窗口内，例如 5 分钟。
- `nonce` 是否已使用，使用 Redis 或数据库短期去重。
- `bodyHash` 是否匹配实际请求体。
- HMAC 比较使用常量时间比较，避免时序攻击。

权限要按最小权限设计。比如“运维可以重放 Outbox”不代表“运维可以改支付金额”。
内部运维接口必须具备：

- 强认证。
- 操作授权。
- 操作审计。
- 双人审批或高危操作确认。
- 可回滚或可补偿。

日志安全也很重要。不要把手机号、地址、token、签名密钥、支付渠道原始报文直接打进日志。

### 可观测性要能支持排障闭环

日志、指标、Trace 不是三套孤立工具，而是一条排障链路。

一次线上故障应能这样定位：

```text
告警显示下单成功率下降。
先看指标确认是 order 服务错误率上升，还是 inventory/payment 下游失败。
再按 traceId 查一条失败请求的完整链路。
最后看对应服务日志和数据库状态，判断是容量、代码、配置还是下游故障。
```

指标设计可以用 RED 和 USE 两套方法：

- RED：Rate、Errors、Duration，适合 HTTP API 和业务接口。
- USE：Utilization、Saturation、Errors，适合 CPU、线程池、连接池、磁盘、Kafka 分区。

核心业务指标不能缺：

- 下单请求量、成功率、失败原因分布。
- 库存预占成功率、库存不足率、库存服务超时率。
- 支付创建量、回调成功率、支付确认订单失败数。
- Outbox 待发布数量、失败数量、最老事件年龄。
- Kafka consumer lag。
- 对账差异数量和未处理时长。

告警不要只按单点阈值。更好的方式是 SLO burn rate：

```text
如果下单可用性目标是 99.9%，月度错误预算约 0.1%。
当 5 分钟窗口和 1 小时窗口都显示错误预算快速燃烧时，才触发高优先级告警。
```

Runbook 要具体到动作，而不是一句“联系研发”：

```text
现象：Outbox 最老事件年龄超过 10 分钟。
检查：Kafka 是否可用，Relay 是否运行，数据库扫描是否慢，失败事件错误原因是什么。
缓解：暂停非核心 topic，扩容 Relay，修复异常 payload，必要时人工重放指定事件。
验证：oldest_age 回落，failed_count 不再增长，下游业务状态恢复。
```

### Kubernetes 生产部署要理解探针、资源和发布

Kubernetes 不是把应用放进容器就结束。生产部署至少要考虑：

- `readinessProbe`：决定实例是否接流量。
- `livenessProbe`：决定实例是否需要重启。
- `startupProbe`：保护慢启动服务，避免启动过程中被误杀。
- `resources.requests`：调度时需要的 CPU 和内存。
- `resources.limits`：防止单实例占满节点资源。
- `PodDisruptionBudget`：节点维护或滚动升级时保证最少可用实例。
- `HorizontalPodAutoscaler`：按 CPU、QPS、队列长度或自定义指标扩缩容。

探针设计不能乱写。比如 `/actuator/health` 如果检查所有下游，
下游 Redis 短暂抖动可能导致所有应用实例都被摘流量，引发雪崩。

推荐拆分：

- liveness：只检查进程是否卡死，不强依赖外部中间件。
- readiness：检查自身是否能接请求，可以包含必要依赖。
- startup：只在启动阶段使用，给应用足够预热时间。

发布要考虑数据库兼容：

```text
第一步：新增可兼容字段，旧代码不使用。
第二步：发布新代码，开始双写或读新字段。
第三步：确认稳定后再清理旧字段。
```

不能先删除字段再发布代码，否则旧 Pod 在滚动过程中还会访问旧字段，导致线上错误。

灰度发布的判断指标：

- HTTP 5xx 是否上升。
- P95/P99 是否劣化。
- 下单、支付、库存核心业务指标是否异常。
- Outbox、Kafka lag、补偿任务是否堆积。
- 日志中是否出现新版本特有异常。

### 测试体系要证明风险被覆盖

单元测试、集成测试、Smoke 测试的价值不同。

单元测试关注纯业务规则，例如：

- 订单状态从 `CREATED` 可以到 `PAID`。
- `PAID` 订单不能直接取消。
- 支付回调金额不一致必须失败。
- 库存不足时不能预占成功。

集成测试关注真实组件，例如：

- Flyway 脚本能创建正确唯一约束。
- Repository SQL 的条件更新能防超卖。
- Outbox 失败后能重试。
- Kafka 消费重复消息不会重复写业务表。
- Testcontainers 中 MySQL、Redis、Kafka 行为符合预期。

Smoke 测试关注部署后的最短核心链路：

```text
创建用户 -> 创建商品和库存 -> 下单 -> 支付回调 -> 查询订单已支付
```

测试不是越多越好，而是要覆盖最高风险路径。L6 面试可以这样讲：

```text
我会把测试按风险分层：领域规则用单元测试快速覆盖，
SQL、事务、唯一键和中间件行为用集成测试覆盖，
部署后的核心链路用 smoke 测试覆盖。
这样既能保持反馈速度，又能证明关键生产风险被验证过。
```

还要能解释 Mock 的边界。Mock 可以验证下游返回超时、失败、重复响应时本服务如何处理，
但 Mock 不能证明 SQL、索引、事务隔离、序列化格式和真实中间件行为正确。

### 成本治理要和容量设计绑定

成本治理不是“少用机器”，而是在满足 SLO 的前提下减少浪费。

电商系统常见成本来源：

- 订单库和日志的长期存储。
- 搜索索引副本和分片。
- Kafka topic 保留时间和副本数。
- 高峰预留机器。
- 过度打点产生的监控和日志成本。
- 冷门服务长期高规格运行。

设计时要做分层：

- 核心交易链路优先保证可用性和延迟。
- 搜索、推荐、广告可以按业务价值弹性扩缩。
- 历史订单转冷存储，在线库只保留近期高频访问数据。
- 日志按级别和采样控制，错误日志完整保留，成功请求可采样。
- 压测结果用于设置 requests、limits 和 HPA 阈值。

L6 面试中，成本治理的好答案不是“用更便宜的机器”，而是：

```text
我会先用指标找成本大头，比如 CPU 利用率、内存水位、Kafka 保留、索引大小和日志量。
然后按业务优先级分层优化。核心链路不牺牲 SLO，非核心链路通过弹性、采样和降级降低成本。
```

### 如何把概念讲成 L6 级别答案

当面试官问一个概念，不要只回答定义。使用下面结构：

```text
1. 这个概念解决什么生产问题。
2. 在我的系统里哪个模块用到。
3. 关键实现是什么。
4. 有什么失败场景。
5. 如何监控和恢复。
6. 这个方案的取舍是什么。
```

例如问“什么是幂等”，初级回答是：

```text
同一个请求执行多次结果一样。
```

L6 回答应该是：

```text
幂等解决的是重试、重复点击、MQ 重投和支付重复回调导致的重复写问题。
在 eMall 中，下单使用 requestId，支付回调用 channelTradeNo，MQ 消费用 messageId。
实现上依赖数据库唯一键和状态机判断，而不是内存 Map。
失败场景包括业务成功但响应丢失、消费者处理成功但 ack 失败、渠道重复回调。
监控上要看幂等冲突数、重复消息数、支付重复回调数。
取舍是需要额外存储幂等记录，但换来可重试和可恢复。
```

再例如问“为什么用 Outbox”，不要只说“防止 MQ 发送失败”，要说：

```text
Outbox 把业务写入和事件记录放在同一个本地事务里，解决业务成功但事件丢失的问题。
它不能保证消费者只处理一次，所以消费者仍然要幂等。
它引入了事件延迟、Relay 扫描和历史清理成本，但换来了可审计、可重试、可人工重放。
```

能把每个概念都讲到这个深度，才接近 L6 面试里的高评价表达。
