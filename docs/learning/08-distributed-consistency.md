## 分布式一致性

### 为什么不用全局大事务

大型电商中，订单、库存、支付通常是不同服务和不同数据库。如果用一个全局事务：

- 性能差。
- 锁持有时间长。
- 可用性差。
- 任一下游慢都会拖垮主链路。
- 跨团队、跨系统难以落地。

所以本工程采用：

```text
本地事务 + 幂等 + Outbox + MQ + 补偿 + 对账
```

### 幂等

幂等表示同一个请求执行一次和执行多次结果一致。

需要幂等的场景：

- 下单。
- 库存预占。
- 库存确认。
- 库存释放。
- 支付回调。
- 退款。
- MQ 消费。
- Outbox 重放。

常见实现：

- `requestId`。
- 业务唯一键。
- 幂等表。
- 数据库唯一约束。
- 状态机判断。

注意：

- 只在内存里记录幂等不可靠。
- 幂等最终要由数据库或持久化记录兜底。

### Outbox

Outbox 解决的问题：

```text
业务数据写成功，但 MQ 发送失败怎么办？
```

方案：

1. 在同一个本地事务中写业务表和 Outbox 表。
2. 后台任务扫描待发送 Outbox。
3. 发送到 Kafka。
4. 发送成功后标记状态。
5. 失败则保留重试状态。

优点：

- 不丢事件。
- 可以重试。
- 可以审计。
- 可以人工重放。

缺点：

- 事件可能重复发送。
- 消费端必须幂等。
- 需要清理历史 Outbox。

### 补偿

补偿用于处理流程中间失败。

示例：

- 订单创建时库存服务不可用，订单进入 `PENDING_RETRY`。
- 补偿任务稍后重试库存预占。
- 如果成功，订单进入 `CREATED`。
- 如果多次失败，保留失败原因，等待人工处理。

补偿设计要点：

- 失败状态必须持久化。
- 失败原因要记录。
- 重试次数要限制。
- 重试要幂等。
- 人工入口要审计。

### 对账

支付对账用于发现本地记录和渠道账单不一致。

常见差异：

- 渠道支付成功，本地未成功。
- 本地成功，渠道无记录。
- 金额不一致。
- 退款状态不一致。
- 重复回调导致状态异常。

对账处理：

1. 导入渠道账单。
2. 按支付单号或渠道交易号匹配。
3. 比较金额、状态、时间。
4. 生成差异记录。
5. 自动修复或人工审核。
