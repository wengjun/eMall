## 数据库和持久化

### 表设计基础

电商核心表需要稳定、可审计、可扩展。

常见表：

- 用户表。
- 商品表。
- SKU 表。
- 库存表。
- 库存预占表。
- 订单表。
- 订单明细表。
- 支付表。
- 支付流水表。
- Outbox 表。
- 幂等记录表。
- 审计表。

设计原则：

- 主键稳定。
- 状态字段用枚举语义。
- 金额使用 `DECIMAL`。
- 时间字段明确创建和更新时间。
- 关键唯一约束必须由数据库兜底。
- 高频查询字段需要索引。

### 索引

索引用于加速查询，但不是越多越好。

适合建索引：

- 订单号。
- 用户 ID。
- SKU ID。
- 支付单号。
- 渠道交易号。
- `request_id`。
- 状态 + 时间组合。

常见错误：

- 在低选择性字段上乱建索引。
- 忽略联合索引最左前缀。
- 对索引字段做函数计算。
- 查询返回太多列和太多行。

### 事务

事务用于保证本服务内部的一组写入要么都成功，要么都失败。

典型场景：

- 创建订单和订单明细。
- 保存支付单和支付流水。
- 写业务表和 Outbox 表。
- 扣减库存和写预占记录。

注意：

- 本地事务不能跨多个服务。
- 不要在事务中做很慢的远程调用。
- 事务范围越大，锁持有时间越长。

### 锁和并发

库存防超卖常见方案：

- 数据库条件更新。
- 乐观锁版本号。
- 悲观锁。
- 库存桶。
- Redis 预扣和数据库确认。

示例思路：

```sql
update inventory
set available = available - 1, reserved = reserved + 1
where sku_id = ? and available >= 1;
```

如果更新行数为 0，说明库存不足或并发竞争失败。

### Flyway

Flyway 用于管理数据库迁移脚本。

常见文件名：

```text
V1__init_schema.sql
V2__add_outbox.sql
V3__add_payment_ledger.sql
```

原则：

- 已执行的迁移不要随便修改。
- 新变更新增版本脚本。
- 每个服务维护自己的迁移。
- 本地、测试、生产保持一致。

### MyBatis Plus 和 Spring JDBC

MyBatis Plus 适合：

- 常规 CRUD。
- Mapper。
- 分页。
- 条件构造。

Spring JDBC 适合：

- 明确 SQL。
- 简单直接的基础设施代码。
- 对 SQL 可控要求高的地方。

项目里如果统一使用 MyBatis Plus，需要注意：

- 复杂事务仍然要自己设计。
- 幂等、Outbox、锁和状态机不是 ORM 自动解决的。
- ORM 只解决数据访问，不解决业务一致性。
