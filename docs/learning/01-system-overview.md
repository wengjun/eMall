## 如何使用本文

建议按下面方式学习：

1. 先通读“核心交易闭环”和“分布式一致性”，建立整体认知。
2. 再按 Java、Spring Boot、Maven、数据库顺序补基础。
3. 然后回到订单、库存、支付，把核心链路手写一遍。
4. 最后补齐中间件、稳定性、测试、部署、可观测性和工程治理。

判断自己是否真正掌握，不看“是否看过”，而看下面问题能否回答：

- 为什么服务要按业务域拆分？
- 为什么下单不能依赖一个全局大事务？
- 重复下单、重复支付回调、重复 MQ 消息如何处理？
- 库存如何防止超卖？
- 下游服务故障时，系统如何限流、熔断、降级和恢复？
- 出现订单、库存、支付不一致时，如何补偿和对账？
- 如何用测试证明系统行为可靠？
- 如何部署、观测、告警和回滚？


## 工程全局认知

eMall 是 Java 17 微服务电商工程，不是单体 CRUD 项目。它按业务域拆成多个模块：

- `gateway`：统一入口、路由、限流、trace ID。
- `user`：用户注册、查询、冻结、敏感信息保护。
- `product`：商品创建、查询、上下架、商品变更事件。
- `pricing`：价格本和下单价格快照。
- `marketing`：优惠券和促销报价。
- `inventory`：库存初始化、预占、确认、释放、防超卖。
- `order`：幂等下单、订单状态机、取消、支付确认、补偿。
- `payment`：支付单、支付回调、退款、流水、渠道对账。
- `search`：商品搜索读模型，异步消费商品事件。
- `fulfillment`：履约、仓库、物流、签收。
- `common`：统一响应、异常、事件、幂等、Outbox、ID、安全工具。
- `governance`：下游平滑恢复、多区域路由等治理能力。

大型电商系统的难点不是写接口，而是处理这些问题：

- 高并发下不超卖。
- 用户重复点击不重复下单。
- 支付渠道重复回调不重复入账。
- MQ 重复投递不重复处理。
- 下游服务失败后可重试、可补偿、可人工恢复。
- 非核心能力故障不影响核心交易。
- 出问题后能通过日志、指标、trace 定位。
