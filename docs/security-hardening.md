# 安全加固

本文说明 eMall 的边缘安全、服务间安全、数据保护、风控、密钥轮换和审计要求。

## 边缘安全

- 网关负责基础限流、路由、trace ID 透传和安全响应头。
- 生产环境应接入 WAF、Bot 防护、IP 信誉和设备指纹。
- 登录、注册、下单、支付、退款、秒杀等接口需要独立风控策略。
- 开放平台请求必须校验签名、时间戳、nonce 和配额。

## 服务间安全

- 内部接口不应默认暴露给公网。
- 内部运维接口需要 `X-Internal-Token`、`X-Operator` 和审计记录。
- 生产环境建议使用 mTLS 或服务网格保护服务间通信。
- 服务账号权限应按最小权限配置。
- Kubernetes 中应使用 NetworkPolicy 限制东西向访问。

## 数据保护

- 手机号等敏感字段使用 AES-GCM 加密存储。
- 精确查询使用 HMAC lookup hash，避免存储明文手机号。
- 日志、指标和 trace 中不能输出明文敏感数据。
- 数据仓库和分析平台需要按数据分级控制访问权限。
- 隐私请求需要有受理、处理、审计和关闭流程。

## 风控

风控模块覆盖以下场景：

- 登录和注册风险。
- 优惠券领取和核销风险。
- 下单和支付风险。
- 退款、售后和拒付风险。
- 秒杀抢购和刷单风险。
- 搜索、评价和广告作弊风险。

风控规则需要版本化、可启停、可审计，并且不能让风控服务故障拖垮核心交易链路。

## 密钥轮换

生产环境应满足：

- 密钥不写死在代码或普通配置文件中。
- 使用密钥管理系统或 Kubernetes Secret 管理敏感配置。
- 支持双密钥窗口，避免轮换时中断服务。
- 轮换动作需要审批、审计和回滚方案。
- 老密钥停用前要验证所有服务已经切换。

## 审计要求

需要审计的动作：

- 内部运维补偿、重放、对账和数据修复。
- 权限授予、权限回收、服务客户端创建。
- 商家开放平台密钥和 Webhook 配置变更。
- 风控规则启停和阈值变更。
- 生产数据库操作、备份、恢复和安全事件处理。

审计记录至少包含操作人、时间、trace ID、目标资源、动作、结果和失败原因。
