# 混沌工程

混沌工程用于验证系统在故障下是否仍能保护核心链路，并且能被观测、恢复和复盘。

## 基线演练

当前混沌目录覆盖：

- Redis 故障。
- MQ 积压。
- 数据库故障切换。
- 下游延迟。
- 部分区域隔离。

Chaos Mesh 示例位于 `ops/k8s/chaos`。

## 安全门禁

混沌演练必须满足：

- 明确演练目标。
- 明确影响范围。
- 明确停止条件。
- 明确回滚方式。
- 有负责人和值班人。
- 不能在未知业务高峰直接执行。

## 执行流程

1. 提交演练计划。
2. 审批演练窗口和影响范围。
3. 执行小范围演练。
4. 观察指标、日志和 trace。
5. 达到停止条件时立即回滚。
6. 复盘故障表现和系统改进点。

## 生产边界

真实生产中的混沌工程不能只是“制造故障”，而是要验证：

- 限流是否生效。
- 熔断是否及时打开。
- 降级是否保护核心交易。
- 补偿是否能恢复失败状态。
- 告警是否准确。
- Runbook 是否能指导处理。
