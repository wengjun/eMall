# 十亿用户电商生产检查清单

本文是生产就绪检查清单。它用于提醒大型电商系统上线前需要验证的能力，不表示当前本地工程已经完成所有真实生产要求。

## 业务正确性

- 下单必须幂等，重复 `requestId` 不能创建重复订单。
- 库存预占、确认和释放必须防止超卖。
- 支付回调必须幂等，重复回调不能重复记账。
- 退款、售后、对账必须能发现并修复不一致。
- 价格和优惠必须保存下单快照，历史订单不能被新规则改写。

## 高并发和性能

- 网关需要限流、鉴权、防刷和热点保护。
- 热点 SKU 需要库存桶、预热、排队或令牌机制。
- 商品、价格、搜索等读多数据需要缓存和失效策略。
- 秒杀流量应与普通下单链路隔离。
- 压测必须覆盖读多、热点 SKU、支付回调和 MQ 积压场景。

## 数据和一致性

- 每个服务拥有自己的数据库 schema 和 Flyway 历史。
- 跨服务流程使用本地事务、Outbox、幂等和补偿。
- MQ 消费者必须可以重复消费。
- 定时任务需要分布式锁或分片调度。
- 关键账务需要对账和人工修复入口。
- 历史数据需要归档、冷热分层和恢复方案。

## 安全和风控

- 敏感字段需要加密或脱敏存储。
- 内部运维接口必须有内部令牌、操作人和审计记录。
- 外部开放平台必须校验签名、时间戳、nonce 和配额。
- 登录、注册、优惠券、下单、支付、退款、秒杀都需要风控。
- 生产密钥必须通过密钥管理系统轮换，不能写死在配置中。

## 发布和运维

- 所有服务需要健康检查、优雅关闭和资源限制。
- 发布需要灰度、回滚和自动化校验。
- 重要 Topic、延迟预算和回放计划需要治理。
- 故障处理必须有 Runbook。
- 生产变更需要审批、审计和责任人。

## 动态配置

- 限流阈值、熔断参数、降级开关、灰度比例应支持动态配置。
- 关键配置变更需要审计。
- 配置下发失败时，服务应继续使用安全的本地默认值。
- 配置中心故障不能影响核心交易链路。

## 灰度发布

- 发布前需要定义成功指标、失败阈值和回滚条件。
- 灰度流量应从小比例开始逐步放大。
- 支持按用户、设备、区域、商家或单元做灰度。
- 灰度期间需要观察错误率、延迟、支付成功率、库存失败率和告警。

## 所有权和流程

- 每个服务需要明确 owner。
- 每个数据库、Topic、缓存 key、配置项需要明确 owner。
- 每个告警需要明确处理人和 Runbook。
- 每次重大发布需要容量评估、风险评估和回滚方案。

## 当前工程状态

当前工程已经具备很多生产基线能力，例如多模块微服务、MySQL/Flyway、Redis、Kafka Outbox、补偿、对账、
可观测、Kubernetes 清单、单元测试和集成测试。但真实生产前仍需要真实基础设施、持续压测、安全合规、
运维流程和团队值守体系。
