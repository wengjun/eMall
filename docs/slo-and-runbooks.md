# SLO 和故障手册

本文说明容量模型、SLO、核心告警和常见故障处理流程。

## 粗略容量模型

设计目标是 1 亿 DAU 和 100 万峰值并发。真实容量不能只靠估算，需要通过持续压测和生产指标验证。

关注指标：

- QPS、并发连接、线程池队列。
- P50、P95、P99 延迟。
- MySQL CPU、连接数、慢 SQL、锁等待。
- Redis 命中率、内存、热 key、慢命令。
- Kafka 生产速率、消费延迟、积压。
- Elasticsearch 查询延迟、写入延迟、索引大小。

## 全局 SLO 目标

建议基线：

- 网关可用性：99.95%。
- 下单接口可用性：99.9%。
- 支付回调处理可用性：99.99%。
- 核心读接口 P99：小于 500 ms。
- 下单 P99：小于 1000 ms。
- 支付回调 P99：小于 800 ms。

这些是目标值，真实生产需要按业务优先级和成本重新校准。

## 核心告警

- 网关 5xx 突增。
- 下单成功率下降。
- 库存预占失败率异常。
- 支付回调失败或积压。
- Outbox 失败数量持续增加。
- Kafka 消费延迟超过阈值。
- Redis 命中率下降或热 key 异常。
- MySQL 慢 SQL、锁等待、连接数耗尽。

## 下单故障处理

1. 查看网关错误率和限流命中。
2. 查看订单服务 P99、线程池和数据库连接。
3. 查看价格、营销、库存依赖状态。
4. 如果营销异常，确认是否已降级为无优惠。
5. 如果库存异常，检查 `PENDING_RETRY` 和补偿任务。
6. 必要时降低非核心流量，保护核心下单。

## 支付故障处理

1. 查看支付回调错误率和积压。
2. 检查渠道交易号去重是否正常。
3. 检查支付流水和订单确认状态。
4. 对失败的订单确认执行内部重试。
5. 对渠道账单和本地流水执行对账。
6. 对资金差异进入人工审核和修复流程。

## 库存故障处理

1. 查看热点 SKU 的预占失败率。
2. 检查库存桶是否初始化。
3. 检查库存行锁、慢 SQL 和数据库连接。
4. 对未支付订单执行过期释放。
5. 对异常预占记录执行补偿。
6. 必要时关闭秒杀或降低热点 SKU 流量。

## 发布护栏

- 发布前必须有回滚方案。
- 灰度期间观察错误率、延迟和业务指标。
- 新版本不能扩大已有告警。
- 关键链路异常时立即停止放量。
- 回滚后确认数据补偿和消息重放状态。

## 可靠性策略

- 核心链路优先，非核心链路可降级。
- 下游故障时快速失败，避免线程耗尽。
- 下游恢复时逐步放量。
- 失败状态必须可查询、可重试、可审计。

## 容灾

真实生产应验证：

- 数据库备份和恢复时间。
- Redis 故障降级策略。
- Kafka 积压和重放能力。
- 单可用区故障切换。
- 多区域路由和单写归属切换。
